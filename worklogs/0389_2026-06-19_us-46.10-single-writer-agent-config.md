# Worklog: US-46.10 — Single-Writer `agent-config.json`

**Date:** 2026-06-19
**Session:** Epic 46 Phase 5 — consolidate four uncoordinated writer paths of agent-config.json into a single AgentConfigWriter.
**Status:** Complete

---

## Objective

Eliminate the highest-risk fragility in the codebase: four independent code paths wrote `agent-config.json` without atomic coordination. The design "worked" via strict boot ordering, `reloadMu` serialisation, and `atomic.Pointer[[]relayModel]` coordination — but a future change could reintroduce relay clobbering. This story implements the single-writer design the README had been deferring.

---

## Work Completed

### AgentConfigWriter (`agent_config_writer.go`, 298 lines)

New type that owns all writes to `agent-config.json` within the agentd process:

```
AgentConfigWriter
├── mu          sync.Mutex      // serialises Rebuild
├── path        string          // agent-config.json path
├── providerRaw json.RawMessage // provider map from FormatOpenCodeConfig
├── model       string          // "providerID/modelID" from workspace-config.json
└── relay       *relaySource    // relay URL + free models from injector
```

**Boot initialisation:** `newAgentConfigWriter(path)` reads the existing file (written by the materialize subcommand) and captures the provider map and model as initial sources. This lets the relay injector merge into them at ~T+7s without re-deriving provider credentials.

**Rebuild:** merges all sources into a complete config (`$schema` + provider map + model + disabled_providers + relay provider) and writes atomically via temp-file + `os.Rename`.

### Materializer.FormatProviders (pkg/agentd/secrets/secrets.go)

New method that formats staged providers without writing to disk. `FlushProviders` now delegates to `FormatProviders` + `atomicWrite`. Callers inside agentd use `FormatProviders` + `writer.SetProviders` + `writer.Rebuild` instead.

## Key Decisions

1. **Boot reads existing file.** `newAgentConfigWriter` reads the file written by the materialize subcommand (separate process) to capture initial provider + model sources. This avoids re-deriving credentials and lets the relay injector merge into existing providers.

2. **Model source NOT updated by SetProviders.** `FormatOpenCodeConfig` sets a default model from provider defaults, but the model from `workspace-config.json` (set at boot) must survive credential reloads. `SetProviders` only updates the provider map.

3. **Writer rebuild failure is graceful (Warn + 200).** The old FlushProviders write failure returned 500. With the writer, a rebuild failure is logged as Warn and the handler returns 200 — credentials are still materialized, and the old config on disk is still valid. This is strictly better behavior.

4. **Materialize subcommand still writes directly.** The subcommand runs as a separate process before agentd starts; it has no access to the writer. `FlushProviders` is kept for this path. Once agentd starts, all writes go through the writer.

5. **Nil guard in reload handler.** If `deps.AgentConfigWriter` is nil (tests only), the handler skips the writer path. Production always passes a non-nil writer.
