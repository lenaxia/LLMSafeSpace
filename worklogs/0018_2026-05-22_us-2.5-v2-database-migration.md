### 2. Critical Finding: `phase` Column Conflict

`database.go` queries, inserts, and updates a `phase` column on the `workspaces` table in multiple places (`GetWorkspace`, `CreateWorkspace`, `UpdateWorkspace`, `ListWorkspaces`). The design doc §5.5a states `phase` is owned by the K8s CRD controller, not PostgreSQL.

**Decision:** The schema follows `database.go` as ground truth per the task spec. The `workspaces` table includes `phase VARCHAR(50) DEFAULT 'pending'`. This is a mirror/cache of the CRD phase used for query performance (consistent with the §5.5a pattern for `user_id` and timestamps). The Go code must compile and run without errors — schema must match queries.

### 3. `workspaces.id` Type Decision

The design doc §15 specifies `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`. However, `database.go` uses `VARCHAR(255)` for workspace IDs throughout (`CheckResourceOwnership` query, `WorkspaceMetadata.ID` is a string in `pkg/types/types.go`), and `workspace_service.go` sets the ID from `uuid.New().String()` — a UUID formatted as a string. Using `UUID` type in PostgreSQL would accept UUID-string values fine, but since the sandboxes table uses `VARCHAR(36)` for its foreign keys and the service uses `uuid.New().String()`, consistency favours `VARCHAR(255)`. The `workspace_id` foreign key on sandboxes was also made `VARCHAR(255)` to match.

## Key Decisions

1. **`phase` column added to `workspaces`** — `database.go` is ground truth; it queries `phase` in 4 places. Not including it would cause runtime panics on scan. Note: this is a denormalized mirror of the K8s CRD status, consistent with §5.5a's pattern for dual ownership.

2. **No `warm_pools`, `execution_history`, `file_operations`, `package_installations`** — V2 removes these tables. They remain only in `001_initial_schema.sql` (V1 reference, not touched).

3. **`sandbox_labels` table added** — `database.go` queries this table in `GetSandbox`, `CreateSandbox`, `UpdateSandbox`, `DeleteSandbox`, and `ListSandboxes`. It was absent from the V1 schema.

4. **`users` gains `active` and `role` columns** — `database.go` selects and inserts both; they were missing from V1.

5. **`sandboxes` gains `name`, `status`, `updated_at`** — `database.go` selects, inserts, and updates all three; they were missing from V1.

6. **`workspaces.id` as `VARCHAR(255)` not `UUID`** — `workspace_service.go` stores `uuid.New().String()` as the ID; `database.go` treats it as a string. Using `VARCHAR(255)` avoids a type mismatch. The FK on sandboxes follows the same type.

7. **`sandbox_labels` has `ON DELETE CASCADE`** — `database.go` manually deletes labels before sandbox deletion, but cascade is the correct safety net.
