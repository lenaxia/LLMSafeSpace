## Summary

16 RT-6.x tests against live cluster. **7 PASS, 8 FAIL, 1 INCONCLUSIVE.**

The standout finding: **the validating webhook does not validate `Spec.Runtime` content at all** — the webhook accepts `runtime: "../../etc/passwd"`, `runtime: "evil.example.com/malicious:latest"`, `storage.size: "999999Gi"`. Confirms the Phase 1/2 prediction (RT-1.2, RT-2.18) at admission level: this is the right layer to fix, and it's currently empty.

## Methodology corrections

### CRD schema discovery
RT-6.1 and RT-6.10 initially failed with `strict decoding error: unknown field "spec.userId"`. The CRD schema uses `spec.owner.userID` (capital ID). Corrected and re-ran.

This is itself a small finding: when an attacker probes the API with the wrong shape, they get a useful schema error message. Not security-relevant, but operationally noisy.

## Forensic deep-dive: webhook gap

The validating webhook `llmsafespace-validate` is configured (via Helm) but its actual code in `controller/internal/webhooks/` apparently doesn't sanity-check `Spec.Runtime` against:
- A registry allow-list
- A path-traversal-safe shape
- A reasonable storage size cap

Cross-referenced with the runtime image-pull flow (per `controller/internal/workspace/controller.go`):
- The controller takes `Spec.Runtime` verbatim and uses it in pod spec `image: ...`
- Kubernetes attempts to pull from whatever registry the string points to
- If the image doesn't exist, the pod errors out, but the workspace was admission-accepted

**Fix priority (combining Phase 2 + Phase 6):**
1. Add allow-list validation to the webhook for `Spec.Runtime`. Reject anything not matching `<allowed-registries>/<image>:<tag>`.
2. Add `Spec.Storage.Size` sanity cap (e.g., 100Gi default, configurable).
3. Add unit test in `controller/internal/webhooks/workspace_webhook_test.go` for both above.
4. Add e2e test that creates a workspace with an evil runtime → expect admission denial.
