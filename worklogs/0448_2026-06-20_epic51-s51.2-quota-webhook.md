# Worklog: Epic 51 S51.2 — Per-Tenant Resource Quota Admission Webhook

**Date:** 2026-06-20
**Session:** Implementation of the per-tenant resource quota validating webhook.
**Status:** PR #317 opened for review.

---

## Objective

Implement S51.2 — the validating admission webhook that prevents noisy-neighbor resource exhaustion in multi-tenant deployments by rejecting workspace pod creation when the tenant's aggregate running resource usage would exceed configured limits.

---

## Work Completed

### PodTenantQuotaValidator handler
- New validating webhook on Pod create, keyed on `llmsafespaces.dev/tenant` pod label (from S51.3)
- Counts existing workspace pods per tenant via controller-runtime client (real-time, not event-based metering)
- Sums CPU (millicores) and memory (MiB) resource requests across all containers
- Enforces three limits: max-workspaces-per-tenant, max-cpu-millis-per-tenant, max-memory-mi-per-tenant
- All limits default to 0 (disabled); webhook only registers when at least one limit > 0

### Controller wiring
- Three new CLI flags: `--max-workspaces-per-tenant`, `--max-cpu-millis-per-tenant`, `--max-memory-mi-per-tenant`
- Conditional webhook registration in main.go (skipped when all limits are 0)

### Helm chart
- New webhook entry in `validating-webhook.yaml` (`vpodtenantquota`) with `objectSelector` on tenant label
- `webhooks.tenantQuota` section in `values.yaml` with documentation
- Controller deployment wires flags from Helm values

### Tests (12 total)
- Allow: no-tenant-label, under-quota, all-limits-zero, non-create, fail-open-on-error, exact-limit-boundary, unspecified-tenant
- Deny: exceeds-workspace-count, exceeds-CPU, exceeds-memory
- Isolation: different tenants don't interfere
- Edge: terminal pods (Succeeded/Failed) excluded from count

---

## Key Decisions

1. **Real-time pod listing over event-based metering** — More accurate for admission decisions. The webhook uses the controller-runtime client to List pods directly, avoiding lag from Epic 12's event-based metering system.

2. **Fail-open on usage errors** — Denying all pods on a transient API server hiccup is worse than allowing a potential overage. The TOCTOU window is already accepted per the Epic 51 design.

3. **Instance-level defaults only** — Org-specific quota overrides (`org_policies` integration) and billing tier → quota mapping are deferred to Epic 43 per the Epic 51 out-of-scope table.

4. **Terminal pods excluded** — Succeeded/Failed pods are releasing resources and shouldn't count toward quota.

5. **Boundary semantics: `projected > limit` denies** — A pod that brings the total to exactly the limit is allowed (at-limit, not over-limit).

## Review Feedback (PR #317)

AI review found a **critical** bug and two test gaps:

1. **CRITICAL: objectSelector matchLabels bug** — The Helm config had `matchLabels: {llmsafespaces.dev/tenant: ""}` which would never match real pods (sanitizeLabelValue never produces ""). Fixed by removing matchLabels, keeping only `matchExpressions: Exists`.
2. **Missing boundary test** — Added `TestS51_2_AllowAtExactLimit` (projected == limit should be allowed).
3. **Missing "unspecified" test** — Added `TestS51_2_AllowWhenTenantIsUnspecified`.
