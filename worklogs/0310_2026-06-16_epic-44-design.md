# Worklog: Epic 44 — Session Reliability & Transparency design

**Date:** 2026-06-16
**Session:** Multi-session investigation, design, validation, and iteration for Epic 44
**Status:** Design complete, ready for implementation

---

## Objective

Design Epic 44 (Session Reliability & Transparency) in response to two production incidents:

1. **Incident A (workspace `a847faa5-19b4-463d-a434-1ce473a16f93`, session `ses_13076538bffeYtLrhoZ2ccRM1E`)**: Pod OOMKilled at 2026-06-16 08:47:44 UTC mid-stream. 16 sessions with ~1.6M total context tokens at OOM. Memory limit 2 GiB exceeded. Assistant turn was lost; session stuck busy in SQLite until next opencode restart.

2. **Incident B (workspace `8154ae86-d7b7-4f53-b046-d8d3b462b972`, session `ses_130c14344ffeVF52UQ6QGPmB0P`)**: User changed workspace secret bindings while a prompt was actively running. workspace-agentd SIGTERMed opencode at 16:04:14 UTC, destroying the in-flight assistant turn.

Both incidents were silent failures — users had no terminal feedback, no retry path, no error message.

---

## Work Completed

This entry covers four sessions of work:

### Session 1: Investigation (produced `INCIDENT-ANALYSIS.md`)

Reproduced both incidents from kubectl logs, opencode SQLite, and pod events:

- **Incident A root cause**: Pod cgroup OOM. Memory at OOM was 1.65 GiB (83%), 4.5 minutes before kill. Default memory limit comes from `controller/internal/workspace/pod_builder.go:249,251` (`defaultMemory = "512Mi"` × `burstFactor = 4` = 2 GiB). Workspace CRD supports `spec.resources.memory`/`memoryLimit` overrides which weren't set.

- **Incident B root cause**: workspace-agentd's secret reconciliation logic (`cmd/workspace-agentd/secrets.go:357,367,398`) restarts opencode unconditionally when env-secret or api-key bindings change. No check for busy sessions; no draining; no user notification.

- **Secret restart matrix**: Analyzed `cmd/workspace-agentd/secrets.go` to determine which secret types require restart vs. which could be hot-reloaded. env-secret and api-key write to env file (require restart since Node.js `process.env` is snapshot at startup); ssh-key, secret-file, git-credential, llm-provider all read filesystem at invocation (no restart needed).

- **Latent bug found**: `cmd/workspace-agentd/secrets.go:398` writes api-key changes to env file but does NOT trigger opencode restart. So api-key changes are silently ignored until next restart-for-other-reason. Aggressive deprecation recommended since no current users.

### Session 3: User Feedback Iteration (produced `QUESTIONS-ANSWERED.md`, `README-UPDATE-SUMMARY.md`, README-v1 → v2)

User clarified critical design points that required restructuring:

1. **Ops monitoring is P0** (was P2): "first-class citizen, not optional"
2. **Memory threshold = 85%** (was 75%)
3. **No timeout for deferred restart**: "agentic flows commonly run for multiple hours" → deferred restart will be VERY common, not rare edge case
4. **UI notification instead of forced restart**: "Pending refresh — click here to initiate (will halt sessions), or will auto-apply when sessions idle"
5. **Aggressive api-key deprecation**: "no api-key secrets currently in use"
6. **Request buffering NOT secret buffering**: My initial design buffered secret changes in agentd memory. User clarified the correct architecture:
   - Secrets written to DB IMMEDIATELY when user edits them
   - Mark workspace as "needs rematerialization"
   - When sessions idle: re-fetch ALL secrets from DB (idempotent)
   - NO in-memory secret buffer needed — DB is source of truth
   - Buffer the user's HTTP requests during restart instead (transparent to user)

This was a significant design correction. The wrong approach was confusing two different buffering concerns (secret state vs. request flow). The right approach treats the database as the source of truth for secret state and only buffers requests, which is what users actually experience.

### Session 4: Final Design (produced `DESIGN-FINAL.md`, current `README.md`)

Consolidated the 11 user stories across 3 phases:

**Phase 1 (P0): Stop Silent Failures** (~8.5 days)
- US-44.1: Terminal SSE events on agent death
- US-44.2: Session-aware restart (check busy sessions, defer until idle)
- US-44.3: Fix api-key restart bug (env changes silently ignored)
- US-44.4: OOM detection via exit code 137 + cgroup memory pressure
- US-44.5: Memory pressure warnings at 85% threshold
- US-44.10: API proxy request buffering during opencode restart

**Phase 2 (P1): User Experience** (~6 days)
- US-44.6: Frontend SSE error event handling
- US-44.7: UI notification for deferred restart (no forced timeout)

**Phase 3 (P2): Observability** (~5.5 days)
- US-44.8: Prometheus metrics — moved to P1 ("ops is first-class citizen")
- US-44.9: api-key deprecation with migration story
- US-44.11: Runbook + alerting rules

Total: ~20 days (originally estimated 14.5; expanded due to request buffering and UI work)

## Lessons Learned

1. **Validate "optional" assumptions too**: I had marked some assumptions as "optional to validate" because they seemed likely correct. User correctly insisted on validating everything. Found 1 wrong assumption (frontend SSE error event handling) and 1 architectural confusion (request vs. secret buffering) that would have caused implementation rework.

2. **Distinguish state buffering from request buffering**: When a system needs to defer an operation, ask: are we buffering data, or buffering action? Confusing these leads to over-engineered solutions. The correct mental model: data should always go to the durable store immediately; actions can be deferred and replayed.

3. **Multi-hour workflows change everything**: Single-prompt LLM interactions and multi-hour agentic workflows have fundamentally different reliability requirements. A 30-second deferred restart is fine in the first model and catastrophic in the second.

4. **Production behavior tells you priorities**: "no api-key secrets currently in use" means we can deprecate aggressively. Always check empirical reality before designing for theoretical use cases.
