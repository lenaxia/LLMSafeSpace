# Worklog: US-38.3: Replace HKDF with Argon2id for Password-Based Key Derivation

**Date:** 2026-06-13
**Session:** Replace single HKDF-based `DeriveKEK` with versioned password-based KDF using Argon2id
**Status:** Complete

---

## Objective

Replace HKDF (a fast KDF with no memory hardness) with Argon2id for password-based key derivation in `pkg/secrets`. Low-entropy passwords must be stretched with a memory-hard KDF to resist offline brute force; high-entropy key material continues to use HKDF (where Argon2id adds cost without security benefit).

---

## Work Completed

### Crypto split

- Split the monolithic `DeriveKEK` into three intent-revealing functions in `pkg/secrets/crypto.go`:
  - `DeriveKEKFromPassword` — Argon2id (memory-hard), `KDFVersionArgon2id = 1`, the new current path.
  - `DeriveKEKFromPasswordV0` — HKDF/SHA-256, `KDFVersionHKDF = 0`, preserved byte-for-byte so historical (pre-deploy) KEKs can still be derived during a future migration.
  - `DeriveKEKFromKey` — HKDF/SHA-256 for high-entropy recovery keys / API keys (unchanged behavior, just a dedicated name).
- Added `KDFVersionHKDF`, `KDFVersionArgon2id`, and `KDFCurrentVersion` constants plus Argon2id parameter constants (`argon2Time=3`, `argon2Memory=64MiB`, `argon2Threads=4`, `argon2KeyLen=32`).

### Call-site migration

All password-based call sites now route through `DeriveKEKFromPassword` (Argon2id); high-entropy paths use `DeriveKEKFromKey`. Touched: `key_service.go`, `root_key.go`, `api/internal/app/secrets_adapters.go`, `api/internal/services/auth/auth.go`, and the corresponding tests (`auth_apikey_dek_test.go`, `crypto_boundary_test.go`).

### Tests

- `pkg/secrets` suite: 136 PASS.
- Strengthened `TestDeriveKEKFromPasswordV0MatchesOldHKDFOutput` to independently recompute the HKDF output via `hkdf.New(sha256.New, …)` and `io.ReadFull`, then compare byte-for-byte — proving true backward compatibility rather than only determinism.

---

## Key Decisions

- **Argon2id over Argon2i/Argon2d.** `id` is the OWASP-recommended hybrid: side-channel resistance from the data-independent first half plus memory-hardness from the data-dependent second half.
- **Preserve `V0` HKDF output exactly.** Even though dispatch/migration is deferred (see below), keeping the `V0` path bit-identical lets us re-derive old KEKs later for a migration without guessing constants.
- **No `kdf_version` dispatch wired yet.** The constants and `V0` path exist as scaffolding; the actual record-version column and on-read migration are deferred to a follow-up story to keep this PR scoped.

---

## Breaking Change

**BREAKING: Existing `user_keys` records and sealed root key files encrypted with HKDF-derived KEKs become inaccessible after deploy.** The password path now derives a different (Argon2id) KEK than the HKDF KEK that wrapped the existing DEKs/root keys, so GCM unwrapping against pre-existing ciphertext will fail. The `kdf_version` dispatch and migration infrastructure are deferred. For development environments, re-seal root keys and re-initialize user keys after deploy.
