## G8 — atomic role assignment

`api/internal/services/database/database.go::CreateUser` rewritten
to do the user-count check + insert in a single SQL statement:

```sql
WITH existing AS (SELECT COUNT(*) AS n FROM users)
INSERT INTO users (...)
SELECT ..., CASE WHEN existing.n = 0 THEN 'admin' ELSE $8 END
FROM existing
RETURNING role
```

Postgres serialises the count + insert under the row-level locks
of the unique index on `(email)`, so two concurrent registrations
cannot both observe count=0. The atomically-assigned role flows
back to the caller via `RETURNING role` and is reflected into
`user.Role` for the auth flow's downstream JWT issuance.

`api/internal/services/auth/auth.go::Register` no longer calls
`CountUsers` separately — that race window is closed at the SQL
layer. The Register flow always passes role="user"; the DB
upgrades to "admin" if and only if count == 0.

Tests in `auth_test.go` updated: 4 tests (Success,
FirstUserBecomesAdmin, SubsequentUsersAreNotAdmin,
CreateUserError_FailsClosed) now mock the DB-side promotion via
`mock.Run` that mutates `u.Role` to mirror the production CTE
behaviour.

`database_test.go` adds `first_user_is_admin_atomically` test
asserting the DB-assigned role flows back into `user.Role`.

## RT-2.4 + RT-2.5 + G13 partial

`api/internal/middleware/rate_limit.go`: `DefaultRateLimitConfig`
flipped `Enabled: false → true`. Pre-fix the chart-default install
shipped with all rate-limiting OFF; pentest confirmed 200 sequential
api-key requests with 0 rate-limited responses. Operators can opt
out via the existing `rateLimiting.enabled=false` instance setting.

This also partially mitigates G13 (email-keyed lockout DoS): an
attacker still needs to authenticate to trigger lockout, and global
per-IP rate-limit caps how many lockout-trigger attempts they can
make per minute.
