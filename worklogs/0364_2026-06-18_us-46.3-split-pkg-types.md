# Worklog: Epic 46 Phase 2 — US-46.3 Split pkg/types/types.go

**Date:** 2026-06-18
**Session:** Split the 932-line `pkg/types/types.go` god file into per-domain files
**Status:** Complete

---

## Objective

Implement US-46.3 (Phase 2 of Epic 46, the first type-safety foundation story):
split the 932-line `pkg/types/types.go` — containing 71 types spanning every
domain — into per-domain files in the same package. Pure file reorganisation;
no import path changes, no type renames, no field changes.

### Deviation from spec (documented)

The US-46.3 spec proposed grouping all 18 workspace-domain types into one
`workspace.go`. Counting the lines, that file would have been ~258 lines —
exceeding the 250-line acceptance criterion ("every file ≤250 lines except
doc.go"). I split the workspace domain into:

- `workspace.go` — the API transfer objects (Workspace, CreateWorkspaceRequest,
  WorkspaceListResult, WorkspaceListItem, WorkspaceStatusResult,
  WorkspaceMetadata, WorkspaceUpdates, WorkspaceConfig,
  WorkspaceNotFoundError, ActivateWorkspaceResponse)
- `container.go` — the pod/resource/security configuration types
  (ResourceRequirements, NetworkAccess, EgressRule, PortRule,
  FilesystemConfig, StorageConfig, SecurityContext, ProfileReference,
  ContainerStateValue+consts, ContainerStatus, NetworkInfo)

Same reasoning applied to the orgs domain (orgs.go + orgs_policy.go) — keeping
orgs.go at a single glance for the core org/member/invitation types, while the
policy types and their 4 methods live alongside. The split point is the natural
domain boundary: "core org entities" vs "org policy enforcement".

## Key Decisions

1. **Pure move — every byte preserved.** No type rename, no field rename, no
   tag change, no comment edit. The only edits to type definitions were
   mechanical: moving them between files. Verified by `diff` of the type list
   against HEAD (zero delta). Rationale: the spec explicitly forbids anything
   but a file move; behaviour preservation is the contract.

2. **No subpackages.** The spec forbids splitting into `pkg/types/auth/` etc.
   because that would break every consumer's import path. Everything stays in
   `package types`. Verified by `go build ./...` passing unchanged. Rationale:
   smaller blast radius than the spec allows; the value is file-level
   navigation, not import-path refactor.

3. **`doc.go` rewritten as the package map.** The old 5-line doc.go said only
   "API Schema definitions". Replaced with a 26-line doc that explains the
   DTO-vs-CRD boundary AND lists every file with its domain. Rationale: a
   reader landing in the package now sees the layout without grepping; this is
   the whole point of the split.

4. **`errors.go` extracted as its own file.** The spec's file list didn't
   explicitly mention errors, but the 4 sentinel error vars (`ErrNotFound`,
   etc.) were sitting at the top of `types.go` unrelated to any domain.
   Promoting them to `errors.go` keeps each domain file clean. Rationale: the
   errors are cross-cutting; parking them in any single domain file would be
   arbitrary.

5. **Workspace domain split (workspace.go + container.go) to satisfy the
   250-line limit.** Documented above. The alternative — keeping one big
   workspace.go at 258 lines — would have failed the acceptance criterion.
