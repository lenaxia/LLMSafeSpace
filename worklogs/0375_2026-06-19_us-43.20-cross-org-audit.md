# Worklog: US-43.20 — Cross-Org Audit View

**Date:** 2026-06-19
**Session:** Implement platform-admin cross-org audit view: `ListAllAudit` store method with filter-driven parameterized queries, AdminGuard-protected endpoint, frontend platform audit tab.
**Status:** Complete — no findings from validation (worklog 0372). The smallest and cleanest of the 4 stories.

---

## Objective

Platform admins (`users.role='admin'`) can view audit events across ALL orgs with filtering by org, actor, domain, and date range. Extends the existing per-org audit infrastructure (US-43.13, migration 034).

---

## Work Completed

### Backend store method
- `pg_org_store.go`: `ListAllAudit(ctx, filters AuditFilters) ([]*types.AuditEntry, *types.PaginationMetadata, error)`.
- `AuditFilters` struct: `OrgID *string`, `ActorID *string`, `Domain *string`, `Limit int`, `Offset int`.
- Query: COUNT then SELECT with conditional `AND` appends + `[]interface{}` args slice. **Fully parameterized** — `$N` placeholder indexes only; no string interpolation of user input (verified by security validator).
- Limit clamped to 500 (defense-in-depth: handler clamps to 500, store also clamps).

### Backend handler + route
- `AuditHandler.ListCrossOrg` — reads query params, constructs filters, returns paginated JSON.
- `GET /api/v1/admin/audit` — AdminGuard-protected (returns 404 for non-admins, hiding existence).

### Frontend
- `PlatformAuditTab.tsx` — table with Action/Actor/Target/Time + filter UI (org, actor, domain).
- Wired into `SettingsPage.tsx` as an admin-only tab.
- `adminAuditApi` added to `frontend/src/api/orgs.ts`.

### Tests
- 10 store tests (sqlmock): no filter, org_id filter, actor_id filter, domain filter, pagination, empty result, count + list WHERE consistency.
- 6 handler tests: 200 with events, non-admin → 404, filter params passed through.
- 2 router wiring tests: AdminGuard protection confirmed.

---

## Key Decisions

1. **`AuditFilters` struct (not individual methods)** — extensible to new filter dimensions without API proliferation. Right abstraction.
2. **Added `ListAllAudit` to the `auditStore` interface (not `orgStore`)** — the existing `AuditHandler` already depends on `auditStore`; adding to `orgStore` would violate ISP. The implementer correctly deviated from the task's literal instruction to follow ISP.
3. **Return signature `([]*types.AuditEntry, *types.PaginationMetadata, error)`** — matches the existing `ListOrgAudit` pattern; provides pagination for limit/offset paging.
