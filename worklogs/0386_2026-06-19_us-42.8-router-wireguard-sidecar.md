# Worklog 0386 — US-42.8 Router WireGuard Sidecar + Network-Agnostic Ingress

**Date:** 2026-06-19
**Epic:** 42 (Multi-Cloud Inference Relay)
**Story:** US-42.8 (Router WireGuard sidecar + network-agnostic ingress + NetworkPolicy)
**Status:** Complete (chart implementation; runtime WG/Talos validation deferred to operator — see Assumptions + Blockers)

---

## Objective

Implement the structural gap identified in worklog 0385's audit: the
relay-router pod ran as a plain HTTP proxy with no `wg0` interface, no public
UDP endpoint, and no NetworkPolicy. Even after the handler fix (PR #273) lets
the `InferenceRelay` CR create on first deploy, the fleet stays 0/N healthy
forever because no WireGuard tunnel terminates in-cluster — cloud relay VMs
cannot dial back in, and the router cannot reach them at `10.42.42.x:8080`.

This PR ships the chart-template implementation of the redesigned Epic 42
Layer 2 spec (worklog 0385 / `design/stories/epic-42-multi-cloud-inference-relay/README.md`):
a WireGuard sidecar that brings up `wg0 (10.42.42.1/24)` in the router pod,
four operator-selectable ingress modes for UDP 51820, and a least-privilege
NetworkPolicy. The chart NEVER installs MetalLB.

TDD: the six chart render-tests from the handoff plan were written first
(RED), then the templates made them GREEN.

## Blockers

- **Runtime WireGuard validation on the target cluster (Talos) is an operator
  step**, not testable from the chart layer. The sidecar needs `NET_ADMIN` /
  `NET_RAW`, access to `/dev/net/tun`, and an image providing `wireguard-tools`,
  `iproute2`, and `jq`. The Talos kernel-module-vs-userspace (`wireguard-go`)
  question (assumption A-IMG below) requires the operator to confirm on their
  node. The chart cannot prove the tunnel actually comes up without cloud VMs
  + a cluster — that is US-42.2 (day-one validation) + a live deploy.
- **US-42.2 (day-one validation)** must confirm AWS/OCI VM IPs are not blocked
  by `opencode.ai/zen/v1` (assumption A22) before the fleet is worth enabling.
  Manual operator step; no code.
- **No purpose-built `relay-wireguard` image is published yet.** The default
  image value (`ghcr.io/lenaxia/llmsafespaces/relay-wireguard`) follows the
  project's ghcr convention (same as `relay-router`, also unpublished); the
  operator must point `wireGuard.image` at an image with the required tools
  before enabling the subsystem. The router subsystem is disabled by default,
  so this never breaks a default install.

None block this PR's merge — the templates are render-validated and the
runtime concerns are documented assumptions for the operator, exactly like
Postgres/Redis infra.

## Assumptions (Rule 7) and validation

- **A-RESTART: "the controller restarts the router on topology changes."**
  **DISPROVED.** The handoff proposed an init container that renders `wg0.conf`
  once at pod start, citing this. A grep of `controller/internal/relay/` shows
  the reconciler only writes the `relay-router-peers` ConfigMap and
  `relay-router-wg` Secret (`reconciler.go` `syncPeerConfigMap` /
  `ensureRouterWGKey`); it never touches the router Deployment. Because the
  router pod starts at chart-install time (before any `InferenceRelay` CR
  exists), an init-only renderer would render an empty config and never pick up
  late-arriving peers. → **This is the strong reason to use a poll-based sidecar
  instead of the handoff's "option 1"** (the handoff explicitly allows option 2
  given a strong reason). Implemented as a polling entrypoint.
- **A-KEYPAIR: the controller's keypair contract.** Validated by reading
  `controller/internal/relay/constants.go` (`routerWGSecret = "relay-router-wg"`,
  keys `publicKey`/`privateKey`, base64 WG format) and `wireguard.go`
  `RenderRouterConfig` (`[Interface]` PrivateKey/Address/ListenPort + one
  `[Peer]` PublicKey/AllowedIPs per relay). The sidecar's renderer matches this
  format; AllowedIPs is `<peerWgIP>/32` so traffic to `10.42.42.x` routes to the
  right tunnel.
- **A-PEERS: the peers ConfigMap shape.** Validated in
  `router_configmap.go`: ConfigMap `relay-router-peers`, key `peers.json`,
  shape `{"relays":[{"id","wgIP","provider","state","publicKey"}]}`. Parsed
  with `jq`; relays without a `publicKey` are skipped (still provisioning).
- **A-WSLABEL: workspace pod selector.** Validated in
  `controller/internal/workspace/constants.go` + `pod_builder.go:43`: all
  workspace pods carry `app: llmsafespaces` + `component: workspace`. The
  NetworkPolicy uses these for the TCP 8080 ingress rule.
- **A-IMG: the WG image provides `wg`/`wg-quick`/`ip`/`jq` and `/dev/net/tun`
  is reachable; on Talos the kernel WG module vs userspace `wireguard-go`
  choice works.** **NOT mechanically validated** (no cluster + no image
  publish). Flagged as an operator-day-one task; the script logs a clear
  warning if `jq` is absent and degrades to interface-only (no peers) rather
  than crashing silently. `/dev/net/tun` is mounted via hostPath.
- **A-FLOAT: YAML numbers decode as `float64`.** Validated the hard way — the
  first test run failed because `sigs.k8s.io/yaml` decodes port numbers as
  `float64`, not `int`; added a `toInt` helper. (Defensive; not a system
  assumption.)

## Work Completed

### values.yaml (`controller.inferenceRelay.router.wireGuard`)
New block with `image` (default `ghcr.io/lenaxia/llmsafespaces/relay-wireguard`),
`address: 10.42.42.1/24`, `listenPort: 51820`, `syncIntervalSeconds: 5`,
`resources`, and the four-mode `ingress` config (`mode` default `external`,
`loadBalancerIP`/`loadBalancerClass`/`annotations`/`nodePort: 31820`/
`hostNetworkNodeSelectorLabel`). Comments document that the chart never installs
MetalLB and that `spec.wireGuard.routerEndpoint` is always operator-authoritative.

### relay-router-deployment.yaml (rewritten)
- Pod-level `securityContext` no longer forces `runAsNonRoot` (that would reject
  the root WG sidecar); `runAsNonRoot`/`runAsUser: 65534` moved to the
  `relay-router` main container, which still drops ALL caps and is read-only.
- New `wireguard` sidecar: `runAsUser: 0`, `capabilities.add: [NET_ADMIN,
  NET_RAW]`, command `/scripts/render-wg.sh`, env `WG_ADDRESS`/`WG_LISTEN_PORT`/
  `WG_SYNC_INTERVAL`, a named `wg-udp` container port (51820/UDP).
- Volumes: `wg-scripts` (the renderer ConfigMap, `defaultMode: 0755`),
  `wg-config` (emptyDir at `/etc/wireguard`), `wg-secret`
  (`relay-router-wg` Secret, `optional: true`), `dev-tun` (hostPath
  `/dev/net/tun`), plus the existing `peer-config` mounted in both containers.
- `hostNetwork` mode: `pod.spec.hostNetwork: true`, `dnsPolicy:
  ClusterFirstWithHostNet`, and a `nodeSelector` keyed on
  `llmsafespaces.dev/relay-router=true`; non-hostNetwork modes keep the
  operator's nodeSelector/tolerations/affinity.

### relay-router-wg-scripts.yaml (new ConfigMap)
Holds `render-wg.sh`, the sidecar entrypoint. It polls every
`WG_SYNC_INTERVAL`: renders `wg0.conf` from the Secret's `privateKey` +
the ConfigMap's peers (via `jq`), brings `wg0` up on first sight of a private
key, and hot-syncs peer changes via `wg syncconf` (against a stripped config
written to writable `/dev/shm`, since the root FS is read-only) without
dropping the tunnel. If the private key is absent it logs "waiting" and
retries — so the pod starts at chart-install time and picks up VMs the
controller provisions later. This is the fix for the disproved A-RESTART
assumption.

### relay-router-wg-service.yaml (new)
Second Service template, conditional on `ingress.mode`:
- `external` / `hostNetwork` → renders nothing.
- `loadBalancer` → `Service{type: LoadBalancer}` on UDP 51820, propagating
  `loadBalancerIP`/`loadBalancerClass`/`annotations`.
- `nodePort` → `Service{type: NodePort}` on UDP 51820 with pinned `nodePort`.

The existing TCP 8080 ClusterIP Service (`relay-router-service.yaml`) is
unchanged — it is the in-cluster HTTP endpoint workspace pods call.

### relay-router-networkpolicy.yaml (new)
Selects relay-router pods; `policyTypes: [Ingress, Egress]`.
- TCP 8080 ingress from workspace pods (`app/component=workspace`, in the
  workspace namespace) and from the controller pod (it scrapes router
  `/metrics`; it is not in the WG mesh).
- UDP 51820 ingress from **anywhere** (relay VMs dial in from public cloud
  IPs that are unknowable ahead of time and rotate; WireGuard public-key
  pinning is the auth).
- Egress: allow all (tunnel to `10.42.42.x`, DNS, Zen-direct fallback).

### chart_test.go (6 new tests + 2 helpers)
`TestRelayRouter_WGSidecar_HiddenWhenInferenceRelayDisabled`,
`…_RendersWhenInferenceRelayEnabled` (sidecar present, NET_ADMIN+NET_RAW, wg
Secret mounted, main container still drops ALL caps, no pod-level
`runAsNonRoot`),
`…_WGIngress_ExternalMode_RendersNoUDPService`,
`…_LoadBalancerMode_RendersLBService` (LB type + IP/class/annotations
propagated),
`…_NodePortMode_RendersNodePortService` (pinned 31820),
`…_HostNetworkMode_RendersHostNetworkPod` (`hostNetwork: true`, nodeSelector,
no WG Service), plus `TestRelayRouter_NetworkPolicy_RendersWhenEnabled`
(TCP 8080 from workspace/controller, UDP 51820 from anywhere, egress
present). Helpers: `podSpecMap`, `capSet`, `toInt` (float64-safe port
coercion), `hasWGUDPPort`, `svcType`.

## Key Decisions

1. **Poll-based sidecar, not init container.** The handoff's "option 1" (init
   renders once, re-render on pod restart) is broken: the controller does not
   restart the router on topology changes (A-RESTART disproved), and the pod
   starts before any peers exist. A polling entrypoint that re-renders +
   `wg syncconf`s is the only correct design for the actual ordering. Documented
   as the "strong reason" the handoff allows.
2. **Sidecar runs as root; main container stays non-root.** `wg-quick`/netlink
   need root; capabilities are scoped to `NET_ADMIN`+`NET_RAW` (no
   `privileged: true`). WG public-key pinning is the real auth boundary, not
   uid. The router main container still drops ALL caps and runs as 65534.
3. **NetworkPolicy: UDP 51820 open to the world.** Relay VM IPs are unknowable
   and rotate on every destroy/recreate; restricting by source would break the
   design. WG rejects unauthenticated handshakes before any app logic runs.
4. **Default image is the project's notional ghcr image, not linuxserver.**
   Matches the existing `relay-router` convention. Because
   `controller.inferenceRelay.enabled` defaults to `false`, a default install
   never pulls it; operators enabling the fleet configure the image then. The
   script warns clearly if `jq` is missing rather than failing opaquely.
5. **`/dev/net/tun` via hostPath.** Required for userspace `wireguard-go`; a
   standard VPN-sidecar pattern. The Talos kernel-vs-userspace question is left
   to operator day-one validation (A-IMG).

## Adversarial Self-Review (Rule 11)

- **Does the root sidecar weaken the pod's security posture vs. the old
  non-root-only pod?** NET_ADMIN+NET_RAW + root is a real privilege increase,
  but it is scoped to the sidecar container only; the main router container is
  unchanged (non-root, ALL caps dropped, read-only). The privilege is required
  for any in-pod WireGuard and is the established Epic-32 VPN-sidecar pattern.
  The NetworkPolicy + the WG-only listen bound to the mesh CIDR bound the
  exposure.
- **Could `wg syncconf` drop in-flight streams on peer change?** No —
  `wg syncconf` updates the peer set without bringing the interface down; only
  `wg-quick up`/`down` touch the link, and those run only on first bring-up.
  In-flight streams to existing peers survive a peer add/remove.
- **What if peers.json is malformed / jq absent?** `render_conf` gates peer
  emission behind `have_jq` and a non-empty file; a parse failure leaves the
  `[Interface]` section intact (interface up, no peers) and the loop retries.
  The script never `set -e`-aborts on a render failure (it logs + returns).
- **Does hostNetwork mode break the TCP 8080 ClusterIP Service?** No — that
  Service still selects the pod by label; `hostNetwork: true` changes the pod's
  IP but the in-cluster Service still routes to it. `dnsPolicy` is set to
  `ClusterFirstWithHostNet` so the router can still resolve upstreams.
- **Is the NetworkPolicy's "egress allow all" too broad?** It is intentional:
  the router dials relay VMs over the WG tunnel (whose peer IPs change), needs
  DNS, and must reach Zen directly in fallback mode. Tightening would require
  enumerating cloud egress IPs that are not knowable. The mesh CIDR + WG are
  the real controls.
- **Do the chart tests assert real behavior or just presence?** They assert
  structural contracts (sidecar caps, mounts, Service type/ports/annotations,
  hostNetwork flag, nodeSelector, NetworkPolicy ports/sources) by parsing
  `helm template` output — the same path operators run. Runtime WG behavior is
  not chart-testable; that is the documented operator day-one validation.
- **Stale `0362` references?** None — new templates reference the Epic 42
  Layer 2 design + this worklog (0386), avoiding the 0362/0385 renumber
  ambiguity from PR #273.

Zero real findings beyond the operator-day-one runtime assumptions already
flagged in Blockers.
