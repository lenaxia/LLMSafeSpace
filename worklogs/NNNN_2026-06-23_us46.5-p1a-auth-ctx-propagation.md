## Objective

Execute P1-a of the #224 audit: add `context.Context` to the auth token
validation fast-path so request deadlines/cancellation reach the Redis cache +
DB calls that run on **every authenticated request**. This is the
highest-payoff item in the recommended migration order posted to #224.

---

## Work Completed

### Signature changes (`api/internal/services/auth/auth.go`)

- `ValidateToken(ctx context.Context, tokenString string) (string, error)`
- `ValidateTokenWithClientIP(ctx context.Context, tokenString, clientIP string) (string, error)`
- `validateAPIKey(ctx context.Context, apiKey, clientIP string) (string, error)`

The three `context.WithTimeout(context.Background(), 5*time.Second)` sites now
derive from the caller's ctx: `context.WithTimeout(ctx, 5*time.Second)`. The 5s
per-call cap is **retained** (safety bound) while gaining request-cancellation
propagation ‚Äî strictly additive, least surprising.

### Callers updated

- `AuthMiddleware` / `OptionalAuthMiddleware` (`auth.go`): pass `c.Request.Context()`.
- `middleware/auth.go`: `authService.ValidateToken(c.Request.Context(), token)`.
- `api/internal/interfaces.AuthService` interface: `ValidateToken(ctx, token)`.
- Mocks: `MockAuthMiddlewareService`, `MockAuthService` ‚Äî accept ctx, match on
  token (existing `On("ValidateToken", token)` setups unaffected).
- All test call sites (`auth_test.go`, `auth_apikey_dek_test.go`,
  `auth_revocation_test.go`): pass `context.Background()`.

### Tests

- New `TestValidateTokenWithClientIP_PropagatesContext` (TDD): injects a
  context-value sentinel and asserts the cache `Get` receives a ctx carrying it
  (a `context.WithTimeout(parent,‚Ä¶)` child preserves parent values, so a
  `context.Background()` regression would leave the sentinel absent and fail the
  matcher). TDD: written first, confirmed red, then implemented.

## Key Decisions

- **Derive from ctx, don't drop the timeout.** `context.WithTimeout(ctx, 5s)`
  preserves the existing per-call safety bound AND adds cancel propagation. Using
  `ctx` directly would remove the cap (an HTTP request with no deadline could
  hang the cache/DB indefinitely). Conservative and correct.
- **Behavioral change (flagged in the PR):** previously, a client disconnect did
  NOT cancel in-flight token validation (it used `context.Background()`). Now it
  does ‚Äî the intended improvement (cancellation short-circuits the work).
- **Match on token in mocks, not ctx.** ctx-matching is rarely useful in mock
  assertions and would have broken every existing `On("ValidateToken", token)`
  setup. Accept ctx, pass only token to `Called`.
