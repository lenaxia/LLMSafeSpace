# Worklog: Epic 30 — Unified Credential Model (Complete)

**Date:** 2026-06-05
**Session:** Full Epic 30 implementation (US-30.1 through US-30.14)
**Status:** Complete

---

## Objective

Replace the three-system credential split (user_secrets for LLM keys, credential_sets for admin, hardcoded OPENCODE_AUTH_CONTENT for free-tier) with a single unified `provider_credentials` table and injection pipeline that merges all sources by priority.

---

## Design Rationale

### Why unify?
- `credential_sets` (Epic 9) had zero runtime effect — dead wiring
- `user_secrets.llm-provider` was the only working LLM path
- `OPENCODE_AUTH_CONTENT` was hardcoded in the controller, requiring redeployment to change
- Three different encryption schemes, three different tables, three injection paths
- Adding org-level credentials (future) would require a fourth system without unification

### Key design decisions:
1. **Single `provider_credentials` table** with `owner_type` discriminator (user/org/admin)
2. **`workspace_credential_bindings`** as the sole source of truth for runtime injection
3. **Two-key priority sort** (source_type then within_priority) — explicit always beats auto
4. **`credential_auto_apply`** is configuration-only; seeding happens at workspace creation
5. **Free-tier key is a real admin credential** seeded at API startup (not SQL migration)
6. **Legacy fallback** when `deriveAdminKey` is nil — backwards compatible deployment

## Architecture After Epic 30

```
provider_credentials (unified table)
├── owner_type='admin' + owner_id='_platform'  → server-KEK encrypted
├── owner_type='user'  + owner_id=user_id      → user-DEK encrypted
└── owner_type='org' (future, Epic 29)         → org-DEK encrypted

workspace_credential_bindings (runtime source of truth)
├── source_type='explicit' (user-bound)        → highest priority
└── source_type='auto' (seeded from auto-apply)→ lower priority

credential_auto_apply (configuration-only)
└── Drives seeding at workspace creation + backfill endpoint

PrepareSecretsForInjection:
  1. Query bindings (ordered by priority)
  2. Decrypt by owner_type (user DEK or admin KEK)
  3. Deduplicate by provider (first wins)
  4. Merge with non-LLM secrets from user_secrets
  5. Return []InjectedSecret JSON for materializer
```
