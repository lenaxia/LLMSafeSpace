## Architecture Notes for the Next Developer

1. **Org DEK cache key:** `"org:<orgID>"` in Redis. Namespace-safe (can't collide with JWT jti UUIDs).

2. **KEK domain separation:** `orgKEKInfo = "llmsafespace-org-kek"` — must differ from user KEK info string `"llmsafespace-kek"`.

3. **Three DEK stores exist now:**
   - `user_keys` — per-user DEK (existing, unchanged)
   - `org_key_members` — per-admin wrapped org DEK (new)
   - Redis `DEKCache` — shared for both, keyed by `sessionID` or `"org:<orgID>"`

4. **verifyOwner / verifyOrgAdmin:** Workspace service has dual ownership checks. Read ops use `verifyOwner` (user owns OR is org member). Destructive ops use `verifyOrgAdmin` (user owns OR is org admin).

5. **SeedWorkspaceCredentials** takes `orgID *string` — nil = personal, non-nil = org. Three SQL blocks: user bindings, org auto-apply, org credentials.

6. **Pending key wrap flow:** When adding an admin, `pending_key_wrap=true` is set. The invited admin calls `POST /orgs/:id/accept-key` with their password. Server wraps org DEK with their KEK, inserts row, clears pending. Member role (`role='member'`) has no key wrap — never gets org_key_members row.

7. **RotateOrgDEK is transactional:** Re-encryption of all org credentials, deletion of old key members, insertion of rotating admin's new key, and setting pending for other admins all happen in one pgx transaction. Rollback if any step fails.
