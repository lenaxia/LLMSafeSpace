## Objective

Implement US-44.10 from `design/stories/epic-44-session-reliability-transparency/README.md`. When opencode (upstream) is restarting, the API proxy buffers `POST /api/v1/workspaces/:id/sessions/:sessionId/message` (the `SendMessage` handler) тАФ holding the HTTP connection open up to 30s, max 10 per workspace тАФ instead of returning 503. ## Work Completed

### Implementation (TDD)

- **New `api/internal/handlers/proxy_request_buffer.go` (198 lines):** `requestBuffer` type with per-workspace FIFO queues and exactly one drainer goroutine per non-empty queue. The drainer attempts to forward the head request; on a connection error it backs off (`pollInterval`, default 500ms) and retries; on success or terminal error it pops and delivers. Sentinel errors: `errBufferTimeout`, `errBufferCommitted` (partial client write тЖТ terminal, not retried), `errBufferRetryLater` (connection pool full тЖТ retryable), `errClientGone` (client disconnected тЖТ terminal). Empty queues are pruned from the map and their gauge metric series deleted.
- **Trigger point (`api/internal/handlers/proxy.go`):** in `proxyToWorkspaceWithErrBody`, AFTER the existing stale-IP retry, on a persistent connection error, gated to `bufferable=true` (only `SendMessage` passes it). All other handlers pass `bufferable=false` and keep the existing 503 behavior.
- **Release-on-park + acquire-in-forward:** a parked request holds no upstream socket, so it releases its connection slot on park; the forward closure re-acquires (briefly, one at a time тАФ the drainer is sequential) per attempt. This makes the buffer the real gate (not `maxConnectionsPerWorkspace`) and removes a workspace-wide availability cliff. The top-level deferred release is guarded by a `slotReleased` flag so `connCount` is decremented exactly once per request.
- **Metrics (`api/internal/services/metrics/metrics.go`, +67):** four package-level `promauto` metrics тАФ `workspace_request_buffer_timeout_total`, `workspace_request_buffer_wait_seconds` (histogram buckets 0.5/1/2/5/10/30), `workspace_request_buffer_size` (gauge), `workspace_request_buffer_full_total` тАФ plus `Record*`/`Set`/`Delete` helpers, matching the agentd `workspace_*` ops-metric family.
- **Config (`api/internal/config/config.go`, +16):** `Proxy` struct section + env overrides `LLMSAFESPACE_PROXY_REQUESTBUFFERSIZEPERWORKSPACE` / `LLMSAFESPACE_PROXY_REQUESTBUFFERTIMEOUTSECONDS`, defaults (10 / 30s) coerced in `SetRequestBufferConfig`.
- **Wiring (`api/internal/app/app.go`, +1):** `proxyHandler.SetRequestBufferConfig(...)` called after construction, before `Start()`.
- **Tests:** `proxy_request_buffer_test.go` (779 lines) тАФ 8 buffer unit tests + 8 HTTP integration tests + config tests, including FIFO ordering, default-config buffer-full reachability (11th request тЖТ 429), the committed-write-no-retry regression, and the no-double-release accounting regression.

### Skeptical validation loop (3 rounds тЖТ zero findings)

**Round 1 тАФ validator found 3 MAJOR:**
- **F1** тАФ drainer retried `forward()` after `doProxy` had already written to `c.Writer` (mid-stream "connection reset" matches `isConnectionError`) тЖТ second write corrupts the response.
- **F2** тАФ handler `select` could pick `ctx.Done` over a delivered success (`nil`) тЖТ skipped metering/activity recording and wrongly removed the active session.
- **F3** тАФ parked requests held a connection slot for up to 30s; since `maxConnectionsPerWorkspace` (10) == default `maxSize` (10), the spec's buffer-full 429 + metric were unreachable in production and 10 parked POSTs starved all other request types.

**Round 2 тАФ remediation:** F1 fixed via `errBufferCommitted` sentinel + `c.Writer.Written()` guard; F2 fixed by restructuring the handler to always learn the drainer's outcome (close `cancelCh`, then block on `<-result`) тАФ during which the **implementer independently caught** that the suggested non-blocking `deliver` would deadlock (~50% drop тЖТ handler hang) and proved a blocking send is deadlock-free (drainer is sole sender, delivers exactly once, always paired with `popHead`); F3 fixed via release-on-park + acquire-in-forward.

**Round 2 re-validation found 1 MAJOR (NF1, my own instruction's fault) + 3 minors:**
- **NF1** тАФ the F3 instruction's claim that the top-level `defer releaseConnection` would "no-op" was wrong (`releaseConnection` only guards against going below 0, not against double-decrement when other connections keep count > 0) тЖТ every parked request's return over-decremented `connCount` тЖТ `maxConnectionsPerWorkspace` could be exceeded.
- NF2 тАФ serial-FIFO head-of-line blocking (bounded, spec-literal).
- NF3/NF4 тАФ missing e2e committed-write test + missing default-config buffer-full test.

**Round 3 тАФ remediation:** NF1 fixed with a `slotReleased`-guarded deferred release; per-request accounting proven 0 for every path; regression test verified to **FAIL against the buggy version and PASS with the fix**. NF3/NF4 tests added; NF2 documented as a known trade-off.

**Round 3 confirmation:** validator returned **PASS тАФ zero real findings**, ready to merge.

## Key Decisions

1. **Connection-error-triggered, not health-probe-triggered.** There is no per-pod health check in the proxy; `isConnectionError` (`proxy_helpers.go:43`) on the forward path is the unhealthy signal. Avoids a new K8s/agentd dependency and matches the actual failure mode (connection refused during opencode's restart).
2. **Release-on-park + acquire-in-forward (F3 fix).** Parked requests hold 0 slots; the sequential drainer holds 1 slot during a forward. Makes the buffer-full path reachable in default config and removes the availability cliff. Required the `slotReleased` guard (NF1) to avoid double-release.
3. **Blocking `deliverBuffered` send, not non-blocking.** The F2 restructure (handler always receives the drainer's outcome) made a non-blocking deliver deadlock-prone; blocking is provably deadlock-free. The drainer is the sole sender of each request's cap-1 `result` channel and delivers exactly once (always paired with `popHead`).
4. **Serial FIFO forwarding.** Spec-literal ("forwards buffered requests FIFO"). Known trade-off (NF2): a long first buffered turn can cause later buffered requests to time out (bounded by the per-request 30s deadline; no infinite hang). Concurrent forwarding once healthy is a documented potential follow-up.
5. **`podIP` captured at park is correct for scope.** In-place opencode restart (env-secret/api-key) keeps podIP stable (same pod); pod-recreating restarts (suspend/resume) go through the not-Active 503 path, never the buffer.
6. **Buffering always enabled** (defaults 10/30s, configurable up). No production kill-switch; `maxSize=0` disable is test-only.
