### Post-Review Adversarial Pass ŌĆö Two Real Findings Fixed

#### F1: S36.3 statusz tests called a hand-rolled handler copy, not `buildStatuszHandler`

**Root cause:** The S36.3 tests constructed anonymous `http.HandlerFunc` closures that replicated the production `statuszHandler` logic independently. If the production closure changed (e.g., a bug in the ContextUsed loop or wrong variable capture), the tests would still pass.

**Fix:** Extracted the production closure into a package-level `buildStatuszHandler(client, cache, tracker, startedAt)` function (`cmd/workspace-agentd/main.go`). `main()` now calls `buildStatuszHandler(...)` rather than defining the closure inline.

Two new integration tests added:
- `TestBuildStatuszHandler_ContextUsed_PerSession` ŌĆö calls `buildStatuszHandler()` directly, verifies ContextUsed=55000 threads from tracker into session
- `TestBuildStatuszHandler_NoContextUsed_WhenTrackerEmpty` ŌĆö cold start, ContextUsed=0

#### F2: Compaction indicator never fires in Active phase (design flaw)

**Root cause:** `useWorkspaceStatus` in `useWorkspaces.ts` had `refetchInterval: false` for Active phase. The compaction detection `useEffect` watches `sessionStatus?.contextUsed`, which only changes if the query re-fetches. With no Active-phase poll, the compaction drop was never seen.

The existing compaction tests were passing only because they called `qc.invalidateQueries()` manually ŌĆö an invalidation event that the production code never generates for an Active workspace.

**Fix:** `useWorkspaceStatus` now polls at 30s for Active phase (`refetchInterval: 30_000`). Rationale: context data is not delivered via SSE ŌĆö it is set by the controller's health loop from agentd statusz. Without a poll, context bars and compaction detection would be permanently stale after initial page load.

Updated `useWorkspaces.ts` and `useWorkspaces.test.tsx`:
- Replaced `"does not poll after fetching Active phase"` test with `"polls every 30s for Active phase to keep context usage fresh"`
- Replaced `"does not poll for any phase"` test with `"polls every 3s for transitioning phases and every 30s for Active phase"`

## Adversarial Self-Review

### Phase 1: Potential Weaknesses

1. **React `enabled: false` stale subscription** ŌĆö `useQuery({enabled: false, staleTime: Infinity})` in Sidebar does NOT make a network fetch. It only reads from the cache. When ChatPage is not mounted (user hasn't opened a workspace), the cache may be empty. **Assessment**: False alarm ŌĆö if cache is empty, `wsStatus` is `undefined`, `contextBySessionId` is an empty map, and no indicators are shown. Graceful degradation.

2. **Compaction detection false positive at session load** ŌĆö on first render `prevContextUsedRef.current` is `undefined`. The `useEffect` only sets compaction when `prev != null`, so the first render never triggers it. **Assessment**: False alarm ŌĆö correct behavior.

3. **Compaction detection on session navigation** ŌĆö `prevContextUsedRef.current` is reset to `undefined` in the `sessionId` effect, so navigating to a new session with a high contextUsed won't trigger a false compaction. **Assessment**: False alarm ŌĆö correctly handled.

4. **`enabled: false` never fetches** ŌĆö the workspace-status query in `WorkspaceSessionList` has `enabled: false` meaning it reads from the cache but never fetches. This is correct: ChatPage's `useWorkspaceStatus` with `staleTime: 0` keeps the cache fresh while the user is on a workspace. However, if the sidebar is shown while ChatPage is NOT mounted (e.g., navigating away), the cache entry may become stale. **Assessment**: Acceptable ŌĆö context indicators are best-effort UX polish, not safety-critical. The DiskUsageBar in ChatPage is the authoritative context display.

5. **`no space left on device` during Go test runs** ŌĆö not a code bug, environment constraint. Individual package tests run fine; running all packages simultaneously exhausts /tmp or workspace tmp.

### Phase 2: Validated Findings

No real bugs found. All findings were false alarms (addressed above).

## Key Decisions

| Decision | Rationale |
|---|---|
| `enabled: false, staleTime: Infinity` in Sidebar query | Read from ChatPage's cache without triggering independent network requests. Sidebar should not generate extra load. |
| Compaction threshold 50% | Consistent with "opencode auto-compact halves the context by summarising history" behaviour. A 30% fluctuation from normal model behaviour should not trigger the banner. |
| Inline formatTokens in Sidebar | Avoids importing from DiskUsageBar (which is a UI component, not a utility). Three-line inline keeps it simple. |
| Dismiss-only compaction banner | Compaction is informational, not actionable. No "undo" button needed. |
