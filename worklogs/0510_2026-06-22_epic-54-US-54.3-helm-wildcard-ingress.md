## Objective

## Work Completed

### values.yaml — new `orgSubdomainRouting` block

Added an opt-in block at the top level (alongside the existing `oidc:` block):

```yaml
orgSubdomainRouting:
  enabled: false          # default off — single-host deploys unaffected
  baseDomain: ""          # e.g. "app.example.com"
  cookieDomain: ""        # e.g. ".app.example.com" (leading dot required)
  wildcardCert:
    tlsSecret: ""         # existing TLS Secret (skips Certificate rendering)
    issuerRef:
      name: ""            # cert-manager Issuer/ClusterIssuer name
      kind: "ClusterIssuer"
```

Documented prerequisites (wildcard DNS, cert-manager, ingress controller) inline. Default is `enabled: false` — backward compatible.

### Templates

**`templates/frontend-wildcard-ingress.yaml`** (new, 48 lines):
- Renders when `orgSubdomainRouting.enabled && frontend.enabled && frontend.ingress.enabled`.
- Single Ingress with `host: "*.<baseDomain>"`, routing `/api` → API service and `/` → frontend service.
- Reuses `frontend.ingress.className` + annotations (security headers, CSP, HSTS — same as the primary ingress).
- TLS block references `<release>-frontend-wildcard-tls` (or the operator-supplied `tlsSecret`).
- Additive to `frontend-ingress.yaml` — the primary host + `additionalHosts` continue to render unchanged.

**`templates/frontend-wildcard-cert.yaml`** (new, 42 lines):
- Renders when `orgSubdomainRouting.enabled && frontend.ingress.tls && !wildcardCert.tlsSecret`.
- cert-manager `Certificate` for `*.<baseDomain>` + `<baseDomain>` (both — the base domain covers root-domain requests).
- Uses `required` on `issuerRef.name` so a misconfigured `enabled=true` without an issuer fails at `helm template` time, not at runtime.
- Duration 8760h (1 year), renewBefore 720h (30 days) — matches the webhook cert pattern.
- When `tlsSecret` is set (operator has an external wildcard cert), the template is skipped and the Ingress references the operator's Secret.

**`templates/configmap-api.yaml`** (modified):
- Emits `orgSubdomainRouting.baseDomain` + `cookieDomain` into `config.yaml` when the block is present.
- Viper reads these into `cfg.OrgSubdomainRouting.BaseDomain` / `.CookieDomain` (struct added in US-54.1, PR #345).

### Cookie domain wiring

**`api/internal/server/router.go`**:
- Added `CookieDomain string` field to `RouterConfig`.
- `setSessionCookie` signature changed: `func setSessionCookie(c *gin.Context, token string, maxAge int, cookieName, cookieDomain string)`. When `cookieDomain` is non-empty, it's passed as the `Domain` attribute to `c.SetCookie`; when empty, behavior is unchanged (host-only cookie — current default).
- `registerAuthRoutes` signature updated to thread `cookieDomain` through to the login/register handlers that call `setSessionCookie`.

**`api/internal/handlers/org_sso.go`**:
- Added `cookieDomain` field to `SSOHandler` struct.
- `NewSSOHandler` signature updated: now accepts `cookieDomain` between `sessionCookie` and `frontendURL`.
- The SSO callback's session-cookie setter (`c.SetCookie(h.sessionCookie, ...)`) now passes `h.cookieDomain`.
- The PKCE state cookie setters (`clearStateCookie`, state cookie issuance) are left host-only — the state cookie is short-lived (10min TTL) and only used within the same-domain IdP→callback flow, so it doesn't need cross-subdomain visibility.

**`api/internal/app/app.go`**:
- `NewSSOHandler` call updated to pass `cfg.OrgSubdomainRouting.CookieDomain`.
- `RouterConfig` construction passes `CookieDomain: cfg.OrgSubdomainRouting.CookieDomain`.

**`api/internal/handlers/org_sso_test.go`**:
- All 4 `NewSSOHandler` test call sites updated with `""` (empty cookieDomain — preserves existing host-only behavior in tests).

### Chart tests (5 new, TDD)

All in `charts/llmsafespaces/chart_test.go`, following the existing `helm template` subprocess pattern:

1. **`TestEpic54_DefaultRender_NoWildcardResources`** — default render produces zero wildcard Ingress or Certificate resources (backward compat).
2. **`TestEpic54_WildcardEnabled_RendersIngressAndCert`** — enabling renders both resources with correct host (`*.<base>`), DNS names (wildcard + base), TLS secret name, and issuerRef.
3. **`TestEpic54_WildcardEnabled_NoCertWhenTlsDisabled`** — when `tls=false`, the Ingress renders without TLS, and no Certificate renders.
4. **`TestEpic54_ExistingTLSSecret_NoCertificateRendered`** — when `tlsSecret` is set, no Certificate renders; the Ingress references the operator's Secret.
5. **`TestEpic54_ConfigMapEmitsOrgSubdomainRouting`** — the API ConfigMap includes `orgSubdomainRouting:` with `baseDomain` and `cookieDomain` when configured.

## Key Decisions

- **`required` on `issuerRef.name`.** A misconfigured `enabled=true` without an issuer (and without `tlsSecret`) fails at `helm template` time via Helm's `required` function — not at pod startup. Better to fail fast.
- **Both `*.<base>` and `<base>` in the Certificate dnsNames.** The base domain covers root-domain requests (`staging.safespaces.dev` without a subdomain). Without it, root-domain HTTPS would get a TLS warning. The spike (worklog 0509) confirmed this is needed.
- **State cookie stays host-only.** The PKCE state cookie (`lsp_sso_state`) is set and consumed within the same SSO start→callback flow on the same domain. Cross-subdomain visibility would be unnecessary and would widen the cookie's scope for no benefit.
- **Additive template, not modifying `frontend-ingress.yaml`.** The wildcard Ingress is a separate resource (`<release>-frontend-wildcard`) alongside the existing `<release>-frontend`. This keeps the primary ingress logic untouched and makes it easy to disable wildcard routing without affecting the primary host.
