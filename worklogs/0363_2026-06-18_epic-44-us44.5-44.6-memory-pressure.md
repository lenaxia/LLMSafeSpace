## Work Completed

### US-44.5 — Memory Pressure Warning

**New file: `cmd/workspace-agentd/memory_pressure.go`**
- `memoryPressureMonitor` — goroutine-safe monitor that periodically reads cgroup v2 `memory.current` / `memory.max` and tracks whether usage exceeds the 85% threshold.
- `check()` — reads cgroup, computes ratio, updates state. Returns true if state changed (for logging).
- `run(ctx)` — periodic loop (60s default, configurable via `MEMORY_CHECK_INTERVAL_MS`). Immediate check on startup. Threshold configurable via `MEMORY_WARNING_THRESHOLD`.
- `snapshot()` — returns `(pressure bool, usedBytes, maxBytes int64)` for statusz enrichment.
- `readCgroupMemoryMax()` — reads `/sys/fs/cgroup/memory.max`, returns 0 for "max" (unlimited).

**Wired into main.go:**
- `pressureMonitor` goroutine started alongside the existing ops-metrics loop.
- `buildStatuszHandler` accepts the monitor and enriches `StatuszResponse.MemoryPressure`.

**New CRD type + condition:**
- `WorkspaceConditionMemoryPressure` — condition type in `pkg/apis/llmsafespace/v1/workspace_types.go`.
- `ReasonMemoryPressure` — reason constant.

**Controller enrichment (`health.go`):**
- `enrichAgentStatus` reads `status.MemoryPressure` from statusz. When true, sets `WorkspaceConditionMemoryPressure=True` with a message like "memory usage high (85%). Consider reducing concurrent sessions or increasing workspace memory limit." When false, auto-clears the condition (mirrors the existing disk-pressure pattern).

**Statusz response (`pkg/agentd/types.go`):**
- Added `MemoryPressure bool` field to `StatuszResponse`.

### US-44.6 — Per-Session Memory Attribution

**`pkg/agentd/types.go`:**
- Added `EstimatedMemoryMB int64` to `SessionInfo`.

**`cmd/workspace-agentd/main.go` (statusz handler):**
- Each session in the statusz response is enriched with `estimateSessionMemoryMB(contextTokens)`.
- Formula: `(contextTokens × 2 bytes) / 1MiB`. The 2 bytes/token factor approximates in-memory representation overhead.
- Token source: `Tokens.Input + Tokens.CacheRead + Tokens.CacheWrite` (falls back to `ContextUsed` if Tokens is nil).

**`cmd/workspace-agentd/memory_pressure.go`:**
- `estimateSessionMemoryMB(contextTokens int64) int64` — pure function, easily testable.

---

## Key Decisions

1. **Monitor lives in agentd, not the controller.** agentd has direct cgroup v2 visibility (`/sys/fs/cgroup/memory.current`). The controller would need a remote call or a new statusz field. By doing the check in agentd (where the helpers already exist), the controller just reads a boolean from statusz — consistent with the disk-pressure pattern.

2. **85% threshold is configurable via env var.** `MEMORY_WARNING_THRESHOLD=0.85` default, overridable without rebuild. This matches the US-44.5 spec.

3. **Per-session memory is an estimate, not a measurement.** opencode is third-party and we can't instrument its allocator. The 2-bytes/token heuristic gives a relative signal — enough for users to identify "which session is heavy" and drop it. The absolute number is intentionally approximate.

4. **Condition mirrors disk-pressure pattern.** `WorkspaceConditionMemoryPressure` auto-clears when pressure drops, never triggers a pod restart, and surfaces a user-actionable message. The frontend can render an amber banner when the condition is True (same UX as disk pressure).

5. **Immediate check on startup.** The monitor does its first `check()` before entering the ticker loop, so statusz has pressure data right after pod boot — not 60s later. This matters for the resume case where the previous pod was OOM-killed.
