## Summary

Built `local/bootstrap.sh` + `local/test.sh` and ran them to green against a fresh kind cluster. Each iteration surfaced one or more real production bugs, all of which were verified from cluster logs/events (not guessed) before being fixed with TDD coverage. Final state: opencode v1.2.27 actually starts inside a sandbox pod, listens on 0.0.0.0:4096, and returns 200 on `/global/health` when authenticated with the credentials LLMSafeSpace's controller mounts from the sandbox-pw secret.

This session uncovered and fixed **14 verified bugs**. None were guessed; every fix was reproduced from a real cluster failure mode (or unblocked by reading the actual code path that was firing). One earlier guess that turned out to be wrong is documented in the "process notes" section as a teaching moment.

## Verified production bugs fixed

Tests written for each (TDD red → green → refactor). Bugs 1–11 covered by commit `a864768`, 12–14 by `c5328ba`, and 15–18 by this session's tail commits.

### Initial discovery while validating runtime image

1. **runtimes/base/Dockerfile pointed at the wrong upstream.** `opencode-ai/opencode v1.0.0` returned 404. The real project is `anomalyco/opencode`, currently v1.2.27 (matching the user's local install). Asset format is `opencode-linux-x64.tar.gz` (not `opencode-linux-amd64`). No `.sha256` files are published — supply-chain verification gap documented in code.
2. **All Go-image Dockerfiles broke without proxy.golang.org.** Added `GOPROXY` build-arg with sane default but overridable, so builds work in environments where the public Go module proxy is blocked.
3. **`pkg/crds/sandbox_crd.yaml` was missing the `workspaceRef` field.** The Go type had it, the CRD schema didn't. K8s CRDs default to structural-schema pruning, so every Sandbox POST silently dropped `spec.workspaceRef` — sandboxes never linked to their workspace PVC. Fixed in CRD + chart copy.

### Discovered while installing the chart

4. **Helm Secret was a regular template; migration Job (a hook) ran first and couldn't find it.** Converted the Secret to a `pre-install,pre-upgrade` hook with weight=-15 (lower than migrations ConfigMap at -10 or migrate Job at -5) and `helm.sh/resource-policy: keep` to avoid rotating JWT/postgres password on upgrade.

### Discovered while watching pods crashloop

5. **API panicked with "Lock identity is empty" on first leader-election attempt.** Viper's `AutomaticEnv` doesn't translate `LLMSAFESPACE_KUBERNETES_PODNAME` → `kubernetes.podname`. Added explicit env-var read, plus `os.Hostname()` fallback so leader election can never panic on a misconfigured deployment.
6. **Controller bound :8080 twice.** Custom `metrics.SetupMetrics` started its own HTTP server on :8080 in addition to controller-runtime's manager metrics server. Removed the custom server; the manager's metrics endpoint already serves all globally-registered Prometheus collectors at the configured `Metrics.BindAddress`.
7. **`pkg/kubernetes/client.go LlmsafespaceV1()` ignored a `rest.RESTClientFor` error and returned a half-initialized client.** Result: `SandboxWatcher.Watch` and the proxy ownership middleware nil-panicked on every CRD operation. Fixed to delegate to the (correct) `newLLMSafespaceV1Client` constructor that sets ContentConfig.GroupVersion, APIPath, and serializer.
8. **Security middleware sent kubelet probes to HTTPS via 301.** Default config has `RequireHTTPS: true`; kubelet uses HTTP and doesn't follow redirects, so liveness/readiness probes always failed and pods entered CrashLoopBackOff. Added `/livez`, `/readyz`, `/health`, `/metrics` to the security middleware skip list (matching the pattern already used in auth/logging/metrics middleware).

### Discovered while reconciling Workspace and Sandbox CRDs

9. **`controller/internal/resources/register.go AddToScheme` didn't call `metav1.AddToGroupVersion`.** Without it, client-go's reflector failed: `v1.ListOptions is not suitable for converting to "llmsafespace.dev/v1"`. The controller silently watched nothing — no reconciles ever fired. Fixed by adding the missing call.
10. **Workspace controller waited forever on PVCs in `WaitForFirstConsumer` mode.** kind's default StorageClass (and most production cloud storage classes) use WFC; PVCs stay Pending until a Pod schedules. The reconciler needed to treat WFC PVCs as "ready" so downstream Sandbox pods could trigger binding. Added `pvcUsesWaitForFirstConsumer` lookup; granted `storage.k8s.io/storageclasses` RBAC.
11. **All three admission webhooks panicked with nil decoder on every request.** `controller-runtime` v0.15+ removed the `InjectDecoder` DI callback; webhook decoders must be set at construction. Fixed `main.go` to call `admission.NewDecoder(mgr.GetScheme())` and pass it into each validator. Exported `Decoder` field on the validator structs (kept `InjectDecoder` as a no-op for backwards compatibility).

### Discovered while creating sandbox pods

12. **K8s pod label values can't contain `:`.** Controller wrote `metadata.labels.runtime = "python:3.11"` directly from `Spec.Runtime`. Pod creation failed with "Invalid value: 'python:3.11'". Added `sanitizeLabelValue` (`:` → `_`).
13. **Controller used `sandbox.Spec.Runtime` directly as the container image.** This pulled `python:3.11` from Docker Hub, ignoring the entire RuntimeEnvironment CRD that had been sitting unused. Added `runtime_resolver.go`: lookup by exact name → `:`-replaced name → `language+version` match, with a fully-qualified-image escape hatch. 9 unit tests cover all four lookup paths including determinism on tie and a helpful error on miss.
14. **Pods rejected by kubelet with "container has runAsNonRoot and image has non-numeric user (sandbox)".** runtime-base's Dockerfile uses `USER sandbox` (a name). Without an explicit numeric uid, kubelet can't pre-flight the runAsNonRoot check. Set `RunAsUser`/`RunAsGroup`/`FSGroup` at the pod-spec level (defaulting to 1000, overridable via `Sandbox.Spec.SecurityContext`).

### Discovered while iterating on workspace re-create + suspend

15. **Workspace re-create raced with controller cache lag for the deleted PVC.** Cache returned a terminating PVC; reconciler treated it as existing and skipped re-creation. New Sandbox pods then failed to schedule because the PVC was gone. Added staleness detection: PVC with `DeletionTimestamp != 0` OR mismatching OwnerReference UID is treated as not-found. Also handles `IsAlreadyExists` on Create as transient. Test fixtures without OwnerReferences continue to work (treated as adopted).
16. **Workspace controller's `deleteWorkspacePods` selector found no pods.** It selects by `llmsafespace.dev/workspace=<name>`, but the sandbox controller wasn't tagging pods with that label. Added `common.LabelWorkspace` constant and propagation to pod labels when `WorkspaceRef` is set.
17. **Workspace controller's `updateSandboxesToSuspended` selector found no Sandboxes.** Same root cause as #16, but the **Sandbox CRD itself** was also missing the workspace label. Sandbox controller now propagates `Spec.WorkspaceRef → metadata.labels[llmsafespace.dev/workspace]` on first reconcile (then requeues so the next reconcile sees the labeled object).
18. **`handleRunningSandbox` raced workspace-suspend → marked Sandbox `Failed` instead of `Suspended`.** When the workspace controller deletes the pod during suspend, the sandbox controller observes the deleted pod via informer and unconditionally marks the sandbox Failed. Added `parentWorkspaceIsSuspending` lookup; if the parent workspace is in `Suspending` or `Suspended` phase, the sandbox transitions to `Suspended` instead.

## Local testing infrastructure (new files)

- **`local/kind-cluster.yaml`** — single-node kind config with port-mapping for the API service.
- **`local/postgres-redis.yaml`** — minimal Postgres + Redis manifests for kind-only testing.
- **`local/bootstrap.sh`** — orchestrates: prerequisite checks, kind cluster create (idempotent), disk hygiene (buildx prune + image prune + drop our own previous tags from the kind node), image build + load (one image at a time to keep `/tmp` use small), cert-manager install, Postgres+Redis install, helm install LLMSafeSpace. Env knobs: `SKIP_BUILD`, `SKIP_RUNTIME_BUILD`, `SKIP_CERT_MANAGER`, `SKIP_DEPS`, `DISABLE_DISK_PRUNE`.
- **`local/test.sh`** — 7-stage e2e:
  1. API probes via port-forward (`/livez`, `/readyz`)
  2. CRDs registered
  3. RuntimeEnvironment for `python:3.11` mapped to the bundled runtime image
  4. Workspace lifecycle (create → Active, PVC bound)
  5. Sandbox lifecycle (create → Running → opencode `/global/health` 200)
  6. Workspace suspend → sandbox pod deleted → Workspace=Suspended → resume → Active
  7. Cleanup
- **`local/teardown.sh`** — `kind delete cluster`.

## What this proves

- **R1 (every sandbox is workspace-backed):** sandbox pod schedules, mounts a real PVC produced by Workspace reconciliation, and the PVC survives sandbox pod deletion (suspend/resume).
- **R2 (every sandbox runs an AI agent as a persistent HTTP server):** opencode v1.2.27 starts inside the pod's main container, finishes its sqlite migration, listens on `0.0.0.0:4096`, and serves `/global/health` and `/doc` (OpenAPI spec) authenticated by the password the controller mounted via `OPENCODE_SERVER_PASSWORD`.
- **Helm chart works:** all 20 resources render and apply cleanly via `helm install`.
- **The full Epic 1 + Epic 2 chain is end-to-end functional.** Epic 3 (proxy/sessions) is also installed and serving requests at the API service level (its handlers are exercised on the existing /api/v1 routes; the e2e test doesn't drive end-to-end proxy traffic in this iteration).

## Process notes — guesses I made and corrected

Per user feedback during this session, I'm calling these out so the next session avoids the same pattern:

- **Wrong:** "leader election lost causing re-attempt" (initial controller crashloop diagnosis). The actual cause was the dual metrics server.
- **Wrong:** "the previous pod's process is still holding port 8080". Container netns isolation makes this impossible. Corrected by reading more carefully.
- **Wrong:** "Watch closing immediately is normal in some cluster states". Said this without evidence. The real cause was the failed liveness probes (security middleware redirecting probes to HTTPS).
- **Unverified:** "the PVC was probably deleted by an earlier delete cascade". Should have either checked the controller log/events or admitted I didn't know. The clean rerun resolved it without me confirming the cause; I later traced it through events to verify the actual race (cache lag + WaitForFirstConsumer) and fixed bug #15.

The corrected pattern: state assumptions explicitly, validate before claiming, and say "I don't know yet" when I don't.
