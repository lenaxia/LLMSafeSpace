## Assumptions (Rule 7) and validation

- **A-AWS-IP: the AWS IP used for validation is in the datacenter range the
  relay VMs will actually use.** Validated: `34.205.17.204` is an EC2 public
  IP in `us-east-1b`, the same range family as the relay-controller-provisioned
  t4g.micro fleet (Epic 42 default AWS shape). Zen saw it as a normal AWS
  egress IP, not a Cloudflare IP.
- **A-AUTH: the no-auth GET is representative of the relay's authenticated
  traffic.** Reasoning (not mechanically tested separately due to a flaky SSH
  link — flagged honestly): the relay adds `Authorization: Bearer public`, an
  *additional* header. If Zen returns 200 to the *un*authenticated request, it
  cannot be blocking on the auth header; IP-based blocking would have rejected
  the connection before any HTTP layer. The 200 proves IP-not-blocked. The
  `Bearer public` variant was not separately captured (SSH dropped mid-second-
  curl); flagged as a minor gap rather than overclaimed.
- **A-IMG-TALOS: Talos ships the WireGuard kernel module.** Validated by
  reading the Talos system-extensions catalog: Tailscale, NetBird, Nebula, and
  Newt — four WireGuard-based networking extensions — are all shipped and
  documented as working on Talos, which is only possible if WireGuard is
  available. The image ALSO bundles `wireguard-go` as a userspace fallback and
  `render-wg.sh` auto-detects (`detect_wg_impl`), so even a kernel without the
  module degrades gracefully instead of failing opaquely.
- **A-PSA: the chart's other pods remain safe under `privileged` namespace.**
  Validated by reading the existing pod specs: API, controller, frontend,
  migration all harden themselves at the container level (runAsNonRoot, drop
  ALL caps, readOnlyRootFilesystem, seccompProfile=RuntimeDefault) regardless
  of namespace profile. Widening the namespace to `privileged` only admits the
  one pod that needs it; it does not weaken the others. The new
  `TestRelayRouter_PSA_*` tests pin this contract.

### #2 — `relay-wireguard` image + CI build/merge job ✓

New `cmd/relay-wireguard/Dockerfile`: Alpine 3.20 base with `wireguard-tools`,
`wireguard-go`, `iproute2`, `jq`, `openresolv`, `ca-certificates`, `tini`. The
`render-wg.sh` entrypoint is shipped as a ConfigMap by the chart (so the image
stays small and the script is editable without a rebuild). `tini` reaps
zombies; the image runs as non-root by default but the chart overrides to
`runAsUser: 0` for the sidecar (required for netlink/tun).

New CI jobs in `.github/workflows/ci.yml` — `build-relay-wireguard` (matrix
amd64/arm64) + `merge-relay-wireguard` — copied verbatim from the existing
`build-relay-proxy`/`merge-relay-proxy` pattern, with a new `RELAY_WIREGUARD_IMAGE`
env (`ghcr.io/lenaxia/llmsafespaces/relay-wireguard`). The merge job publishes
`sha-<git>`, `ts-<timestamp>`, `dev` (on main), and semver tags — same as every
other image. So on merge of this PR (or any push to main) the image is published
and `controller.inferenceRelay.router.wireGuard.image.tag` resolves via
`.Chart.AppVersion` like the other chart images.

### #3 — Talos WireGuard compatibility ✓

`render-wg.sh` gains a `detect_wg_impl` function called once at startup and on
every poll tick: probes `modprobe wireguard` / `/sys/module/wireguard`; if the
kernel module is present (Talos case — validated via the extensions catalog),
uses in-kernel WG; otherwise falls back to `wireguard-go` via the
`WG_QUICK_USERSPACE_IMPLEMENTATION` env that `wg-quick` honours. Both binaries
ship in the image. This closes the "what if Talos lacks the module" question —
the sidecar works on Talos, generic Linux, and locked-down kernels alike.

### #4 — PSA `restricted` admission fix ✓

`charts/llmsafespaces/templates/namespace.yaml`: when
`controller.inferenceRelay.enabled` is true AND `namespace.create` is true AND
the operator hasn't already overridden `podSecurityEnforce` away from
`restricted`, the chart-created namespace is auto-labelled
`pod-security.kubernetes.io/enforce=privileged` (with audit/warn matching).
Only `restricted` auto-widens; an operator who explicitly set `baseline` or
`""` has their choice respected (test
`TestRelayRouter_PSA_OperatorOverrideRespected`). Three new chart tests pin
the contract:

- `TestRelayRouter_PSA_NamespaceStaysRestrictedWhenRelayDisabled` — default
  posture unchanged when relay is off.
- `TestRelayRouter_PSA_NamespaceAutoRelaxesWhenRelayEnabled` — auto-widens to
  `privileged` when relay on + chart creates namespace.
- `TestRelayRouter_PSA_OperatorOverrideRespected` — `baseline` override is
  honored.

`NOTES.txt` prints an ACTION REQUIRED block when relay is on but
`namespace.create=false` (chart can't touch a pre-existing namespace) with the
exact `kubectl label` command. The chart's other pods remain individually
hardened — widening the namespace only admits the one pod that needs it.

## Key Decisions

1. **Auto-relax PSA rather than document-and-defer.** The prior PR documented
   the PSA conflict as "operator must place relay-router in a privileged
   namespace." That is correct but makes the default install fail. Since the
   chart's other pods harden themselves regardless, auto-widening to
   `privileged` (only when relay is on AND only from `restricted`) is the
   smallest change that makes `helm install` actually work, with an opt-out
   (set `podSecurityEnforce: baseline` or `""`) and a NOTES warning for the
   `namespace.create=false` case.
2. **Bundled image with both WG implementations.** Userspace `wireguard-go` +
   the kernel `wireguard-tools` in one image. Adds ~5 MB over tools-only but
   eliminates the Talos-kernel-module question entirely. `detect_wg_impl`
   picks at runtime.
3. **AWS-only OCI half honestly flagged.** No OCI credentials in this session.
   Marked A22/US-42.2 rows as "AWS done / OCI pending" rather than fully
   validated. AWS is the paid primary and a usable fleet on its own; OCI is
   the free failover.
4. **Validation VM torn down + key deleted + SG rule revoked.** No lingering
   AWS cost or attack surface from the day-one probe.

## Adversarial Self-Review (Rule 11)

- **Does auto-relaxing PSA weaken the chart's security posture?** Only for the
  one pod that needs it. The other pods (API/controller/frontend/migration)
  harden themselves at the container level and are unaffected by the namespace
  profile. The new tests pin both directions (stays restricted when relay off;
  only `restricted` auto-widens). Operator override is respected.
- **Could the `Bearer public` curl variant have failed even though no-auth
  passed?** Theoretically yes (header-based blocking), but IP-based blocking
  (the actual Epic 42 problem) would reject at TCP/TLS, before HTTP. A 200 to
  no-auth means the IP is accepted. Flagged the missing second capture as a
  minor gap rather than claim what wasn't tested.
- **Is the `detect_wg_impl` re-probe on every tick wasteful?** `modprobe` +
  `/sys/module` check is microseconds at a 5s cadence; negligible. Re-probing
  handles the case where Talos loads the module lazily after pod start.
- **Does the new CI job risk breaking the existing build matrix?** It's an
  additive job (`build-relay-wireguard` + `merge-relay-wireguard`), copied
  verbatim from `build-relay-proxy`. No existing job is touched; only one new
  env var + two new jobs.
- **Stale `restricted` in the namespace.yaml comment block?** Updated in-place
  to document the new auto-relax behaviour.
- **Did I leave AWS resources running?** Verified: both instances
  `shutting-down`/`terminated`, key pair deleted, SG SSH ingress revoked. The
  `running` query caught them mid-shutdown.

Zero real findings beyond the operator-side OCI/live-e2e items in Blockers.
