# Worklog: Epic 46 US-46.2 Amendment — Retain annotateModels Guard

**Date:** 2026-06-18

**Status:** Complete

---

## Objective

The user questioned whether removing the `annotateModels` remap guard (originally scoped in US-46.2) was the right call. This session re-evaluated the decision by reading the code author's documented reasoning against the README's classification, traced the guard's history through prior worklogs, and amended US-46.2 to retain the guard.

---

## Work Completed

### Re-evaluation of the annotateModels removal

Read `api/internal/handlers/models.go:440-469` and the surrounding comment block. The code author's comment at lines 450-453 explicitly argues for defense-in-depth:

> *"The guard is kept as defense-in-depth: if an opencode version ever keeps the built-in provider in connected[] despite `disabled_providers`, this code correctly remaps it rather than silently routing to a disabled provider."*

This argument is **stronger** than `README-LLM.md:517`'s opposing classification ("dead code (tech debt to remove)"). The README presents an assertion; the code comment presents a reasoned trade-off with a specific failure mode it guards against.

### Historical context discovered

Traced the guard's evolution through worklogs:

- **Worklog 0178** (`0178_2026-06-07_model-selector-free-models-relay-race.md`): The remap was specifically *narrowed* to its current form to fix a `ProviderModelNotFoundError` bug where models already mapped to `opencode-relay` were being remapped again. The `p.ID == "opencode"` predicate was added at this point.
- **Worklog 0189** (`0189_2026-06-08_bug3-relay-personal-key-routing-fix.md`): The comment block was rewritten after a follow-up audit found the original comment described the wrong behaviour. The current Phase 1/Phase 2/personal-key explanation was authored here.

The guard is the **result of two prior incident cycles**, not accidental cruft. Removing it discards the hardening from both.

### Comparison: GCPDriver vs annotateModels

| | GCPDriver stub | annotateModels guard |
|---|----------------|----------------------|
| Satisfies interface for no purpose | Yes | No (it's a runtime guard, not interface fulfilment) |
| Defense-in-depth argument | None (internal stub) | Yes (guards against opencode ignoring `disabled_providers`) |
| External dependency | None | Yes (opencode — upstream we don't control) |
| Cost of keeping | 28 lines of pure stub | ~20 LoC (4-line conditional + 2 tests + 14-line comment) |
| Cost of removing | None | Silent mis-routing of free-tier models if opencode changes |
| History of prior incidents | None | Two prior bug cycles (0178, 0189) |

**Conclusion:** GCPDriver removal is clearly correct. annotateModels removal is clearly **incorrect** — the README's classification is wrong, and the epic should not have followed it.

## Key Decisions

1. **Retain the annotateModels guard as intentional defense-in-depth.** The code author's documented reasoning at `models.go:450-453` outweighs `README-LLM.md:517`'s classification. The guard costs ~20 LoC and protects against a real failure mode in an external dependency we don't control.

2. **Keep the GCPDriver removal in US-46.2.** That half of the original story is unaffected — the stub is genuinely dead, has no defense-in-depth argument, and satisfies an interface for no purpose.

3. **Effort estimate reduced 0.5d → 0.25d.** Only GCPDriver + its test references remain. Simpler change.

4. **Did not retroactively rewrite worklog 0337.** Per Rule 5 (Worklog discipline rule 7: *"Never retroactively rewrite a worklog — worklogs are append-only history"*), the original worklog stands. This worklog (0338) records the correction.

5. **Did NOT update README-LLM.md:517 in this PR.** That edit changes the README's narrative about the relay config subsystem and is a separate concern. Flagged as a follow-up in US-46.2's acceptance criteria: *"Optional, separate PR: README-LLM.md:517 updated to reclassify the annotateModels guard as intentional defense-in-depth."*
