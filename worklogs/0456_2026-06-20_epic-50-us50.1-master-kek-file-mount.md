# Worklog: Epic 50 — US-50.1 Master KEK File Mount

**Date:** 2026-06-20
**Session:** Implement US-50.1 (Phase 2 of Epic 50) — deliver the master KEK as a read-only file mount instead of an env var, eliminating `/proc/1/environ` exposure (finding H1).
**Status:** Complete (implementation + validation); PR pending.

---

## Objective

Close finding H1 from the Epic 50 security assessment: the master KEK was
delivered to the API pod as the `LLMSAFESPACES_MASTER_SECRET` env var
(`api-deployment.yaml:63-67`), exposing it via `/proc/1/environ`. Project it as
a read-only file mount instead, with a one-release legacy-env fallback for
non-Helm deployments.

## Work Completed

### Go (`api/internal/app/secrets_adapters.go` + `app.go`)
- New `LLMSAFESPACES_MASTER_SECRET_FILE` source (colon-separated for the US-50.4
  rotation window; last file = highest/active version).
- `decodeMasterRaw` (hex/raw decode, no length filter) + `decodeMasterMaterial`
  (adds the 32-byte minimum). Split so the loader can surface present-but-short
  files for precise diagnostics.
- `loadMasterSecretMaterials()` reads the file mount(s); includes short files so
  `validateMasterSecret` can report "too short" rather than "no readable file".
- `activeMasterSecret()` prefers the file mount (active = last, ≥32 bytes);
  falls back to the legacy value env vars. `deriveServerKey()` uses it unchanged
  in signature (side-effect-free).
- `validateMasterSecret()` validates whichever source is configured: file path
  set + no readable file → startup error; file present but short → precise
  "too short" error; legacy env → works + deprecation Warn. Also warns when the
  file is healthy but a legacy value env is still set (redundant + still leaks).

### Helm (`api-deployment.yaml` + `values.yaml`)
- `masterSecret.deliveryMethod` (default `file`): removes the
  `LLMSAFESPACES_MASTER_SECRET` env block; adds `LLMSAFESPACES_MASTER_SECRET_FILE`
  + a `master-secret` Secret volume (defaultMode `0440`) mounted via
  `subPath: master-secret` at `/etc/llmsafespaces/master-secret`. `subPath`
  avoids conflict with the existing `/etc/llmsafespaces` config mount.
- `deliveryMethod=env` (legacy opt-in) restores the env block, no volume.
- New top-level `masterSecret:` values block (`deliveryMethod`, `fileMountPath`).

### Adversarial review + remediation
Independent skeptical validator found 5 issues; all remediated:
- **MEDIUM (dead code / lost diagnostic):** the loader pre-filtered `<32`-byte
  materials, making `validateMasterSecret`'s too-short branch unreachable and
  producing a misleading "no readable file" error for short key files. Fixed by
  splitting decode (raw vs ≥32) so the loader includes short files; the
  too-short diagnostic is now reachable and regression-tested.
- **MEDIUM (AC1 wording):** the design's acceptance "env | grep MASTER returns
  empty" conflicts with its mandated `LLMSAFESPACES_MASTER_SECRET_FILE` env var.
  Resolved as a documentation note: the security *goal* is met (only the PATH is
  in env, like `CONFIG_PATH`; the KEK value is absent). AC1's literal grep should
  be `env | grep -E '^LLMSAFESPACES_MASTER_SECRET='`. Not a code change.
- **LOW (orphaned doc comment):** removed the stale pre-US-50.1 doc block.
- **LOW (test isolation):** pre-existing tests now clear the file env var.
- **LOW (overlap diagnostic):** added a Warn when file is set + legacy env still present.
- (LOW note-only: `os.ReadFile` per `deriveServerKey` call — acceptable for a
  tmpfs-backed Secret; US-50.2 folds the derivation into `RootKeyProvider`.)

## Key Decisions

1. **subPath mount, not nested volume.** The config ConfigMap already mounts
   `/etc/llmsafespaces`; mounting the secret volume at the same path would
   conflict. `subPath: master-secret` mounts just the `master-secret` key as a
   single file into that directory — the standard K8s pattern, verified in the
   rendered manifest.
2. **Loader includes short files for diagnostics.** A security-sensitive boot
   path must distinguish "file missing" from "file present but too short"; the
   loader returns present+decodable materials regardless of length, and callers
   enforce the 32-byte minimum where it matters.
3. **AC1 reconciliation is a doc note, not a rename.** The design explicitly
   names `LLMSAFESPACES_MASTER_SECRET_FILE`; renaming it would violate the
   design. The var holds a PATH (not the secret), so the H1 goal is achieved.
4. **Multi-file at the US-50.1 layer.** The loader + `deriveServerKey` handle
  colon-separated paths now (active = last); the multi-*key decrypt provider*
  is correctly deferred to US-50.4.
