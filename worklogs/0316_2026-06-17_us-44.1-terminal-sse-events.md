# Worklog: US-44.1 — Terminal SSE Events for Agent Death (proxy-side)

**Date:** 2026-06-17
**Session:** Implement US-44.1a (proxy-side emission of `agent_died` SSE event when an opencode-backed SSE stream dies mid-flight). First of two foundational PRs for Epics 44/45.
**Status:** Complete (proxy-side only; broker bridge split to US-44.1c)

---

## Objective

Prevent the silent-SSE-close failure mode documented in Incidents A (OOMKill) and B (unsafe restart, 2026-06-16): when opencode dies mid-stream the proxy's `doProxy` streaming loop hit `io.EOF` and broke out silently, leaving SDK clients with a truncated body and no signal that the agent was gone. Implement the proxy-side half of US-44.1 per the epic design.

---

## Work Completed

### Implementation
- `api/internal/handlers/proxy.go` (`doProxy` streaming loop): added a `bytesReceived int64` counter incremented on every successful `Read`, plus an `isSSEStream` guard (`strings.HasPrefix` on `Content-Type: text/event-stream`). On `io.EOF`/`io.ErrUnexpectedEOF` after data on an SSE response, the proxy now appends a canonical `event: error\ndata: {"type":"agent_died","reason":"unknown"}\n\n` frame to the client writer and flushes. Non-SSE responses and empty bodies stay silent. Non-EOF errors keep the existing Epic 25 B2 wire format.
- `api/internal/handlers/proxy_test.go`: rewrote the backend of `TestProxy_B2_CleanStreamEnd_NoSSEError` to emit `application/json` instead of `text/event-stream` so the test name (non-SSE passthrough → no error event) still asserts what its name claims.
- `api/internal/handlers/proxy_terminal_events_test.go` (new): 13 tests covering happy paths (EOF-after-data on SSE, partial-chunk-then-EOF), unhappy/inverse paths (EOF with zero bytes, non-SSE JSON passthrough, mid-stream non-EOF), edge cases (multi-chunk accumulation, `text/event-stream; charset=utf-8` suffix, single Read returning `(n>0, EOF)`), and design-pinned contracts (acceptable false positive on clean SSE close, both `event: error` shapes documented).

## Key Decisions

1. **Scope-limit heuristic to `text/event-stream` responses.** The literal design says "any data before EOF". Adversarial review proved this breaks JSON/REST passthrough — every normal HTTP response completes via EOF after data, and appending an SSE event into a JSON body corrupts downstream parsers. The design's TITLE ("Terminal **SSE** Events") and PROBLEM STATEMENT ("SSE proxy silently closes") both support the scoped reading. The refinement is documented in code comments and in the design README acceptance criterion.

2. **Keep asymmetric `event: error` shapes.** B2 (network failure: `{"error":"upstream connection lost"}`) and US-44.1 (process death: `{"type":"agent_died","reason":"unknown"}`) describe semantically distinct failure modes. Unifying them would either break existing B2 clients or lose the structured agent_died signal. Asymmetry is intentional; both shapes pinned by `TestProxy_US44_1_ErrorShapesAreDocumented`.

3. **Split US-44.1b into US-44.1c (broker bridge).** The original 0.5-day estimate for "frontend display" assumed the frontend read the proxied SSE body. It does not. Delivering the user-facing goal requires broker-side emission from `sseTracker.connectAndRead` plus a ChatPage.tsx handler — non-trivial work that does not belong in this proxy-only PR. Honest rescoping per Rule 5 (faithful to the ask) and Rule 11 (validate, don't assume).

4. **`strings.HasPrefix` over `==` for SSE detection.** Most SSE producers (opencode included) emit `text/event-stream; charset=utf-8`. Equality check would silently miss these. Pinned by `TestProxy_US44_1_CharsetSuffix_MatchesSSE`.

5. **Counter type `int64`.** A 2 GiB workspace OOMs long before saturating even int32. int64 chosen for future-proofing and zero cost.
