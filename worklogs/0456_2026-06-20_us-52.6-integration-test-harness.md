## Work Completed

### Validated the story's assumptions (Rule 7) — and found the premise false

US-52.6 is written around testcontainers ("testcontainers-go is already in the
dependency tree, used by `pkg/secrets/pg_integration_test.go`"). Validated
against source:

- **testcontainers is NOT in any `go.mod`.** `pg_integration_test.go` uses a
  14-line `getTestPool` helper that connects to an **external** Postgres via
  `TEST_DATABASE_URL` and `t.Skip`s if unreachable.
- **No migration runner exists in the codebase.** Migrations are applied
  out-of-band by the external `migrate` CLI (`api/Makefile` `migrate-up`) and
  in CI by raw `psql -f api/migrations/*.up.sql`.
- The story's 5-file inventory was inaccurate: `handlers/secrets_integration_test.go`
  and `settings_integration_test.go` are httptest + mock-store tests (no
  Postgres); `services/database/credflow_integration_test.go` is a go-sqlmock
  unit test misnamed `_integration_test`.
- Docker is unavailable in the dev environment, and no local Postgres runs.

**User decision (Rule 6):** wrap the actual external-Postgres pattern rather
than introduce testcontainers.

### Implemented `api/internal/testharness`

- `New(t *testing.T) *Harness` — connects to `TEST_DATABASE_URL` (skip-if-
  unreachable, so `go test ./...` stays green without Docker), applies the
  embedded migration set idempotently, starts an isolated miniredis, and
  registers `t.Cleanup` teardown.
- Handles: `Pool() *pgxpool.Pool`, `SQLDB() *sql.DB` (for database/sql stores
  like `PgEmailTokenStore`), `Redis() *redis.Client`, `Miniredis()`.
- `MigrateUp()/MigrateDown()` via **golang-migrate** (the project's existing
  migration tool) reading migrations embedded by the new `api/migrations`
  package (`//go:embed *.up.sql *.down.sql`). The migrate lifecycle uses a
  dedicated `*sql.DB` so it cannot affect the exposed handles.
- `Reset()` — `TRUNCATE … RESTART IDENTITY CASCADE` of every user table; never
  touches `schema_migrations`.
- `NewContext()` (30s deadline, cancel wired to `t.Cleanup`), `Logger()`/`Logs()`
  (in-memory zap capture), `Seed(table, row)`, `DSN()`, `ID()`.

### Created `api/migrations` embed package

`api/migrations/embed.go` exposes `var FS embed.FS` over the golang-migrate-
format files only (`*.up.sql *.down.sql`). This deliberately excludes the
legacy `001_initial_schema*.sql` files and `README.md` so golang-migrate's
iofs source sees an unambiguous migration set. (The legacy files are inert
dead weight — out of scope for this story; noted for a future cleanup.)

- `harness_integration_test.go` — 9 DB-backed contract tests

## Key Decisions

- **External-Postgres, not testcontainers.** The story's testcontainers
  premise was false; introducing it would add a Docker dependency the project
  does not require. The harness wraps the validated `TEST_DATABASE_URL`
  pattern + embedded golang-migrate + miniredis. (User-approved reframing.)
- **Auto-`MigrateUp` in `New()` is safe.** Verified all 41 migrations are
  idempotent (`IF NOT EXISTS`, `ON CONFLICT`, guarded `DO` blocks; no
  `CREATE TYPE`/`EXTENSION`; all `DROP`s `IF EXISTS`). CI pre-migrates via
  `psql` (no `schema_migrations` table), so the first `MigrateUp` re-applies
  idempotently and creates the version table; subsequent calls return
  `ErrNoChange` (instant). Concurrent `MigrateUp` serializes via golang-
  migrate's `pg_advisory_lock`.
- **Isolation model = shared DB + unique per-test IDs (the project's existing
  convention)**, with an opt-in `Reset()`. Per-instance database isolation
  (the testcontainers path) was deliberately not added; documented in the
  README and deferred.
- **`api/internal/testharness` location (per the story).** This means `pkg/`
  cannot use it (internal visibility); the pkg/secrets duplication is left in
  place and documented.
- **golang-migrate (not a hand-rolled applier)** — it is the project's existing
  migration tool, handles dollar-quoting/version-table/locking correctly, and
  avoids reinvention.

## Adversarial Review (Rule 11)

Independent skeptical review (separate sub-agent) found **2 real findings**,
both fixed:

1. **MAJOR (Rule 0):** the harness's own contract tests had no CI run — added
   the CI step.
2. **MINOR:** `*sql.DB` handle leak in `New()` on the `PingContext`-fail path
   (`h.sqlDB` assigned after the check) — moved the assignment before the
   check so `h.Close()` closes it.

Many other concerns were investigated and confirmed clean: migration
idempotency (the would-be CI-breaker), skip-from-goroutine deadlock,
`email_tokens` CI coverage retention, internal-visibility constraint,
golang-migrate lifecycle (`Close` returns two errors), `rootStop` non-leak,
`Close` concurrency, `Logs` nil-safety, iofs subdir handling, concurrent
`MigrateUp` advisory-locking.
