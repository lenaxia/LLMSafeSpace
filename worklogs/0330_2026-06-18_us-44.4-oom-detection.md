## Objective

Incident A (2026-06-16): 16 concurrent sessions exhausted 2 GiB memory limit, pod OOMKilled mid-task. Session stuck "busy" with no indication. US-44.4 adds OOM detection to the managedProcess supervisor so ops can observe OOM events via Prometheus and the marker file persists for post-mortem.

## Work Completed

### Detection (`oom_detection.go`)
- `classifyExit(waitErr)` — classifies process exit as normal/SIGKILL/SIGTERM/crash using Go's `exec.ExitError` → `syscall.WaitStatus.Signaled()` + `Signal()`
- `isOOMExit(kind)` — returns true for SIGKILL (the OOM killer's signal)
- `writeOOMMarker(path, memoryLimit)` — writes JSON marker `{"reason":"oom","exitCode":137,"timestamp":"...","memoryLimit":"2Gi"}` to `/workspace/.opencode-oom-marker` (PVC-backed, persists across restarts)
- `handleOOMExit(workspaceID, markerPath)` — orchestrates marker write + Prometheus counter increment + warning log
- `getMemoryLimit()` — reads cgroup v2 `/sys/fs/cgroup/memory.max` for the marker
- `workspaceIDFromEnv()` — reads `WORKSPACE_ID` env var set by controller pod builder

### Supervisor integration (`main.go`)
- Crash path in `supervise()` now calls `classifyExit(waitErr)` before the generic crash log
- If `isOOMExit` → `handleOOMExit(workspaceIDFromEnv(), OOMMarkerPath)`

### Prometheus
- `workspace_oom_kills_total{workspace_id}` counter registered package-level
- Exposed on existing `:9090/metrics` endpoint (already wired for gate timings)

### Deferred items (documented in design README)
- SSE emission: opencode is third-party — agentd writes the marker but cannot force opencode to read it. SSE emission requires US-44.1c broker bridge.
- API proxy OOM notice: follow-up after broker bridge
- Frontend warning: follow-up after broker bridge
