## Why these particular deliverables, in this shape

A real-human red team would `kind create cluster` on their workstation and improvise. For an LLM-driven engagement (or a multi-operator engagement with rotating staff), Phase 0 must produce **artefacts** that codify the environment, not just point to a one-off cluster:

1. **Cluster bootstrap script** with the right CNI. The default `kind` kindnet does NOT enforce NetworkPolicy. If we used kindnet, the G16 fix would render YAML but have zero runtime effect — the pentest would silently produce false-negative results on G16-related test cases. Phase 0 therefore deploys Cilium.
2. **Control fixture**. A deliberately-vulnerable pod whose bugs are KNOWN. Run kube-hunter / trivy / kubeaudit against it; if they don't detect the planted bugs, the tool has a false-negative and Phase 1 results from that tool are unreliable. This is the single most important Phase 0 deliverable: it's the tool-calibration step that distinguishes "Phase 1 found nothing because the platform is secure" from "Phase 1 found nothing because the scanner is broken".
3. **Test accounts** with the same role hierarchy the platform uses (admin + 2 regular). Hardcoded into the fixture so cross-tenant tests in Phase 2/3/7 have a stable attacker persona.
4. **Logging-baseline check** that synthesises an event and verifies all three log streams (K8s audit, API, controller) capture it within 30 seconds. "Logs were flowing yesterday" is useless mid-pentest.
5. **Baseline snapshot** so the post-test diff is mechanical: anything that changed during the pentest is enumerable from the snapshot delta.
6. **Tools manifest** with pinned versions so any tool-version bump is a deliberate decision recorded in this file. New scanner releases that add checks would otherwise silently turn "no findings" into "real findings" with no signal in our codebase.

## Design decisions worth recording

**1. Multi-node kind cluster (1 control plane + 2 workers).**
Single-node was tempting but several Phase-5 / Phase-6 test cases exercise pod-to-pod NetworkPolicy enforcement across nodes (CNI bugs can be node-affinity sensitive). The cost of an extra worker in kind is negligible.

**2. `kubeProxyMode: none` + Cilium replaces kube-proxy.**
Cilium with kube-proxy disabled exercises the modern eBPF-based dataplane. If we left kube-proxy on, the cluster would have *both* iptables-based and eBPF-based service routing; conflicting findings would be hard to attribute. Pentest cluster needs to mirror the realistic production deployment shape.

**3. Audit policy is `RequestResponse` for security verbs, `None` for system noise.**
Without filtering, the audit log fills with kubelet watch heartbeats and real signal is hard to find. The policy escalates to `RequestResponse` on Secrets / ServiceAccount tokens / pods/exec / NetworkPolicy / RBAC mutations and on `llmsafespace.dev` group resources, drops `system:kube-controller-manager` etc. reads, and floors everything else at `Metadata`. This means "no audit entry exists for this pod's SA → it never called the API" is a true negative.

**4. Control fixture has THREE planted bugs, not one.**
Different scanners detect different classes of bug. A fixture with only one bug would let a tool pass calibration by luck. We plant:
- **Bug 1:** privileged + hostNetwork + hostPID + SYS_ADMIN/NET_ADMIN caps. Anything claiming pod-security awareness must catch this (kubeaudit, kube-hunter, kube-bench).
- **Bug 2:** busybox:1.30 base image with multiple HIGH-rated CVEs. SBOM/CVE scanners (trivy, grype, syft) must report ≥1 HIGH+.
- **Bug 3:** ServiceAccount with ClusterRoleBinding to cluster-admin and `automountServiceAccountToken` left at default true. RBAC analyzers must flag the privilege escalation path.

The `02-verify-control-fixture.sh` script asserts that EACH expected finding type appears. Missing ANY tool's expected finding fails Phase 0 — the kit refuses to gate Phase 1 with broken tooling.

**5. Test account passwords derive from the cluster context name.**
`pentest-$(echo "${KUBE_CONTEXT}" | sha256sum | cut -c1-32)`. Stable across re-runs (so re-running 03 doesn't change the password and break later phases) but unguessable. The accounts.json file is the only place the password lives, mode 0600, gitignored.

**6. JWT first-user-admin race (G8) is an EXPECTED auto-promotion, not a bypass.**
The first registration in `03-provision-accounts.sh` becomes admin. The script warns rather than fails if `role != admin` because re-runs against an existing DB will not promote. Phase 2 RT-2.7 tests the race condition explicitly; Phase 0 just wants a working admin.

**7. Phase 0 does NOT run pentest tools yet.**
Phase 0 only validates that the tools produce expected outputs against the control fixture. Actual pentest scans (against the LLMSafeSpace surface) start in Phase 1. This separation prevents Phase-0 calibration runs from being mistaken for real findings.

**8. Phase 0 does NOT run a real cloud cluster.**
Cloud-specific findings (cloud metadata endpoint blocking, IRSA confusion, EKS-specific RBAC) are tested in a follow-on cloud-pentest session. The kind environment covers the platform's own surface area; cloud config is operator responsibility.

## TDD discipline

Pure-bash and pure-YAML deliverables; the validation surface is:

- `bash -n` syntax check on every script — passes.
- `python3 -c 'yaml.safe_load_all'` on every YAML — passes.
- Each script exits non-zero on failure (`set -Eeuo pipefail`); meaningful error messages on every failure path.
- The exit-criteria check (`exit-check.sh`) is the integration test: it runs every other Phase-0 invariant and gates Phase 1.

I did NOT run the kit against a live kind cluster from this session. That requires the operator to have docker + kind + cilium-cli on their machine. The kit is authored for someone who does.
