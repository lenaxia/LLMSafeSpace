### 4c. SSE Events ‚Äî **PASS**

Connected to `/event` on the pod directly. Observed events:

| Event Type | Has `id` field | Wire format |
|-----------|---------------|-------------|
| `server.connected` | `evt_e75249f3e0019evwI4Eo0Pnqux` | `data: {...}\n\n` |
| `session.created` | `evt_e7524a70f001fu0dZRjZ2gQ6Aq` | `data: {...}\n\n` |
| `session.updated` | `evt_e7524a711001awBXzZRe4NvYxz` | `data: {...}\n\n` |

**Findings:**

- **`id` field present on all events** ‚Äî confirmed. Format: `evt_<ascending-id>`. Silently ignored by Go `json.Unmarshal` (no `id` field in the struct), as predicted by worklog 0070 Impact 1.
- **No `event: message\n` prefix** ‚Äî contrary to the Impact 2 analysis prediction. The SSE wire format is `data: {JSON}\n\n` with no `event:` line. The Effect `Sse.encode()` may only emit `event:` under specific conditions not triggered here. The SSETracker parser handles both cases correctly regardless.
- **`session.status` events** ‚Äî not captured during this test because the session transition (busy‚Üíidle) completed faster than the SSE capture window (no LLM credentials on this workspace). The `session.created` and `session.updated` events confirm the SSE stream is functional.
- **Browser-facing SSE** (`/api/v1/workspaces/<id>/events`): Events arrive as `opencode.event` type from the WorkspaceEventBroker, wrapping the raw opencode events. Three events captured during a session creation: `session.updated`, `session.created`, `session.updated`.

### EA5: Mid-Stream History Returns Partial Messages ‚Äî **CONFIRMED**

Tested by sending a long prompt via `prompt_async`, then fetching `GET /session/:id/message` at intervals while the assistant was still streaming.

**T+5s (mid-stream):**
- 2 messages returned (user + assistant)
- Assistant message had 4 parts: `step-start`, `reasoning`, `tool`, `tool` ‚Äî still accumulating

**T+10s (still streaming):**
- 3 messages returned (user + 2 assistant turns)
- First assistant message grew to 6 parts (added `tool` + `step-finish`)
- Second assistant message started with 1 part (`step-start`)

**Final (after completion):**
- 3 messages, assistant messages fully populated

**Implications:** `GET /session/:id/message` returns messages as they are being built. The proxy's `GetHistory` handler streams this response through unmodified, so frontend consumers that poll for history during an active session will see partial/incomplete assistant messages. Each part carries an `id` field (`prt_...`).
