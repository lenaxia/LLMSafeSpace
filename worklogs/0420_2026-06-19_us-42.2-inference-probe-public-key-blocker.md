# Worklog 0420 — US-42.2 Inference Probe: `public` key no longer authorizes Zen inference (A23 BLOCKER)

**Date:** 2026-06-19
**Epic:** 42 (Multi-Cloud Inference Relay)
**Status:** Finding (no code change; surfaces an architecture blocker pending operator decision)

---

## Objective

PR #287 ticked US-42.2 / A22 as "AWS validated" based on `GET /v1/models` returning HTTP 200 from an AWS t4g.micro. The operator correctly pushed back: that proved IP-reachability, not that the relay could actually do **inference**. This worklog records the follow-up inference probe and the blocker it surfaced.

---

## Blockers

- **The relay-forwarding-`public` architecture cannot produce inference.** `public` returns 401 "Missing API key" on `/chat/completions` regardless of source IP or header format. This is independent of the Epic 42 IP-diversity work (which is now necessary-but-insufficient).
- **Architecture decision pending operator input** (see Key Decisions). This worklog does NOT pick a direction.

---

## Assumptions (Rule 7) and validation

- **A-HEADER: the 401 "Missing API key" is a header-format mismatch, not a key rejection.** **DISPROVED.** Tested `Authorization: Bearer public`, `x-api-key: public` (Anthropic style), `api-key: public` (Azure style), Bearer+x-api-key combined, and no-auth — all return the identical 401 "Missing API key" in ~0.2s. The error shape is Anthropic (`{"type":"error","error":{"type":"AuthError",...}}`) but the rejection is header-agnostic. `public` is treated as "no key" on the inference path.
- **A-RELAY-INJECTS: the old CF Worker relay injected a real key behind the path-secret.** **DISPROVED by code.** `workers/inference-relay/src/index.ts:61-71` forwards `request.headers` unchanged after validating `RELAY_SECRET`; the new `cmd/relay-proxy/proxy.go:134` copies non-hop-by-hop headers (`Authorization` is not hop-by-hop) unchanged. Grep for `ZEN_API_KEY`/`UPSTREAM_API_KEY`/etc. = zero hits. No key injection exists in either relay.
- **A-IP-INDEPENDENT: the 401 is the same from any IP.** **VALIDATED.** Identical 401 from residential `24.18.52.209` and AWS Ashburn `13.221.103.1` (captured via webhook — SSH to the second instance was banner-throttled from this sandbox after the first test session).
- **A-MODELS-STILL-WORKS: `/models` still accepts `public`.** **VALIDATED.** `Bearer public` → 200, 4020-byte list, from both IPs. So `public` is now scoped to read-only listing, not inference.

### Control: a real key works

`POST ai.thekao.cloud/v1/chat/completions` + `Authorization: Bearer sk-ItSC…` → **200**, real `glm-5.2` completion. The operator-owned gateway + real key is a working inference path.

### Code audit: both relays forward `public` unchanged

- Old CF Worker: `workers/inference-relay/src/index.ts:64` — `headers: request.headers` (passthrough after path-secret validation).
- New relay-proxy: `cmd/relay-proxy/proxy.go:134` — `copyNonHopByHopHeaders(upstreamReq.Header, r.Header)`; `Authorization` is not in the hop-by-hop set (`proxy.go:18-30`).
- No upstream-key injection anywhere in `cmd/relay-proxy/`, `cmd/relay-router/`, `controller/internal/relay/`, or `workers/`.

### Historical trajectory (from worklog 0184)

- **2026-06-07:** `public` → **429 `FreeUsageLimitError`** (`retry-after: ~22876s`) — key accepted, free-tier quota exhausted.
- **2026-06-13:** worklog 0184 correction — owner verified `public` "works without issue from a residential IP"; concluded Zen throttles CF egress IPs, not the key.
- **2026-06-19 (this worklog):** `public` → **401 "Missing API key"** on inference, everywhere. Key now scoped to listing.

Between 2026-06-13 and 2026-06-19, Zen changed: `public` no longer authorizes inference at all (not just rate-limited — rejected as if absent).

## Key Decisions

**No architecture decision is made here — it is the operator's call.** The three options, with honest costs:

1. **Inject a real Zen key into the relay** (router injects + strips the client's `public`, or relay VMs carry it). Restores claude-* via Zen. **Reopens the "no secrets on relay VMs" threat model Epic 42 explicitly avoided** (Layer 2 §1: "No secrets on relay VMs. The relay proxies `apiKey: public` requests — nothing worth stealing."). Requires a real Zen key that is not present in the env. Real design conversation.
2. **Re-point the relay at an operator-owned gateway** (`ai.thekao.cloud` + the real key already returns 200). Near-zero cost: the relay changes `UPSTREAM_URL` and injects the operator key. The operator's working path is ALREADY this (the validating agent runs as `thekao cloud/glm-5.2`). Loses free-tier `claude-*` (those models exist only at Zen via `public`, which is dead either way). Epic 42's WG/IP-diversity machinery stays useful for whatever upstream is chosen.
3. **Confirm with Zen/opencode** whether `public` inference was intentionally retired or is a transient outage. Only the operator can pursue this (cannot be done from code). If transient, Epic 42 stands as-designed.

The deciding product question (operator's): **is free-tier `claude-*` a hard requirement?** If no → (2) is strictly better. If yes → only (1) keeps claude, at the cost of the secret-on-VM model.

## Adversarial Self-Review (Rule 11)

- **Could the 401 be a transient Zen outage rather than a policy change?** Possible — but the trajectory (429 → 200-residential → 401) and the precise scoping (`/models` 200, `/chat/completions` 401, header-agnostic) point to a deliberate change, not a blip. Flagged as "confirm with Zen" (option 3) rather than asserted as permanent.
- **Did I actually exercise inference, or just listing?** Both. `/models` 200 (listing works); `/chat/completions` 401 (inference rejected). The 401 is the load-bearing result.
- **Is the "no key injection" audit complete?** Grepped the four relay-related dirs for key-injection patterns; zero hits. The two forward paths were read in full. High confidence, but a real Zen key could live in a CF Worker secret / env not in the repo — the audit only covers repo code, which is what the shipped relays run.
- **Did I overclaim in PR #287's US-42.2 tick?** Yes — "/models 200" was framed as "AWS validated" without an inference probe. Corrected in this worklog and in the A22/US-42.2/Phase-0 rows of the Epic 42 README.
- **Am I pre-deciding the architecture?** No. This worklog records the finding and the options; the operator picks.
