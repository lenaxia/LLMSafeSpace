## Assumptions (Rule 7) and validation

- **A-HEADER: the 401 "Missing API key" is a header-format mismatch, not a key rejection.** **DISPROVED.** Tested `Authorization: Bearer public`, `x-api-key: public` (Anthropic style), `api-key: public` (Azure style), Bearer+x-api-key combined, and no-auth — all return the identical 401 "Missing API key" in ~0.2s. The error shape is Anthropic (`{"type":"error","error":{"type":"AuthError",...}}`) but the rejection is header-agnostic. `public` is treated as "no key" on the inference path.
- **A-RELAY-INJECTS: the old CF Worker relay injected a real key behind the path-secret.** **DISPROVED by code.** `workers/inference-relay/src/index.ts:61-71` forwards `request.headers` unchanged after validating `RELAY_SECRET`; the new `cmd/relay-proxy/proxy.go:134` copies non-hop-by-hop headers (`Authorization` is not hop-by-hop) unchanged. Grep for `ZEN_API_KEY`/`UPSTREAM_API_KEY`/etc. = zero hits. No key injection exists in either relay.
- **A-IP-INDEPENDENT: the 401 is the same from any IP.** **VALIDATED.** Identical 401 from residential `24.18.52.209` and AWS Ashburn `13.221.103.1` (captured via webhook — SSH to the second instance was banner-throttled from this sandbox after the first test session).
- **A-MODELS-STILL-WORKS: `/models` still accepts `public`.** **VALIDATED.** `Bearer public` → 200, 4020-byte list, from both IPs. So `public` is now scoped to read-only listing, not inference.

## Key Decisions

**No architecture decision is made here — it is the operator's call.** The three options, with honest costs:

1. **Inject a real Zen key into the relay** (router injects + strips the client's `public`, or relay VMs carry it). Restores claude-* via Zen. **Reopens the "no secrets on relay VMs" threat model Epic 42 explicitly avoided** (Layer 2 §1: "No secrets on relay VMs. The relay proxies `apiKey: public` requests — nothing worth stealing."). Requires a real Zen key that is not present in the env. Real design conversation.
2. **Re-point the relay at an operator-owned gateway** (`ai.thekao.cloud` + the real key already returns 200). Near-zero cost: the relay changes `UPSTREAM_URL` and injects the operator key. The operator's working path is ALREADY this (the validating agent runs as `thekao cloud/glm-5.2`). Loses free-tier `claude-*` (those models exist only at Zen via `public`, which is dead either way). Epic 42's WG/IP-diversity machinery stays useful for whatever upstream is chosen.
3. **Confirm with Zen/opencode** whether `public` inference was intentionally retired or is a transient outage. Only the operator can pursue this (cannot be done from code). If transient, Epic 42 stands as-designed.

The deciding product question (operator's): **is free-tier `claude-*` a hard requirement?** If no → (2) is strictly better. If yes → only (1) keeps claude, at the cost of the secret-on-VM model.

## CORRECTION (2026-06-20) — A23 is FALSE

**What was wrong:** The original finding (sections above) concluded `public` "no longer authorizes Zen inference" — a universal, IP-independent, header-agnostic rejection — based on 401s from `claude-fable-5`, `claude-sonnet-4`, `gemini-3.5-flash`, `claude-haiku-4-5`, `claude-opus-4-8`. That conclusion is **false**. Those specific 401s were an artifact of probing only models that happen to have `allowAnonymous` unset in Zen's current deploy-time config. Other free models with `allowAnonymous: true` succeed with the identical `Bearer public` from the identical IP.

**Definitive re-probe (residential IP `24.18.52.209`, same IP used for the original 401s):**

| Request | Auth | Result |
|---|---|---|
| `POST /v1/chat/completions` model=`big-pickle` | `Authorization: Bearer public` | **HTTP 200** — real completion, `model: deepseek-v4-flash`, 111 tokens, finish_reason=`length` |
| `POST /v1/chat/completions` model=`claude-fable-5` | `Authorization: Bearer public` | **HTTP 401** "Missing API key" (same as original finding) |
| `GET /v1/models` | `Authorization: Bearer public` | **HTTP 200** (unchanged) |

`big-pickle` is in the public `/models` list and its response is a genuine LLM completion (the prompt was "Reply with exactly: PUBLIC KEY WORKS"; the model produced reasoning_content then truncated at max_tokens=20). This is not a listing artifact.

**Actual mechanism (from `github.com/anomalyco/opencode`, our fork):**

The Zen handler (`packages/console/app/src/routes/zen/util/handler.ts`) gates free-model inference on a **per-model flag**, not on the key or the source IP:

```ts
// handler.ts:104 — "public" is normalized to "no key"
const zenApiKey = rawZenApiKey === "public" ? undefined : rawZenApiKey
// ...
// handler.ts:599-603
async function authenticate(modelInfo: ModelInfo, zenApiKey?: string) {
  if (!zenApiKey) {                                    // public → undefined here
    if (modelInfo.allowAnonymous) return               // model flagged allowAnonymous → OK
    throw new AuthError(t("zen.api.error.missingApiKey")) // else → 401 "Missing API key"
  }
  // ... real-key path
}
```

`allowAnonymous` is a per-model field in ZenData (`packages/console/core/src/model.ts:26`: `allowAnonymous: z.boolean().optional()`), loaded from deploy-time SST secrets (`Resource.ZEN_MODELS1..30`, set via `packages/console/core/script/pull-models.ts`) — **not in the repo**, so the 2026-06-19 probe could not see which models carried it and wrongly generalized from its 401 sample.

**Implications:**

1. **A0 (original Epic 26/42 premise) is restored**: Zen throttles by source IP for anonymous free-tier traffic, not by the `public` key. The relay's IP-distribution purpose is valid for any model Zen flags `allowAnonymous`.
2. **The router upstream-auth injection (#297) and default→thekao (#298) were built on A23.** They are not wrong as *mechanisms* (router-side key swap is a legitimate capability), but their stated *rationale* — "public 401s everywhere, so we must inject a real key" — is unfounded. Revisit whether those changes should remain the default.
3. **Relay scope (operator-confirmed ground truth): the relay carries free-model traffic only; paid-model traffic goes direct to the user's provider.** This is already enforced in agentd's `annotateModels` (remaps only `ModelFreeTier` to `opencode-relay`) and is unchanged.
4. **Free-tier `claude-*` specifically**: still 401 via `public` today, but that is a per-model `allowAnonymous` config choice in Zen, not a key/IP block. It could change without notice; the relay architecture should not assume either way.
