### 3. Implementation

`controller/internal/workspace/controller.go`:
- `WorkspaceReconciler` struct with `client.Client` and `Scheme`
- `WorkspaceFinalizer` constant: `"workspace.llmsafespace.dev/finalizer"`
- Full state machine via `Reconcile` → phase handlers:
  - `handlePending`: adds finalizer, creates PVC (idempotent), transitions to Active
  - `handleActive`: updates activeSessions, auto-suspend logic with requeue at 20% of remaining idle time
  - `handleSuspending`: race condition check (reverts to Active if recent activity), deletes pods, sets Suspended
  - `handleSuspended`: TTL expiry check (→ Terminating) or requeue after remaining TTL
  - `handleResuming`: waits for all referenced sandboxes to be Running, then → Active or requeue 5s
  - `handleTerminating`: deletes PVC (best-effort), deletes Sandbox CRDs, removes finalizer, sets Terminated
  - `handleDeletion`: (deletionTimestamp) deletes Sandbox CRDs, deletes PVC, removes finalizer
  - Failed/Unknown phases: log only, no requeue
- `buildPVC`: constructs PVC with correct labels, access mode, storage size, optional StorageClassName
- Helper methods: `listSandboxesForWorkspace`, `deleteSandboxCRDs`, `deleteWorkspacePods`
- `SetupWithManager` using `ctrl.NewControllerManagedBy`

## Key Decisions

1. **Race condition threshold**: The spec says "if activity happened since we queued this reconcile, cancel suspend." Interpreted as: in `handleSuspending`, if `lastActivityAt` is non-nil and `time.Since(lastActivityAt) < idleTimeout`, revert to Active. This matches §5.6 exactly.

2. **Requeue formula for Active/auto-suspend**: Spec says "requeue after `remaining idle time * 0.2`". Implemented as `remaining * 0.2` with a 1s minimum floor to avoid zero-duration requeues.

3. **Suspended TTL reference time**: When `lastActivityAt` is nil (workspace was never active), falls back to `creationTimestamp` as the reference for TTL expiry. This is a safe assumption — a workspace that never had activity should still be garbage-collected after TTL.

4. **Terminating vs Deletion**: Both paths clean up PVC and Sandbox CRDs. `handleTerminating` is driven by the phase field (lifecycle-driven); `handleDeletion` is driven by `deletionTimestamp` (Kubernetes delete call). Both converge on removing the finalizer.

5. **No `Owns(&corev1.PVC{})` in SetupWithManager**: The sandbox reconciler owns Pods; the workspace reconciler creates PVCs with owner references but does not watch them as owned objects. This avoids spurious reconciles on PVC status updates and keeps the controller simple. The owner reference on the PVC still ensures garbage collection.
