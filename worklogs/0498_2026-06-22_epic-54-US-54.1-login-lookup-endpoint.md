# Worklog: Epic 54 US-54.1 — Login Discovery Endpoint (TDD)

**Date:** 2026-06-22

**Status:** Complete (PR opening next)

---

## Objective

Ship the backend half of Epic 54's login discovery flow: a public endpoint that resolves an email to a single `redirectUrl` pointing at the user's org. Enumeration-safe by construction (uniform 200 + uniform body shape, DB errors masked). This is the first implementation story after the design PR (#341) merged.

The frontend (US-54.2) and Helm chart (US-54.3) stories follow; the spike (S54-0, wildcard subdomain routing viability) is an operator task requiring staging cluster access and is not blocked by this work — the endpoint falls back to the direct SSO start URL when subdomain routing is disabled.

### Handler implementation

`login_discovery.go` (~165 lines including comments). Structure mirrors `password_reset.go`:

- Small focused interfaces declared in the handler file (`loginDiscoveryUserLookup`, `loginDiscoveryOrgLookup`, `loginDiscoveryLogger`) — same pattern as `passwordResetUserLookup` etc.
- `LoginDiscoveryHandler.Lookup` — binds JSON, normalizes email, walks the resolver, returns uniform 200.
- `subdomainFor(slug, base)` — pure function; empty base falls back to direct SSO start URL (never tells a found user "not found").
- `maxLoginDiscoveryBodySize = 1 << 20` — 1 MiB body limit (matches `maxPasswordResetBodySize`).

**Design deviation (documented):** the epic README placed the resolver in a new `services/authlookup` package. Implementation revealed that was overkill — the resolver is 2 DB calls + URL construction, and the `password_reset.go` precedent keeps equivalent logic in the handler. Kept it in `handlers/` to match the codebase convention. If the logic grows (e.g. multi-org `default_org_id` resolution), extraction is a trivial follow-up.

### Test iteration

Two test failures during TDD iteration, both informative:

1. **`captureLogger` name collision** with `email_test.go:57`. Renamed to `lookupCaptureLogger` / `lookupLogEntry`. (In-package test files share the namespace.)
2. **`TestLoginLookup_NormalizesEmail` initially tested whitespace trimming.** Gin's `binding:"email"` validator rejects whitespace-padded input *before* the handler sees it — same behavior as `password_reset.go`. Updated the test to verify case normalization only (the realistic scenario); whitespace rejection is already covered by `TestLoginLookup_InvalidEmail_Returns400`.
3. **`TestSubdomainFor_WithLeadingDotBase_Normalizes` initially failed** because the leading-dot trim was in the constructor, not in `subdomainFor`. Moved the `strings.TrimPrefix(base, ".")` into the helper itself (defense-in-depth — the helper normalizes its own input regardless of caller).

## Key Decisions

- **No separate service package** — handler-only, matching `password_reset.go`. Documented as a deviation from the epic README; extraction is trivial if multi-org logic later warrants it.
- **No timing pad** — matches `password_reset.go:119` exactly. The benchmark confirms the resolver adds negligible overhead (~1.8µs with fakes). Real Postgres will add ~0.5-2ms/query but the asymmetry stays bounded (~1-1.5ms divergence on 2 extra indexed lookups), well under network jitter. The acceptance criterion mandates real integration measurement; this worklog records the lower bound.
- **Global rate limiter is sufficient for now** — the existing middleware (default 100/min/IP, burst 20) protects the endpoint. The epic recommended "10/min/IP, 5/hour/email" but per-endpoint custom limits require config plumbing not warranted for the first cut. Documented as a follow-up if abuse patterns emerge.
- **`subdomainFor` fallback to direct SSO URL** — when `baseDomain` is empty (subdomain routing disabled, the default), the endpoint redirects found users to `/api/v1/auth/sso/<slug>/start`. This means US-54.1 has standalone value even before the spike (S54-0) passes: it resolves email → direct SSO URL, skipping the broken domain-match step on `LoginPage.tsx:42`.
