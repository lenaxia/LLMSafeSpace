## Objective

Ship US-29.5 — the highest-leverage handler split. The `SecretsHandler` god object had 19 methods + 11 fields. Models accounted for 2 public methods, 5 private helpers, and 4 struct fields (`wsUpdater`, `relayActive`, `metricsRecorder`, `policyChecker`) — all models-only. - `SetModel` — PUT /workspaces/:id/model. Validates against catalog via `agentClient.ListModels`; persists to DB + K8s Secret; pushes to agent via `agentClient.PatchConfig`.

The relay check needs the agentd admin port (4098, Bearer auth) which is separate from AgentClient (port 4096, Basic auth). Keeping it as an injected function avoids leaking pod/password concerns into the handler.

## Key Decisions

1. **AgentClient replaces passwordGetter + podIPResolver for opencode calls.** The handler never resolves podIP or password directly — AgentClient.ListModels and AgentClient.PatchConfig handle that internally. This is the US-29.1 principle applied at the consumer level.

2. **RelayStateChecker is a separate injected function.** The relay check needs the agentd admin port (4098, Bearer auth), not the opencode port (4096, Basic auth). Rather than expanding AgentClient to cover two ports + two auth schemes, the relay check stays as a narrow `func` injection. The handler remains clean.

3. **Catalog fetching consolidated.** The old code had triplicated catalog fetch logic (inline in ListModels, fetchCatalog, resolveModelIDFromCatalog). The new code has `agentClient.ListModels` as the single fetch point, with `modelExistsInRawCatalog` and `resolveModelFromCatalog` as pure functions that operate on the pre-fetched raw bytes.

4. **SetModel fetches catalog once.** The old code fetched /provider up to 3 times in SetModel (existence check, resolveModelID, patchAgentModel). The new code fetches once via `agentClient.ListModels` and reuses the bytes for both validation and resolution.
