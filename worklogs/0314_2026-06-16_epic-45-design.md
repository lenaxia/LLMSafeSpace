**Date:** 2026-06-16

## Objective

Design Epic 45 to eliminate the bug class that caused today's stuck sessions. While Epic 44 addresses session-lifecycle observability (terminal events, OOM detection, restart mechanics), Epic 45 addresses the deeper architectural issue: **the API server holds critical state in process-local maps and treats opencode's SSE stream as the source of truth**. This works at single-replica scale but breaks at multi-replica.

Today's stuck-session bug (`ses_13076538bffeYtLrhoZ2ccRM1E`) is the symptom; per-replica `activeSess` map is the disease.

## Work Completed

### Architectural Analysis

Investigated the codebase for state-management weak points using a targeted explore agent. Key findings:

#### State sources of truth conflicts

| Fact | Store A | Store B | Store C | Conflict? |
|---|---|---|---|---|
| Session busy/idle | opencode SQLite (runtime) | agentd `sessionStatusTracker` | API `activeSess` map | **YES** — empty after restart, no sync between replicas |
| Session title | opencode SQLite | API `session_index` table | — | Race on `RenameSessionInAgent` |
| Workspace phase | Workspace CRD `.status.phase` | Watcher `knownPhases` | ProxyHandler `priorPhase` | Bookmarks + 290s reconnect window |
| Workspace password | k8s Secret | `pwCache` map | — | Per-replica invalidation, no cross-replica notify |
| Connected providers | opencode `/provider` | agentd `providerCache` (15s TTL) | CRD condition | Three-way 60s skew |

#### `ProxyHandler` god object inventory

`api/internal/handlers/proxy.go:50-102` defines a 32-field struct with 7 in-memory maps (the maps are the multi-replica concern; the rest are mutexes, injected services, and once-guards), all per-replica:

| Map | Authority | Move to Redis? | Priority |
|-----|-----------|----------------|----------|
| `activeSess` | API server | ✅ Yes | P0 (today's bug) |
| `deletedSessions` | API server | ✅ Yes | P0 |
| `pwCache` | k8s Secret (cache) | ✅ Yes | P1 |
| `wsConfig` | Workspace CRD (cache) | ✅ Yes | P1 |
| `priorPhase` | Workspace CRD | ✅ Yes | P2 |
| `parentBackfilled` | API server | ✅ Yes | P2 |
| `connCount` | Per-replica resource | ❌ No (correctly local) | N/A |

`connCount` should remain local because it represents per-replica resources (file descriptors, memory). Each replica should enforce its own connection ceiling.

#### Failure mode analysis

**Today's incident chain** (validated against source):

1. opencode emits `session.status: busy` for `ses_13076538...`
2. API replica's `onSessionActive()` adds to `activeSess["a847faa5..."]["ses_13076538..."]`
3. opencode OOMKilled at 08:47:44 UTC **before** emitting `idle`
4. Pod restarts ~11h later, SSE tracker reconnects, runs `reconcileStrandedQueues()` (the **pre-PR-#197** function name and behavior — see note below)
5. **Pre-PR-#197 bug**: At commit `c06feff8` and earlier, `proxy_events.go` had logic that skipped sessions with no queued messages:
   ```go
   // Pre-PR-#197 code:
   if err != nil || n == 0 { continue }
   ```
6. Result: `activeSess` map keeps stale entry forever → POST returns 409 Conflict

> **Note on line numbers:** This worklog was written immediately after PR #197 (`c06feff8` + `4ea4e96d`) shipped. PR #197 renamed `reconcileStrandedQueues` → `reconcileSessionState` and added stale `activeSess` cleanup, so the line numbers in this analysis reference the **pre-#197** state. The current code (post-#197) has different line numbers and includes the stale-cleanup fix. This worklog documents what the bug was, not what the code looks like now.

**Multi-replica corollary**: With 2 replicas (current production), each has independent `activeSess`. One replica's reconcile doesn't help the other. The user's request lands on whichever replica the load balancer picks; on the stuck one, they get 409.

### Design Decisions (User-Approved)

Used the `question` tool to gather decisions on architectural tradeoffs:

| # | Question | Decision | Rationale |
|---|----------|----------|-----------|
| Q1 | Rollback strategy: env var switch vs. forward-fix only? | **Forward-fix only** | No `WSSTATE_BACKEND` toggle. Bugs resolved via PRs. |
| Q2 | Valkey HA: prerequisite or follow-up? | **Ship without HA** | Fail-open behavior + k8s fallback tolerate brief Valkey outages |
| Q3 | L1 cache: pre-emptive or measure-first? | **Measure first** | Avoid premature optimization; trivial to add later via decorator pattern |

Additional decisions made during design:

- **Hash tags from day one**: Use `{workspace_id}` in keys (e.g., `ws:{abc123}:active`). Free now, enables future Valkey cluster migration painlessly.
- **TTL on every key**: 30 minutes for activeSess/deletedSessions, 1 hour for password cache, 5 minutes for workspace config. TTL acts as safety net for bugs we haven't found yet.
- **Lua scripts for atomicity**: Prevent race conditions in multi-replica check-and-add operations.
- **Asymmetric failure modes**:
  - `activeSess` → fail-open (allow request) — better UX
  - `deletedSessions` → fail-closed (suppress event) — data integrity
  - Caches (`pwCache`, `wsConfig`) → fall back to k8s — slower but correct

### Why Not Address Everything Now

The architectural analysis surfaced 4 categories of issues:
- **Reliability** (R1-R5): Stuck sessions, abort-all-on-restart, state drift
- **Performance** (P1-P5): Synchronous k8s GETs, /v1/statusz thundering herd
- **Maintainability** (M1-M5): Code duplication, god objects, magic numbers
- **Scalability** (S1-S5): Multi-replica SSE duplication, no leader election for side effects

Epic 45 focuses narrowly on **R2 (state externalization)** because:
1. It directly fixes the bug class that caused today's incident
2. It's a prerequisite for fixing S1 (distributed SSE tracker) and S3 (watcher coordination)
3. It can ship incrementally (story by story) without breaking changes

The other categories will be addressed in future epics:
- **R3** (durable session busy state in agentd) — separate epic, addresses opencode-crash data loss (out of scope: incident A wasn't actually data loss, just state confusion)
- **S1** (distributed SSE tracker) — depends on Epic 45 finishing first
- **M1-M5** (maintainability) — ongoing, no single epic

## Lessons Learned

1. **Bug class > bug instance**: Today's stuck session was the symptom. The disease is the architectural pattern of holding distributed state in per-process memory. PR #197 fixes the symptom; Epic 45 addresses the disease.

2. **Document the decisions made, not just the decision tree**: User answered Q1-Q3 with one-letter responses. Recording those decisions inline with their rationale prevents "wait, why did we choose forward-fix-only?" questions in 6 months.

3. **Hash tags are free; cluster migrations are expensive**: Using `{workspace_id}` in Redis keys costs nothing at single-instance Valkey scale and saves a future redesign when migrating to cluster mode. Always include constraints that future-proof at zero present cost.

4. **Decorator pattern enables measure-first**: By defining `WorkspaceStateStore` as an interface from day one, adding L1 cache later is a 4-hour task: write `cachedStateStore` that wraps `redisStateStore`, swap dependency injection. This eliminates the "we should add L1" anxiety because adding it later is cheap.

5. **Failure mode asymmetry matters**: `activeSess` and `deletedSessions` look superficially similar (both are sets of session IDs in Redis), but their failure modes are opposite. Getting this right requires explicit thinking about each operation's failure semantics, not just generic "Redis-backed cache" patterns.
