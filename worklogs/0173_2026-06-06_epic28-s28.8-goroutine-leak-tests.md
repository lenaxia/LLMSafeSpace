## Background

`StreamUserEvents` (`stream_user_events.go`) is the user-scoped SSE endpoint that delivers workspace phase events to browser clients. The zombie-connection problem it solves:

- Mobile networks silently drop TCP connections; the server doesn't notice for minutes
- Without a write deadline the goroutine blocks in `fmt.Fprint(c.Writer, ...)` indefinitely
- The subscription stays registered, consuming broker memory and channel capacity

The fix (already shipped): `rc.SetWriteDeadline(time.Now().Add(30s))` on every write. Write failure → `streamCancel()` → live loop exits → `defer UnsubscribeUser` fires.

The handler also spawns two goroutines: `snapshotUserWorkspaces` and `heartbeatLoop`. Both must exit when the client disconnects, or they leak.

Prior test suite covered 10 happy-path scenarios — none of these exit paths had protection.

## Tests Added

### `TestStreamUserEvents_GoroutineExitsOnClientDisconnect`

**What it proves:** When the TCP connection drops:
1. `c.Request.Context()` is cancelled by the HTTP server
2. `streamCtx` (derived via `context.WithCancel`) is cancelled
3. Both `snapshotUserWorkspaces` and `heartbeatLoop` exit (both select on `ctx.Done()`)
4. `defer h.userBroker.UnsubscribeUser(uid, s)` fires
5. The broker's subscriber count returns to zero — no zombie

**Mechanism:**
- `httptest.NewServer` gives a real TCP connection
- `brokerSubCount` helper polls `UserEventBroker.shards[userShard].userSubs` directly (same package, test-only — no production API surface added)
- `resp.Body.Close()` drops the connection; server detects via context cancellation
- `require.Eventually(..., 2s, 20ms)` asserts count reaches zero

### `TestStreamUserEvents_WriteErrorCancelsStream`

**What it proves:** When a write to the response body fails (write deadline exceeded, broken pipe):
1. `fmt.Fprint(c.Writer, ...)` or `fmt.Fprintf(...)` returns a non-nil error
2. `streamCancel()` is called (`stream_user_events.go:109`, `123`, or `133`)
3. `defer UnsubscribeUser` fires — subscription unregistered

**Mechanism:**
- Publishes a workspace.phase event to force the live loop into a write attempt
- Closes `resp.Body` immediately after publish — server's write to dead connection fails
- Same `brokerSubCount` assertion

Both tests pass with `-race` and `-count=3`.

## Helper Added

```go
// brokerSubCount is a test helper that returns the number of active subscribers
// for a user by reading the broker's internal shard directly.
func brokerSubCount(b *UserEventBroker, userID string) int {
    sh := b.userShard(userID)
    sh.mu.Lock()
    defer sh.mu.Unlock()
    return len(sh.userSubs[userID])
}
```

Lives in `stream_user_events_test.go` (same package). Reads the shard directly rather than exposing a count method on the production type.

## Remaining Weak Points (not addressed here)

The `TestStreamUserEvents_WriteErrorCancelsStream` test exercises write failure via connection close, which triggers the `streamCancel()` path at the OS level. It does **not** exercise the `rc.SetWriteDeadline` path directly — that requires a writable but artificially slow connection where the deadline expires before the write completes. This is hard to test without a custom `net.Conn` that blocks writes. The behavior is correct and the connected code path (`SetWriteDeadline → write returns error → streamCancel`) is the same; only the trigger mechanism differs.
