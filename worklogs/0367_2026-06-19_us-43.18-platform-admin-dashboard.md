# Worklog: US-43.18 Platform Admin Dashboard

**Date:** 2026-06-19
**Session:** Implement US-43.18 (platform admin dashboard) — org/user list endpoints, handlers, routes, and frontend tabs — building on the already-landed US-43.19 (suspension) and US-43.20 (cross-org audit).

---

## Objective

Deliver the remaining gaps of US-43.18 on top of US-43.19/43.20: platform-wide
list endpoints (`GET /admin/orgs`, `GET /admin/users`), paginated/filtered store
queries with aggregated counts, a dashboard frontend (Organisations + Users
tabs with suspend/unsuspend wired to US-43.19 endpoints), the API client, and
tests at every layer.

## Key Decisions

1. **`ListAllUsers` lives on `*database.Service` (not `PgOrgStore`)** — matches
   the existing `SetUserStatus` location, so `platformAdminUserStore` (already
   satisfied by `dbSvc` in `app.go`) gains the method with no DI change.
2. **`ListAllOrgs` on `PgOrgStore`** — matches `UpdateOrgStatus`/audit surface
   already used by `platformAdminOrgStore`; `app.go` wiring unchanged.
3. **Dashboard DTOs return org_id/org_name for users, not just a count** — under
   D8 (single-org) a count is degenerate (0/1); the resolved name is more useful
   for operators at the same query cost. `OrgCount` retained for clients that
   only care about membership existence.
4. **Single SELECT with correlated subqueries for org counts** — avoids N+1
  round trips; acceptable for the admin-dashboard cardinality (<1k orgs).
5. **Frontend follows the existing SettingsPage tab pattern** (not
   `/admin/platform/*` routes) per the deliverable instruction — consistent UX,
   less new code.
6. **`OrgSettingsTab.tsx` rewritten in place** (not a new file) — it was only
   ever mounted as the admin-only `admin-orgs` tab, so "replaces" is the
   cleanest reading; avoids a second orgs tab and dead code.
