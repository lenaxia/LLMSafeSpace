## Objective

D17: org members can sign in via the org's OIDC provider (Azure AD, Okta, Keycloak, Google). First-time SSO users are auto-provisioned (configurable). Group claims map to org roles. Email-domain routing shows "Sign in with [Org]" on the login page.

### SSO config handler (org admin CRUD)
- `GET/PUT/DELETE /api/v1/orgs/:id/sso` — OrgAdminGuard. Client secret encrypted via server KEK before store; never returned (`hasSecret` only).

### OIDC login flow
- `GET /api/v1/auth/sso/:orgSlug/start` — creates OIDC provider, generates PKCE verifier + HMAC-signed state cookie (10min TTL), redirects to IdP.
- `GET /api/v1/auth/sso/:orgSlug/callback` — verifies state (constant-time), exchanges code with PKCE verifier, verifies ID token (RS256 + JWKS via `coreos/go-oidc`), extracts email/sub/groups, looks up or auto-provisions user, applies group-claim→role mapping, issues JWT.
- `GET /api/v1/auth/sso/domains` — public endpoint for login-page domain discovery.

### Group-claim role mapping
- `resolveRole`: admin outranks member; unmapped → member default. Refreshed on every login. Last-admin demotion prevention.

### Auto-provisioning
- Creates user with email + random un-verifiable bcrypt hash (blocks password login). Ensures org membership.

### Frontend
- `OrgSSOTab.tsx` — OIDC config form.
- Login page domain detection + "Sign in with [Org]" button.

## Key Decisions

1. **`coreos/go-oidc/v3`** — the de facto Go OIDC library. `golang.org/x/oauth2` promoted from indirect to direct.
2. **Server KEK (not org DEK) for client-secret encryption** — per D17-S4. The org DEK was removed in migration 000035; server KEK is the right key for service-owned secrets.
3. **PKCE state in HMAC-signed cookie, not server-side session** — API is stateless; cookie is the correct mechanism.
4. **`memberOf` fallback for Azure AD** — Azure uses `memberOf` instead of `groups`. **THIS IS UNTESTED (F9)** — see Blockers.

---

## Blockers

**3 findings from worklog 0372:**

- **F8 (MEDIUM):** OIDC does not check `email_verified` before binding email to an existing account. A user who registers `victim@example.com` at a permissive IdP could be matched to the victim's account. **Fix: require `email_verified == true` for account binding.**
- **F9 (MEDIUM):** `memberOf` fallback is untested. Azure AD (a major enterprise IdP) uses `memberOf`; the code path exists but no test exercises it. **Fix: add a test with `memberOf` claims.**
- **F10 (MEDIUM):** PKCE verifier is not validated by the fake IdP test. The client sends `code_verifier`, but the fake IdP's `/token` handler records it without checking `SHA256(verifier) == challenge`. **Fix: have the fake IdP validate the binding.**
- **F11 (LOW):** Redirect URL trusts `X-Forwarded-Proto/Host` when `RedirectBaseURL` is unset. Mitigated by IdP redirect-URI registration. **Fix: document the production requirement to set `RedirectBaseURL`.**
