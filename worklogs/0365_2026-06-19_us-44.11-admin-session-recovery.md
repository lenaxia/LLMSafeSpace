## Objective

Add `POST /api/v1/admin/workspaces/:workspaceId/sessions/:sessionId/force-abort` ‚ÄĒ an admin-only endpoint that clears a stuck session from `wsstate.Store.activeSess` without requiring the workspace pod to be reachable. Used for incident recovery when a workspace was deleted but sessions persist in "busy" state (the root cause of the 2026-06-16 incidents A & B).

## Key Decisions

1. **Endpoint takes both workspaceId + sessionId** (not sessionID-only as in the design). `wsstate.Store` is workspace-scoped; there's no global session‚Üíworkspace index (session_index only has sessions that received messages). All existing session endpoints take both IDs.
2. **Operates purely on local state** ‚ÄĒ no opencode proxy call. The pod may be deleted; force-abort is exactly for when `reconcileSessionState` can't run (no pod to reconnect to).
3. **`*sql.DB` dependency** (not `interfaces.DatabaseService`). `DatabaseService` doesn't expose `ExecContext`; matches the `UsageHandler` audit-log pattern (`usage.go:241`).
4. **Audit failure is non-fatal.** Incident recovery must not be blocked by a DB hiccup. The force-abort succeeds even if the audit INSERT fails (logged server-side). Differs from `AdminDiscardDLQ` which returns 500 on audit failure ‚ÄĒ that operation is not incident-recovery-critical.
5. **Session-index cleanup deliberately omitted.** Force-abort clears only the stuck active-session marker. The session may still be live in opencode. Calling `DeleteSession` on the index would corrupt a live session's sidebar entry.
