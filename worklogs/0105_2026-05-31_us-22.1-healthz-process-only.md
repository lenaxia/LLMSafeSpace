# Worklog 0104 — US-22.1: Process-Only `/v1/healthz` Liveness

**Date:** 2026-05-31
**Author:** OpenCode (autonomous coding agent)
**Epic:** 22 — agentd Health-Endpoint Redesign
**Story:** US-22.1
**Status:** Shipped

## TL;DR

`/v1/healthz` no longer calls opencode. The kubelet liveness probe will not kill workspace pods because opencode is busy under SSE load (the failure mode documented in worklog 0096 and reproduced multiple times this session in worklogs 0099/0100).

## Context

Pre-fix, `/v1/healthz` made an HTTP call to opencode's `/global/health` on every probe. When opencode was back-pressured by SSE traffic, the call timed out. After kubelet's `FailureThreshold=6` consecutive 5-second-timeout liveness failures (~3 minutes of slowness), the kubelet killed the pod — even though agentd itself was perfectly healthy.

The verified incident pattern (worklog 0100): agentd's `/v1/statusz` timed out 3× under SSE load → controller deleted the pod → dying pod misclassified as Failed (Epic 23 Story 1) → workspace stuck. The kubelet liveness fuse was the slowest of the three race conditions but eventually kills the pod independently of the controller. US-22.1 fixes the kubelet liveness fuse.

## Change

`/v1/healthz` is now process-only:

```go
func healthzHandler(startedAt time.Time) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Content-Type", "application/json")
        _ = json.NewEncoder(w).Encode(agentd.HealthzResponse{
            Healthy:       true,
            Version:       buildVersion,
            UptimeSeconds: int(time.Since(startedAt).Seconds()),
        })
    }
}
```

If the handler runs, the agentd process is alive and serving HTTP. That's the entire signal kubelet's liveness probe needs. Anything more conflates agentd's liveness with opencode's responsiveness.

The response shape is unchanged (`agentd.HealthzResponse`) for kubelet/consumer compatibility. `Version` now reports the agentd build version (defaulting to `"dev"`, overridable via `-ldflags "-X main.buildVersion=$VERSION"`) instead of opencode's version. `Healthy` is always `true` when the handler executes — by definition, a dead process cannot respond.

## Files

- `cmd/workspace-agentd/healthz.go` — new file, factory function `healthzHandler`.
- `cmd/workspace-agentd/healthz_test.go` — new file, 11 unit tests + benchmark.

No other files touched. The change is purely additive: factory function and tests are new, `buildVersion` is new, the inline handler in main is removed and replaced with a factory invocation.

## Test coverage

11 unit tests assert the contract:

1. `TestHealthzHandler_ReturnsHealthyWithoutOpencode` — happy path; response shape, status, content type, body fields.
2. `TestHealthzHandler_NeverCallsOpencode` — central contract: opencode mock fails the test if it receives any request.
3. `TestHealthzHandler_LatencyUnderOpencodeStarvation` — regression for worklog 0096; opencode mock hangs 30s, handler must respond < 100ms over 50 probes.
4. `TestHealthzHandler_ConcurrentRequestsAreRaceFree` — 100 concurrent requests under `-race`.
5. `TestHealthzHandler_IgnoresRequestBodyAndMethod` — table-driven over GET/POST/HEAD/PUT/DELETE.
6. `TestHealthzHandler_UptimeAdvances` — uptime monotonically advances.
7. `TestHealthzHandler_DoesNotConstructOpenCodeClient` — defensive; signature MUST NOT take an `*OpenCodeClient`.
8. `TestHealthzHandler_ContextCancellationIgnored` — handler doesn't consult ctx.
9. `TestHealthzHandler_ResponseShapeIsExactlyAgentdHealthzResponse` — exactly 3 documented fields, no extras.
10. `TestHealthzHandler_BurstThroughput` — 10,000 concurrent requests in < 2s, no goroutine leak.
11. `BenchmarkHealthzHandler` — measured at 950 ns/op on dev hardware. SLO is < 100ms (100,000× headroom).

Race-detector clean. Lint clean. Existing tests pass (`go test ./cmd/workspace-agentd/... ./controller/... ./pkg/agentd/...` — all green).

## What this does NOT change

- `/v1/readyz` still calls opencode 4× (cachedState + IsHealthy). Fixed in US-22.2 + US-22.3 (eager-refresh cache + cached readyz).
- `/v1/statusz` still calls opencode multiple times under a mutex. Documented as expensive in US-22.4; controller's deep-status poll uses it on a slower cadence per US-22.6.
- Pod spec probe paths unchanged. US-22.8 will later relocate `/v1/healthz` from the user port to the admin port.
- Controller's `checkAgentHealth` still hits `/v1/statusz`. US-22.5 introduces a separate `checkAgentLiveness` targeting `/v1/healthz`.

## Production impact

This change ships as part of the next agentd image build. Pod cycle propagates the fix per workspace. No mixed-version concerns within a single pod. Old workspace pods continue to use the pre-fix handler until they cycle.

Post-deploy expectations:
- Kubelet liveness probe latency drops to sub-millisecond (was: up to 5s timeout under load).
- Pods no longer restart due to opencode busyness. Workspaces become more resilient to SSE bursts.
- The cause of "workspace went Failed despite agentd being healthy" decreases by one independent failure mode (the other two — controller's `/v1/statusz` timeout and Story 1's dying-pod misclassification — are addressed by Epic 22 follow-on stories and Epic 23 Story 1 respectively).
