## Objective

Workspace passwords are stable (only change on workspace recreate), so they cache well. Moving to Redis eliminates the multi-replica staleness where one replica could clear its cache on a 401 while another kept serving the old password and got 502s. With a shared cache, all replicas agree.

### Key design decisions
1. **Fail-through-to-K8s** (third policy in the Epic 45 fail matrix): Redis is purely a performance optimization; the source of truth is the K8s Secret. On Redis error, GetCachedPassword returns `("", false)` so the caller falls through to the K8s Secret fetch. NOT fail-open (no return true) and NOT fail-closed (no false data).
2. **TTL = 1 hour** (longer than activeSess/deletedSessions at 30min) â€” passwords are stable, longer caching is safe.
3. **Hash-tagged key** `ws:{workspace_id}:pw` â€” forces co-location with activeSess and deletedSessions keys for future cluster migration.
4. **InvalidatePassword = DEL** â€” no pubsub needed per design (replicas hit Redis on every request anyway).
