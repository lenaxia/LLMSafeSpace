## Objective

Ship US-29.1 — the foundational interface that unifies all opencode HTTP communication. Before this, 5 separate password-getter injection points existed across the API (SecretsHandler, sse.Tracker, ProxyHandler, AgentReloadHandler, BulkReloadHandler), each with the same `func(ctx, workspaceID) (string, error)` signature but wired independently. US-29.1 provides the single interface that future handler splits (US-29.2/29.3/29.5) will consume.

## Work Completed

### 1. Extended `opencode.Client` with `ListModels` and `PatchConfig`

**`pkg/agent/opencode/client.go`** — added two methods to the existing low-level Client:
- `ListModels(ctx) ([]byte, error)` — GET /provider, returns raw JSON (1 MiB limit)
- `PatchConfig(ctx, config map[string]any) error` — PATCH /global/config

Both use the same Basic auth pattern as the existing methods (`agentd.AuthUsername` + password).

### 2. Defined `AgentClient` interface + `WorkspaceClient` implementation

**NEW file: `pkg/agent/opencode/agent_client.go`**
- `AgentClient` interface — 5 methods, all workspace-scoped (resolve podIP + password internally):
  - `ListModels(ctx, userID, workspaceID) ([]byte, error)`
  - `PatchConfig(ctx, userID, workspaceID, config) error`
  - `DisposeInstance(ctx, userID, workspaceID) error`
  - `GetSessionStatuses(ctx, userID, workspaceID) (map[string]string, error)`
  - `StageCredentials(ctx, userID, workspaceID, providers) error`
- `WorkspaceClient` — implements AgentClient by resolving workspace → podIP + password on each call, then delegating to the low-level Client. Constructed with `PasswordResolver` and `PodIPResolver` injected.
- `PasswordResolver` type — `func(ctx, workspaceID) (string, error)` — matches the existing 5 injection points.
- `PodIPResolver` interface — `GetWorkspacePodIP(ctx, userID, workspaceID) (string, error)` — matches existing `handlers.PodIPResolver`.
- Compile-time check: `var _ AgentClient = (*WorkspaceClient)(nil)`

## Key Decisions

1. **Interface lives in `pkg/agent/opencode/`, not `api/internal/handlers/`.** The interface is consumed by handlers but defined in the agent package because it describes the opencode API contract, not the handler contract. This matches the existing `Client` location.

2. **WorkspaceClient wraps the low-level Client, doesn't replace it.** The existing `Client` (with resolved baseURL + password) is still used directly by `AgentReloadHandler` and `agentd`. The new `WorkspaceClient` adds the workspace-resolution layer on top. No existing callers change.

3. **No handler migration in this PR.** US-29.1 defines the interface; wiring handlers to consume it is US-29.5 (ModelsHandler extraction). This keeps the PR focused on the foundational type, not a massive refactor. The interface is ready for incremental adoption.

4. **`resolve` creates a fresh Client per call.** This ensures podIP and password are always current (no stale cache). The Client is lightweight (just URL + password + http.Client). For hot paths that call multiple methods on the same workspace, a future optimization could cache the resolved Client per-request.
