## Objective

Replace raw user-ID member addition with email-based invitations using AWS SES (D2). Implement token-based invitation flow with accept/decline, concurrent-accept protection, rate limiting, and bounce handling.

---

## Work Completed

### Schema (migration 031)
- `org_invitations` table: id, org_id (FK CASCADE), email, role, invited_by (FK users), token_hash (UNIQUE), expires_at, accepted_at/by, declined_at, bounce_type/bounced_at, created_at, updated_at.
- Indexes on pending invitations (org_id, email).

### Email package (`pkg/email/`)
- `EmailProvider` interface with `Send(ctx, Message)`.
- `SESProvider` (AWS SDK v2), `NoopProvider` (stderr logging for dev/test).
- Config: `email.provider` (ses/noop), `email.sesRegion`, `email.fromAddress`, `email.baseUrl`.

### Invitation store (`pg_org_store.go`)
- CreateInvitation, ListPendingInvitations, GetInvitationByTokenHash, GetInvitationByID, AcceptInvitationTx (FOR UPDATE + ON CONFLICT DO NOTHING), DeclineInvitation, DeleteInvitation, CountInvitationsLastHour.

### Invitation handler (`invitations.go`)
- Create (batch emails, rate limit 50/hr, per-email token generation).
- List (pending only), Delete (revoke), Resend (new token, invalidates old).
- GetByToken (public — org name, inviter username, role, expiry).
- Accept (JWT auth, expiry/membership/taken checks, FOR UPDATE).
- Decline (JWT auth).

### Token security
- 32 bytes crypto/rand → base64url (~43 chars, 256 bits entropy).
- SHA-256 hash stored; plaintext never in DB.
- Lookup via hash only; timing-safe comparison not needed (hash is constant-length).

### Validator remediations
- **C-1**: Cross-org IDOR on Resend/Delete → verify `existing.OrgID == orgID` (404 otherwise).
- **H-2**: Email send failures now logged at Error level.
- **H-3**: Startup validation rejects `provider=ses` without `fromAddress`/`baseUrl`.
- **H-4**: Resend rejects already-accepted/declined invitations (409).
- **L-1**: Expiry re-checked under FOR UPDATE lock (`AND expires_at > NOW()`).
- **L-2**: Inviter fallback changed from raw user ID to "An administrator".
- **L-4**: N+1 GetOrg eliminated (org fetched once in Create, passed to sendInvitationEmail).
- **L-6**: AcceptInvitationTx fetches org_id in the FOR UPDATE query (single SELECT).
