## Part 1: Benchmark Infrastructure — Issues Found and Fixed

### Benchmark script development (`hack/in-cluster-benchmark.sh`)

The benchmark was run from an in-cluster pod (`llmsafespace-benchmark`) because
the cluster network (pod CIDR 10.69.x.x, service CIDR 10.96.x.x) is not routable
from the dev host.

**Issues encountered and resolved before getting valid numbers:**

| Issue | Root cause | Fix |
|-------|-----------|-----|
| `date +%s%3N` returned seconds, not ms | BusyBox `date` in Alpine does not support `%3N` (GNU extension) | Replaced `ts_ms()` with `python3 -c 'import time; print(int(time.time()*1000))'` |
| Port 4098 (agentd admin) unreachable cross-node | `llmsafespace-workspace-default-deny-ingress` NetworkPolicy only allows ingress from pods with `app.kubernetes.io/name=llmsafespace` + `app.kubernetes.io/component=api` | Labelled benchmark pod with those labels |
| API calls returning 403/redirect | Security middleware redirects plain HTTP to HTTPS unless `X-Forwarded-Proto: https` header is present | Added header to all in-cluster API calls |
| API path wrong | Routes are at `/api/v1/auth/login`, `/api/v1/workspaces/…` not `/auth/login`, `/workspaces/…` | Corrected paths |
| Auth: benchmark user couldn't own workspaces | `bench@benchmark.local` user was inserted directly into DB but existing suspended workspaces were owned by `mike@kao.family` | Created fresh workspace under bench user |
| Near-zero gate times despite rewrite | `wait_new_pod_running` used `jq` `strptime`/`mktime` to filter new pods — not available in alpine `jq 1.7.1` | Rewrote to use `awk` string comparison on ISO8601 timestamps (`$2 >= t0`) which works lexicographically |
| Still near-zero after jq fix | `RESUME_WS_ID="${3:-}"` on line 12 overwrote the env var of the same name with empty `$3` | Changed to `RESUME_WS_ID="${RESUME_WS_ID:-${3:-}}"` |
| Still near-zero after env fix | Gate timestamps captured with `ts_ms()` AFTER wait functions returned, not INSIDE polling loops; if event already happened before first poll all times collapse | Rewrote all gate loops to capture `t_now=$(ts_ms)` inside the loop body at the moment the condition is first observed |
| Near-zero even after loop fix | `ts_ms()` still returning seconds (the BusyBox issue not yet fixed at that point) — 13 seconds displayed as `13ms / 1000 = 0.013s → "0.01s"` | See timer fix above — fixed by switching to python3 |
| POSIX sh quoting: `curl -H "Authorization: Bearer $token"` not working | `${token:+-H "Authorization: Bearer $token"}` expands as one argument; word-splitting doesn't split `-H` from the header value | Changed to explicit `if [ -n "$admin_token" ]; then curl -H ...; fi` |

**Validated assumptions before each run:**
- `date -u '+%Y-%m-%dT%H:%M:%SZ'` — confirmed Alpine supports this format
- `awk '$2 >= t0'` on ISO8601 strings — confirmed lexicographic comparison works correctly for same-second granularity  
- Port 4098 reachability — confirmed 200 response from benchmark pod after labelling
- Auth token retrieval — confirmed `kubectl get secret workspace-pw-<id>` and `base64 -d` works in pod
- `RESUME_WS_ID` env var passing — confirmed `export VAR=val; exec sh script` passes vars to child

## Part 2: Measured Baseline

**Methodology:**
- Benchmark user: `bench@benchmark.local` (inserted directly into DB)
- API key: `lsp_5e086054...` (created via `/api/v1/auth/api-keys`)
- Benchmark pod: `llmsafespace-benchmark` in `default` namespace, labelled with network policy labels
- Script: `hack/in-cluster-benchmark.sh` v1 (in-cluster variant)
- Timer precision: `python3 time.time()*1000` — millisecond precision, wall-clock
- Gate definitions:
  - **G1 API→Creating:** `ts(resume API call)` → first poll returning `phase=Creating` or `Active`
  - **G2 Creating→Pod Running:** first G1 observation → first new pod (creationTimestamp ≥ t0_iso) with `phase=Running`
  - **G3 Pod Running→readyz 200:** G2 observation → first HTTP 200 from `http://<pod_ip>:4098/v1/readyz` with Bearer token
  - **G4 readyz 200→Active:** G3 observation → first poll returning `phase=Active`
- Iterations: 3 per path (small sample; directional baseline only)
- Notes on readyz: deployed version `sha-b867cc6` has old readyz logic requiring `providers_connected > 0`. After S18.11 ships, G3 will only require opencode to be up (not provider connected).

### Resume baseline (Suspended → Active)

| Gate | run 1 | run 2 | run 3 | p50 | p99 |
|------|-------|-------|-------|-----|-----|
| G1 API → Creating | 0.22s | 0.29s | 0.18s | 0.22s | 0.29s |
| G2 Creating → Pod Running | 5.83s | 10.21s | 10.76s | 10.21s | 10.76s |
| **G3 Pod Running → readyz 200** | **5.89s** | **6.59s** | **5.06s** | **5.89s** | **6.59s** |
| G4 readyz 200 → Active | 0.10s | 0.07s | 0.09s | 0.09s | 0.10s |
| **TOTAL** | **12.05s** | **17.15s** | **16.09s** | **16.09s** | **17.15s** |

**Observation:** The "2 minutes" previously reported was unverified. Measured p99 is 17s.
G2 (pod scheduling) is the dominant gate at up to 10.8s. G3 (readyz/opencode startup
+ provider connect) is second at up to 6.6s.

### Create baseline (Pending → Active, no packages, no initScript)

| Gate | run 1 | run 2 | run 3 | p50 | p99 |
|------|-------|-------|-------|-----|-----|
| G1 API → Creating (PVC) | 5.35s | 5.44s | 5.44s | 5.44s | 5.44s |
| G2 Creating → Pod Running | 14.48s | 15.32s | 13.80s | 14.48s | 15.32s |
| **G3 Pod Running → readyz 200** | **5.02s** | **7.82s** | **7.79s** | **7.79s** | **7.82s** |
| G4 readyz 200 → Active | 0.07s | 0.20s | 0.07s | 0.07s | 0.20s |
| **TOTAL** | **24.92s** | **28.79s** | **27.11s** | **27.11s** | **28.79s** |

**Observation:** PVC provisioning (G1 ≈ 5.4s) is a consistent ~20% overhead on create
vs resume. Pod scheduling (G2) is slower on create (14–15s) vs resume (6–11s) — likely
because resume pods reuse a cached image and known PVC location, while create may
involve different scheduling decisions.

### Caveats on these numbers

1. **3 iterations is a small sample.** p99 = p90 = max in all cases. More iterations
   needed for statistical confidence. The benchmark script supports `--iterations N`.

2. **Single-tenant cluster.** No concurrent workspace pressure. Production numbers with
   many concurrent workspaces may differ.

3. **Image already cached on worker-00.** All workspace pods scheduled to `worker-00`
   (192.168.3.20). Image pull cost was not measured (`--fresh-node` variant not run).

4. **readyz includes provider connectivity (old code).** After S18.11 deploys, G3 is
   expected to drop from ~5–8s to ~2–4s (opencode startup only, no provider check).

5. **UX-perceived latency is not yet measured.** The benchmark measures backend
   phase transitions. Frontend latency (SSE propagation, React render, user-visible
   spinner dismissal) is additive and currently unknown. Frontend timing
   instrumentation was added in this worklog to measure it.

## Part 3: Frontend Timing Instrumentation

### What was added

All timing log lines use the prefix `[ws-timing]` — filter by this string in the
browser DevTools Console to isolate them.

**Format:**
```
[ws-timing] <event> | ws=<id_short> | t=<perf_ms>ms | wall=<iso> | <extra>
```
- `t=` is `performance.now()` — sub-millisecond, monotonic, relative to page load
- `wall=` is `Date.toISOString()` — for correlation with backend/benchmark logs

**Events logged:**

| Event | Where | What it marks |
|-------|-------|---------------|
| `sse.connecting` | `useEventStream` | SSE fetch initiated |
| `sse.connected` | `useEventStream` | First successful SSE connection |
| `sse.reconnected` | `useEventStream` | Reconnect after drop (also fires `onReconnect`) |
| `sse.connect_failed` | `useEventStream` | HTTP error on SSE connect (with status code) |
| `sse.disconnected` | `useEventStream` | SSE stream ended (with will_retry flag) |
| `sse.error` | `useEventStream` | Non-abort error during streaming |
| `sse.teardown` | `useEventStream` | Effect cleanup (component unmounted / workspaceId changed) |
| `sse.workspace_phase` | `useEventStream` | `workspace.phase` SSE event received (with `phase=`) |
| `status.fetch_start` | `useWorkspaceStatus` | React Query fires a status GET |
| `status.fetch_done` | `useWorkspaceStatus` | Status GET returned (with `phase=`) |
| `activate.called` | `useActivateWorkspace` | Activate mutation fired (API call started) |
| `activate.success` | `useActivateWorkspace` | Activate API call returned 2xx |
| `ui.phase_changed` | `ChatPage` | React rendered with a new phase value (prev→next logged) |
| `ui.workspace_ready` | `ChatPage` | Phase changed to `Active` from non-Active — spinner dismissed |
| `ui.user_clicked_activate` | `ChatPage` | User clicked "Activate" on SuspendedBanner |

### Files modified

| File | Change |
|------|--------|
| `frontend/src/hooks/useEventStream.ts` | Added `wsLog()` helper (exported), instrumented connect/reconnect/disconnect/phase events |
| `frontend/src/hooks/useWorkspaces.ts` | Wrapped `queryFn` to log `status.fetch_start` and `status.fetch_done` with phase |
| `frontend/src/hooks/useActivateWorkspace.ts` | Logged `activate.called` and `activate.success` with workspaceId |
| `frontend/src/pages/ChatPage.tsx` | Imported `wsLog`; added `useEffect` with `prevPhaseRef` to log every phase change and the `ui.workspace_ready` moment; added `ui.user_clicked_activate` on SuspendedBanner |

### How to use

1. Open browser DevTools → Console
2. In the filter box, type: `[ws-timing]`
3. Trigger a resume (click "Activate") or navigate to a resuming workspace
4. Observe the sequence:
   - `activate.called` → `sse.workspace_phase phase=Resuming` → `sse.workspace_phase phase=Creating` → ... → `sse.workspace_phase phase=Active` → `status.fetch_start` → `status.fetch_done phase=Active` → `ui.phase_changed Active` → `ui.workspace_ready`
5. The gap between `sse.workspace_phase phase=Active` and `ui.workspace_ready` is the
   frontend processing overhead (React Query invalidate → fetch → render cycle)
6. The gap between `activate.called` and `ui.workspace_ready` is the full UX latency

### Expected UX latency breakdown

Based on the architecture:
- `activate.called` → backend sets phase=Resuming → SSE emits `workspace.phase Resuming` → UI shows spinner (fast, ~100ms)
- Spinner shown until backend reaches `Active` and SSE emits `workspace.phase Active`
- React Query then fires `status.fetch_start` (invalidation-triggered), fetches, renders
- Expected overhead from `sse.workspace_phase Active` to `ui.workspace_ready`: 1 HTTP round-trip (~50–200ms)

The gap between backend `Active` (measured by benchmark at ~16–17s p99) and
`ui.workspace_ready` is what the frontend timing will reveal.
