# US-24.17: Degraded Detection

**Domain:** Observability
**Depends On:** US-24.1
**Priority:** Medium

## Summary

Detect application-level health issues (disk pressure, credential loss, readiness failure) and surface them as CRD conditions. The controller does NOT restart pods for these — it signals the problem and lets the user or environment self-heal.

## Signals Detected

| Signal | Source | Condition Set | Auto-Recoverable? |
|---|---|---|---|
| Disk >95% full | `enrichAgentStatus` → `statusz.Disk` | `DiskPressure = True` | No (user must free space) |
| No providers connected | `enrichAgentStatus` → `statusz.Connected = []` | `AgentHealthy = False` (reason: AgentDegraded) | Sometimes (provider reconnects) |
| Pod NotReady while Active | Pod condition `Ready = False` | `AgentHealthy = Unknown` | Usually (load drops) |

## Implementation

In `enrichAgentStatus` (runs every 60s on Active workspaces):

```go
func (r *WorkspaceReconciler) enrichAgentStatus(ctx context.Context, ws *v1.Workspace) {
    // ... existing statusz fetch ...

    // Disk pressure detection
    if status.Disk != nil {
        ratio := float64(status.Disk.UsedBytes) / float64(status.Disk.TotalBytes)
        if ratio > 0.95 {
            r.setCondition(ws, WorkspaceConditionDiskPressure, "True",
                "DiskPressure", fmt.Sprintf("disk %.0f%% full (%s/%s)",
                    ratio*100, humanBytes(status.Disk.UsedBytes), humanBytes(status.Disk.TotalBytes)))
        } else {
            r.removeCondition(ws, WorkspaceConditionDiskPressure)
        }
    }

    // Provider connectivity (existing AgentDegraded logic already handles this)
    // ... no change needed, already sets AgentDegraded condition ...
}
```

In proxy handler (for NotReady pods):

```go
// Before proxying, check if pod is Ready
if !isPodReady(pod) {
    c.JSON(http.StatusServiceUnavailable, gin.H{
        "error":      "workspace temporarily unreachable",
        "reason":     "readiness_probe_failing",
        "retryAfter": 5,
        "message":    "The workspace is running but temporarily unable to serve requests. This usually resolves automatically.",
    })
    return
}
```

## Condition Auto-Clear

Conditions are cleared when the signal resolves:
- Disk drops below 95% → `DiskPressure` removed
- Providers reconnect → `AgentDegraded` reason cleared, `AgentHealthy = True`
- Pod becomes Ready → proxy routes normally (no condition to clear — it's a transient proxy behavior)

## Key Principle

**Degraded does NOT restart the pod.** The pod is running. The user may have partial functionality. Restarting would destroy their session for no benefit. The controller's job is to SIGNAL the problem, not fix it.

## Existing State (validated against code)

- The API's `WorkspaceStatusResult` already includes `Conditions []WorkspaceConditionResult` (pkg/types/types.go:283) — conditions are already returned by the GET /status endpoint.
- The controller already sets `AgentHealthy = False` with reason `AgentDegraded` when no providers are connected (controller.go:1364).
- The frontend TypeScript `WorkspaceStatus` type does NOT include `conditions` (frontend/src/api/types.ts:89) — this is the gap US-24.18 fills.
- No `removeCondition` helper exists — only `setCondition`. US-24.1 adds `removeCondition`.
- No `DiskPressure` condition type exists yet — must be added to the CRD types.

## Acceptance Criteria

1. Disk >95% → `DiskPressure = True` with human-readable message
2. Disk <95% → `DiskPressure` removed (auto-clear)
3. No providers + configured > 0 → `AgentHealthy = False` (already exists, verify)
4. Providers reconnect → condition cleared
5. Pod NotReady → proxy returns structured 503 (not raw error)
6. Controller does NOT restart pods based on Degraded conditions
7. Conditions include actionable message for the user
