# US-24.7: Controller-Initiated Restarts Don't Count as Failures

**Domain:** Recovery State Machine
**Depends On:** US-24.5
**Priority:** Medium

## Summary

When `checkAgentHealth` deletes a pod (health-check threshold exceeded), this is a controlled restart — not a failure. It should increment `RestartCount` and `ControllerRestartCount` but NOT `ConsecutiveFailures`.

## Current Behavior

`checkAgentHealth` (L1247-1256):
- Deletes pod
- Sets phase=Creating
- Increments `RestartCount`
- Resets `ConsecutiveHealthFailures` to 0

Next reconcile enters `handleCreating` → if new pod fails → `enterRecovery` increments `ConsecutiveFailures`. This is correct — the NEW pod's failure counts. The restart itself doesn't.

## Changes

1. Add `ControllerRestartCount int32` to WorkspaceStatus
2. `checkAgentHealth` increments both `RestartCount` and `ControllerRestartCount`
3. If `ControllerRestartCount` exceeds 5 without any stability period between restarts → trigger safe mode (A13 fix: persistent unreachability)
4. Stability reset (US-24.8) resets `ControllerRestartCount` to 0

## Counter Semantics (F9)

| Counter | Meaning | Incremented by |
|---|---|---|
| `RestartCount` | Total pod restarts (all causes) | Health restart, restartGeneration bump, architecture drift |
| `ControllerRestartCount` | Health-check initiated restarts only | Health restart only |
| `ConsecutiveFailures` | Failures in the recovery loop | `enterRecovery` only (NOT health restarts) |

## Acceptance Criteria

1. Health-check pod delete increments `RestartCount` and `ControllerRestartCount`
2. Health-check pod delete does NOT increment `ConsecutiveFailures`
3. `ControllerRestartCount > 5` without stability → enters safe mode
4. Stability reset clears `ControllerRestartCount`
5. `restartGeneration` bump clears `ControllerRestartCount`
