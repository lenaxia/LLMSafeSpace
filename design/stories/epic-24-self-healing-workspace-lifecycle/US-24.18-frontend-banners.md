# US-24.18: Frontend SafeMode and Degraded Banners

**Domain:** Frontend
**Depends On:** US-24.13, US-24.17
**Priority:** Medium

## Summary

Frontend displays actionable banners when workspace conditions indicate SafeMode or Degraded state. Users see what's wrong and what they can do about it.

## TypeScript Type Changes

```typescript
// api/types.ts — add conditions to WorkspaceStatus
// NOTE: The backend API (WorkspaceStatusResult in pkg/types/types.go) already
// returns conditions in the JSON response. The frontend type just doesn't
// consume them. This is a frontend-only change.
export interface WorkspaceStatus {
  phase: string;
  // ... existing fields ...
  conditions?: WorkspaceCondition[];  // NEW — already in API response, just not typed
  safeMode?: boolean;                 // NEW — convenience field (also derivable from conditions)
}

export interface WorkspaceCondition {
  type: string;    // "SafeMode" | "DiskPressure" | "AgentHealthy"
  status: string;  // "True" | "False" | "Unknown"
  reason?: string;
  message?: string;
}
```

## Banner Components

### Safe Mode Banner (Red)

Shown when: `conditions` includes `{type: "SafeMode", status: "True"}`

```
┌─────────────────────────────────────────────────────────────────┐
│ ⚠️  Safe Mode                                                    │
│                                                                   │
│ Your workspace entered safe mode after repeated failures.         │
│ Reason: {condition.message}                                       │
│                                                                   │
│ Your files are accessible via terminal (read-only).               │
│                                                                   │
│ [Retry Normal Mode]  [Download Files]  [Open Terminal]            │
└─────────────────────────────────────────────────────────────────┘
```

- "Retry Normal Mode" → calls `POST /workspaces/:id/restart` (bumps restartGeneration)
- "Download Files" → calls `GET /workspaces/:id/files/download` (streams tar.gz)
- "Open Terminal" → opens existing terminal tab

### Disk Pressure Banner (Yellow)

Shown when: `conditions` includes `{type: "DiskPressure", status: "True"}`

```
┌─────────────────────────────────────────────────────────────────┐
│ ⚠️  Disk Almost Full                                             │
│                                                                   │
│ {condition.message}                                               │
│ Open a terminal and delete unnecessary files to free space.       │
│                                                                   │
│ [Open Terminal]                                                    │
└─────────────────────────────────────────────────────────────────┘
```

### Agent Degraded Banner (Yellow)

Shown when: `conditions` includes `{type: "AgentHealthy", status: "False", reason: "AgentDegraded"}`

```
┌─────────────────────────────────────────────────────────────────┐
│ ⚠️  LLM Provider Disconnected                                    │
│                                                                   │
│ No LLM provider is connected. Check your credentials.             │
│ Terminal and file access still work.                               │
│                                                                   │
│ [Manage Credentials]                                              │
└─────────────────────────────────────────────────────────────────┘
```

## Banner Behavior

- Banners appear at the top of the workspace view (above chat)
- Banners auto-dismiss when the condition clears (SSE workspace.phase event triggers re-fetch of status)
- Multiple banners can stack (e.g., SafeMode + DiskPressure)
- SafeMode banner takes priority (shown first/most prominently)

## Acceptance Criteria

1. `WorkspaceStatus` TypeScript type includes `conditions` array
2. Red banner shown for SafeMode condition with Retry/Download/Terminal buttons
3. Yellow banner shown for DiskPressure with Terminal button
4. Yellow banner shown for AgentDegraded with Manage Credentials link
5. Banners auto-dismiss when condition clears (via SSE-driven status re-fetch)
6. Multiple banners can coexist
7. Buttons trigger correct API calls
8. Banners are accessible (ARIA roles, keyboard navigable)
