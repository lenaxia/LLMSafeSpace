# US-11.10: Password Change — Re-wrap Org DEK

**Epic:** 11 (Organizations)
**Depends On:** US-11.2 (OrgKeyService.RewrapOrgDEKForAdmin)
**Estimated Effort:** ~2 hours

---

## Goal

When an org admin changes their password, their `org_key_members` rows must be re-wrapped with the new KEK. Without this, the next login attempt will fail to unwrap the org DEK (stale wrap) and the admin will need org DEK rotation to recover.

---

## Files

- `api/internal/handlers/secrets.go` — extend `RotateKeyHandler.ChangePassword`
- `api/internal/handlers/secrets_test.go` — extend

**Note:** The `ChangePassword` logic is NOT in `auth.go`. It is in `RotateKeyHandler` in `api/internal/handlers/secrets.go` (around line 782). `auth.Service` has no `ChangePassword` method. The `orgKeyService` dependency must be added to `RotateKeyHandler`, not `auth.Service`.

---

## Design

### `RotateKeyHandler.ChangePassword` extension

Current flow in `handlers/secrets.go` (around line 782): authenticate → verify old password via `keyService.ChangePassword(ctx, userID, sessionID, oldPassword, newPassword)` — this internally re-wraps the user DEK and re-wraps API key DEKs.

After the existing `keyService.ChangePassword` call succeeds, add:

```go
// Re-wrap org DEKs for all orgs where this user is an admin.
// Uses the same batch approach as login: one GetOrgKeyMembersForUser query.
// Non-fatal per org: if re-wrap fails, the org_key_members row becomes stale.
// Stale wrap is detected at next login (UnlockOrgDEK fails with ErrOrgKeyStale)
// and the admin is directed to trigger POST /orgs/:id/rotate-key.
//
// CACHE ORDERING NOTE: ChangePassword (via KeyService.ChangePassword internally)
// evicts the user's session-specific DEK cache key ("sessionID"), but does NOT
// evict the org DEK cache key ("org:<orgID>"). RewrapOrgDEKForAdmin reads from
// "org:<orgID>" — which is still present. This ordering must be preserved: never
// evict org DEK cache keys in the ChangePassword flow.
if h.orgKeyService != nil {
    // Use OrgKeyService directly (not via KeyServiceInterface) — rotation is
    // a handler-level concern, not an auth-service concern.
    if err := h.orgKeyService.RewrapAllOrgDEKsForAdmin(ctx, userID, newPassword); err != nil {
        h.logger.Warn("ChangePassword: org DEK re-wrap failed; admin should run org DEK rotation",
            "userID", userID, "error", err.Error())
        // Non-fatal: ChangePassword already succeeded. Log and return success.
    }
}
```

**`RewrapAllOrgDEKsForAdmin`** is a new method on `OrgKeyService` (not a new `KeyServiceInterface` method — this is handler-level logic, not auth-service logic):

```go
// RewrapAllOrgDEKsForAdmin re-wraps the org DEK for every org this user admins.
// Uses GetOrgKeyMembersForUser (batch query) then calls RewrapOrgDEKForAdmin per org.
// Non-fatal per org: ErrOrgDEKUnavailable or RewrapOrgDEKForAdmin errors are logged
// as Warn and skipped. Returns nil even if all orgs fail — ChangePassword has already
// succeeded; the caller should not receive a failure.
func (s *OrgKeyService) RewrapAllOrgDEKsForAdmin(ctx context.Context, userID string, newPassword []byte) error
```

**`orgKeyService` dependency on `RotateKeyHandler`:**

```go
// api/internal/handlers/secrets.go
type RotateKeyHandler struct {
    // ... existing fields ...
    orgKeyService *secrets.OrgKeyService  // optional; nil disables org re-wrap
}

func (h *RotateKeyHandler) SetOrgKeyService(svc *secrets.OrgKeyService) {
    h.orgKeyService = svc
}
```

Wired in `app.go`:
```go
rotateKeyHandler.SetOrgKeyService(orgKeyService)
```

**Cache ordering invariant (documented here per F-10):**

`KeyService.ChangePassword` evicts `"sessionID"` (the user's personal DEK cache key) BEFORE updating the DB (see `key_service.go:304-309` — pre-commit eviction for ordering correctness). It does NOT evict `"org:<orgID>"`. `RewrapOrgDEKForAdmin` reads from `"org:<orgID>"` — a different Redis key that is still present.

This ordering dependency must be documented and protected:
- A comment in `key_service.go:ChangePassword` must state: "intentionally does NOT evict org DEK cache keys (`org:*`); see US-11.10 for the ordering dependency"
- A test `TestChangePassword_OrgDEKCacheNotEvicted` must verify that `"org:<orgID>"` is still in cache after `ChangePassword` returns

---

## Acceptance Criteria

- Admin changes password: `org_key_members` rows re-wrapped for all admin orgs via `RotateKeyHandler`, not `auth.Service`
- After password change and re-wrap: next login with new password succeeds in unlocking org DEK
- Admin changes password with expired org DEK session (org DEK not in cache): Warn logged; password change still succeeds; admin directed to run rotation
- Non-admin user changes password: `GetOrgKeyMembersForUser` returns empty; no org operations attempted
- `orgKeyService == nil` on `RotateKeyHandler`: org re-wrap block skipped entirely (backward compatible for pre-US-11 deployments)
- `"org:<orgID>"` Redis key is still present after `ChangePassword` returns (not evicted)
