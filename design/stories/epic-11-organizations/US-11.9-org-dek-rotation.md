# US-11.9: Org DEK Rotation

**Epic:** 11 (Organizations)
**Depends On:** US-11.2 (OrgKeyService), US-11.6 (org credentials exist)
**Estimated Effort:** ~3 hours

---

## Goal

Allow an org admin to rotate the org DEK. This generates a new DEK, re-encrypts all org-owned `provider_credentials` rows, re-wraps the new DEK for every current admin, and caches the new DEK for the session. Use case: removing a suspected-compromised admin (removal alone is insufficient; the removed admin had the old DEK in their session cache).

---

## Files

- `pkg/secrets/org_key_service.go` — implement `RotateOrgDEK`
- `api/internal/handlers/orgs.go` — add `POST /api/v1/orgs/:id/rotate-key`
- `api/internal/handlers/orgs_test.go` — extend

---

## Route

```
POST /api/v1/orgs/:id/rotate-key
```

Request body:
```json
{ "password": "admin-plaintext-password" }
```

The password is required for two reasons:
1. To verify the admin is authorised (not just a token replay)
2. To re-derive their KEK for re-wrapping (though the org DEK must already be in cache — if not, the admin cannot proceed)

---

## Design

### `OrgKeyService.RotateOrgDEK`

The correct signature (as defined in US-11.2) uses narrow interfaces and includes the required password and adminUserID parameters:

```go
// RotateOrgDEK generates a new org DEK and re-wraps it for the rotating admin.
// All other admins have their org_key_members rows deleted and pending_key_wrap
// set to true — they must call accept-key to re-wrap with the new DEK.
//
// The current org DEK must be in cache (ErrOrgDEKUnavailable if not).
// adminUserID identifies the rotating admin; adminPassword derives their KEK.
//
// After rotation, the new org DEK is cached under OrgCacheKey(orgID) with a 7-day TTL,
// NOT the session TTL. This is intentional: other admins need to call accept-key
// AFTER the rotating admin finishes, which may be hours or days later. Using the
// session TTL (24h) risks a deadlock where the rotating admin logs out before all
// other admins complete the handshake. With a 7-day TTL, the new org DEK remains
// available in Redis for accept-key calls even after the rotating admin's session expires.
//
// If the rotating admin is also removing themselves from the org (e.g. offboarding),
// they should complete all other admin re-wraps first, or ensure at least one other
// admin completes accept-key before the rotating admin is removed.
func (s *OrgKeyService) RotateOrgDEK(
    ctx context.Context,
    orgID, adminUserID string,
    adminPassword []byte,
    reEncryptor OrgCredentialReEncryptor,
    orgMemberUpdater OrgMemberPendingUpdater,
) error
```

Implementation:

1. `currentOrgDEK, err := cache.GetDEK(ctx, OrgCacheKey(orgID))` → `ErrOrgDEKUnavailable`
2. `newOrgDEK, err := GenerateDEK()` — two-return form, handle error; `defer zeroBytes(newOrgDEK)`
3. `adminSalt = store.GetUserSalt(ctx, adminUserID)`
4. `adminKEK = DeriveKEK(adminPassword, adminSalt, orgKEKInfo)`; `defer zeroBytes(adminKEK)`
5. `newWrappedDEK = WrapDEK(adminKEK, newOrgDEK)`
6. `newKeyVersion = 1 + (max key_version across ListOrgKeyMembers)`
7. In a single transaction:
   a. `reEncryptor.ReEncryptOrgCredentials(ctx, orgID, currentOrgDEK, newOrgDEK)` — re-encrypt all org `provider_credentials`
   b. Delete ALL `org_key_members` rows for this org (`store.DeleteAllOrgKeyMembers(ctx, orgID)`)
   c. Insert new `org_key_members` row for rotating admin: `{orgID, adminUserID, newWrappedDEK, newKeyVersion}`
   d. `orgMemberUpdater.SetPendingKeyWrapForOtherAdmins(ctx, orgID, adminUserID)` — sets `pending_key_wrap=true` for all other admins
8. `cache.CacheDEK(ctx, OrgCacheKey(orgID), newOrgDEK, 7*24*time.Hour)` — 7-day TTL (not session TTL)
9. Zero `currentOrgDEK`, `adminKEK`

**Note on step 7 transaction:** Steps 7a–7d must all be in a single `pgx.Tx`. `ReEncryptOrgCredentials` accepts an optional tx parameter for this purpose. If any step fails, the entire transaction rolls back: old `org_key_members` rows are restored (via FK cascade the rows are deleted, but since PostgreSQL FK cascades run within the same transaction, a rollback restores them), credentials stay at old DEK, pending_key_wrap flags unchanged.

### `OrgCredentialReEncryptor` interface

```go
// OrgCredentialReEncryptor is a narrow interface for re-encrypting org credentials
// within a transaction. The pgx.Tx parameter is required: RotateOrgDEK needs steps
// 7a–7d to be atomic, and re-encryption must participate in the same transaction
// as the key member delete/insert.
//
// Implemented by PgSecretStore. pgx.Tx is from pgx/v5.
type OrgCredentialReEncryptor interface {
    ReEncryptOrgCredentials(ctx context.Context, tx pgx.Tx, orgID string, oldDEK, newDEK []byte) (int, error)
}
```

**How `RotateOrgDEK` opens the transaction:** `OrgKeyService` holds a `pgxpool.Pool` reference (passed to `PgOrgKeyStore` and accessible via the store). `RotateOrgDEK` opens a `pgx.Tx` via `pool.Begin(ctx)`, passes it to `ReEncryptOrgCredentials`, and uses it for the `DeleteAllOrgKeyMembers` and `UpsertOrgKeyMember` calls. All four operations share the same `pgx.Tx`. On success: commit. On any error: rollback.

Alternatively, `OrgKeyStore` can expose a `BeginTx(ctx) (pgx.Tx, error)` method so `OrgKeyService` never directly holds the pool. This is cleaner:

```go
// Add to OrgKeyStore:
BeginTx(ctx context.Context) (pgx.Tx, error)
```

Then `RotateOrgDEK` calls `store.BeginTx(ctx)` to get a tx, passes it to all store operations and to `reEncryptor.ReEncryptOrgCredentials`. This keeps `OrgKeyService` free of direct pool references.

### `POST /api/v1/orgs/:id/rotate-key` handler

1. `OrgAdminGuard`
2. Bind `{ "password": string }`
3. `orgKeySvc.RotateOrgDEK(ctx, orgID, callerUserID, []byte(req.Password), reEncryptor, orgMemberUpdater)`
   - `reEncryptor` is `pgSecretStore` (implements `OrgCredentialReEncryptor`)
   - `orgMemberUpdater` is `pgOrgStore` (implements `OrgMemberPendingUpdater` via `SetPendingKeyWrapForOtherAdmins`)
4. Return 200 `{ "message": "Org DEK rotated. Other admins must log in and call /accept-key to regain access to org credentials." }`

## Acceptance Criteria

- After rotation: rotating admin can decrypt org credentials (new DEK in cache under `"org:<orgID>"`)
- After rotation: new org DEK is cached with 7-day TTL (not session TTL)
- After rotation: other admins have `pending_key_wrap=true`; their `org_key_members` rows deleted
- After rotation: other admins can call `POST /accept-key` even after the rotating admin's session expires (7-day TTL ensures org DEK available)
- After rotation: compromised/removed admin has no `org_memberships` row → cannot call `accept-key`
- All org `provider_credentials` re-encrypted with new DEK atomically; old DEK cannot decrypt after rotation
- Transaction is atomic: re-encryption error → all changes rolled back; old state intact, old credentials still decryptable
- Old DEK zeroed from memory after rotation
- Password field not logged or returned in any response
