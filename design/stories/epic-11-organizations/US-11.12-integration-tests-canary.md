# US-11.12: Integration Tests + Canary

**Epic:** 11 (Organizations)
**Depends On:** All prior stories complete

## Goal

End-to-end integration tests that exercise the full org lifecycle (create → invite admin → accept key → create org credential → create org workspace → credential injected into pod → rotate DEK → remove admin). Plus a canary scenario that can run against a live cluster.

## Integration Test Scenarios

### `TestOrgLifecycle_FullFlow`

Uses real PostgreSQL + `miniredis` for cache. No K8s required.

```
1. Create user A (founding admin), user B (future admin), user C (member)
2. User A logs in → A's user DEK unlocked
3. User A creates org "test-org":
   - POST /api/v1/orgs
   - Verify org row in DB
   - Verify org_memberships row for A with role='admin', pending_key_wrap=false
   - Verify org_key_members row for A (wrapped DEK)
   - Verify "org:<orgID>" in cache
4. User A invites B as admin:
   - POST /api/v1/orgs/:id/members {userId: B, role: "admin"}
   - Verify org_memberships row for B with pending_key_wrap=true
   - Verify NO org_key_members row for B yet
5. User B logs in → B's user DEK unlocked; ListAdminOrgIDs returns empty (pending=true excluded)
6. User B calls accept-key:
   - POST /api/v1/orgs/:id/accept-key {password: B's password}
   - Verify org_key_members row created for B
   - Verify org_memberships.pending_key_wrap=false for B
7. User B logs out and logs in again:
   - UnlockOrgDEK called for test-org
   - "org:<orgID>" in cache with B's session TTL
8. User A creates org credential (anthropic):
   - POST /api/v1/orgs/:id/credentials {name: "shared-anthropic", provider: "anthropic", apiKey: "sk-test"}
   - Verify provider_credentials row: owner_type='org', owner_id=<orgID>
   - Verify encrypted with org DEK (decrypt with A's cached org DEK → correct)
9. User A adds auto-apply rule:
   - POST /api/v1/orgs/:id/credentials/:credID/auto-apply
   - Verify credential_auto_apply row: target_type='org', target_id=<orgID>
10. User A invites C as member:
    - POST /api/v1/orgs/:id/members {userId: C, role: "member"}
    - Verify NO org_key_members row for C (members don't get key access)
11. Create org workspace under user C:
    - POST /api/v1/workspaces {orgId: <orgID>}
    - Verify workspaces.org_id set
    - Verify SeedWorkspaceCredentials ran: workspace_credential_bindings has org credential row
12. PrepareSecretsForInjection for the org workspace (simulated inject):
    - Call PrepareSecretsForInjection with A's session → returns org credential in secrets JSON
    - Call PrepareSecretsForInjection with C's session (no org DEK) → org credential skipped; falls back to admin platform credential
    - Call PrepareSecretsForInjection with no session (controller-initiated restart) → org credential skipped; admin credential used
13. User A changes password:
    - PUT /api/v1/account/password {oldPassword, newPassword}
    - Verify org_key_members row for A updated (new wrapped DEK)
    - A logs in with new password → UnlockOrgDEK succeeds
14. Rotate org DEK:
    - POST /api/v1/orgs/:id/rotate-key {password: A's new password}
    - Verify org credential re-encrypted (old DEK cannot decrypt)
    - Verify B's org_key_members row deleted; B's membership pending_key_wrap=true
    - B calls accept-key → B can again access org DEK
15. Remove user B:
    - DELETE /api/v1/orgs/:id/members/B
    - Verify org_memberships row for B deleted
    - Verify org_key_members row for B deleted
    - B logs in → ListAdminOrgIDs returns empty → no org DEK cached for B
```

### `TestOrgIsolation_CrossOrgAccess`

```
Two orgs, two admins, two org workspaces.
- Admin of Org A cannot list Org B's workspaces (403)
- Admin of Org A cannot read Org B's credentials (403)
- Member of Org A can list Org A's workspaces but not delete them
```

### `TestOrgLastAdminProtection`

```
- Org with one admin
- Admin attempts self-removal → 409
- Admin attempts to rotate key and then there are zero non-pending admins → rotation still works (they re-wrapped for themselves)
```

### `TestSeedWorkspaceCredentials_OrgVsPersonal`

```
- Org auto-apply rule for org credential
- Create personal workspace (no orgID) → org rule NOT applied
- Create org workspace (orgID set) → org rule applied
- Create workspace for different org → different org's rules NOT applied
```

## Canary Scenario

**File:** `tests/canary/org_credential_canary_test.go`

Runs against live cluster (requires admin credentials and a test org).

```
Scenario: Org credential injection end-to-end

Prerequisites:
  - A test org exists (created during cluster setup or by the test itself)
  - The test org has an anthropic credential with a valid API key
  - The test org has an auto-apply rule for all org workspaces

Steps:
  1. Create a workspace under the test org
  2. Activate the workspace; wait for Active phase
  3. Send a message to the workspace: "What is 2+2? Reply with just the number."
  4. Verify response is "4" (confirms org credential was injected and provider is reachable)
  5. Delete the workspace
```
