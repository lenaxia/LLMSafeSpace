# US-11.3: Login — Unlock Org DEKs for Admin Members

**Epic:** 11 (Organizations)
**Depends On:** US-11.2 (OrgKeyService), US-11.4 (org_memberships DB queries)
**Estimated Effort:** ~3 hours

---

## Goal

Extend the login flow so that when an org admin logs in, the org DEK for each of their orgs is unlocked and cached alongside their personal DEK. Non-admins and users with no org memberships are unaffected.

---

## Files

- `api/internal/services/auth/auth.go` — extend `Login` and `KeyServiceInterface`
- `api/internal/services/auth/auth_test.go` — new tests

## Design

### `KeyServiceInterface` extension

The interface in `auth/auth.go:37-43` must be extended. Following the existing pattern (no breaking changes to existing methods):

```go
type KeyServiceInterface interface {
    // Existing methods — unchanged
    InitializeUserKeys(ctx context.Context, userID string, password []byte) (recoveryKeyHex string, err error)
    UnlockDEK(ctx context.Context, userID string, password []byte, sessionID string, ttl time.Duration) error
    HasKeys(ctx context.Context, userID string) (bool, error)
    GetDEK(ctx context.Context, sessionID string) ([]byte, error)
    CacheDEK(ctx context.Context, sessionID string, dek []byte, ttl time.Duration) error

    // New — batch org DEK unlock. Fetches all org key records for userID in a single
    // DB query and unlocks each. userSalt is from user_keys (already in memory during
    // login). Non-fatal: returns nil even if individual orgs fail.
    UnlockAllOrgDEKs(ctx context.Context, userID string, userSalt []byte, password []byte, ttl time.Duration) error
}
```

**`UnlockAllOrgDEKs` is added to the interface, not `OrgKeyService` directly.** The auth service calls `s.keyService.UnlockAllOrgDEKs(...)`. The concrete `KeyService` struct does not implement org keys; instead, a thin adapter `OrgAwareKeyService` wraps both `KeyService` and `OrgKeyService`:

```go
// pkg/secrets/org_aware_key_service.go

// OrgAwareKeyService wraps KeyService (user DEKs) and OrgKeyService (org DEKs)
// to satisfy KeyServiceInterface. This is the concrete type wired in app.go.
type OrgAwareKeyService struct {
    *KeyService
    org *OrgKeyService
}

func NewOrgAwareKeyService(ks *KeyService, org *OrgKeyService) *OrgAwareKeyService

func (s *OrgAwareKeyService) UnlockAllOrgDEKs(ctx context.Context, userID string, userSalt []byte, password []byte, ttl time.Duration) error {
    return s.org.UnlockAllOrgDEKs(ctx, userID, userSalt, password, ttl)
}
```

This avoids modifying `KeyService` itself and keeps the interface clean.

**`fakeKeyService` in `auth_test.go`:** The existing hand-rolled fake at `auth_test.go:629-684` implements the current 5-method interface. Adding `UnlockAllOrgDEKs` to `KeyServiceInterface` breaks compilation of every test using `fakeKeyService`. The story must add the following method to `fakeKeyService`:

```go
func (f *fakeKeyService) UnlockAllOrgDEKs(ctx context.Context, userID string, userSalt []byte, password []byte, ttl time.Duration) error {
    f.unlockAllOrgDEKsCalls = append(f.unlockAllOrgDEKsCalls, unlockAllOrgDEKsCall{userID: userID, ttl: ttl})
    return f.unlockAllOrgDEKsErr
}
```

Also add `unlockAllOrgDEKsErr error` and `unlockAllOrgDEKsCalls []unlockAllOrgDEKsCall` fields to `fakeKeyService`. **The mock in `mocks/` must also add `UnlockAllOrgDEKs`.**

### `Login` extension

The org DEK unlock must be efficient: a single batch DB query, not one per org. The user's `user_keys.salt` is already retrieved during `UnlockDEK` — thread it through rather than re-querying.

Current login flow (`auth.go:739-753`) already calls `UnlockDEK` which internally reads `user_keys.salt`. The `KeyService.UnlockDEK` method reads the record at line 178. To avoid a second query, `UnlockDEK` should return the salt (or the full record) to the caller. However, changing `UnlockDEK`'s return type would break the interface.

**Accepted approach:** `UnlockAllOrgDEKs` calls `store.GetOrgKeyMembersForUser` (the batch query defined in US-11.2) which does NOT need `user_keys.salt` for its DB query. The salt is fetched once inside `OrgKeyService` using `store.GetUserSalt(ctx, userID)` — a single additional query for the user's salt, shared across all orgs. This is at most 2 DB queries total (one for org records, one for the salt) regardless of org count.

After the existing `UnlockDEK` block, add:

```go
// Unlock org DEKs for any orgs where this user is an admin.
// Uses a single batch query for all org key records + one query for the user salt.
// Non-fatal: org DEK failures never block login.
if s.keyService != nil {
    if err := s.keyService.UnlockAllOrgDEKs(ctx, user.ID, nil, []byte(req.Password), tokenDur); err != nil {
        s.logger.Warn("Login: failed to unlock org DEKs", "userID", user.ID, "error", err.Error())
        // Non-fatal — login proceeds
    }
}
```

Note: `userSalt` is passed as `nil` here; `UnlockAllOrgDEKs` fetches it internally via `GetUserSalt`. This keeps the login call site simple.

### Removed: `orgMemberStore` dependency on auth service

The previous design had the auth service call `ListAdminOrgIDs` to get org IDs, then loop calling `UnlockOrgDEK` per org. This is replaced by `UnlockAllOrgDEKs` which handles the batch internally in `OrgKeyService`. The auth service no longer needs `OrgMemberStore` or `SetOrgMemberStore`. This simplifies the auth service's dependency surface.

### Wiring in `app.go`

```go
// After existing keyService wiring:
orgKeyService := secrets.NewOrgKeyService(pgOrgKeyStore, dekCache)
orgAwareKS := secrets.NewOrgAwareKeyService(keyService, orgKeyService)
authSvc.SetKeyService(orgAwareKS)   // replaces keyService with org-aware wrapper
// No SetOrgMemberStore needed — UnlockAllOrgDEKs handles this internally
```

## Acceptance Criteria

- Login as non-admin user: no org DEK operations, behaviour identical to today
- Login as org admin (pending_key_wrap=false): org DEK cached under `"org:<orgID>"` for each admin org
- Login as org admin (pending_key_wrap=true, key handshake not complete): `GetOrgKeyMembersForUser` returns no record for that org; no unlock attempted; no error
- Login as org admin with stale wrap: Warn logged, org DEK not cached, login still succeeds
- `GetOrgKeyMembersForUser` DB error: Warn logged, org unlock skipped, login still succeeds (non-fatal)
- `KeyServiceInterface` mock AND `fakeKeyService` in `auth_test.go` both compile with `UnlockAllOrgDEKs` method added
- Login causes at most 2 additional DB queries regardless of how many orgs the user admins: one `GetOrgKeyMembersForUser`, one `GetUserSalt`
