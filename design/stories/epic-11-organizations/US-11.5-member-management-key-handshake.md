# US-11.5: Org Member Management + Key Handshake

**Epic:** 11 (Organizations)
**Depends On:** US-11.4 (org CRUD + OrgStore), US-11.2 (OrgKeyService)
**Estimated Effort:** ~4 hours

---

## Goal

Implement member listing, adding, and removing. Implement the two-step admin key handshake: Step 1 (invite) creates the membership row; Step 2 (accept-key) wraps the org DEK for the new admin using their password.

## Routes

```
GET    /api/v1/orgs/:id/members              ‚Äî list members (OrgMemberGuard)
POST   /api/v1/orgs/:id/members              ‚Äî add member/admin (OrgAdminGuard)
PUT    /api/v1/orgs/:id/members/:userID      ‚Äî change member role (OrgAdminGuard)
DELETE /api/v1/orgs/:id/members/:userID      ‚Äî remove member (OrgAdminGuard)
POST   /api/v1/orgs/:id/accept-key           ‚Äî complete admin key handshake (authMiddleware only ‚Äî any member invited as admin)
```

## Design

### `POST /api/v1/orgs/:id/members` ‚Äî Add Member

Request body:
```json
{ "userId": "user-xyz", "role": "admin" | "member" }
```

1. `OrgAdminGuard` ‚Äî caller must be org admin
2. Validate `userId` exists in `users` table
3. Check user not already a member ‚Üí 409 if already present
4. If `role == "member"`:
   - `orgStore.AddOrgMember(ctx, orgID, req.UserID, OrgRoleMember, pendingKeyWrap=false)`
   - Return 201; no key wrap needed
5. If `role == "admin"`:
   - `orgStore.AddOrgMember(ctx, orgID, req.UserID, OrgRoleAdmin, pendingKeyWrap=true)`
   - Return 201 with `{ "pendingKeyWrap": true, "message": "Admin must call POST /orgs/:id/accept-key to complete key setup" }`
   - Do NOT attempt to wrap org DEK here ‚Äî the invited user's password is not available

### `POST /api/v1/orgs/:id/accept-key` ‚Äî Complete Admin Key Handshake

Request body:
```json
{ "password": "user-plaintext-password" }
```

1. `authMiddleware` ‚Äî caller must be authenticated (their own JWT)
2. Get caller's `orgMembership` for this org ‚Üí 403 if not a member
3. Check `orgMembership.role == "admin"` ‚Üí 403 if not admin
4. Check `orgMembership.pending_key_wrap == false` ‚Üí 409 "key wrap already complete ‚Äî no action needed"
5. `orgKeySvc.WrapOrgDEKForNewAdmin(ctx, orgID, callerUserID, []byte(req.Password))`
   - Returns `ErrOrgDEKUnavailable` ‚Üí 409 `{ "error": "org DEK not currently available ‚Äî an org admin must be logged in to complete key setup" }`
   - Other error ‚Üí 500
6. `orgStore.SetPendingKeyWrap(ctx, orgID, callerUserID, false)` ‚Äî clear the flag
7. Return 200 `{ "message": "Key wrap complete. Org credentials will be available on your next login." }`

**Note on why 200 not 204:** The response includes a guidance message. Clients should display this to the user.

**Note on immediate DEK caching for current session:** After step 5, the new admin's org DEK is not yet cached for their current session (the `WrapOrgDEKForNewAdmin` writes to DB but does not cache for the caller ‚Äî they used the cached org DEK from another admin's session to do the wrap). The new admin will get the org DEK cached on their NEXT login. This is documented. A follow-up improvement could cache it immediately, but it requires the new admin's DEK to be in the calling session ‚Äî which it isn't (this endpoint is called by the invited admin using their own JWT, not the inviting admin's JWT). Design decision: correct behavior > clever optimisation.

### `PUT /api/v1/orgs/:id/members/:userID` ‚Äî Change Role

Request body: `{ "role": "admin" | "member" }`

**Promotion (member ‚Üí admin):**
1. `OrgAdminGuard`
2. Verify target user is a current member ‚Üí 404 if not
3. Verify target user is currently `member` ‚Üí 409 if already admin
4. `orgStore.UpdateOrgMemberRole(ctx, orgID, targetUserID, OrgRoleAdmin)` ‚Äî sets `role='admin'`, `pending_key_wrap=true`
5. Return 200 with guidance: `{ "pendingKeyWrap": true, "message": "Admin must call POST /orgs/:id/accept-key to complete key setup" }`

**Demotion (admin ‚Üí member):**
1. `OrgAdminGuard`
2. Verify target user is a current admin ‚Üí 404/409 as appropriate
3. Cannot demote yourself ‚Üí 409 "org admins cannot demote themselves"
4. Check `CountOrgAdmins(orgID)` ‚Üí 409 "cannot demote the last org admin" if count == 1
5. In a transaction:
   a. `orgStore.UpdateOrgMemberRole(ctx, orgID, targetUserID, OrgRoleMember)` ‚Äî sets `role='member'`, `pending_key_wrap=false`
   b. `orgStore.DeleteOrgKeyMember(ctx, orgID, targetUserID)` ‚Äî removes wrapped DEK; CASCADE would not fire here since we're only changing role, not removing membership. Must be explicit.
   ‚Äî Wait: the composite FK `org_key_members ‚Üí org_memberships` cascades on DELETE of the membership row, NOT on UPDATE. Role change does NOT cascade. So step 5b is explicitly needed.
6. Return 200

### `DELETE /api/v1/orgs/:id/members/:userID` ‚Äî Remove Member

1. `OrgAdminGuard` ‚Äî caller must be org admin
2. Cannot remove yourself if you are the last admin: `CountOrgAdmins` ‚Üí if 1 and userID == callerID ‚Üí 409 `{ "error": "cannot remove the last org admin" }`
3. Cannot remove yourself if you are admin removing yourself: 409 `{ "error": "org admins cannot remove themselves; transfer admin role first" }` ‚Äî prevents accidental lockout
4. `orgStore.RemoveOrgMember(ctx, orgID, req.UserID)` ‚Äî cascades to `org_key_members`
5. Return 204

**`org_key_members` cascade:** The `org_key_members` table has `ON DELETE CASCADE` from `org_memberships` FK on `user_id`. Removing the membership row automatically removes the wrapped DEK row. Confirmed: `ON DELETE CASCADE` is on `org_key_members(user_id)` FK, not `org_memberships`.

Wait ‚Äî these are separate tables without a direct FK between them. The cascade must be explicit:

```go
// RemoveOrgMember must delete BOTH:
// 1. org_memberships row
// 2. org_key_members row (if exists)
// Both in a transaction.
```

The `PgOrgStore.RemoveOrgMember` implementation wraps both DELETEs in a transaction.

### `GET /api/v1/orgs/:id/members` ‚Äî List Members

Returns all `OrgMember` items for the org. Includes `pendingKeyWrap` field so the inviting admin can see who has not yet completed setup. No sensitive data exposed.

---

## Acceptance Criteria

- `POST /members` with `role='member'`: membership created, `pending_key_wrap=false`, no key wrap
- `POST /members` with `role='admin'`: membership created, `pending_key_wrap=true`, 201 response includes guidance
- `POST /accept-key` with org DEK in cache: `org_key_members` row created, `pending_key_wrap` cleared
- `POST /accept-key` with org DEK NOT in cache: 409 with specific message
- `POST /accept-key` when already wrapped (pending_key_wrap=false): 409 "already complete"
- `POST /accept-key` as non-member: 403
- `POST /accept-key` as member (not admin): 403
- `DELETE /members/:id` as last admin: 409
- `DELETE /members/:id` as self-removing admin: 409
- `DELETE /members/:id` as non-admin: 403
- Remove admin: `org_key_members` row also deleted (transactional)
- `GET /members` shows `pendingKeyWrap: true` for uninitiated admins
- Password never logged or returned in any response
