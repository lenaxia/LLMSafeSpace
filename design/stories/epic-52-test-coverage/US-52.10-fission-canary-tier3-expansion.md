# US-52.10: Fission Canary Tier 3 Expansion

**Epic:** 52 — Comprehensive Test Coverage
**Layer:** Canary (production)
**Effort:** Large (3–4d)
**Depends on:** —
**Status:** Proposed

---

## Problem

The canary layer (`sdks/canary/`) is the strongest part of the test
pyramid, but it has structural gaps. The current two-tier model
(`TESTPLAN.md` §2) covers:

- **Tier 1 (shallow, 1 min)** — request-shape conformance: is the API up,
  do endpoints return the right status, do auth paths work.
- **Tier 2 (deep, 5–15 min)** — workflow conformance: does a workspace
  reach Active, does an LLM message round-trip, does suspend/resume
  preserve history.

What is missing:

1. **Tier 3 — failure-injection / chaos.** No canary deliberately breaks
   something and verifies the system self-heals. We find out the controller
   doesn't recover a dead pod only when a real pod dies in prod.
2. **Controller-plane canaries.** Every existing canary exercises the SDK
   → API path. No canary exercises the controller directly: CRD apply,
   webhook rejection, status observation, finalizer cleanup.
3. **MCP server canaries.** Only 8 scenarios in `sdks/canary/mcp/`, all
   shallow; no deep canaries for the MCP-server code paths that the SDK
   canaries don't hit (the MCP server's internal `prompt_async` + SSE
   plumbing).
4. **Inference-relay worker canaries.** Zero. The worker is the LLM cost
   surface; a regression ships silently.
5. **Multi-SDK drift detection.** 40 Go scenarios, 39 Python, 39
   TypeScript — `s-cred-provider-precedence` is Go-only. Drift between SDK
   implementations is invisible.
6. **Canary-result aggregation.** Each Fission function emits a `Result{}`
   JSON; nothing aggregates them. A regression that fails 6 of 40
   scenarios shows up as 6 individual pages, not as "the workspace
   lifecycle is broken."

## Scope

**In:**
- Tier 3 canaries: 8 new scenarios that inject failure and assert
  self-healing.
- Controller-plane canaries: 6 new scenarios that exercise CRDs + webhooks
  directly via `kubectl` (executed from inside the Fission function).
- MCP server deep canaries: 4 new scenarios.
- Inference-relay worker canaries: 4 new scenarios (HTTP probes against
  the worker URL).
- Close the SDK drift: port `s-cred-provider-precedence` to Python + TS.
- Result aggregation: a new `sdks/canary/aggregator/` that polls Fission
  function URLs and emits Prometheus metrics (`canary_scenario_status`,
  `canary_scenario_duration_seconds`, `canary_scenario_drift`).
- Update `TESTPLAN.md` (v4.0) and `canary-functions.yaml` together.

**Out:**
- Real chaos engineering at the infrastructure level (network partition,
  node drain) — that requires admin cluster privileges the canary runtime
  doesn't have. Tier 3 here is *application-level* failure injection.
- Synthetic long-lived traffic — that is US-52.11 (different shape, not a
  discrete pass/fail assertion).
- Load testing — still `hack/benchmark-*.sh`.
- Rewriting the existing 40 scenarios — they are extended (Python + TS
  ports), not refactored.

## Definitions (normative for this story)

These definitions extend TESTPLAN.md §2 and are cited by every new
scenario. Tier 3 is the new tier.

### Tier 1 — Shallow (existing)

**What it tests:** Request-shape conformance. Is the API up? Do
endpoints return documented status codes and field shapes?

**Failure signature:** "The API is broken" — wrong status, missing field,
serialization regression, auth bypass.

**Schedule:** 1 minute. **Target duration:** < 30s.

**Alert policy:** Alert on 1st failure; page on 3rd consecutive.

### Tier 2 — Deep (existing)

**What it tests:** Workflow conformance. Does a workspace reach Active?
Does an LLM round-trip work? Does suspend/resume preserve history?

**Failure signature:** "The product is broken" — controller regression,
agent startup failure, credential injection broken, LLM integration
broken.

**Schedule:** 5–15 minutes. **Target duration:** 3–10 min.

**Alert policy:** Alert on 2nd consecutive failure; page on 4th.

### Tier 3 — Failure injection / chaos (new)

**What it tests:** Self-healing. Deliberately break something — kill a
pod, expire a credential, fill a PVC, saturate a connection pool — and
assert the system recovers without human intervention within a defined
SLA.

**Failure signature:** "Self-healing is broken" — a transient failure
became a permanent outage. The most expensive class of regression because
it surfaces only under real load.

**Schedule:** 15–30 minutes (failure injection is disruptive; running it
every minute would create noise). **Target duration:** 5–15 min including
recovery wait.

**Alert policy:** Alert on 1st failure (these scenarios are high-signal);
page on 2nd consecutive. Each Tier 3 scenario has a documented
`Recovery SLA` it asserts; a failure means "recovery took longer than
SLA" or "recovery did not happen at all."

**Quarantine:** Tier 3 scenarios run against a dedicated canary namespace
(`llmsafespaces-canary`) so injected failures don't affect real users.
The canary service account is scoped to that namespace.

**Shared-infra isolation (Redis + Postgres):** K8s namespace isolation
does **not** extend to the shared Redis and Postgres instances.
T3-REDIS-FLUSH and T3-DB-POOL-SATURATE must isolate at the application
layer:

- **Redis:** all canary keys must use a `canary:` key prefix. The flush
  scenario flushes only `canary:*` keys via Redis SCAN+DEL (not FLUSHDB).
  The canary API key + JWT are separate from production keys by user
  identity (canary accounts), so rate-limit and JWT-cache keys are
  inherently isolated.
- **Postgres:** the DB pool saturation scenario opens connections against
  a dedicated `canary_pool` database user with a `max_connections=10`
  Postgres-level cap, so it cannot exhaust the production connection pool.
  Alternatively, the scenario can be gated to run only when
  `LLMSAFESPACES_CANARY_DEDICATED_DB=true` is set, skipping gracefully
  on shared-pool deployments.

### Synthetic traffic (separate — US-52.11)

**What it tests:** Long-horizon behavioural drift. Continuous user-like
sessions with reconnects, idle periods, and mid-session failures.

**Why distinct from canaries:** Canaries are discrete assertions
(pass/fail per check). Synthetic traffic is continuous measurement
(session-duration distribution, reconnect-success-rate, events-per-second)
that emits time-series metrics, not pass/fail.

## Intent-Declared Scenario Inventory

Every new scenario states intent in its `Handler` doc comment, matching
the format of existing scenarios.

### Tier 3 — Failure injection (8 scenarios)

| ID | Scenario | What breaks | Self-heal assertion | Recovery SLA |
|---|---|---|---|---|
| T3-POD-KILL | `d-t3-pod-kill` | Canarically `kubectl delete pod` on a canary workspace | Controller recreates pod; phase returns Active | ≤ 90s |
| T3-CRED-EXPIRE | `d-t3-cred-expire` | Bound credential's underlying K8s Secret is deleted | agentd detects missing cred; status reflects unavailable; reload re-injects after recreate | ≤ 60s after Secret recreated |
| T3-PVC-DETACH | `d-t3-pvc-detach` | PVC temporarily unmounted (simulate via pod restart with missing volume) | Pod restarts loop until PVC re-attached; no data loss | ≤ 120s |
| T3-REDIS-FLUSH | `d-t3-redis-flush` | Flush canary namespace keys in Redis (rate-limit + JWT cache) | No 500s to ongoing requests; rate-limit cold-starts gracefully | ≤ 30s |
| T3-DB-POOL-SATURATE | `d-t3-db-pool-saturate` | Open N=2×pool_size idle connections to saturate pool | API rejects excess with 503 (not hang) within 5s SLA; existing requests complete. **Per second-pass REWORK-10.1:** "no hang" is non-deterministic; the 5s SLA + 503 assertion is the deterministic check. A hang manifests as a test timeout that fails loudly. | Immediate (5s SLA) |
| T3-SSE-CLIENT-DROP | `d-t3-sse-client-drop` | Open 50 SSE streams then kill clients | Goroutine count returns to baseline; no leak | ≤ 60s |
| T3-AGENT-CRASH | `d-t3-agent-crash` | Send `SIGKILL` to opencode inside canary pod | agentd detects, restarts opencode, session resumes | ≤ 30s |
| T3-LLM-UPSTREAM-500 | `d-t3-llm-upstream-500` | Point canary workspace at a stub upstream returning 500 | User-facing error surfaced; no cascade; workspace stays Active | Immediate |

Each Tier 3 scenario has these check rows:

```
P1 — Failure injected (verified via API)
P2 — System observed the failure (status / log / metric)
P3 — System recovered within Recovery SLA
P4 — No side-effects on other canary namespaces
N1 — Recovery took > 2× SLA (alert-worthy)
N2 — Recovery did not happen within hard cap (page-worthy)
```

### Controller-plane canaries (6 scenarios)

These exercise the K8s control plane directly, not the SDK. The Fission
function uses the in-cluster service account (scoped to
`llmsafespaces-canary` namespace).

| ID | Scenario | What | Failure mode protected |
|---|---|---|---|
| C-CRD-APPLY | `s-c-crd-apply` | `kubectl apply` a Workspace YAML in canary ns | CRD schema regression |
| C-WEBHOOK-REJECT | `s-c-webhook-reject` | Apply invalid Workspace (storage > 1024Gi) → rejected | Webhook bypass |
| C-STATUS-OBSERVE | `s-c-status-observe` | Apply ws, poll status.phase until Active | Status observation broken |
| C-FINALIZER-CLEAN | `d-c-finalizer-clean` | Delete ws, verify PVC/Secret/NP all gone within 60s | Finalizer leak |
| C-EVENT-DELIVERED | `s-c-event-delivered` | Apply ws, watch for `WorkspaceCreated` event | Watch/event delivery broken |
| C-OWNERREF-GC | `d-c-ownerref-gc` | Delete ws, verify child objects GC'd via ownerRefs | Orphan resources |

> **Note (second-pass REWORK-10.3):** C-EVENT-DELIVERED uses a two-tier
> assertion: (a) the event appears within 60s (asserted); (b) if it
> doesn't, the scenario records `result.skipped=true` with reason
> `event-best-effort` rather than failing. Kubernetes events are
> documented best-effort; over-asserting delivery flakes. The scenario's
> value is detecting *complete* event-delivery breakage, not transient
> drops.

### MCP server deep canaries (4 scenarios)

Extends the existing 8 (S-MCP-TOOLS, S-MCP-AUTH-NEG, S-MCP-CRED,
S-MCP-INPUT-NEG, D-MCP-WORKSPACE, D-MCP-SESSION, D-MCP-PROMPT-ASYNC,
D-MCP-MODEL).

| ID | Scenario | What | Failure mode protected |
|---|---|---|---|
| D-MCP-SESSION-ABORT | `d-mcp-session-abort` | `session_message` then abort via API | MCP server doesn't handle abort |
| D-MCP-CONCURRENT | `d-mcp-concurrent` | 10 concurrent `session_message` | MCP server serialisation bug |
| D-MCP-RECONNECT | `d-mcp-reconnect` | Drop SSE mid-stream, reconnect | MCP SSE resumption broken |
| D-MCP-BODY-LIMIT | `d-mcp-body-limit` | Send body at the documented 1 MiB MCP limit; assert at-limit succeeds, over-limit returns "too large" | MCP body-size limit broken |

> **Note (second-pass REWORK-10.4):** `D-MCP-LONG-MESSAGE` (100KB) was
> renamed to `D-MCP-BODY-LIMIT` and reworked — 100KB asserted nothing
> because the actual limit is 1 MiB. The reworked scenario tests the
> boundary precisely.

### Inference-relay worker canaries (4 scenarios)

HTTP probes against the worker URL (separate from the SDK API URL).

| ID | Scenario | What | Failure mode protected |
|---|---|---|---|
| S-WORKER-HEALTH | `s-worker-health` | GET `/health` on worker. **Per second-pass REWORK-10.2:** assert body has `status: "ok"` and `version` matching deployed SHA — not just 200 (every healthy service returns 200; the regression risk is the response shape) | Worker down OR response shape drift |
| S-WORKER-AUTH | `s-worker-auth` | Request with bad secret → 403 | Auth bypass |
| D-WORKER-PROXY | `d-worker-proxy` | Full inference round-trip via worker (LLM-gated) | Worker proxy broken |
| D-WORKER-RATELIMIT | `d-worker-ratelimit` | Burst requests, verify 429 after limit | Rate-limit broken |

### Aggregator

New `sdks/canary/aggregator/` (Go binary, deployed as a Fission function
or standalone Deployment):

| Metric | Labels | What |
|---|---|---|
| `canary_scenario_status` | scenario, sdk, tier | 0 = pass, 1 = fail, 2 = skipped |
| `canary_scenario_duration_seconds` | scenario, sdk, tier | Wall-clock of last run |
| `canary_scenario_checks_total` | scenario, sdk, result | Per-check pass/fail count |
| `canary_scenario_drift` | scenario | 0 = parity across SDKs, 1 = drift |
| `canary_aggregator_scrape_total` | — | Self-health |

The aggregator polls each Fission Function URL on the schedule defined in
`TESTPLAN.md` §8, parses `Result{}`, and emits the metrics. Alert rules
defined in TESTPLAN.md alert on `canary_scenario_status{tier="1"} == 1`
for 1m, etc.

## Test Infrastructure (idioms + isolation)

- **Same scenario contract** (`Handler(w, r)` + `main()` + `Result{}`) as
  existing scenarios — three SDK implementations per scenario.
- **Quarantine namespace**: Tier 3 scenarios run in `llmsafespaces-canary`
  namespace, isolated from real users. Service account scoped via RBAC.
- **Cleanup discipline**: every scenario creates resources with
  `canary-t3-<scenario>-<timestamp>` prefix and cleans them up in
  `defer`, even on failure. TESTPLAN.md §1's self-contained rule applies.
- **LLM-gating**: scenarios that need real LLM (`T3-LLM-UPSTREAM-500`,
  `D-WORKER-PROXY`, all `D-MCP-*`) skip cleanly when
  `LLMSAFESPACES_LLM_API_KEY` is unset and the skip is visible in the
  `Result{}` JSON.
- **Aggregator isolation**: the aggregator polls, it does not mutate. It
  has no service account beyond reading Fission function URLs.

> **Open question (second-pass infra audit):** Before implementing the
