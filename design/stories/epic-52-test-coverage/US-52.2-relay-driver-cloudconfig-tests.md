# US-52.2: Relay Driver + Cloud-Config Tests

**Epic:** 52 — Comprehensive Test Coverage
**Layer:** Unit + integration
**Effort:** Medium (2–3d)
**Depends on:** —
**Status:** Proposed

---

## Problem

`controller/internal/relay/` is the multi-cloud VM provisioning layer
(Epic 32, Epic 42). It has 11 source files; **6 have no tests**:

| File | What it does | Why untested is dangerous |
|---|---|---|
| `cloudinit.go` | Generates cloud-init user-data for VPN VMs | Bad user-data = VM boots but never joins the mesh |
| `oci_driver.go` | Oracle Cloud Infrastructure driver | One of three clouds; bugs only surface on OCI deploys |
| `router_configmap.go` | Builds the relay-router ConfigMap | Wrong ConfigMap = router misroutes all traffic |
| `rsa.go` | RSA key generation for WireGuard | Bad key gen = silent crypto failure |
| `health.go` | Health check polling for provisioned VMs | False-positive health = routing to dead VMs |
| `constants.go` | Magic numbers / strings | Untyped drift between code and reality |

`aws_driver.go`, `driver.go`, `reconciler.go`, `wireguard.go` have happy-path
tests only — no failure-path coverage. A regression in any cloud driver
costs real money (orphaned VMs bill per-hour) and breaks user inference
paths silently.

## Scope

**In:**
- Unit tests for the 6 untested files above, using the existing `stubDriver`
  pattern from `controller/internal/relay/reconciler_test.go:43`.
- Unhappy-path additions to `aws_driver_test.go`, `driver_test.go`,
  `reconciler_test.go`, `wireguard_test.go`.
- One integration test (`relay_cloudinit_integration_test.go`) that renders
  a real cloud-init template and verifies it boots in a throwaway container
  (Docker, only in CI with `integration` tag).

**Out:**
- Real cloud API calls — drivers are tested with stubbed HTTP clients, not
  fake cloud accounts.
- Performance tests of cloud-init rendering.
- Refactoring drivers to be more testable (only if a test is impossible
  without it; otherwise leave the code alone).

## Intent-Declared Test Inventory

### `cloudinit_test.go`

| Test | What | Value added | Failure mode protected | Expected |
|---|---|---|---|---|
| `TestCloudInit_RendersMinimalUserData` | Minimum valid input | Smoke test; bug here = every VM fails to boot | Missing required cloud-init sections (not YAML syntax errors) | Output is valid YAML **and** contains the keys `write_files`, `runcmd`, and `systemd.units` — the three contract sections every cloud-init for relay VMs must have |
| `TestCloudInit_ContainsWireGuardConfig` | With WG config | WireGuard must reach the user-data or VPN never comes up | WG section dropped | Output contains `[Interface]` and `PrivateKey` |
| `TestCloudInit_ContainsRelayRouterService` | With router binary URL | Router must install as systemd unit or VM has no purpose | Service section missing | Output contains `systemd.units` with relay-router |
| `TestCloudInit_NoSecretsInOutput` | Any input with secrets | cloud-init user-data is stored unencrypted on the cloud provider's metadata service | Cloud credentials leaked to cloud provider | No API key, private key, or token in rendered output (regex assertion) |
| `TestCloudInit_TemplateInjectionEscaped` | User-controlled field with `{{ }}` | Template injection via spec fields | Arbitrary template execution | User input rendered as literal text, not evaluated |
| `TestCloudInit_SizeWithinMetadataLimit` | Maximum valid config | Cloud providers reject user-data > 16 KB (AWS) / 32 KB (GCP) | User-data truncated → silent boot failure | `len(rendered) < driver.Limit()`; `driver.Limit()` returns the documented constant per cloud (AWS 16384, GCP 262144, OCI 16384). Catches render bloat AND limit-constant drift. |

## Failure-Mode Coverage Matrix

| Failure mode | Covered by |
|---|---|
| VM fails to boot (bad cloud-init) | cloudinit × 3 + integration × 1 |
| Orphaned VM (partial provision, failed destroy) | oci × 2, aws × 2 |
| Crypto failure (bad RSA, key reuse) | rsa × 2 |
| Routing to dead VM (false-positive health) | health × 2 |
| Goroutine leak in health loop | health × 1 |
| Lost route on update (destructive CM update) | router_configmap × 1 |
| Cloud credential leak via user-data | cloudinit × 1 |
