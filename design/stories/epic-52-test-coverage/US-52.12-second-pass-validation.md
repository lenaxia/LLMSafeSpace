# US-52.12: Second-Pass Validation — Meaningfulness Audit

**Epic:** 52 — Comprehensive Test Coverage
**Layer:** Process (applies to all stories US-52.1–52.11)
**Effort:** Small (1–2d per pass; multiple passes until clean)
**Depends on:** All of US-52.1 through US-52.11
**Status:** Proposed

---

## Problem

Every story in this epic adds tests. The risk of a "comprehensive test
plan" is that it produces **tests that pass but do not protect** — tests
that:

- Assert what the code already does, not what the contract requires
  (tautology tests).
- Test trivially-true conditions (`assert.NotNil(t, x)` after a function
  that returns `&Foo{}`) without asserting any behaviour.
- Duplicate coverage at the wrong layer (a unit test that spins up
  Postgres, hiding the integration-layer gap).
- Document existing behaviour that has no specification (crystallising a
  bug as "expected").
- Cannot fail because the assertion is unreachable or vacuously true.
- Cover a path that is already covered identically elsewhere.

These tests add maintenance burden without adding protection. They make
the coverage number go up while leaving the actual gap open. They are
worse than no test because they create the *illusion* of coverage.

The user's explicit ask: *"After you have finished writing a test plan
then make a second pass to validate that all the tests you defined are
actually meaningful tests and add positive value."* This story is that
second pass. It runs after US-52.1–52.11 are written and before any of
them are merged.

## Scope

**In:**
- A structured audit of every test proposed in US-52.1 through US-52.11
  against the meaningfulness criteria below.
- A `findings.md` document under
  `design/stories/epic-52-test-coverage/second-pass/` recording every
  audited test, the audit result (KEEP / REWORK / DROP), and the rationale.
- For every REWORK: a concrete change to the test's intent or assertion
  that makes it meaningful.
- For every DROP: a deletion from the originating story's plan + rationale
  for why the test added no value.
- Re-application of the audit after REWORKs are applied, looping until
  zero findings remain (the same loop the orchestrator workflow's
  Validator Loop mandates, applied to test plans instead of code).
- Validation that the test infra conventions proposed across stories are
  idiomatic, isolated, and reusable — i.e., the *infrastructure* of the
  test plan is also meaningful.

**Out:**
- Implementing the tests — that's US-52.1–52.11. This story audits the
  *plan*, then re-audits the *implementation* via the existing Rule 11
  adversarial review at PR time.
- Adding new tests not in the original plan — if the audit reveals a
  missing test, that's a finding against the originating story, fixed by
  amending the story (not by adding it here).

## Meaningfulness Criteria (normative)

A test is **meaningful** iff it satisfies all six criteria. A test that
fails any one criterion is REWORKed or DROPped.

### M1. Failure-mode-protective

The test asserts something that, if the assertion were inverted, would
indicate a real production bug. Equivalently: there exists a plausible
code change that would (a) be a real regression and (b) fail this test.

**Anti-pattern:** `assert.NotNil(t, result)` where `result` is set by
`return &Foo{}` on every path. The only way to fail this test is to
return `nil`, which no plausible change would do.

**Audit question:** *Describe the regression that this test catches.* If
the answer is "the function returns nil" or "the function panics" or
"someone changes a constant," the test is too weak. Real regressions are
logic changes — wrong status code, missing field, wrong owner ref,
mis-classified OOM.

### M2. Non-tautological

The test does not assert a property that is true by construction. A
table-driven test where every row triggers the same code path with the
same outcome is one test masquerading as N.

**Anti-pattern:**
```go
{"empty name", "", true},
{"nil name", nil, true},
{"whitespace name", "  ", true},
```
If all three exercise the same `if name == ""` branch, two of the three
add no value. Collapse to one row + add a *different* failure mode
(too-long name, special characters, etc.).

**Audit question:** *What does each table row cover that the others do
not?* If two rows cover the same branch, drop one.

### M3. Right layer

The test is in the layer (unit / integration / e2e / canary / synth) whose
failure signal matches the test's purpose. A unit test that boots Postgres
is in the wrong layer — it's an integration test pretending to be a unit
test, and as a result it's slow, flaky, and the unit-layer gap remains
open.

**Audit question:** *If this test failed in CI, would the diagnosis start
at "the function is wrong" (unit), "these components don't compose"
(integration), or "the deployed system is broken" (e2e)?* The answer
determines the layer; mismatch is a REWORK.

### M4. Idiomatic

The test uses the project's sanctioned patterns (per epic P1–P6):
`fixture{}` constructor, fake client/testcontainers/miniredis, named
subtests with `t.Run`, no `time.Sleep` for synchronization, no module-level
mutable state, no `os.Exit` in test code, etc.

**Audit question:** *Does this test look like
`api/internal/services/workspace/workspace_service_test.go:33` (the
canonical fixture) or like nothing else in the codebase?* Idiom drift is
REWORKed to match.

### M5. Isolated and reusable

The test does not depend on state from other tests, does not leak state to
other tests, and uses shared helpers (US-52.6 harness, US-52.7 runner,
`tests/e2e/helpers`) rather than reinventing setup.

**Audit question:** *Can this test run with `-run TestThisOne` and pass?*
If not, it has a hidden dependency — REWORK to remove the dependency.

### M6. Intent-declared

The test (or table row) carries the inline intent comment required by epic
P1: what / value / failure-mode / expected. A test without this is
unreadable to the next reviewer.

**Audit question:** *Could a reviewer who didn't write this test state its
purpose in one sentence from the test alone?* If not, add the comment.

## Audit Process

The audit is a single pass per story, recorded as
`design/stories/epic-52-test-coverage/second-pass/<story>-audit.md`.
Each audit file has this structure:

```markdown
# Second-Pass Audit: <Story>

**Audited:** <date>
**Auditor:** independent (not the story author)
**Result:** <PASS | REWORK | FAIL>

## Per-test findings

### TestX (from <story>.md)

**Criterion check:**
- M1 Failure-mode-protective: <yes/no> — <rationale>
- M2 Non-tautological: <yes/no> — <rationale>
- M3 Right layer: <yes/no> — <rationale>
- M4 Idiomatic: <yes/no> — <rationale>
- M5 Isolated/reusable: <yes/no> — <rationale>
- M6 Intent-declared: <yes/no> — <rationale>

**Decision:** KEEP | REWORK | DROP
**Rework (if applicable):** <concrete change to the story's plan>
```

After every story has an audit file, the cross-cutting infrastructure
proposals (the US-52.6 harness, US-52.7 runner, US-52.10 aggregator,
US-52.11 synth runner) get a second audit specifically for idiom and
isolation — these are the highest-leverage pieces and a bad pattern here
propagates everywhere.

## Infrastructure Audit (cross-cutting)

Beyond per-test findings, the audit evaluates the four infra stories
(US-52.6 harness, US-52.7 runner, US-52.10 aggregator, US-52.11 synth
runner) for:

| Criterion | Question |
|---|---|
| Reuse | Does the infra reduce duplication, or just move it? |
| Idiom | Does it match existing patterns or invent new ones? |
| Isolation | Can two consumers use it concurrently without interference? |
| Failure mode | If the infra itself is broken, how does it surface? |
| Maintenance cost | Is the LoC removed > LoC added? |
| Documentation | Is there a README that answers "when do I use this vs. the alternative?" |

A finding of "this infra reinvents what testcontainers already does" or
"this runner duplicates `local/test.sh` without replacing it" is REWORKed
or the story is dropped.

## Loop Termination

The second pass loops until:

1. Every per-test audit returns KEEP.
2. Every predicted finding is either confirmed-and-fixed or refuted-with-
   rationale.
3. The infra audit returns zero real findings.
4. The total test count added by the epic, minus DROPs, minus REWORKs that
   collapsed duplicate rows, is recorded in the closing worklog.

Per Rule 11: no compromises, no "good enough." The loop continues until
the audit returns zero real findings.

## Verification

The verification of this story is the audit itself. The artifact is
`design/stories/epic-52-test-coverage/second-pass/` containing the audit
files. Each file links back to its story; each story links forward to its
audit. The epic README's Acceptance Criteria checkbox for "adversarial
second-pass records zero real findings" is checked only when every audit
file ends with `**Result:** PASS`.

```bash
# After the audit:
ls design/stories/epic-52-test-coverage/second-pass/
# Should list one -audit.md per story + an infra-audit.md

# Sanity: grep for unresolved findings
grep -r "REWORK\|DROP" design/stories/epic-52-test-coverage/second-pass/
# Each match must trace to an amended story or a recorded rationale
```
