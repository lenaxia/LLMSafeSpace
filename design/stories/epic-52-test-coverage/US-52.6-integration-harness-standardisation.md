# US-52.6: Integration-Test Harness Standardisation

**Epic:** 52 — Comprehensive Test Coverage
**Layer:** Infra
**Effort:** Medium (2d)
**Depends on:** —
**Status:** Proposed

---

## Problem

The codebase has multiple Postgres/Redis-backed integration tests, each
reinventing the same setup:

| File | Reinvents |
|---|---|
| `pkg/secrets/pg_integration_test.go` | Postgres testcontainer + migration runner |
| `pkg/secrets/credential_store_integration_test.go` | Postgres testcontainer + migration runner (duplicate) |
| `api/internal/handlers/secrets_integration_test.go` | Postgres + miniredis + migration runner (triplicate) |
| `api/internal/handlers/settings_integration_test.go` | Same again |
| `api/internal/services/database/*_test.go` | Same again |

This is a **tax on writing integration tests**: every new test takes longer
to author, drifts from the others, and breaks independently when a
migration changes. The tax is high enough that several services have *no*
integration tests at all (US-52.3). This story removes the tax so the rest
of the epic's integration tests can be written quickly and consistently.

## Scope

**In:**
- A single package: `api/internal/testharness/` (new).
- Exports:
  - `New(t) *Harness` — returns a harness with a migrated Postgres
    testcontainer + miniredis + cleanable tables, registered with
    `t.Cleanup` for automatic teardown.
  - `(*Harness).DB() *pgxpool.Pool` — pooled Postgres connection.
  - `(*Harness).Redis() *redis.Client` — real Redis client (backed by
    miniredis).
  - `(*Harness).Miniredis() *miniredis.Miniredis` — direct access for
    advanced assertions (TTL, key scan).
  - `(*Harness).MigrateUp() error` / `MigrateDown() error` — apply/revert
    migrations using the existing migration runner.
  - `(*Harness).NewContext() context.Context` — context with timeout +
    trace ID for test assertions.
  - `(*Harness).Logger() *zap.Logger` — captured logger; tests assert on
    captured log lines via `Logger().Memory()` or similar.
  - `(*Harness).Seed(model, values)` — typed seeding helper for common
    tables (users, workspaces, orgs).
- Migration of all 5 existing integration test files (above) to use the
  harness.
- Documentation: `api/internal/testharness/README.md` explaining when to
  use the harness vs. unit-test mocks (per epic P5).

**Out:**
- A harness for K8s integration tests — that is US-52.1's envtest path and
  US-52.7's e2e kind path. The K8s fake client + envtest patterns stay
  separate.
- A harness for end-to-end tests — that's US-52.7.
- Removing go-sqlmock — it's still appropriate for narrow unit tests; the
  harness is for true integration tests.

## Design

### When to use the harness vs. mocks

| Test needs | Use |
|---|---|
| Single function, no I/O | Plain unit test, no harness |
| Single function, needs DB or Redis semantics | Harness |
| Multiple services wired together | Harness |
| Full HTTP request through router → service → DB | Harness + `httptest` |
| K8s client interactions | `controller-runtime/client/fake` (not this harness) |

### Idioms the harness enforces (per epic P4)

1. **Per-test isolation**: `New(t)` constructs a fresh Postgres + miniredis
   per call. Database is migrated up on construction, dropped on
   `t.Cleanup`. No shared state across tests, even in the same package.
2. **Parallel-safe**: each harness instance uses unique schema names
   (Postgres testcontainer is per-instance, so this is automatic). Miniredis
   uses an isolated DB index per instance.
3. **No `TestMain`**: each test calls `New(t)` in its body, not in
   `TestMain`. This makes `t.Parallel()` safe and avoids the global-state
   trap.
4. **Time-bounded**: `New(t)` uses `ctx` with a 30s deadline for
   testcontainer startup; failures surface as test errors, not hangs.
5. **Logger capture**: the harness logger writes to an in-memory buffer;
   tests can assert "no ERROR-level log" or "this substring appeared in
   logs" without parsing stderr.

### Why testcontainers (not embedded-postgres)

`testcontainers-go` is already in the dependency tree (used by
`pkg/secrets/pg_integration_test.go`). Embedded-postgres would add a new
dependency and platform-specific binaries. The harness uses what's there.

## Intent-Declared Behaviour Inventory

The harness itself is tested by `api/internal/testharness/harness_test.go`.
Tests of the harness assert the *contract* of the harness, not its
internals.

| Test | What | Value added | Failure mode protected | Expected |
|---|---|---|---|---|
| `TestHarness_New_ReturnsWorkingDB` | Construct harness, run a SELECT | Harness smoke test | Harness construction broken = every dependent test broken | SELECT 1 returns 1 |
| `TestHarness_New_ReturnsWorkingRedis` | Construct, SET + GET | Same for Redis | Same | GET returns set value |
| `TestHarness_MigrationsApplied` | Construct, query a known table | Migrations must run automatically | Tests assume schema that isn't there | `information_schema.tables` contains expected tables |
| `TestHarness_CleanupDropsData` | New → INSERT → cleanup → New → SELECT | Per-test isolation | Test pollution across runs | New harness sees no rows |
| `TestHarness_Parallel_NoInterference` | Two harnesses in parallel | `t.Parallel()` safety | Cross-test data corruption | Each harness inserts a row with unique marker (`email = 'test-<harness_id>@test'`); each harness queries its own marker and asserts exactly 1, not 2. The marker makes isolation explicit. |
| `TestHarness_Logger_Captures` | Log via `h.Logger()` | Log assertions work | Log buffer broken | Captured entry matches input |
| `TestHarness_NewContext_HasTimeout` | Ctx from harness | Ctx always has deadline | Hang on slow query | `ctx.Deadline()` non-zero |
| `TestHarness_Seed_InsertsAndReturns` | `h.Seed("user", values)` | Typed seeding helper | Drift between seeders | Row inserted; returned with ID |

## Migration plan for existing tests

| File | Change |
|---|---|
| `pkg/secrets/pg_integration_test.go` | Replace inline testcontainer setup with `harness.New(t)`; remove ~60 lines of boilerplate |
| `pkg/secrets/credential_store_integration_test.go` | Same |
| `api/internal/handlers/secrets_integration_test.go` | Same; additionally switch from inline miniredis to `h.Redis()` |
| `api/internal/handlers/settings_integration_test.go` | Same |
| `api/internal/services/database/*_test.go` | Same |

Migration is one PR per file (per Rule 5: no mixing unrelated changes), or
one PR if the diff is small and the commit splits cleanly. Either way, each
migrated file must pass its tests before the next is touched.

## Failure-Mode Coverage Matrix (for the harness itself)

| Failure mode | Covered by |
|---|---|
| Test pollution across runs | CleanupDropsData |
| Cross-test data corruption in parallel | Parallel_NoInterference (with unique markers) |
| Migration silently skipped | MigrationsApplied |
| Log assertions silently pass | Logger_Captures |
| Hang on slow testcontainer startup | NewContext_HasTimeout |

> **Note (second-pass audit):** `TestHarness_NoDuplicatePortBinding` was
> dropped (DROP-6.1) — port allocation is testcontainers-go's job, not
> the harness's. The parallel test (REWORK-6.1) implicitly covers it.
