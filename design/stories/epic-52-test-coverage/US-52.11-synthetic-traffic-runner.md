# US-52.11: Synthetic-Traffic Runner

**Epic:** 52 — Comprehensive Test Coverage
**Layer:** Synthetic traffic (production)
**Effort:** Large (3–4d)
**Depends on:** US-52.10 (uses the same Fission runtime + canary accounts)
**Status:** Proposed

---

## Problem

The codebase has `sdks/canary/go/cmd/seed-accounts/` — a one-shot CLI that
provisions the canary test accounts. There is **no runner that exercises
the system with sustained, user-like traffic**. The gap:

- **Canaries are discrete.** They assert contract conformance on isolated
  endpoints. They do not exercise long-lived sessions, reconnect
  behaviour, mid-session failures, or session-duration drift.
- **No reconnect-backoff signal.** Epic 15 (streaming reconnect) shipped
  with unit tests but no production signal. We find out the reconnect
  backoff regressed only when users report stale UIs.
- **No session-duration distribution.** The README claims ~3s resume; we
  have no time-series of actual resume durations to detect slow drift.
- **No idle-timeout / reconnection storm signal.** A user opening 50
  browser tabs generates 50 SSE streams; we don't know how the system
  behaves under that load pattern in production.
- **No "fresh-eyes" probe.** Canaries authenticate with a long-lived API
  key. A new-user flow (register → first workspace → first message) is
  not exercised continuously.

This story adds a synthetic-traffic runner that fills these gaps. It is
**not** a load test (that's `hack/benchmark-*.sh`); it is a behavioural
drift detector.

## Scope

**In:**
- A new `sdks/canary/synth/` directory (Go-only — the synth runner is a
  state machine, not a contract check; three SDK implementations add no
  value, per epic P-canaries).
- A configurable runner that executes user-journey scripts against the
  API.
- Five user-journey scripts, each emitting time-series metrics.
- A "fresh-eyes" sub-runner that registers a new user, performs the
  first-time flow, and tears down — exercised every 30 minutes.
- Prometheus metrics emitted for: session duration, reconnect-success-rate,
  events-per-second, message-latency, registration-duration.
- Deployment as a Fission TimeTrigger (matching the canary pattern) plus a
  Deployment for the long-running continuous journeys.

**Out:**
- Load / throughput benchmarking — separate concern, separate tool.
- Real-LLM-cost-avoidance: synthetic traffic that calls real LLMs is
  expensive; journeys use the `T3-LLM-UPSTREAM-500` stub from US-52.10 or
  a deterministic stub upstream.
- UI testing — synth exercises the API; UI is covered by Playwright.
- Replacing the canary two/three-tier model — synth is a separate axis
  (continuous vs discrete), not a tier.

## Definitions (normative)

### Synthetic traffic vs. canary

| Property | Canary (Tier 1–3) | Synthetic traffic |
|---|---|---|
| Shape | Discrete (pass/fail per check) | Continuous (time-series metric) |
| Duration | < 30s (T1) to < 15 min (T3) | Hours to days |
| Output | `Result{}` JSON | Prometheus metrics + structured log |
| Asserts | "Does X conform to contract?" | "How does X drift over time?" |
| Alert on | Boolean failure | SLO burn rate, percentile shift |
| Reuses state | Never (self-contained per run) | Within a journey (session persists across steps) |
| Recovery | Built into each run | Long-running; reconnects on transient failure |

### User-journey script

A user-journey script is a Go function with this signature:

```go
type Journey func(ctx context.Context, env *Env, metrics *Metrics) error
```

- `env` provides authenticated clients (API key, JWT, SSE) plus the
  workspace ID the journey owns.
- `metrics` is the per-journey metric recorder (duration histograms,
  counters).
- Returning nil = journey completed cleanly; returning error = journey
  failed (logged + `synth_journey_failed_total` incremented, then the
  runner recreates the journey's workspace and continues — synth is
  continuous, not exit-on-failure).

Journeys are *state machines with reconnect*. A network blip does not end
the journey; the journey reconnects and continues, recording the
reconnect in metrics.

## Intent-Declared Journey Inventory

Every journey declares: what user behaviour it simulates, what drift it
detects, what metrics it emits, and what failure mode it would surface
that canaries miss.

### J1 — Long-lived chat session

**File:** `sdks/canary/synth/journeys/long_chat.go`

**Simulates:** A user who opens a workspace in the morning and chats
throughout the day — 1 message every 5 minutes for 6 hours.

**Detects drift in:**
- Message-latency percentile shift (LLM path slow-down)
- SSE-stream lifetime (broker holding connections for hours)
- Session-history growth pattern
- Workspace-status refresh rate

**Metrics emitted:**

| Metric | Type | Labels |
|---|---|---|
| `synth_longchat_message_duration_seconds` | histogram | journey |
| `synth_longchat_history_entries` | gauge | journey |
| `synth_longchat_sse_reconnects_total` | counter | journey, reason |
| `synth_longchat_active_connections` | gauge | journey |

**Failure mode surfaced that canaries miss:** A memory leak in the SSE
broker that only appears after holding a connection for 4+ hours. Canaries
run for 30s max; this journey would surface the leak as a
`synth_longchat_sse_reconnects_total` spike after the broker kills the
connection.

### J2 — Reconnect storm

**File:** `sdks/canary/synth/journeys/reconnect_storm.go`

**Simulates:** A flaky-network user (mobile, train) — connect, exchange
one message, disconnect, reconnect every 30 seconds for 30 minutes.

**Detects drift in:**
- Reconnect-success-rate (Epic 15's backoff)
- Reconnect-duration distribution
- Server-side goroutine cleanup after disconnect
- Session-resume correctness across reconnects

**Metrics emitted:**

| Metric | Type | Labels |
|---|---|---|
| `synth_reconnect_success_total` | counter | journey |
| `synth_reconnect_failure_total` | counter | journey, reason |
| `synth_reconnect_duration_seconds` | histogram | journey |
| `synth_reconnect_goroutine_delta` | gauge | journey |

**Failure mode surfaced that canaries miss:** Reconnect backoff regressed
to immediate retry (D6 from Epic 25's bug list). Canaries don't reconnect;
this journey would surface it as `synth_reconnect_failure_total` spikes.

### J3 — Multi-tab SSE subscriber

**File:** `sdks/canary/synth/journeys/multitab.go`

**Simulates:** A user with 10 open browser tabs — 10 concurrent SSE
streams on one workspace, each receiving events for 1 hour.

**Detects drift in:**
- Connection-limit enforcement (10/connection cap)
- Event-fan-out correctness (all 10 receive same event)
- Memory-per-connection

**Metrics emitted:**

| Metric | Type | Labels |
|---|---|---|
| `synth_multitab_active_streams` | gauge | journey |
| `synth_multitab_events_received_total` | counter | journey, stream_id |
| `synth_multitab_event_fanout_lag` | gauge | journey (max-min across streams) |

**Failure mode surfaced that canaries miss:** Event-broker shard
imbalance that only appears at >5 subscribers (sharding tested at 100 in
US-52.3 but only briefly).

### J4 — Suspend/resume SLA probe

**File:** `sdks/canary/synth/journeys/suspend_resume_sla.go`

**Simulates:** A user who suspends and resumes their workspace every 10
minutes for 24 hours, measuring the actual resume duration.

**Detects drift in:**
- Resume-duration percentile shift (README claims ~3s)
- PVC-reattach consistency
- Session-history survival rate

**Metrics emitted:**

| Metric | Type | Labels |
|---|---|---|
| `synth_resume_duration_seconds` | histogram | journey |
| `synth_resume_history_entries_before` | gauge | journey |
| `synth_resume_history_entries_after` | gauge | journey |
| `synth_resume_history_lost_total` | counter | journey |

**Specific drift assertion (per second-pass REWORK-11.2):**
`history_entries_after ≥ history_entries_before` on every resume. If any
resume loses history, increment `synth_resume_history_lost_total` and
mark the journey failed. The metric is the long-horizon drift signal; the
journey-failure surfaces acute regressions.

**Failure mode surfaced that canaries miss:** PVC-reattach slow-down
that only manifests after Longhorn has many snapshots (D-SUSPEND-RESUME
canary runs every 10 min but creates fresh PVCs; this journey reuses
one PVC across 144 suspend/resumes).

### J5 — Fresh-eyes registration

**File:** `sdks/canary/synth/journeys/fresh_eyes.go`

**Simulates:** A brand-new user — register → confirm → create first
workspace → wait Active → first message → delete workspace → delete
account. Runs every 30 minutes; each run is independent.

**Isolation (per second-pass REWORK-11.1):**
- Deterministic email: `synth-fresheyes-<unix_ts>@canary.test`
- Cleanup in `defer` **and** a startup sweep that deletes any
  `synth-fresheyes-*@canary.test` older than 1 hour (in case the previous
  run crashed). The sweep makes the journey self-cleaning across runner
  restarts.

**Detects drift in:**
- Registration-duration percentile
- Time-to-first-workspace-Active
- First-message success rate
- Account-deletion cleanup completeness

**Metrics emitted:**

| Metric | Type | Labels |
|---|---|---|
| `synth_fresheyes_register_duration_seconds` | histogram | run |
| `synth_fresheyes_first_workspace_active_seconds` | histogram | run |
| `synth_fresheyes_first_message_duration_seconds` | histogram | run |
| `synth_fresheyes_cleanup_complete` | gauge | run (0/1) |

**Failure mode surfaced that canaries miss:** Registration path uses
long-lived API keys. A regression in the registration → first-workspace
flow (e.g., a default-setting change in the webhook) is invisible to
existing canaries.

## Aggregation and SLOs

The runner emits raw metrics to Prometheus. SLO burn-rate alerts are
defined separately (in the observability stack owned by Epic 33). This
story's responsibility is *producing* the signal, not alerting on it.

For each journey, this story documents a *baseline expectation*:

| Journey | Baseline |
|---|---|
| J1 long-chat | 0 reconnects/hour on a healthy cluster |
| J2 reconnect-storm | ≥ 95% reconnect success within 5s |
| J3 multi-tab | 0 event-fanout-lag > 1s |
| J4 suspend/resume | p50 resume ≤ 5s; p99 ≤ 15s |
| J5 fresh-eyes | ≥ 99% end-to-end success |

Baselines are recorded in `sdks/canary/synth/README.md` and validated
during the staging soak (see Acceptance Criteria).

## Test Infrastructure (idioms + isolation)

- **Go-only** — synth is a state-machine runner; multi-SDK adds no value.
- **Per-journey workspace isolation:** each journey owns its workspace in
  the `llmsafespaces-canary` namespace; no journey touches another's
  resources.
- **Continuous, not exit-on-failure:** the runner logs failures, records
  metrics, and continues. A failed journey's workspace is recreated and
  the journey restarts.
- **Configurable via env vars** matching canary conventions
  (`LLMSAFESPACES_SYNTH_*`).
- **No real LLM cost:** journeys stub the LLM upstream (using the
  US-52.10 T3-LLM-UPSTREAM-500 stub pattern). The stub returns a
  deterministic canned response, so message-latency measures the API +
  agent path, not LLM provider variance.
- **Idempotent cleanup:** the runner's `t.Cleanup`-equivalent loop
  ensures no `synth-*` resources leak across restarts of the runner
  itself.
- **Prometheus exposition:** `/metrics` endpoint on `:9091`; scraped by
  the existing Prometheus instance (Epic 33).

## Deployment

Two deployment shapes:

1. **Long-running journeys (J1–J4)** — a Kubernetes Deployment in the
   `llmsafespaces-canary` namespace. Replica count: 1 (synth is a
   measurement, not a load generator). Restart policy: always (the
   runner is self-healing).

2. **Periodic journey (J5)** — a Fission TimeTrigger firing every 30
   minutes, matching the canary pattern. Each invocation is a fresh
   registration + cleanup.

Helm values added under `synth:` block in `charts/llmsafespaces/values.yaml`,
disabled by default, enabled in staging + prod via `values-cluster.yaml`.

## Verification

```bash
# Build
go build -o /tmp/synth ./sdks/canary/synth/

# Run locally against a dev API
LLMSAFESPACES_URL=http://localhost:8080 \
LLMSAFESPACES_SYNTH_JOURNEYS=long_chat,reconnect_storm \
  /tmp/synth --duration=10m

# Metrics endpoint
curl localhost:9091/metrics | grep synth_

# Deploy
helm upgrade --install llmsafespaces charts/llmsafespaces \
  --values values-cluster.yaml --set synth.enabled=true
```
