# US-10.13: API Key DEK Wrapping & At-Rest Encryption

**Epic:** 10 (Multi-Tenant Trust & Secret Management)
**Status:** Design
**Priority:** High (security + enables programmatic secret management)
**Date:** 2026-06-02

---

## Problem

Two issues with the current API key implementation:

1. **API keys stored in plaintext.** The `api_keys.key` column stores raw `lsp_...` values. Any database breach exposes every API key verbatim. The `GetUserByAPIKey` query does a direct `WHERE k.key = $1` match.

2. **API key users can't manage secrets.** Operations requiring DEK (create/update/decrypt secrets) only work with JWT auth because the DEK unlock flow requires a password. API key users (SDK, MCP, automation) are locked out of credential management.

---

## Design

### Part 1: API Key At-Rest Encryption

API keys are encrypted in the database using a **Root Key** that is never stored alongside the encrypted data. The Root Key is accessed through a pluggable `RootKeyProvider` interface, supporting any infrastructure (cloud KMS, Vault, or self-hosted sealed key).

```
Core Invariant:
  The Root Key MUST NOT exist in the database.
  A database breach alone MUST yield only useless ciphertext.

Storage per row:
  api_keys.key_hash       = SHA-256(raw_api_key) â€” for O(1) indexed lookup
  api_keys.key_ciphertext = AES-256-GCM(RootKey, raw_api_key)
  api_keys.key_prefix     = first 8 chars of raw key (for display: "lsp_a3b2...")
```

**RootKeyProvider interface:**

```go
// RootKeyProvider abstracts root encryption key access.
// Implementations MUST keep key material out of the database and out of
// application config files in production.
type RootKeyProvider interface {
    Encrypt(ctx context.Context, plaintext []byte) ([]byte, error)
    Decrypt(ctx context.Context, ciphertext []byte) ([]byte, error)
}
```

**Implementations:**

| Environment | Provider | Key resides in |
|-------------|----------|---------------|
| AWS | `KMSProvider` â€” calls AWS KMS Encrypt/Decrypt | KMS hardware (never leaves HSM) |
| GCP | `CloudKMSProvider` â€” calls GCP Cloud KMS | Google's KMS infrastructure |
| Self-hosted (Vault) | `VaultTransitProvider` â€” calls Vault transit/encrypt, transit/decrypt | Vault's seal (memory or HSM) |
| Self-hosted (no Vault) | `SealedKeyProvider` â€” key encrypted by operator passphrase, unsealed at startup, held in memory only | Operator-provided passphrase at deploy time |
| Local dev/test | `StaticKeyProvider` â€” raw key from env var | Env var (logged as WARNING at startup, forbidden in production) |

**SealedKeyProvider (self-hosted without KMS/Vault):**

```
Deploy time:
  1. Operator: openssl rand -hex 32 â†’ root_key
  2. Operator: seal(passphrase, root_key) â†’ sealed_root_key
  3. Store sealed_root_key in K8s Secret (safe â€” useless without passphrase)
  4. Passphrase injected via separate K8s Secret / external-secrets-operator / env var

Runtime:
  1. Pod reads sealed_root_key from mounted Secret
  2. Pod reads passphrase from separate mount / env
  3. root_key = unseal(passphrase, sealed_root_key) â€” held in memory only
  4. Never written to disk, never in DB, never logged
```

**Performance:** The `StaticKeyProvider` and `SealedKeyProvider` perform local AES operations (~microseconds). `KMSProvider` and `VaultTransitProvider` make network calls (~5-20ms). For network-backed providers, cache a KMS-generated data key in memory:

```
Startup:
  data_key = KMS.GenerateDataKey()   // returns plaintext + encrypted copy
  Store encrypted copy for crash recovery
  Hold plaintext data_key in memory

Per-request:
  Use data_key for local AES encrypt/decrypt (no network call)

Rotation (configurable interval, e.g., 24h):
  Generate new data_key, re-encrypt with KMS
  Old data_key kept briefly for in-flight decrypts
```

This gives O(1) per-request performance regardless of backend, with one KMS call per rotation interval.

**Lookup flow (authentication):**
```
1. Client sends: Authorization: Bearer lsp_<64hex>
2. Server computes: hash = SHA-256(raw_key)
3. Query: SELECT * FROM api_keys WHERE key_hash = $1 AND active = true
4. Decrypt: stored_raw = rootKeyProvider.Decrypt(key_ciphertext)
5. Verify: constant_time_compare(stored_raw, raw_key_from_request)
6. Zero(stored_raw) â€” scrub from memory immediately
7. If match: authenticated as key.user_id
```

**Why SHA-256 for lookup instead of bcrypt:**
- bcrypt is slow by design (100ms+). Cannot be indexed. Would require scanning all rows.
- SHA-256 is deterministic and indexable. Combined with decryption verification (step 4-5), provides fast lookup + proof of possession.
- At-rest protection comes from AES encryption via RootKeyProvider, not hash strength.
- 32 random bytes (256 bits) are not brute-forceable against SHA-256.

**Operational Requirements:**

| Requirement | Rationale |
|-------------|-----------|
| Root Key MUST NOT be in the database | Core invariant â€” DB breach yields only ciphertext |
| Production MUST NOT use `StaticKeyProvider` | Env vars leak via /proc, crash dumps, container inspect |
| Root Key access SHOULD be audit-logged (CloudTrail, Vault audit) | Detect unauthorized decrypt attempts |
| Root Key MUST survive pod restarts (persisted sealed copy or KMS key ID) | Loss of Root Key = all API keys irrecoverable |
| NEVER log decrypted API keys or Root Key material | Only key_hash and key_prefix in logs |

### Part 2: API Key DEK Wrapping

At API key creation time, if the user opts in (`decrypt_access: true`), wrap the user's DEK with a KEK derived from the raw API key:

```
API Key Creation (user already has active session with DEK):
  1. Generate raw_key = "lsp_" + 32 random bytes (hex)
  2. key_hash = SHA-256(raw_key)
  3. key_ciphertext = AES-256-GCM(SMK, raw_key)
  4. If decrypt_access requested:
     a. Retrieve DEK from session cache
     b. api_kek_salt = random 32 bytes
     c. api_kek = HKDF-SHA256(raw_key, api_kek_salt, "llmsafespace-apikey-kek")
     d. wrapped_dek = AES-256-GCM(api_kek, DEK)
     e. Store (wrapped_dek, api_kek_salt) on the api_keys row
  5. Store key_hash, key_ciphertext, key_prefix, wrapped_dek, api_kek_salt
  6. Return raw_key to user (shown once, never stored in plaintext)
```

**Authentication with DEK unlock:**
```
1. Client sends: Authorization: Bearer lsp_<raw_key>
2. Lookup via key_hash (Part 1 flow)
3. If key has wrapped_dek (decrypt_access = true):
   a. api_kek = HKDF-SHA256(raw_key, api_kek_salt, "llmsafespace-apikey-kek")
   b. dek = UnwrapDEK(api_kek, wrapped_dek)
   c. Cache DEK in Redis under a session ID derived from the key hash
   d. Set sessionID on gin context (same as JWT path)
4. User now has full access including secret decrypt/encrypt operations
```

## Schema Migration

```sql
-- 000012_api_key_encryption.up.sql

ALTER TABLE api_keys
  ADD COLUMN key_hash         BYTEA,           -- SHA-256(raw_key), indexed for lookup
  ADD COLUMN key_ciphertext   BYTEA,           -- AES-256-GCM(RootKey, raw_key)
  ADD COLUMN key_prefix       VARCHAR(12),     -- "lsp_a3b2" for display
  ADD COLUMN wrapped_dek      BYTEA,           -- AES-256-GCM(api_kek, user_dek), nullable
  ADD COLUMN kek_salt         BYTEA,           -- HKDF salt for api_kek derivation, nullable
  ADD COLUMN decrypt_access   BOOLEAN NOT NULL DEFAULT FALSE,
  ADD COLUMN dek_synced       BOOLEAN NOT NULL DEFAULT TRUE,
  ADD COLUMN allowed_cidrs    TEXT[];          -- optional IP allowlist (CIDR notation)

CREATE UNIQUE INDEX idx_api_keys_key_hash ON api_keys(key_hash) WHERE active = true;
```

**Migration strategy:**
1. Deploy code that reads from BOTH `key` (plaintext, legacy) and `key_hash` (new)
2. Run backfill script: encrypt each raw key via RootKeyProvider, compute hash, store
3. Verify all rows have `key_hash` + `key_ciphertext` populated
4. **Same transaction:** NULL out the plaintext `key` column for all backfilled rows
5. Deploy code that reads ONLY from `key_hash`
6. Drop `key` column in a follow-up migration after soak period

## Security Analysis

| Threat | Before | After |
|--------|--------|-------|
| DB breach (read access) | All API keys exposed in plaintext | Ciphertext only â€” useless without Root Key (which is NOT in DB) |
| DB breach + application server compromise | All API keys in plaintext | Attacker has RootKeyProvider access (same privileges as the app). API keys recoverable. This is the accepted boundary â€” if attacker IS the app, no crypto helps. |
| Leaked `decrypt_access: true` key from wrong IP | N/A | Rejected at auth â€” CIDR check fails before any decryption |
| Leaked `decrypt_access: true` key from allowed IP | N/A | Full secret access until revoked. Mitigated by: short DEK session TTL, audit alert, immediate revocation capability. |
| Leaked `decrypt_access: false` key | Auth only (current behavior) | Same â€” can manage workspaces, chat, but can't read secret values |
| Backup tape / DB snapshot theft | All API keys exposed | Ciphertext only (Root Key not in backups) |
| Container filesystem inspection | Plaintext in DB files on disk | Encrypted â€” Root Key only in memory (SealedKeyProvider) or external service (KMS/Vault) |

**Defense-in-depth layers:**
1. **Encryption at rest** â€” DB breach alone is insufficient (Root Key separate from data)
2. **IP allowlisting** â€” leaked key useless from wrong network
3. **`decrypt_access` default off** â€” most keys can't decrypt even if leaked
4. **Short DEK session TTL** for API key sessions (1h default)
5. **Audit logging** on decrypt operations via API key
6. **Immediate revocation** â€” delete row â†’ key gone, no grace period
7. **Provider audit trail** â€” KMS CloudTrail / Vault audit log detects unusual access patterns

**What this does NOT protect against (honest assessment):**
- Full application server compromise (attacker has RootKeyProvider access)
- Cloud account root compromise (attacker can access KMS/Vault)
- Memory dump of the running process (Root Key or data key in memory)

These require different controls (network isolation, IAM least-privilege, runtime integrity monitoring, TEEs) that are outside the scope of application-level encryption.

## API Changes

### Create API Key (updated)

```
POST /api/v1/auth/api-keys
{
  "name": "my-automation-key",
  "decrypt_access": true,              // opt-in: key can unlock DEK
  "allowed_cidrs": ["10.0.0.0/8"]     // optional: restrict source IPs
}
```

Response (raw key shown once):
```json
{
  "id": "uuid",
  "name": "my-automation-key",
  "key": "lsp_a3b2...full-key-shown-once",
  "prefix": "lsp_a3b2",
  "decrypt_access": true,
  "allowed_cidrs": ["10.0.0.0/8"],
  "warning": "This key has decrypt_access enabled. It can decrypt all your secrets. Store it securely â€” it cannot be recovered after this response.",
  "created_at": "..."
}
```

**IP allowlisting enforcement:** When `allowed_cidrs` is set, authentication is rejected if the request source IP is not within any of the specified CIDR ranges. Returns 403 with `"request source IP not in allowed ranges for this key"`. Checked BEFORE DEK unlock (fail fast, don't waste KMS calls).

### List API Keys (updated)

```json
[
  {
    "id": "uuid",
    "name": "my-automation-key",
    "prefix": "lsp_a3b2",
    "decrypt_access": true,
    "allowed_cidrs": ["10.0.0.0/8"],
    "dek_synced": true,
    "active": true,
    "created_at": "..."
  }
]
```

`dek_synced: false` indicates the key needs re-sync after a DEK rotation (see below).

## DEK Rotation Interaction

When `POST /api/v1/account/rotate-key` generates a new DEK:

**Primary path (immediate re-wrap â€” covers all cases):**
1. User triggers rotation (they have an active session â€” DEK is in cache)
2. New DEK generated, wrapped with password-derived KEK (existing flow)
3. New DEK wrapped with recovery-key KEK (existing flow)
4. **New:** For each of the user's API keys with `decrypt_access: true`:
   - Decrypt `key_ciphertext` via `rootKeyProvider.Decrypt()` â†’ raw API key
   - Re-derive `api_kek = HKDF-SHA256(raw_key, kek_salt, "llmsafespace-apikey-kek")`
   - Wrap NEW DEK: `new_wrapped_dek = AES-256-GCM(api_kek, new_dek)`
   - Update row with `new_wrapped_dek`, set `dek_synced = true`
   - Zero all intermediate key material

This works because the server can always recover the raw API key (via RootKeyProvider) to re-derive the api_kek. No lazy migration needed.

**Fallback (transient RootKeyProvider failure):**
If the RootKeyProvider call fails during rotation (network timeout to KMS/Vault):
- Mark affected keys as `dek_synced = false`
- Rotation still succeeds for password + recovery KEKs
- Background job retries every 5 minutes until provider is reachable
- Stale key auth for decrypt operations returns: `403 "API key DEK re-sync in progress. Retry shortly."`
- User sees `"dek_synced": false` in `GET /api/v1/auth/api-keys`

## Out of Scope

- HSM-backed SMK (future â€” enterprise tier)
- Per-API-key permissions/scopes (separate epic)
- API key expiry enforcement changes (already exists via `expires_at`)
- Migrating from SHA-256 lookup to constant-time scan (SHA-256 on 32 random bytes is not brute-forceable; no timing attack surface on the hash lookup itself)
