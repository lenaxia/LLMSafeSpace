# US-21.2: Implement Backoff Schedule in `recoverFromTransientPodLoss`

**Epic:** 21 — Workspace State Machine
**Priority:** Critical
**Depends on:** US-21.1 (status fields)
**Blocks:** US-21.6 (cap), US-21.9 (per-cause policy)
**Layer:** Controller

## User Story

As the workspace controller, when I detect a transient pod loss (pod missing, pod not Running, pod in CrashLoopBackOff, or agentd-unreachable beyond threshold), I need to schedule the next retry attempt according to an exponential-backoff schedule rather than counting strikes against a hard cap, so that short transient flakes (the worklog 0100 incident class) self-heal without consuming the workspace's failure budget.

## Acceptance Criteria

- [ ] `controller/internal/workspace/constants.go` defines:
  ```go
  var BackoffSchedule = []time.Duration{
      5 * time.Second,
      30 * time.Second,
      2 * time.Minute,
      5 * time.Minute,
      15 * time.Minute,
      1 * time.Hour,
      4 * time.Hour,
      12 * time.Hour,
  }
  const MaxConsecutiveFailures = 8
  ```
- [ ] `recoverFromTransientPodLoss` (controller.go:456-477) is rewritten to:
  - Increment `Status.ConsecutiveFailures` by 1.
  - If `ConsecutiveFailures > MaxConsecutiveFailures`, defer to US-21.6 for terminal-Failed transition (in this story, leave a TODO stub that returns `ctrl.Result{}, nil` without writing terminal — US-21.6 fills it).
  - Otherwise, compute `wait := BackoffSchedule[ConsecutiveFailures-1]`.
  - Set `Status.NextRetryAt = metav1.NewTime(time.Now().Add(wait))`.
  - Set `Status.Phase = WorkspacePhaseCreating`.
  - Clear `Status.PodIP`, `Status.Endpoint`.
  - Persist via `r.Status().Update(ctx, workspace)` (Story 23.2 will add retry-on-conflict; for now, single-shot is acceptable).
  - Return `ctrl.Result{RequeueAfter: wait}, nil` (so the requeue arrives at the right time even if the watch doesn't fire).
- [ ] `BackoffSchedule[i]` is **clamped** at index `MaxConsecutiveFailures-1` (no out-of-bounds access for ConsecutiveFailures values that exceed the schedule length, defensively — should never happen in practice once US-21.6 caps).
- [ ] Test fixtures use a `clock.Clock` interface (or `metav1.NowFunc`) so tests can drive time deterministically.
- [ ] All existing transient-recovery tests are updated to match the new schedule (some assertions that referenced `TransientFailureCount` get rewritten as `ConsecutiveFailures`).
- [ ] No regression in the health-check path's call to `recoverFromTransientPodLoss` (controller.go:303 inside handleActive's CrashLoopBackOff check).

## Technical Details

### New `recoverFromTransientPodLoss` body

```go
// recoverFromTransientPodLoss is invoked when handleActive detects the pod
// is missing, not Running, or in CrashLoopBackOff. It increments the
// ConsecutiveFailures counter, schedules the next retry per BackoffSchedule,
// and transitions the workspace back to Creating so the next reconcile
// (after NextRetryAt) recreates the pod.
func (r *WorkspaceReconciler) recoverFromTransientPodLoss(ctx context.Context, workspace *v1.Workspace) (ctrl.Result, error) {
    workspace.Status.ConsecutiveFailures++

    // Defer terminal-cap handling to US-21.6.
    // For now, if we ever exceed the schedule length, just clamp the wait
    // to the last entry. US-21.6 will replace this with markFailed().
    n := workspace.Status.ConsecutiveFailures
    if n > int32(MaxConsecutiveFailures) {
        // TODO US-21.6: transition to terminal Failed with FailureReason=TooManyFailures.
        // For now: re-clamp so we don't out-of-bounds and let the next reconcile
        // observe whatever state we leave.
        n = int32(MaxConsecutiveFailures)
    }

    wait := BackoffSchedule[n-1]
    nextRetry := metav1.NewTime(time.Now().Add(wait))

    workspace.Status.NextRetryAt = &nextRetry
    workspace.Status.Phase = v1.WorkspacePhaseCreating
    workspace.Status.PodIP = ""
    workspace.Status.Endpoint = ""

    if err := r.Status().Update(ctx, workspace); err != nil {
        return ctrl.Result{}, err
    }
    return ctrl.Result{RequeueAfter: wait}, nil
}
```

### Time injection for tests

`controller/internal/workspace/clock.go`:

```go
package workspace

import "time"

// nowFunc is overridable in tests via SetNowForTesting.
var nowFunc = time.Now

// SetNowForTesting sets the clock used by the controller for the duration
// of a test. Returns a restore function the test must defer.
func SetNowForTesting(t time.Time) (restore func()) {
    prev := nowFunc
    nowFunc = func() time.Time { return t }
    return func() { nowFunc = prev }
}
```

Replace `time.Now()` in `recoverFromTransientPodLoss` with `nowFunc()`. Same for any other failure-path code paths in the same commit. Tests use `SetNowForTesting` to drive the clock.

## Test Plan

### Unit tests

**Happy path (table-driven):**
- For N in `[1, 2, 3, 4, 5, 6, 7, 8]`: workspace with `ConsecutiveFailures=N-1` → invoke `recoverFromTransientPodLoss` → assert:
  - `Status.ConsecutiveFailures == N`.
  - `Status.NextRetryAt == fixedNow + BackoffSchedule[N-1]` (within 1s tolerance).
  - `Status.Phase == Creating`.
  - `Status.PodIP == "" && Status.Endpoint == ""`.
  - `ctrl.Result.RequeueAfter == BackoffSchedule[N-1]`.

**Unhappy paths:**
- Update conflict (mock `r.Status().Update` returns Conflict error): assert function returns the conflict error; status writes are NOT committed; counter does NOT advance in etcd. Note: this test confirms the single-shot behavior; Story 23.2 wraps with retry.
- Workspace already at `ConsecutiveFailures = MaxConsecutiveFailures (8)`: invoke → ConsecutiveFailures becomes 9, clamp triggers, schedule index uses 7 (last entry, 12h). No terminal-Failed write yet (US-21.6 owns that).
- Workspace at `ConsecutiveFailures = 100` (defensive — should never happen): no out-of-bounds; clamp to 12h wait; no panic.

**Edge cases:**
- `ConsecutiveFailures` is `omitempty`, so a fresh workspace has `Status.ConsecutiveFailures == 0`. First call: post-increment is 1; uses `BackoffSchedule[0] = 5s`. ✅
- `time.Now()` returns the same value within a single reconcile (clock injection ensures determinism).
- Workspace is being deleted (`DeletionTimestamp != nil`): handleActive's earlier guard short-circuits before reaching `recoverFromTransientPodLoss`. Confirm by test (regression).
- Concurrent reconcile is impossible (controller-runtime serializes per-object), so no atomicity concern. Test: invoke twice in sequence; second invocation increments to 2.

### Integration tests (envtest)

- **Real Status update:** create a Workspace, simulate pod missing, invoke reconcile, fetch from K8s API → assert ConsecutiveFailures=1, NextRetryAt set.
- **Sequential failures:** simulate three back-to-back failures over a short period (with clock injection). Assert ConsecutiveFailures progresses 1→2→3 and NextRetryAt matches schedule.
- **Cap behavior:** simulate 9 failures. Assert ConsecutiveFailures=9 (cap not yet enforced — US-21.6 does it). Schedule clamped at attempt 8.

### E2E tests (envtest with controller running, simulated agentd)

- Agentd mock that fails health check 4 times, then succeeds on the 5th. Workspace recovers without entering terminal Failed; final ConsecutiveFailures=4 (not reset until US-21.4 ships).

### Security tests

- **No secret leaks in error messages:** even if Status update returns an error containing the workspace name, no password/token material leaks. Verified by: when `Update` returns an error, the error string does not contain any field from `workspace.Status` other than identifying metadata.
- **No privilege escalation:** the path doesn't call `r.Create`/`r.Delete` on any new resource; it only writes the workspace's own status.

### Performance tests

- **Backoff write latency:** benchmark `recoverFromTransientPodLoss` end-to-end (including a fake `Status.Update`). Acceptance: < 5ms per call (the existing call is < 1ms; we're not adding network calls).
- **Memory:** no per-failure allocation of large objects. The schedule is a package-level slice; `metav1.NewTime` allocates one struct.

### Robustness tests

- **Clock skew:** simulate clock running backwards between two reconciles (system time fix-up). Assert `NextRetryAt` is still honored on the next reconcile (the elapsed-time check uses `time.Now().Before(NextRetryAt)`, which is correct under both forward and backward skew).
- **Controller crash mid-update:** kill the controller pod between in-memory mutation and `Status().Update`. Replacement controller's next reconcile observes the un-incremented counter (no double-count). Verified via envtest scenario.
- **Schedule overflow defense:** `BackoffSchedule` is package-level immutable; tests verify length matches `MaxConsecutiveFailures`.

### Negative tests

- Invoke `recoverFromTransientPodLoss` on a workspace with `Phase=Suspended`: this is an illegal call (handleSuspended doesn't invoke it). Confirm by reviewing call sites; document that the function assumes the caller has already determined the workspace is in `Active` state with a pod problem.
- Invoke with a nil workspace: panics (caller is responsible). Acceptable; matches existing convention.

## Dependencies

- US-21.1 must be merged first (status fields exist).

## Risk Assessment

- **Medium risk.** The function is in a hot path triggered by every transient pod-loss event. A bug here would either fail to recover (workspace stuck) or recover too aggressively (busy-loop).
- **Mitigation:** comprehensive table-driven tests over the schedule; envtest scenarios with real K8s API; clock injection makes timing deterministic.

## Rollout

Single commit. Deploy via Helm. Existing Workspaces with `ConsecutiveFailures=0` immediately benefit on first failure (5s backoff instead of 1-of-3 strike toward terminal). Workspaces currently in Failed are NOT auto-recovered (Change A's manual restart is the recovery path); they'll start using the new schedule on next bump.
