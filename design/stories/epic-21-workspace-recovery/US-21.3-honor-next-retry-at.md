# US-21.3: Honor `NextRetryAt` in `handleCreating` and `handleActive`

**Epic:** 21 — Workspace State Machine
**Priority:** Critical
**Depends on:** US-21.1 (status fields)
**Blocks:** US-21.4 (reset), US-21.5 (RestartGeneration interaction)
**Layer:** Controller

## User Story

As the workspace controller, when `Status.NextRetryAt` is set on a workspace and that time has not yet elapsed, I need to skip all pod-state-checking work and simply wait until the retry time, so that the backoff schedule produced by `recoverFromTransientPodLoss` (US-21.2) is actually respected and we don't spin in a tight loop trying to recreate or re-probe a pod we just decided to back off from.

## Acceptance Criteria

- [ ] At the top of `handleCreating` (controller.go:184-ish, before any pod-fetch logic): if `Status.NextRetryAt != nil`, evaluate the backoff:
  - If `time.Now().Before(NextRetryAt)`: return `ctrl.Result{RequeueAfter: time.Until(NextRetryAt)}, nil`. No pod operations.
  - If `time.Now().After(NextRetryAt) || time.Now().Equal(NextRetryAt)`: clear `NextRetryAt = nil`, persist via `r.Status().Update`, fall through to existing `handleCreating` logic.
- [ ] Same logic at the top of `handleActive` (controller.go:256-ish, before the pod-fetch). Identical guard.
- [ ] The clear-on-elapse update is a single `r.Status().Update` call. If it fails (conflict, etc.), the function returns the error so controller-runtime requeues; the next reconcile will re-attempt.
- [ ] Mock-call assertions in tests: while `NextRetryAt` is in the future, `r.Get(pod)` is NEVER called, `r.Create(pod)` is NEVER called, `client.IsHealthy` (health check) is NEVER called.
- [ ] Existing `handleCreating` / `handleActive` happy-path tests continue to pass (the guard short-circuits only when `NextRetryAt` is set and in the future; for a fresh workspace with `NextRetryAt=nil`, behavior is unchanged).
- [ ] The `nowFunc` clock injection from US-21.2 is reused — no new clock abstraction.

## Technical Details

### Guard implementation (helper function)

```go
// shouldSkipForBackoff returns RequeueAfter and true when the workspace is
// still in its backoff window. The caller should return the result
// immediately. When the backoff has elapsed, this function clears
// NextRetryAt (persisting via Status().Update) and returns false so the
// caller proceeds with normal logic.
func (r *WorkspaceReconciler) shouldSkipForBackoff(ctx context.Context, workspace *v1.Workspace) (ctrl.Result, bool, error) {
    if workspace.Status.NextRetryAt == nil {
        return ctrl.Result{}, false, nil
    }
    now := nowFunc()
    if now.Before(workspace.Status.NextRetryAt.Time) {
        wait := workspace.Status.NextRetryAt.Time.Sub(now)
        return ctrl.Result{RequeueAfter: wait}, true, nil
    }
    // Backoff elapsed; clear and proceed.
    workspace.Status.NextRetryAt = nil
    if err := r.Status().Update(ctx, workspace); err != nil {
        return ctrl.Result{}, true, err  // controller-runtime requeues
    }
    return ctrl.Result{}, false, nil
}
```

### Call sites

```go
// In handleCreating:
func (r *WorkspaceReconciler) handleCreating(ctx context.Context, workspace *v1.Workspace) (ctrl.Result, error) {
    if result, skip, err := r.shouldSkipForBackoff(ctx, workspace); skip {
        return result, err
    }
    // ... existing logic ...
}

// In handleActive: identical guard at the top.
```

### Why a helper instead of inline?

DRY. The same logic runs in two places. A helper:
- Makes the intent clear at each call site (one line: "honor backoff").
- Centralizes the clear-on-elapse persistence so a future bug fix only edits one location.
- Is unit-testable in isolation.

### Files modified

- `controller/internal/workspace/controller.go` — new `shouldSkipForBackoff` helper; call from `handleCreating` and `handleActive`.
- `controller/internal/workspace/controller_test.go` — new tests for the helper and integration tests for both call sites.

## Test Plan

### Unit tests for `shouldSkipForBackoff`

**Happy path — backoff active:**
- Workspace with `NextRetryAt = now + 10s`. Invoke. Assert: `result.RequeueAfter == 10s`, `skip == true`, `err == nil`. Assert workspace.Status.NextRetryAt is unchanged.

**Happy path — backoff elapsed:**
- Workspace with `NextRetryAt = now - 10s`. Invoke. Assert: `result == empty`, `skip == false`, `err == nil`. Assert `workspace.Status.NextRetryAt == nil` (cleared). Assert `r.Status().Update` was called once.

**Happy path — no backoff:**
- Workspace with `NextRetryAt == nil`. Invoke. Assert: `result == empty`, `skip == false`, `err == nil`. Assert `r.Status().Update` was NOT called.

**Unhappy paths:**
- Backoff elapsed but `Update` returns conflict error. Assert: `skip == true`, `err == conflict`. Caller will return the error and controller-runtime requeues.
- Backoff elapsed but `Update` returns Forbidden (RBAC). Same: `skip == true`, `err != nil`.
- `NextRetryAt = now` exactly (boundary). Assert: `time.Now().Before(NextRetryAt) == false` → backoff elapsed → cleared.

**Edge cases:**
- `NextRetryAt = now + 1ns` (sub-millisecond future). `RequeueAfter` is 1ns; controller-runtime caps at some minimum but the function returns the literal value.
- `NextRetryAt = year 2099`. Far future — `RequeueAfter` is huge but valid. The reconcile won't fire that long; if controller restarts before then, the new controller observes `NextRetryAt` and re-applies the wait.
- Workspace passed in is a deep copy; mutations to `workspace.Status.NextRetryAt = nil` don't affect the caller's view (controller-runtime convention: pass owned pointers).

### Unit tests for `handleCreating` and `handleActive` integration

**handleCreating with active backoff:**
- Setup: workspace `Phase=Creating`, `NextRetryAt = now + 5s`. Mock `r.Get(pod)` to error if called.
- Invoke `handleCreating`. Assert: returns `RequeueAfter==5s`. Assert: `r.Get` NOT called, `r.Create` NOT called.

**handleCreating with elapsed backoff:**
- Setup: `Phase=Creating`, `NextRetryAt = now - 1s`. Mock `r.Get(pod)` to return NotFound. Mock `r.Create(pod)` to succeed.
- Invoke. Assert: `NextRetryAt` cleared via Update; falls through to existing logic; `r.Get` and `r.Create` are called per existing `handleCreating` code.

**handleActive with active backoff:**
- Setup: `Phase=Active`, `NextRetryAt = now + 30s`, `PodIP=""` (was cleared in US-21.2 transition). Mock `r.Get(pod)` to error if called. Mock `client.IsHealthy` to error if called.
- Invoke. Assert: `RequeueAfter==30s`. Assert: no pod fetch, no health check.

**handleActive with elapsed backoff:**
- Setup: `Phase=Active`, `NextRetryAt = now - 1s`. Mock the rest of the path.
- Invoke. Assert: `NextRetryAt` cleared; existing handleActive logic runs.

**Regression:**
- handleCreating with `NextRetryAt == nil`. Mock the rest of the path. Assert: behavior identical to pre-US-21.3 (the guard short-circuits to `skip=false` immediately).
- handleActive with `NextRetryAt == nil`. Same.

### Integration tests (envtest)

- **End-to-end backoff window:** create a Workspace, simulate a transient failure (US-21.2 sets `ConsecutiveFailures=1`, `NextRetryAt=now+5s`), let envtest's controller manager run a reconcile. Use a mock clock — advance to `now+2s`, observe NO pod creation. Advance to `now+6s`, observe pod creation.
- **Sequential backoffs:** trigger 3 failures in sequence with growing schedule. Assert: each backoff window is honored before the next attempt; total wall-clock matches schedule sum.
- **Controller restart mid-backoff:** kill controller pod with workspace at `NextRetryAt = now + 1m`. Restart controller. Assert: replacement controller's reconcile observes the existing `NextRetryAt` and respects the remaining wait (does not reset it).

### E2E tests

- **Real cluster, real flake:** with Epic 22 unshipped, induce an agentd flake (e.g., kill opencode briefly inside a pod). Workspace's controller log shows `NextRetryAt` set, the next reconcile waits, then retries. Workspace recovers without operator intervention.

### Security tests

- **No secret material in `NextRetryAt`:** the field is purely a `metav1.Time` (timestamp). No leakage.
- **No DoS vector:** an attacker setting `NextRetryAt = year 2099` would cause the workspace to never reconcile until that time. This is only writable by the controller (status subresource RBAC); no cross-tenant exposure.
- **Cross-tenant isolation:** the helper operates on a single workspace's status; no read/write to other workspaces.

### Performance tests

- **Hot-path overhead:** the guard adds two field reads (nil check + time comparison) and possibly one `Update`. Bench: a no-backoff reconcile is < 1µs slower than baseline.
- **Backoff window: zero useless work:** when `NextRetryAt` is in the future, the controller does NO pod operations, NO HTTP calls. Verified by mock-call assertions; documented as the acceptance criterion.

### Robustness tests

- **Clock skew:** if system clock jumps backward by an hour (NTP correction), `NextRetryAt` (which was relative to the old "now") may now be in the future when it shouldn't be. The reconcile honors the skewed time and waits longer. Acceptable — once the clock stabilizes, the next reconcile fires correctly.
- **Clock jump forward:** same — `NextRetryAt` becomes in-the-past, the next reconcile clears and proceeds.
- **Status update conflict:** if `r.Status().Update` (when clearing NextRetryAt) returns Conflict, the function returns the error; controller-runtime requeues; on retry, the freshly-fetched workspace may have `NextRetryAt` already cleared by another path (e.g., RestartGeneration bump from US-21.5). Idempotent — no double-clear, no panic.
- **NextRetryAt is "zero" struct:** `metav1.Time{}` (not nil pointer, but zero value). `Time.IsZero()` returns true. The check `now.Before(zero-time)` returns false (any positive time is after zero). So the guard treats a zero NextRetryAt as "elapsed" and clears it. Acceptable — defensive against malformed CRDs.

### Negative tests

- Workspace in `Phase=Suspended` with `NextRetryAt` set. The guard is only at `handleCreating` and `handleActive`; suspend handler is untouched. NextRetryAt would persist in the CRD until the next non-suspended reconcile clears it (or until US-21.5's RestartGeneration handler clears it). **Acceptable** — Suspended workspaces are not running pods; the backoff is irrelevant.
- Workspace in `Phase=Failed` with `NextRetryAt` set. Same — the Failed handler doesn't honor the backoff. If a workspace flapped and ended up Failed (e.g., the cap eventually fires in US-21.6), the backoff field is dead state. US-21.5's RestartGeneration recovery clears it.

## Dependencies

- US-21.1 must be merged (status fields exist).

## Risk Assessment

- **Medium risk.** Adds a guard at the top of two hot reconcile paths. A bug here either (a) makes the controller spin (if guard fails to short-circuit), or (b) makes the controller never reconcile (if guard returns wrong RequeueAfter and never re-fires).
- **Mitigation:** the helper's logic is small and well-tested; envtest validates against a real reconciler manager which would surface (b)-class bugs immediately.

## Rollout

Single commit. Deploy via Helm. Workspaces currently in any phase don't have `NextRetryAt` set (US-21.1 introduced the field; no prior writes), so the new guard is a no-op for all existing workspaces. New transient failures (post-US-21.2 deploy) populate the field and the guard kicks in.
