# US-21.10: Surface `FailureReason` in API Status Endpoint

**Epic:** 21 — Workspace State Machine
**Priority:** Medium
**Depends on:** US-21.7 (FailureReason field on CRD), US-21.8 (markFailed populates the field)
**Blocks:** Frontend per-reason rendering (separate epic)
**Layer:** API service + API types

## User Story

As a frontend developer, when I call `GET /api/v1/workspaces/:id/status` for a Failed workspace, I need the response payload to include the `failureReason` enum value alongside `phase` and `message`, so that the UI can render per-cause copy and per-cause affordances (e.g., hide retry button during auto-retry, show storage-admin contact for PVCBindTimeout) without parsing free-form `Status.Message` text.

## Acceptance Criteria

- [ ] `WorkspaceStatusResult` (in `api/internal/types/workspace.go` or wherever it lives) gains a `FailureReason string \`json:"failureReason,omitempty"\`` field. String type (not the enum) since this is the JSON wire format; the value is one of the enum's string forms.
- [ ] `Service.GetWorkspaceStatus` (workspace_service.go around line 540-560) populates `result.FailureReason = string(crd.Status.FailureReason)`.
- [ ] If `crd.Status.FailureReason` is empty (workspace not Failed, or pre-Epic-21 state), the response field is omitted (`omitempty`).
- [ ] If the workspace is also auto-retrying (Phase=Creating with NextRetryAt set), the response also includes `nextRetryAt *time.Time \`json:"nextRetryAt,omitempty"\`` so the UI can render a countdown.
- [ ] OpenAPI / contract documentation is updated (api/openapi/workspace.yaml or similar).
- [ ] Existing API tests still pass; new tests assert the new field appears for Failed workspaces and is absent for non-Failed.

`ConsecutiveFailures` is exposed as observability — a UI could render "retrying (attempt 3 of 8)" if it knows both the current count and the schedule length.

### Service code

```go
func (s *Service) GetWorkspaceStatus(ctx context.Context, userID, workspaceID string) (*WorkspaceStatusResult, error) {
    if err := s.verifyOwner(ctx, userID, workspaceID); err != nil {
        return nil, err
    }
    crd, err := s.k8sClient.LlmsafespaceV1().Workspaces(s.config.Namespace).Get(workspaceID, metav1.GetOptions{})
    if err != nil {
        return nil, apierrors.NewInternalError("workspace_get_failed", err)
    }
    result := &WorkspaceStatusResult{
        Phase:               string(crd.Status.Phase),
        Message:             crd.Status.Message,
        FailureReason:       string(crd.Status.FailureReason),  // NEW
        ConsecutiveFailures: crd.Status.ConsecutiveFailures,    // NEW
    }
    if crd.Status.NextRetryAt != nil {                          // NEW
        t := crd.Status.NextRetryAt.Time
        result.NextRetryAt = &t
    }
    if crd.Status.LastActivityAt != nil {
        t := crd.Status.LastActivityAt.Time
        result.LastActivityAt = &t
    }
    // ... existing field copies ...
    return result, nil
}
```

## Test Plan

### Unit tests

**Happy path — Failed workspace:**
- CRD with `Status.Phase=Failed`, `Status.FailureReason=PodBuildFailed`, `Status.Message="..."`. Invoke `GetWorkspaceStatus`. Assert response has `Phase="Failed"`, `FailureReason="PodBuildFailed"`, `Message="..."`.

**Happy path — auto-retry workspace:**
- CRD with `Phase=Creating`, `NextRetryAt=&future`, `ConsecutiveFailures=3`. Invoke. Assert response has `Phase="Creating"`, `NextRetryAt=&future`, `ConsecutiveFailures=3`. `FailureReason` omitted.

**Happy path — Active workspace:**
- CRD with `Phase=Active`, no failure fields. Invoke. Assert response has only `Phase="Active"`; failure-related fields all omitted.

**Wrong owner:**
- User `u2` requests workspace owned by `u1`. Assert: 403 Forbidden. (Existing authz logic, regression test.)

**Workspace not found:**
- CRD doesn't exist. Assert: 404 with structured error. Existing behavior, regression test.

**Edge cases:**
- `Status.FailureReason = ""` (zero value) and `Status.Phase = "Failed"` (semantically inconsistent but possible if a controller bug or pre-Epic-21 data). Response has `Phase="Failed"`, `FailureReason` omitted (since empty). UI should render a generic "Failed" state. ✅
- `NextRetryAt` in the past (race between API read and controller's clear). Response includes the past timestamp; UI could either render "retrying now" or refresh the status. Acceptable; document.

### Integration tests

- Spin up envtest controller; create a Workspace; transition through Failed via the controller; call the API service's `GetWorkspaceStatus`. Assert response matches CRD state.
- Round-trip via HTTP: POST workspace → induce failure → GET /api/v1/workspaces/:id/status → assert JSON structure.

### Contract tests

- API contract test (e.g., consumer-driven contract or OpenAPI validator): the response schema includes `failureReason: string` (optional). The frontend's TypeScript type matches.
- Schema validates: if `failureReason` is present, its value is one of the enum strings (or empty if forwards-compat needed).

### E2E tests

- Real cluster: configure a workspace to fail (bad image). Trigger via UI or API. `curl GET /api/v1/workspaces/:id/status -H "Authorization: ..."`. Assert response JSON contains `"failureReason":"PodBuildFailed"`.

### Security tests

- **Authz:** ownership check is the first line of `GetWorkspaceStatus`. Wrong-owner returns 403 before any CRD fetch. (Existing behavior.)
- **No data leakage in error paths:** errors include workspace ID but no other workspace's data.
- **No status injection:** the `FailureReason` field is read-only via the API (no PATCH endpoint for status). Operators with CRD-level access can write status (controller's RBAC), but the API surface is read-only.

### Performance tests

- API response latency: < 50ms p99 (existing baseline). New fields add negligible serialization cost (3 fields).
- No N+1: the service makes a single CRD fetch.

### Robustness tests

- CRD has unexpected `FailureReason` value (e.g., a future enum value the API binary doesn't know about). The service passes it through as-is (string-type field). Forwards-compatible. Frontend receiving an unknown value can fall back to a generic display.
- Concurrent reads while controller writes: K8s client returns a consistent snapshot; no torn reads.

### Negative tests

- Caller passes empty workspace ID. Existing 400 Bad Request behavior. Regression test.

## Dependencies

- US-21.7 (CRD field exists).
- US-21.8 (controller actually populates the field).

## Risk Assessment

- **Low risk.** Pure read-side addition. Field is `omitempty`; absent values are gracefully handled.

## Rollout

Single commit. API redeploys. Existing GET /status calls receive the new fields when the underlying workspace has them populated. Old Failed workspaces (pre-Epic-21) continue to receive the existing `phase` + `message` without the new fields (omitempty).

Frontend doesn't yet render the new fields; that's a separate frontend epic. The API just makes them available.
