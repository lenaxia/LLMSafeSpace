# US-21.4: Stability-Reset in `maybeResetTransientCounter`

**Epic:** 21 — Workspace State Machine
**Priority:** High
**Depends on:** US-21.1 (status fields), US-21.3 (NextRetryAt honored)
**Blocks:** none (the cap in US-21.6 works without reset, but reset prevents premature termination)
**Layer:** Controller

## User Story

As the workspace controller, when a workspace has been running `Active` continuously for at least the `TransientFailureResetWindow` (5 minutes), I need to reset `Status.ConsecutiveFailures` to 0 and clear `Status.NextRetryAt`, so that a workspace that has self-healed and is operating normally for a meaningful period doesn't carry forward stale failure counts that would prematurely trip the cap on a fresh independent failure hours later.

## Acceptance Criteria

- [ ] `maybeResetTransientCounter` (existing function at controller.go:481-490) is extended to:
  - On every `handleActive` reconcile that observes the workspace running healthy (Phase=Active, PodIP non-empty, post all early-return guards), update `Status.LastSuccessAt = metav1.NewTime(now)`.
  - If `Status.LastSuccessAt != nil && now.Sub(LastSuccessAt) >= TransientFailureResetWindow`:
    - Set `Status.ConsecutiveFailures = 0`.
    - Set `Status.NextRetryAt = nil` (defensive — should already be nil if we're in handleActive successfully).
- [ ] The function is called from `handleActive` at the appropriate point — AFTER all "the workspace is sick" branches (CrashLoopBackOff, agent-unreachable, RestartGeneration bump) have returned, so it only runs when the workspace genuinely is healthy on this reconcile.
- [ ] `LastSuccessAt` is set on EVERY successful `handleActive` reconcile, not just when the reset condition fires. This is what the reset condition checks.
- [ ] The reset and the `LastSuccessAt` write happen as a single `Status().Update` call (no double-write).
- [ ] If `LastSuccessAt` was never set (first-ever Active reconcile), set it to the current time and skip the reset condition check (no LastSuccessAt yet means nothing to compare to).
- [ ] Existing `maybeResetTransientCounter` tests are updated to use the new field names.

## Technical Details

### Rewritten `maybeResetTransientCounter`

```go
// maybeResetTransientCounter is called once per successful handleActive
// reconcile (after all "workspace is sick" branches have returned). It
// updates LastSuccessAt to now, and if the workspace has been Active
// continuously for >= TransientFailureResetWindow, it resets the
// ConsecutiveFailures counter and any lingering NextRetryAt.
//
// Mutations are persisted by the caller (handleActive) as part of its
// normal Status().Update. This function does NOT call Update itself —
// it only mutates the in-memory object.
func (r *WorkspaceReconciler) maybeResetTransientCounter(workspace *v1.Workspace) {
    now := nowFunc()

    if workspace.Status.LastSuccessAt == nil {
        // First-ever successful Active reconcile. Bootstrap the timestamp.
        ts := metav1.NewTime(now)
        workspace.Status.LastSuccessAt = &ts
        return
    }

    elapsed := now.Sub(workspace.Status.LastSuccessAt.Time)
    if elapsed >= TransientFailureResetWindow {
        if workspace.Status.ConsecutiveFailures > 0 || workspace.Status.NextRetryAt != nil {
            workspace.Status.ConsecutiveFailures = 0
            workspace.Status.NextRetryAt = nil
        }
    }

    // Refresh the timestamp on every healthy reconcile so a transient blip
    // followed by sustained recovery resets the counter on the next call.
    ts := metav1.NewTime(now)
    workspace.Status.LastSuccessAt = &ts
}
```

### When the function runs

In `handleActive` (controller.go:256-355 today), the call site is at the end of the "happy path" — after pod is verified Running with PodIP, after RestartGeneration check, after `checkAgentHealth` succeeds, after `spec.timeout` check. Specifically: **right before** the final `r.Status().Update` that persists per-reconcile state (StartTime, Conditions, etc.).

```go
// In handleActive, after all the "is the workspace sick?" branches:
r.maybeResetTransientCounter(workspace)

// existing per-reconcile housekeeping ...

return ctrl.Result{RequeueAfter: requeueActive}, r.Status().Update(ctx, workspace)
```

### Why update LastSuccessAt on every reconcile?

The function name says "maybe reset" — but the timestamp is updated unconditionally. This is intentional:
- The timestamp captures the **latest** observed-healthy moment.
- The reset condition checks **how long ago** the previous moment was.
- Without per-reconcile updates, `LastSuccessAt` would be the FIRST observed-healthy time, not the most recent. A workspace that's been healthy for 2 minutes, fails, recovers in 5 seconds, and runs healthy for another 4 minutes would have `LastSuccessAt = first time` — total elapsed 6+ min — and the reset would fire even though it hasn't been continuously healthy.

The "continuously healthy" semantic comes from: **`LastSuccessAt` is updated only when the function is called, and the function is called only on reconciles that pass all "sick" checks**. If a reconcile detects sickness, it returns early; LastSuccessAt is NOT updated; the next reconcile's elapsed-time computation correctly reflects the gap.

## Test Plan

### Unit tests for `maybeResetTransientCounter`

**Happy path — first reconcile:**
- Setup: `LastSuccessAt = nil`, `ConsecutiveFailures = 3`, `NextRetryAt = nil`. Set `nowFunc` to `t0`.
- Invoke. Assert: `LastSuccessAt == &t0` (bootstrap), `ConsecutiveFailures` unchanged (no reset on first call).

**Happy path — within window:**
- Setup: `LastSuccessAt = t0`, `ConsecutiveFailures = 2`. Set `nowFunc` to `t0 + 2 minutes`.
- Invoke. Assert: `LastSuccessAt == t0+2m` (refreshed), `ConsecutiveFailures` unchanged (only 2m elapsed; window is 5m).

**Happy path — window elapsed, reset fires:**
- Setup: `LastSuccessAt = t0`, `ConsecutiveFailures = 5`, `NextRetryAt = t0+10m` (stale future). Set `nowFunc` to `t0 + 6 minutes`.
- Invoke. Assert: `ConsecutiveFailures == 0`, `NextRetryAt == nil`, `LastSuccessAt == t0+6m`.

**Happy path — no failures to reset:**
- Setup: `LastSuccessAt = t0`, `ConsecutiveFailures = 0`. `nowFunc = t0+10m`.
- Invoke. Assert: `LastSuccessAt = t0+10m`, `ConsecutiveFailures = 0` (no-op reset is fine).

**Edge cases:**
- `LastSuccessAt = t0 - 1h` (extremely stale, possibly from a controller-restart scenario). `nowFunc = t0`. Elapsed = 1h. Reset fires. ✅ — defensive against long gaps.
- `LastSuccessAt == &t0` and `nowFunc == t0` (zero elapsed). Reset does NOT fire. ✅
- `nowFunc < LastSuccessAt` (clock jumped backward). `time.Sub` returns negative. `negative >= 5m` is false. No reset. `LastSuccessAt` is updated to the new (earlier) time. Acceptable — clock skew handling is documented. ✅
- `ConsecutiveFailures == 0 && NextRetryAt == nil` already (clean state). Window elapsed. Function still updates `LastSuccessAt` but the reset block is a no-op (the conditional `if ConsecutiveFailures > 0 || NextRetryAt != nil` skips). ✅

**Negative tests:**
- `maybeResetTransientCounter(nil)` — caller bug; panics. Document that the caller is responsible. (Same convention as other helpers.)

### Integration tests for `handleActive` call site

**Workspace healthy for window, failure occurs, recovery, reset:**
- Setup: workspace in steady-state Active for 6 minutes (envtest).
- Inject a transient failure (e.g., delete the pod). US-21.2 sets `ConsecutiveFailures=1`. Wait for backoff. Pod recreates, becomes healthy.
- Run for another 6 minutes Active. Assert: `ConsecutiveFailures = 0` (reset fired during the second healthy period).

**Workspace healthy briefly, fails, recovers, fails again — counter does NOT reset:**
- Setup: workspace Active for 1 minute. Failure 1 (ConsecutiveFailures=1). Recovery. Active for 2 minutes. Failure 2.
- Assert at second failure: `ConsecutiveFailures = 2` (not reset; window = 5m, only 2m elapsed during recovery).

**Sick branch returns early, no reset call:**
- Mock `handleActive` with the early-return path triggered (e.g., pod NotFound). Assert `maybeResetTransientCounter` is NOT called.
- This is verified by mock-call assertion on the test fixture or by tracing the code path.

### E2E tests

- Long-running E2E (envtest with virtual time): workspace runs Active for 10 simulated minutes; triggers ConsecutiveFailures bump; recovers; runs healthy for 6 minutes. Assert ConsecutiveFailures resets. ✅

### Security tests

- **No secret material:** `LastSuccessAt` is a timestamp. No leakage.
- **No privilege escalation:** function reads/writes only its own workspace's status.

### Performance tests

- **Reconcile overhead:** the function is called once per Active reconcile. Bench: < 1µs. (Pure in-memory math.)
- **No allocation in the hot path:** `metav1.NewTime` allocates a struct; can be optimized later if profiling shows a hotspot. Out of scope for this story.

### Robustness tests

- **Clock skew backward:** documented above. No correctness issue, just slightly slower reset eventually.
- **Clock skew forward:** `LastSuccessAt = old`, `now` jumps forward by 10m. Reset fires immediately even if the workspace was actually only healthy for 1m. **Imperfect** — this is a real edge case but the workspace IS currently healthy (the function only runs after handleActive's "healthy" path). A premature reset is benign — the next failure will be counted from 0, which is what we want. Acceptable.
- **Controller restart:** `LastSuccessAt` is in the CRD. Replacement controller picks it up. The 5-minute window persists across controller restarts. ✅
- **NextRetryAt set when LastSuccessAt was never set:** the function bootstraps `LastSuccessAt` and skips the reset; the stale `NextRetryAt` would persist. **But:** US-21.3's `shouldSkipForBackoff` runs BEFORE handleActive's body, so a workspace with a NextRetryAt would have backed off and not reached `maybeResetTransientCounter` in the first place. Documented invariant. Test: ensure `NextRetryAt = nil` whenever `maybeResetTransientCounter` is called.

### Negative tests

- Function called on a workspace in `Phase=Suspended`: caller error; should never happen. The function doesn't check phase. If called accidentally, it would update LastSuccessAt and possibly reset — both relatively benign. Document the precondition.

## Dependencies

- US-21.1, US-21.3 must be merged.

## Risk Assessment

- **Low-medium risk.** The function is small, clearly defined, and has a clear precondition (called only from a known healthy code path). Worst-case bug: spurious reset (benign — next failure starts from 0, which is still bounded).
- **Mitigation:** integration test in envtest with virtual time covers the multi-minute scenarios that unit tests can't easily cover.

## Rollout

Single commit. Existing workspaces have `LastSuccessAt = nil` (US-21.1 added the field with no writes). On the first Active reconcile after deploy, each workspace bootstraps its `LastSuccessAt`. From then on, the reset window kicks in. Existing transient failures (which set `ConsecutiveFailures` from US-21.2) get reset on the first 5-minute stable period.
