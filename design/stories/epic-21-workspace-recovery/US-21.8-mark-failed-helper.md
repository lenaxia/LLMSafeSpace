# US-21.8: `markFailed` Helper + Wire All 5 Failed-Write Sites

**Epic:** 21 — Workspace State Machine
**Priority:** Critical
**Depends on:** US-21.7 (FailureReason enum exists)
**Blocks:** US-21.6 (cap uses markFailed), US-21.9 (per-cause policy reads FailureReason)
**Layer:** Controller

## User Story

As the workspace controller, I need a single `markFailed` helper function that all five Failed-transition sites use, so that every Failed write consistently sets `Phase`, `FailureReason`, and `Message`, the test surface is single-source-of-truth, and adding new failure reasons in the future is one-edit-per-call-site instead of a sweep.

## Acceptance Criteria

- [ ] `controller/internal/workspace/controller.go` defines:
  ```go
  func (r *WorkspaceReconciler) markFailed(
      workspace *v1.Workspace,
      reason v1.FailureReason,
      format string,
      args ...any,
  ) {
      workspace.Status.Phase = v1.WorkspacePhaseFailed
      workspace.Status.FailureReason = reason
      workspace.Status.Message = fmt.Sprintf(format, args...)
      // Defensive: clear backoff fields since we're terminal.
      workspace.Status.NextRetryAt = nil
      // ConsecutiveFailures is preserved as observability — it tells us
      // how many attempts we made before giving up.
      // LastSuccessAt is preserved for the same reason.
  }
  ```
- [ ] All 5 Failed-write sites in `controller.go` are migrated to call `markFailed` with the appropriate reason:
  | Line (today) | New reason |
  |---|---|
  | controller.go:138 | `FailureReasonPendingTimeout` |
  | controller.go:169 | `FailureReasonPVCBindTimeout` |
  | controller.go:203 | `FailureReasonPodBuildFailed` |
  | controller.go:244 | `FailureReasonPodFailedDuringCreation` |
  | controller.go:467 (`recoverFromTransientPodLoss` cap path, US-21.6) | `FailureReasonTooManyFailures` |
- [ ] After migration, **no other code path writes `Status.Phase = WorkspacePhaseFailed` directly**. Verified by `grep -rn "Status.Phase = v1.WorkspacePhaseFailed" controller/internal/workspace/` returning only the body of `markFailed`.
- [ ] All existing tests that asserted on `Status.Message` content still pass (the messages may be slightly tweaked but should remain human-readable; tests should be updated to assert on `FailureReason` enum where appropriate, then on Message substrings as a secondary check).
- [ ] New tests assert that each migrated site sets the matching `FailureReason`.

## Technical Details

### Helper signature rationale

```go
markFailed(workspace, reason, format, args...)
```

- `format + args` matches `fmt.Sprintf` so call sites can interpolate values cleanly without concatenation.
- The helper does NOT call `r.Status().Update`. The caller's existing Update call commits the changes. This keeps the helper pure (easier to test) and avoids accidental double-writes.

### Migration of each site

**controller.go:138 (handlePending timeout):**
```go
// Before:

workspace.Status.Message = "workspace timed out in Pending phase"

// After:

```

**controller.go:203 (handleCreating buildPod error):**

**controller.go:244 (handleCreating PodFailed branch):**

## Test Plan

### Unit tests for `markFailed` helper

**Happy path:**
- `markFailed(ws, FailureReasonTransientPodLoss, "lost pod %s", "p1")`. Assert: `ws.Status.Phase == Failed`, `FailureReason == TransientPodLoss`, `Message == "lost pod p1"`, `NextRetryAt == nil`.

**Backoff fields cleared:**
- Workspace with `NextRetryAt = &future`. `markFailed`. Assert NextRetryAt cleared.
- ConsecutiveFailures NOT cleared (preserved as observability of how many attempts before terminal).

**No side effects:**
- Helper does NOT call `r.Status().Update`. Verified by mock-call assertion.

**Edge cases:**
- Empty format string: `markFailed(ws, reason, "")`. Message is empty. Acceptable; caller's responsibility to provide useful diagnostics.
- Nil workspace: panics. Caller error.
- `FailureReasonNone` ("") with Phase set to Failed: this is technically permitted by the type but semantically wrong. Document that callers must pass a non-empty reason. Tests should NEVER call `markFailed(ws, "", ...)`.

### Unit tests per migrated site

For each of the 5 sites, set up the trigger condition, invoke the relevant handler, assert:
- `Status.Phase == Failed`.
- `Status.FailureReason == <expected enum>`.
- `Status.Message` contains the expected substring (e.g., "PVC", "pod build failed", etc.).

**handlePending timeout:**
- Workspace with `Status.Phase = Pending`, `StartTime` more than `pendingPhaseTimeout` ago. Reconcile.
- Assert: Phase=Failed, FailureReason=PendingTimeout, Message contains "timed out in Pending phase".

**handlePending PVC not bound:**
- Workspace with PVC reference but PVC stuck Pending in K8s for > timeout.
- Assert: Phase=Failed, FailureReason=PVCBindTimeout, Message contains the PVC name and storage class.

**handleCreating buildPod error:**
- Workspace with bad runtime image (mock buildPod to return error).
- Assert: Phase=Failed, FailureReason=PodBuildFailed, Message contains the error string.

**handleCreating PodFailed (post-Epic-23-Story-1 guard):**
- Workspace with pod in K8s phase=Failed, DeletionTimestamp=nil (genuine creation failure).
- Assert: Phase=Failed, FailureReason=PodFailedDuringCreation, Message contains pod name.

**recoverFromTransientPodLoss cap:**
- Workspace at ConsecutiveFailures=8, transient failure causes 9th increment.
- Assert: Phase=Failed, FailureReason=TooManyFailures, Message contains "exhausted 8 retry attempts" and the most recent cause.

### Integration tests (envtest)

- Apply each scenario end-to-end. Fetch workspace via K8s API. Assert FailureReason field is correctly populated and matches the helper's output.

### E2E tests

- Real cluster: trigger each scenario (e.g., bad image for PodBuildFailed). Assert frontend or `kubectl get` shows the FailureReason. (Frontend rendering is a separate epic.)

### Security tests

- **Format string safety:** `fmt.Sprintf` with attacker-controlled format string from a CRD field would be a vulnerability. Audit: the `format` argument is always a static string literal in the call sites. The `args` may contain dynamic values (pod names, error messages). No format-string injection possible.
- **Cause string truncation (cross-reference US-21.6):** for `recoverFromTransientPodLoss`, the cause might be derived from external pod state. Ensure truncation at e.g. 200 chars to prevent log/CRD bloat from a malicious pod.

### Performance tests

- `markFailed` invocation: < 1µs (string formatting + struct field writes). No allocation pressure.

### Robustness tests

- **Concurrent terminal-Failed writes:** controller-runtime's serialization makes concurrent reconciles of the same workspace impossible (epic 21 A1, A8). No race within `markFailed`.
- **Test-time fixture safety:** `markFailed` mutates the in-memory workspace; tests should pass deep copies to avoid cross-test pollution. Document.

### Negative tests

- Calling `markFailed(ws, FailureReasonNone, ...)`: forbidden by convention but not enforced at compile time. Add a defensive runtime check that logs a warning if `reason == ""` is passed:
  ```go
  if reason == "" {
      log.FromContext(...).V(0).Info("markFailed called with empty FailureReason; this is a bug",
          "workspaceID", workspace.Name)
  }
  ```
  The function still proceeds (writes the empty reason); the warning surfaces the bug for debugging.

## Dependencies

- US-21.7 (enum exists).

## Risk Assessment

- **Low-medium risk.** The helper is small. The 5 migrations are mechanical. Risk is regression from a typo in one of the message format strings or a wrong reason mapping.
- **Mitigation:** per-site tests assert both Phase=Failed AND the exact FailureReason value. Code review checks the table.

## Rollout

Single commit. Existing workspaces in Failed phase don't have `FailureReason` set (US-21.7 added the field with no writes). New Failed transitions populate the field correctly. Old Failed workspaces' messages are still in `Status.Message`; consumers parsing those continue to work.
