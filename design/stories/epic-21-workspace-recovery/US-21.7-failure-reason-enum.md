# US-21.7: Add `FailureReason` Enum + Status Field

**Epic:** 21 — Workspace State Machine
**Priority:** Critical (blocks US-21.8, US-21.9, US-21.10)
**Depends on:** none
**Blocks:** US-21.6, US-21.8, US-21.9, US-21.10
**Layer:** API (CRD types) + API service types

## User Story

As a platform engineer, I need a typed `FailureReason` enum on `WorkspaceStatus` so that operators, the frontend, and metric consumers can switch on a stable machine-readable value instead of pattern-matching free-form `Status.Message` text. The enum captures the semantic distinctions across the five Failed-write sites that exist today.

## Acceptance Criteria

- [ ] `pkg/apis/llmsafespace/v1/workspace_types.go` defines:
  ```go
  type FailureReason string

  const (
      FailureReasonNone                    FailureReason = ""
      FailureReasonTransientPodLoss        FailureReason = "TransientPodLoss"
      FailureReasonPodFailedDuringCreation FailureReason = "PodFailedDuringCreation"
      FailureReasonPodBuildFailed          FailureReason = "PodBuildFailed"
      FailureReasonPVCBindTimeout          FailureReason = "PVCBindTimeout"
      FailureReasonPendingTimeout          FailureReason = "PendingTimeout"
      FailureReasonTooManyFailures         FailureReason = "TooManyFailures"
  )
  ```
- [ ] `WorkspaceStatus` adds `FailureReason FailureReason \`json:"failureReason,omitempty"\``.
- [ ] `pkg/apis/llmsafespace/v1/zz_generated.deepcopy.go` regenerated.
- [ ] CRD manifest schema reflects the new field; OpenAPI schema includes the enum string values for validation.
- [ ] All existing tests continue to compile and pass (no semantic change yet — this story is purely the type addition).

## Technical Details

### Field placement

```go
type WorkspaceStatus struct {
    Phase              WorkspacePhase       `json:"phase,omitempty"`
    FailureReason      FailureReason        `json:"failureReason,omitempty"`  // NEW (US-21.7)
    PVCName            string               `json:"pvcName,omitempty"`
    // ... rest unchanged ...
}
```

Placed adjacent to `Phase` because they're conceptually paired (Phase=Failed implies a non-empty FailureReason; Phase != Failed implies FailureReason="").

### CRD validation

Add OpenAPI v3 schema enum constraints:

```yaml
status:
  properties:
    failureReason:
      type: string
      enum:
        - ""
        - "TransientPodLoss"
        - "PodFailedDuringCreation"
        - "PodBuildFailed"
        - "PVCBindTimeout"
        - "PendingTimeout"
        - "TooManyFailures"
```

This catches typos at admission time. The empty string is valid (any non-Failed phase has empty reason).

### Why an enum and not free-form

The five existing Failed sites have distinct semantics (worklog 0100):
- `PendingTimeout`: PVC create couldn't even attempt within 5min
- `PVCBindTimeout`: PVC stuck non-bound
- `PodBuildFailed`: spec invalid
- `PodFailedDuringCreation`: pod observed in K8s phase=Failed
- `TooManyFailures`: backoff cap exhausted (US-21.6)

An enum forces every Failed site to declare its semantic class; consumers can switch on it. Free-form messages stay (in `Status.Message`) for human consumption.

### Files modified

- `pkg/apis/llmsafespace/v1/workspace_types.go` — type + constants + status field.
- `pkg/apis/llmsafespace/v1/zz_generated.deepcopy.go` — regenerate.
- `charts/llmsafespace/templates/crd-workspace.yaml` (or wherever CRD lives) — schema enum.

## Test Plan

### Unit tests

**Happy path:**
- Construct `WorkspaceStatus{FailureReason: FailureReasonTransientPodLoss}`. Marshal to JSON. Assert: `"failureReason":"TransientPodLoss"`.
- Construct with `FailureReason: ""` (none). Assert: JSON omits the field (`omitempty`).
- Round-trip: marshal → unmarshal → assert equality including the new field.
- DeepCopy: verify the field is copied (not aliased).

**Enum sanity:**
- Sweep over all 7 enum values. Each marshals to its string. Each unmarshals back. ✅

**CRD admission:**
- Apply a Workspace with `status.failureReason="TransientPodLoss"`. Should succeed.
- Apply with `status.failureReason="garbage"`. Should fail with admission error (kube-apiserver enforces the enum).
- Apply with no `failureReason` (omitted). Should succeed.

### Integration tests (envtest)

- Create a Workspace via the typed client with `Status.FailureReason = FailureReasonPodBuildFailed`. Fetch it back. Assert `FailureReason == "PodBuildFailed"`.
- Create with an invalid enum string via raw YAML. Assert kube-apiserver rejects.

### E2E tests

- Apply CRD chart. Verify on a kind cluster: `kubectl get workspace -o yaml` displays `status.failureReason` correctly. `kubectl explain workspace.status.failureReason` shows the enum values (per OpenAPI schema).

### Security tests

- **No secret material:** the enum and field are pure metadata. No leakage.
- **Validation prevents injection:** the enum constraint at admission time prevents an attacker (with status-update RBAC, which controller alone has) from injecting arbitrary strings.

### Performance tests

- **Marshal/unmarshal:** benchmark vs pre-US-21.7. Acceptance: < 1% latency increase.
- **DeepCopy:** same.

### Robustness tests

- **Forwards-compat:** a controller built on this story can read/write the new field. Subsequent stories (US-21.8, US-21.9) populate it from real code paths.
- **Empty-string treatment:** `FailureReasonNone` ("") is the zero value of `FailureReason`. Code that checks `if status.FailureReason == ""` (or `== FailureReasonNone`) works either way. Consumers that switch on the enum should include the empty case.

### Negative tests

- Set `Status.FailureReason = "TransientPodLoss"` while `Status.Phase = "Active"` (semantically inconsistent — Active workspaces shouldn't have a failure reason). The CRD schema doesn't enforce this cross-field constraint. Document: enum is **observational** about why a Failed phase fired; setting it without Phase=Failed is a controller bug, not a CRD violation. Tests should not depend on this consistency at the type level.

## Rollout

Single commit. CRD update applied via chart. Existing workspaces have no `failureReason` (zero value); their Phase=Failed messages are still in `Status.Message` and consumers parsing those continue to work.

US-21.8 then wires actual writes. Until US-21.8 ships, the field stays empty even on Failed phases — that's intentional staging.
