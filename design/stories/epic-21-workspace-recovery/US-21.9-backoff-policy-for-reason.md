# US-21.9: `BackoffPolicyForReason` Helper + Per-Cause Retry Policy

**Epic:** 21 — Workspace State Machine
**Priority:** High
**Depends on:** US-21.2 (backoff in recoverFromTransientPodLoss), US-21.7 (FailureReason enum), US-21.8 (markFailed wires FailureReason at sites)
**Blocks:** none
**Layer:** Controller

## User Story

As the workspace controller, the backoff schedule applied to a transient failure should depend on the **reason** the failure occurred. `TransientPodLoss` (an agentd flake) deserves the full 8-attempt schedule. `PodFailedDuringCreation` (likely an image issue) is unlikely to recover after 3 attempts. `PodBuildFailed` (spec is wrong) shouldn't auto-retry at all. `PendingTimeout` (upstream K8s outage) deserves longer initial waits. Per-cause policy ensures we don't waste retry budget on hopeless cases nor give up too quickly on recoverable ones.

## Acceptance Criteria

- [ ] `controller/internal/workspace/constants.go` defines per-reason backoff schedules:
  ```go
  // Default schedule for TransientPodLoss (8 attempts, 17h total).
  var BackoffSchedule = []time.Duration{
      5 * time.Second, 30 * time.Second, 2 * time.Minute, 5 * time.Minute,
      15 * time.Minute, 1 * time.Hour, 4 * time.Hour, 12 * time.Hour,
  }

  // Tighter schedule for PodFailedDuringCreation (3 attempts).
  var BackoffSchedulePodFailedDuringCreation = []time.Duration{
      5 * time.Second, 30 * time.Second, 2 * time.Minute,
  }

  // Slower schedule for PendingTimeout (5 attempts, 14h+ total).
  var BackoffSchedulePendingTimeout = []time.Duration{
      30 * time.Second, 5 * time.Minute, 30 * time.Minute, 2 * time.Hour, 12 * time.Hour,
  }
  ```
- [ ] A new helper `BackoffPolicyForReason(reason FailureReason) []time.Duration` returns the correct schedule:
  - `TransientPodLoss` → `BackoffSchedule` (8 attempts)
  - `PodFailedDuringCreation` → `BackoffSchedulePodFailedDuringCreation` (3 attempts)
  - `PendingTimeout` → `BackoffSchedulePendingTimeout` (5 attempts)
  - `PodBuildFailed`, `PVCBindTimeout`, `TooManyFailures`, `FailureReasonNone` → `nil` (no auto-retry)
- [ ] `recoverFromTransientPodLoss` is extended to take a `reason FailureReason` parameter and use `BackoffPolicyForReason(reason)` for the schedule and cap. Schedule length is the per-reason cap (no global `MaxConsecutiveFailures` for cap; each reason has its own).
- [ ] If `BackoffPolicyForReason` returns nil for a given reason, the function does NOT retry — it transitions directly to terminal `Failed` with that reason. (For `PodBuildFailed` and `PVCBindTimeout`, this is the existing behavior at sites :203 and :169 which already terminal-Fail; for `TooManyFailures` it's a defensive no-op since that's already terminal.)
- [ ] All callers of `recoverFromTransientPodLoss` pass the appropriate reason:
  - `handleActive` pod-NotFound branch → `FailureReasonTransientPodLoss`
  - `handleActive` non-Running branch → `FailureReasonTransientPodLoss`
  - `handleActive` CrashLoopBackOff branch → `FailureReasonTransientPodLoss`
  - `checkAgentHealth` agent-unreachable-beyond-threshold → `FailureReasonTransientPodLoss`
  - `handleCreating` PodFailed branch (after Epic 23 Story 1's DeletionTimestamp guard) → `FailureReasonPodFailedDuringCreation`
- [ ] Tests cover each policy and confirm `PodBuildFailed`/`PVCBindTimeout` do NOT auto-retry.

## Technical Details

### `BackoffPolicyForReason`

```go
// BackoffPolicyForReason returns the backoff schedule for a given failure
// reason, or nil if the reason should not be auto-retried (callers should
// transition straight to terminal Failed).
func BackoffPolicyForReason(reason v1.FailureReason) []time.Duration {
    switch reason {
    case v1.FailureReasonTransientPodLoss:
        return BackoffSchedule
    case v1.FailureReasonPodFailedDuringCreation:
        return BackoffSchedulePodFailedDuringCreation
    case v1.FailureReasonPendingTimeout:
        return BackoffSchedulePendingTimeout
    case v1.FailureReasonPodBuildFailed,
         v1.FailureReasonPVCBindTimeout,
         v1.FailureReasonTooManyFailures,
         v1.FailureReasonNone:
        return nil
    default:
        // Unknown reason: defensive — treat as no-retry to surface the bug.
        return nil
    }
}
```

### `recoverFromTransientPodLoss` final form (US-21.6 + US-21.9 combined)

```go
func (r *WorkspaceReconciler) recoverFromTransientPodLoss(
    ctx context.Context,
    workspace *v1.Workspace,
    reason v1.FailureReason,
    cause string,
) (ctrl.Result, error) {
    schedule := BackoffPolicyForReason(reason)
    if schedule == nil {
        // Reason is non-retriable. Transition straight to terminal Failed.
        r.markFailed(workspace, reason, "%s (no auto-retry policy for %s)", cause, reason)
        if err := r.Status().Update(ctx, workspace); err != nil {
            return ctrl.Result{}, err
        }
        metrics.WorkspaceTerminalFailuresTotal.WithLabelValues(string(reason)).Inc()
        return ctrl.Result{}, nil
    }

    workspace.Status.ConsecutiveFailures++
    n := workspace.Status.ConsecutiveFailures

    if n > int32(len(schedule)) {
        // Cap reached for this reason. Terminal Failed.
        r.markFailed(workspace, v1.FailureReasonTooManyFailures,
            "exhausted %d retry attempts for %s; most recent cause: %s",
            len(schedule), reason, cause)
        if err := r.Status().Update(ctx, workspace); err != nil {
            return ctrl.Result{}, err
        }
        metrics.WorkspaceTerminalFailuresTotal.WithLabelValues(string(v1.FailureReasonTooManyFailures)).Inc()
        return ctrl.Result{}, nil
    }

    wait := schedule[n-1]
    nextRetry := metav1.NewTime(nowFunc().Add(wait))
    workspace.Status.NextRetryAt = &nextRetry
    workspace.Status.Phase = v1.WorkspacePhaseCreating
    workspace.Status.PodIP = ""
    workspace.Status.Endpoint = ""
    if err := r.Status().Update(ctx, workspace); err != nil {
        return ctrl.Result{}, err
    }
    return ctrl.Result{RequeueAfter: wait}, nil
}
```

### Why per-reason caps instead of global `MaxConsecutiveFailures`

The original epic design had a single 8-attempt cap. US-21.9 supersedes that: each reason has its own cap derived from the schedule length. `MaxConsecutiveFailures` constant is removed.

The terminal-failure reason for "exceeded the cap" is always `TooManyFailures` regardless of which schedule produced it — operators can correlate via the message which triggering reason ran out of retries.

## Test Plan

### Unit tests for `BackoffPolicyForReason`

- Sweep over all 7 enum values. Assert correct schedule (or nil) for each. Table-driven.
- Defensive: an unknown enum value (cast from a string the controller doesn't recognize) returns nil. Document this is the failure mode for forward-compat.

### Unit tests for `recoverFromTransientPodLoss` per reason

**TransientPodLoss — full 8-attempt schedule:**
- 8 sequential invocations. Assert schedule applied. 9th invocation triggers TooManyFailures terminal.

**PodFailedDuringCreation — 3-attempt schedule:**
- 3 invocations. Assert each backoff matches the tight schedule. 4th invocation triggers TooManyFailures.

**PendingTimeout — 5-attempt schedule:**
- 5 invocations with the longer-ramp schedule. 6th triggers TooManyFailures.

**PodBuildFailed — no auto-retry:**
- Single invocation. Assert: workspace transitions to terminal Failed (PodBuildFailed) with no backoff. ConsecutiveFailures stays 0 (no increment because we early-return).

**PVCBindTimeout — no auto-retry:**
- Same as PodBuildFailed.

**TooManyFailures — no auto-retry:**
- Caller passes TooManyFailures (defensive — shouldn't normally happen since TooManyFailures is itself the cap-result reason). Assert: function transitions terminal again (idempotent — already at terminal). ConsecutiveFailures stays.

**Edge case — schedule swap mid-cycle:**
- Workspace with `ConsecutiveFailures = 3` from earlier TransientPodLoss-class failures. New reconcile invokes with reason `PodFailedDuringCreation` (3-attempt schedule, length 3). `n = 4 > 3` → TooManyFailures terminal. **Acceptable** — mixing reasons within a backoff cycle is unusual and terminating early is correct (the workspace has a pattern of varied failures, suggesting genuine brokenness).

### Integration tests (envtest)

- Real-K8s scenario: workspace with bad image. First reconcile → `handleCreating` → buildPod returns error → `recoverFromTransientPodLoss` invoked with PodBuildFailed → terminal Failed in one shot, no retry.
- Workspace with PVC bind timeout. handlePending → recoverFromTransientPodLoss with PVCBindTimeout → terminal Failed.
- Workspace with agentd flake (TransientPodLoss). 5 transient failures. Workspace stays alive through all 5 attempts. 9th failure terminates with TooManyFailures.

### E2E tests

- Real cluster: configure a workspace with intentionally bad image. Assert: `kubectl get workspace -o jsonpath='{.status.failureReason}'` returns `"PodBuildFailed"` within seconds (no retries).
- Configure a workspace with valid image but inject 9 agentd-restart flakes. Assert: workspace eventually shows `failureReason=TooManyFailures`.

### Security tests

- Per-reason policy doesn't introduce new attack surface.

### Performance tests

- Schedule lookup: O(1) (map / switch). < 100ns.
- No additional Status writes vs US-21.6.

### Robustness tests

- **Reason changes mid-cycle:** documented above (edge case test). Cap fires early; acceptable.
- **Caller passes empty reason:** `BackoffPolicyForReason("")` → nil → no-retry. Defensive; the empty reason is treated as a non-recoverable failure. The terminal-Failed message will say "no auto-retry policy for ". Confused but bounded.

### Negative tests

- Caller passes a reason but no `cause`: cause is empty string. Terminal message has empty cause. Acceptable; document.
- Caller passes a non-enum string (e.g., raw `FailureReason("xyz")`): `BackoffPolicyForReason` returns nil → no-retry → terminal. Defensive; surfaces the bug.

## Risk Assessment

- **Medium risk.** Behavior depends on which reason each call site passes. Wrong reason at a site means wrong policy applied (e.g., a real PodBuildFailed retried 8 times wastes budget; a real TransientPodLoss treated as non-retriable terminates immediately).
- **Mitigation:** explicit table mapping call site → reason; per-site test asserts correct reason.

## Rollout

Single commit. Removes `MaxConsecutiveFailures`. New per-reason behavior takes effect on first reconcile post-deploy.
