# US-21.6: Cap at Attempt 9 → Terminal Failed (`FailureReason = TooManyFailures`)

**Epic:** 21 — Workspace State Machine
**Priority:** High
**Depends on:** US-21.1 (status fields), US-21.2 (backoff increments ConsecutiveFailures), US-21.7 (FailureReason enum), US-21.8 (markFailed helper)
**Blocks:** none
**Layer:** Controller

## User Story

As the workspace controller, when `ConsecutiveFailures` reaches `MaxConsecutiveFailures + 1` (the 9th attempt), I need to transition the workspace to terminal `Failed` with `FailureReason = TooManyFailures` so that genuinely-broken workspaces don't loop forever consuming retry budget, and operators / users have a clear signal that the auto-recovery budget is exhausted.

## Acceptance Criteria

- [ ] The TODO stub in `recoverFromTransientPodLoss` (left from US-21.2) is replaced with a real terminal-Failed transition when `ConsecutiveFailures > MaxConsecutiveFailures`:
  ```go
  if workspace.Status.ConsecutiveFailures > int32(MaxConsecutiveFailures) {
      r.markFailed(workspace, v1.FailureReasonTooManyFailures,
          "exhausted %d retry attempts; last failure: %s",
          MaxConsecutiveFailures, lastFailureMessage)
      return ctrl.Result{}, r.Status().Update(ctx, workspace)
  }
  ```
- [ ] `lastFailureMessage` is the most recent transient-failure description (e.g., "pod NotFound after 3 health-check failures", "agent unreachable beyond threshold"). Wired through from `recoverFromTransientPodLoss`'s caller.
- [ ] The terminal-Failed write uses Change C's `markFailed` helper (US-21.8) to ensure consistent FailureReason population.
- [ ] After the cap fires, the workspace stays in `Failed` until an explicit `RestartGeneration` bump (Change A / US-21.5) — auto-recovery is intentionally over.
- [ ] Recovery via RestartGeneration resets `ConsecutiveFailures` to 0 (already covered by US-21.5); the next failure starts a fresh backoff cycle from attempt 1.
- [ ] Tests cover the cap-firing behavior, the post-cap recovery via RestartGeneration, and the boundary (attempt 8 = last allowed retry, attempt 9 = terminal).

## Technical Details

### `recoverFromTransientPodLoss` final form (after US-21.6)

```go
func (r *WorkspaceReconciler) recoverFromTransientPodLoss(
    ctx context.Context,
    workspace *v1.Workspace,
    cause string, // human-readable description of the transient failure
) (ctrl.Result, error) {
    workspace.Status.ConsecutiveFailures++
    n := workspace.Status.ConsecutiveFailures

    if n > int32(MaxConsecutiveFailures) {
        // US-21.6: cap reached. Transition to terminal Failed.
        r.markFailed(workspace, v1.FailureReasonTooManyFailures,
            "exhausted %d retry attempts; most recent cause: %s",
            MaxConsecutiveFailures, cause)
        // Note: markFailed clears Phase/Message/FailureReason in-memory.
        // Persist via the same Status().Update below.
        if err := r.Status().Update(ctx, workspace); err != nil {
            return ctrl.Result{}, err
        }
        metrics.WorkspaceTerminalFailuresTotal.WithLabelValues(string(v1.FailureReasonTooManyFailures)).Inc()
        return ctrl.Result{}, nil
    }

    // Backoff path (US-21.2 unchanged).
    wait := BackoffSchedule[n-1]
    nextRetry := metav1.NewTime(nowFunc().Add(wait))
    workspace.Status.NextRetryAt = &nextRetry
    workspace.Status.Phase = v1.WorkspacePhaseCreating
    workspace.Status.PodIP = ""
    workspace.Status.Endpoint = ""
    if err := r.Status().Update(ctx, workspace); err != nil {
        return ctrl.Result{}, err
    }
    return ctrl.Result{RequeueAfter: wait}, nil
}
```

### Caller signature change

`recoverFromTransientPodLoss` now takes a `cause string` argument so the terminal message has a meaningful explanation. Existing callers (today: `handleActive` at controller.go:280, 303) pass:

- `"pod NotFound"` — from the NotFound branch.
- `"pod not Running (phase=X)"` — from the non-Running branch.
- `"agent unreachable beyond threshold"` — from the health-check branch (controller.go:1015).
- `"pod CrashLoopBackOff"` — from the CrashLoopBackOff branch.

These strings are the building blocks of the terminal-Failed `Status.Message`.

### Why a strict-greater comparison

`n > MaxConsecutiveFailures` means: when ConsecutiveFailures becomes 9 (the 9th increment), terminal fires. This implements the table from the epic:

| Attempt | n value when reached | Action |
|---|---|---|
| 1 | n=1 | backoff 5s |
| 2 | n=2 | backoff 30s |
| ... | | |
| 8 | n=8 | backoff 12h |
| 9 | n=9 (=MaxConsecutiveFailures+1) | terminal Failed |

Off-by-one safety check: the schedule has 8 entries (indices 0-7); `BackoffSchedule[7]` is the last valid index. The cap fires AT 9, so we never read `BackoffSchedule[8]`. ✅

## Test Plan

### Unit tests

**Happy path — cap fires at attempt 9:**
- Workspace `ConsecutiveFailures = 8` (last allowed). Invoke `recoverFromTransientPodLoss(cause="pod NotFound")`.
- Assert: `ConsecutiveFailures = 9`. `Phase = Failed`. `FailureReason = TooManyFailures`. `Message contains "exhausted 8 retry attempts"` and `"pod NotFound"`. `NextRetryAt = nil` (markFailed should clear backoff fields). `r.Status().Update` called once. Returns `ctrl.Result{}` with no requeue.

**Boundary — attempt 8 is still backoff:**
- `ConsecutiveFailures = 7`. Invoke. Assert: `ConsecutiveFailures = 8`. `Phase = Creating`. `NextRetryAt = now + 12h` (last schedule entry). NO terminal-Failed.

**Cap respects strict comparison:**
- Workspace `ConsecutiveFailures = 9` (defensive — should never happen if cap is consistent, but test defense). Invoke. Assert: `ConsecutiveFailures = 10`, > MaxConsecutiveFailures, terminal-Failed fires. Schedule index would be out of bounds but we never reach the schedule code. ✅
- Workspace `ConsecutiveFailures = MaxConsecutiveFailures (8)`. Invoke. Assert: `ConsecutiveFailures = 9`, terminal-Failed fires. ✅

**Post-cap recovery via RestartGeneration:**
- Workspace at terminal Failed (FailureReason=TooManyFailures, ConsecutiveFailures=9, FailureMessage="exhausted 8 retry attempts").
- `Spec.RestartGeneration` bumped to 1. Invoke `handleFailed`.
- Assert: `honorRestartGenerationBump` (US-21.5) handles it. Phase=Pending. ConsecutiveFailures=0. NextRetryAt=nil. RestartCount incremented. ObservedRestartGeneration=1.

**Cause string propagation:**
- Caller passes `cause="agent unreachable beyond threshold"`. Cap fires. Assert: `Message` contains the cause string verbatim.

**Update conflict:**
- Mock `r.Status().Update` returns Conflict. Invoke at cap. Assert: returns the conflict error; status writes are NOT committed; counter does NOT advance further. controller-runtime requeues. ✅

**Edge cases:**
- `ConsecutiveFailures = -1` (illegal). Invoke. Becomes 0. Schedule index 0-1 = -1 → out of bounds → would panic. **Defensive fix:** the function should treat negatives as 0 and proceed with attempt 1. Alternatively, the field validation at US-21.1 should reject negatives. Choose: defensive treatment in the function (safer; doesn't depend on validation infrastructure).
  ```go
  if n < 1 { n = 1 }  // defensive
  ```
- `ConsecutiveFailures = MaxInt32`. Invoke. Becomes overflow → very large negative. Defensive treatment kicks in. Acceptable.
- Empty `cause` string. Cap fires; Message contains "...; most recent cause: " (empty). Document; not a bug — just less helpful diagnostic.

### Integration tests (envtest)

**End-to-end cap firing:**
- Create workspace, simulate 9 transient failures using clock-injection to skip backoff windows.
- After the 9th failure: assert workspace is in terminal Failed with FailureReason=TooManyFailures.
- Bump RestartGeneration. Assert workspace transitions back through Pending → Creating → Active.

**Cap respects backoff:**
- Workspace fails 8 times spaced over 17 hours (real backoff). 9th failure triggers cap. Assert message contains the 9th failure's cause.

**Cap doesn't fire prematurely:**
- Workspace fails 5 times, recovers, runs Active for 6 minutes (US-21.4 reset fires), then fails 5 more times. ConsecutiveFailures should reach 5 (after reset) on the 10th total failure but only 5 since reset. Cap does NOT fire.

### E2E tests

- Real cluster: configure a workspace with a deliberately-broken image (e.g., bad image:tag). Pod fails to pull. ConsecutiveFailures climbs to 9. Cap fires. User sees "Repeatedly failing" badge with FailureReason=TooManyFailures (post-frontend-epic).

### Security tests

- **No secrets in cause string:** review caller sites; cause strings are static phrases or pod-name references. No password/token material.
- **No DoS via cause-string injection:** if cause is derived from external input (e.g., pod's containerStatuses[].State.Waiting.Reason), an attacker controlling the pod could inject long strings. **Mitigation:** the caller should sanitize/truncate cause strings before passing. Cap at e.g. 200 chars. Add to the helper.

### Performance tests

- **Cap path latency:** invocation time < 1ms (single Status().Update). Same as the non-cap path.
- **Metric increment overhead:** prometheus counter increment is < 100ns. Negligible.

### Robustness tests

- **Cap-then-restart-then-cap loop:** operator bumps RestartGeneration → workspace recovers → fails 9 times → cap → operator bumps again. Each cycle is fully bounded; no leak. Verified by 3 cycles of envtest.
- **Cap during a clock skew:** if `nowFunc` returns a stale time when computing the schedule index (irrelevant — we don't compute the index when capping). ✅
- **Status update conflict storm:** simulate persistent conflicts on the cap's Update. controller-runtime requeues; subsequent retries hit conflicts too; eventually one succeeds (after a few seconds). Workspace ends up correctly Failed.

### Negative tests

- **Caller passes nil ctx:** panics. Caller error.
- **markFailed not yet implemented:** if US-21.8 hasn't merged, this story can't merge (dependency declared). The compile error is the test.

## Risk Assessment

- **Medium risk.** Adds the only path that **terminates** a workspace via the new code. A bug here either (a) never terminates (workspace loops forever consuming retries), or (b) terminates too eagerly (workspace fails on first hiccup, defeating the whole epic).
- **Mitigation:** boundary tests at attempt 8 vs 9; envtest end-to-end; manual review of the strict-greater comparison.

## Rollout

Single commit. Deploy. Existing workspaces with `ConsecutiveFailures > MaxConsecutiveFailures` (shouldn't happen — US-21.2's clamp prevented it; defensive) will trigger the cap on their first reconcile post-deploy. Workspaces newly hitting cap conditions get the proper terminal-Failed behavior. RestartGeneration recovery is the escape hatch for all of them.
