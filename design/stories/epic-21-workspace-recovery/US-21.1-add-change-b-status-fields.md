# US-21.1: Add Change B Status Fields; Remove Deprecated Fields

**Epic:** 21 — Workspace State Machine
**Priority:** Critical (blocks all other Epic 21 stories)
**Depends on:** none

## User Story

## Technical Details

### Field semantics

```go
// ConsecutiveFailures counts how many consecutive transient failures have
// occurred without an intervening 5-minute Active stable period. Reset to 0
// by maybeResetTransientCounter when LastSuccessAt is older than
// TransientFailureResetWindow. Capped by MaxConsecutiveFailures = 8;
// reaching 9 triggers terminal Failed with FailureReason=TooManyFailures.
ConsecutiveFailures int32 `json:"consecutiveFailures,omitempty"`

// NextRetryAt is the absolute time at which handleCreating / handleActive
// may resume work after a transient failure. The reconcile loop honors this
// field by returning ctrl.Result{RequeueAfter: time.Until(NextRetryAt)} and
// performing no pod-state operations while time.Now().Before(NextRetryAt).
// nil means "no backoff active; proceed normally."
NextRetryAt *metav1.Time `json:"nextRetryAt,omitempty"`

// LastSuccessAt is the most recent time the controller observed the workspace
// in a fully-Running state (PodIP non-empty, Phase=Active). Used by
// maybeResetTransientCounter to decide when ConsecutiveFailures may reset.
// nil means the workspace has never reached Active.
LastSuccessAt *metav1.Time `json:"lastSuccessAt,omitempty"`
```

### Removal rationale

`TransientFailureCount` is replaced by `ConsecutiveFailures` (semantically identical name, but explicitly named to match the new backoff design). `LastTransientFailureAt` is replaced by `LastSuccessAt` (the inverse — instead of "when did we last fail?" we track "when did we last succeed?", which is what the reset condition actually needs).

Per Epic 21 A7 (no production users), removing the old fields directly is acceptable. No migration code, no dual-write period.

This keeps the build green during the transition.

Specifically:

The TODO stubs ARE the migration path: each subsequent story replaces one stub with real logic. Tests that asserted specific behavior of the old fields are either deleted (if they test removed behavior) or rewritten to match the stub (placeholder tests that assert "stub behavior"; replaced by real tests in subsequent stories).

## Test Plan

### Unit tests

- **Happy path:** new struct can be constructed with new fields; JSON round-trip preserves values; deepcopy preserves values.
  - Test: marshal `WorkspaceStatus{ConsecutiveFailures: 5, NextRetryAt: &t, LastSuccessAt: &t}` → unmarshal → assert equal.
  - Test: deepcopy of a Workspace with these fields populated produces equal-but-not-aliased struct.
- **Default values:** zero-value struct has `ConsecutiveFailures=0`, `NextRetryAt=nil`, `LastSuccessAt=nil`. JSON marshal of zero values omits all three (per `omitempty`).
- **Removed fields:** synthesizing a struct with the old field names fails to compile. Verified by attempting `WorkspaceStatus{TransientFailureCount: 1}` in a test stub that should fail compilation; CI uses `go vet` to catch.
- **Edge case:** `omitempty` correctness — `ConsecutiveFailures=0` produces no JSON key; `ConsecutiveFailures=1` produces `"consecutiveFailures":1`. Same for the timestamp pointers.

### Integration tests

- **CRD apply:** `kubectl apply -f charts/llmsafespace/templates/crd-workspace.yaml` succeeds against a kind cluster; `kubectl get workspace -o yaml` returns objects with the new fields available.
- **Round-trip via K8s API:** a test that creates a Workspace via the typed client with new field values, fetches it back, asserts equality. Catches deepcopy/JSON tag bugs.

### E2E tests

- Not applicable for this story (pure type addition; the wiring lands in US-21.2+).

### Security tests

- **No secrets in new fields:** the three new fields hold timestamps and an int. No string fields that could leak data. No PII. Confirmed by review.
- **No privilege escalation:** field additions don't grant any new capability. RBAC unchanged.

### Performance tests

- **Marshal/unmarshal cost:** benchmark `json.Marshal(workspace)` before and after adding fields. Acceptance: < 5% latency increase. (Realistically negligible — three fields are trivial.)
- **Deepcopy cost:** same benchmark for `workspace.DeepCopy()`. Same threshold.

### Robustness tests

- **Forwards-compatibility:** a controller built from this story can read and write Workspaces with these fields. Subsequent stories' wiring tests verify behavior.
- **Stub behavior is correct:** the temporary stubs in `recoverFromTransientPodLoss` etc. don't break the test suite. Specifically: an integration test that triggers transient pod loss should not panic and should leave the workspace in `Creating` (matching the pre-stub behavior, since cap is disabled).

### Negative tests

- Construct a Workspace with `NextRetryAt = &metav1.Time{}` (zero time, not nil); reconcile; assert no panic. (Defensive; a misconfigured fixture shouldn't crash.)
- Construct a Workspace with `ConsecutiveFailures = -1` (illegal); reconcile; assert it's treated as "no failures" (functional equivalence to 0). Negative values should not occur in practice; the controller doesn't crash on them.

## Risk Assessment

- **Low risk.** Pure additive change to the type system, with deletions of two unused-after-this-story fields. The temporary stubs ensure the build stays green; subsequent stories replace stubs with real logic.
- **Mitigation:** the stubs are commented "TODO US-21.X" with the linking story; CI lint catches any orphaned TODOs (Epic 19's lint job).

## Rollout

Single commit lands in `main`. CI builds new images. Deploy via standard Helm upgrade. CRD reconciliation updates the schema in-cluster. Existing workspaces are unaffected because the fields they don't have just default to zero values, which the new controller treats as "no backoff active, no failure count."
