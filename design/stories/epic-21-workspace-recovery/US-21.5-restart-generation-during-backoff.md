# US-21.5: RestartGeneration Bump During Backoff Clears State

**Epic:** 21 — Workspace State Machine
**Priority:** High
**Depends on:** US-21.1 (status fields), US-21.3 (NextRetryAt honored), Change A (already shipped — worklog 0100)
**Blocks:** none
**Layer:** Controller

## User Story

As a platform operator, when I bump `spec.restartGeneration` on a workspace that is currently mid-backoff (Phase=Creating with `Status.NextRetryAt` set in the future), I need the controller to immediately end the backoff and start a fresh recovery cycle rather than waiting for the scheduled retry time, so that operator-driven recovery overrides automatic backoff and the user sees their explicit retry honored without delay.

## Acceptance Criteria

- [ ] The Change A handler in `handleFailed` (controller.go:84-99) is extended to clear all three Change B fields when transitioning Failed → Pending:
  - `Status.ConsecutiveFailures = 0`
  - `Status.NextRetryAt = nil`
  - `Status.LastSuccessAt = nil`
- [ ] A new equivalent handler is added at the top of `handleCreating` and `handleActive` to detect `spec.RestartGeneration > status.ObservedRestartGeneration` and treat it as an end-backoff signal:
  - Clear all three Change B fields.
  - Increment `Status.RestartCount`.
  - Update `Status.ObservedRestartGeneration = Spec.RestartGeneration`.
  - Transition phase appropriately:
    - From `Creating` (mid-backoff or otherwise): stay in `Creating` but clear backoff so the next reconcile recreates the pod immediately.
    - From `Active` (existing Change A behavior): transition to `Pending` (existing logic at controller.go:258-266).
- [ ] The bump-handler check runs **before** `shouldSkipForBackoff` (US-21.3's guard). Otherwise an operator bump would be blocked by the very backoff window they're trying to override.
- [ ] After clearing, the function returns `ctrl.Result{Requeue: true}, r.Status().Update(ctx, workspace)` so the next reconcile fires immediately with the new state.
- [ ] The bump-handler is idempotent: if the bump is already-observed (`Spec.RestartGeneration == Status.ObservedRestartGeneration`), it does nothing and falls through to normal logic.
- [ ] Tests cover all three call sites (handleFailed, handleCreating, handleActive) for the Change A + Change B field-clearing semantics.

## Technical Details

### Helper function

```go
// honorRestartGenerationBump returns (handled, error). When handled is true,
// the caller MUST return the (ctrl.Result{}, error) from the helper to the
// controller-runtime; backoff and other state has been cleared and a new
// reconcile cycle has been requested. When handled is false, the caller
// proceeds with normal logic for its phase.
func (r *WorkspaceReconciler) honorRestartGenerationBump(
    ctx context.Context,
    workspace *v1.Workspace,
    targetPhase v1.WorkspacePhase,
) (handled bool, result ctrl.Result, err error) {
    if workspace.Spec.RestartGeneration <= workspace.Status.ObservedRestartGeneration {
        return false, ctrl.Result{}, nil
    }

    log := log.FromContext(ctx)
    log.Info("RestartGeneration bumped; clearing backoff and restarting",
        "workspaceID", workspace.Name,
        "fromPhase", workspace.Status.Phase,
        "toPhase", targetPhase,
        "spec.restartGeneration", workspace.Spec.RestartGeneration,
        "status.observedRestartGeneration", workspace.Status.ObservedRestartGeneration)

    workspace.Status.Phase = targetPhase
    workspace.Status.Message = ""
    workspace.Status.PodName = ""
    workspace.Status.PodNamespace = ""
    workspace.Status.PodIP = ""
    workspace.Status.Endpoint = ""
    workspace.Status.ConsecutiveFailures = 0
    workspace.Status.NextRetryAt = nil
    workspace.Status.LastSuccessAt = nil
    workspace.Status.RestartCount++
    workspace.Status.ObservedRestartGeneration = workspace.Spec.RestartGeneration

    if err := r.Status().Update(ctx, workspace); err != nil {
        return true, ctrl.Result{}, err
    }
    return true, ctrl.Result{Requeue: true}, nil
}
```

### Why three call sites

A bump can land while the workspace is in any of {Creating, Active, Failed}. Each existing branch needs the same logic:
- **Failed:** Change A's existing path. Just extend it to clear the new fields.
- **Creating:** new in this story. Specifically catches the bump-during-backoff case the user complained about.
- **Active:** existing Change A path. Same extension.

Suspended/Resuming/Terminating/Terminated don't honor the bump (consistent with existing behavior).

## Test Plan

### Unit tests

**handleFailed (Change A regression):**
- Workspace `Phase=Failed`, `Spec.RestartGeneration=1`, `Status.ObservedRestartGeneration=0`. Other fields populated (Message, PodName, etc.).
- Invoke `handleFailed`. Assert: helper handles it. `Phase=Pending`, all stale fields cleared (including 3 new Change B fields), `RestartCount=1`, `ObservedRestartGeneration=1`. `r.Status().Update` called once. Returns `Requeue: true`.

**handleFailed — bump already observed (idempotent):**
- `Spec.RestartGeneration=1`, `Status.ObservedRestartGeneration=1`. Invoke. Assert: helper returns false; existing Failed handler runs (cleanup secrets, V(1) log, return).

**handleCreating mid-backoff with bump:**
- Workspace `Phase=Creating`, `Status.NextRetryAt=now+1h` (deep into backoff), `ConsecutiveFailures=5`. `Spec.RestartGeneration=1`, `Status.ObservedRestartGeneration=0`.
- Invoke `handleCreating`. Assert: helper handles it. `Phase=Creating` (target=Creating), `NextRetryAt=nil`, `ConsecutiveFailures=0`, `LastSuccessAt=nil`, `RestartCount` incremented, `ObservedRestartGeneration=1`. Helper returns `Requeue: true`. **Critically:** `shouldSkipForBackoff` is NEVER called (the bump-handler runs first).

**handleCreating no bump:**
- `Spec.RestartGeneration=0`, `Status.ObservedRestartGeneration=0`. Invoke. Assert: helper returns false. `shouldSkipForBackoff` runs next per its own logic.

**handleActive (Change A regression):**
- Workspace `Phase=Active`, `Spec.RestartGeneration=2`, `Status.ObservedRestartGeneration=1`. Pod running.
- Invoke. Assert: helper handles it. `Phase=Pending`, all fields cleared, `RestartCount` incremented, `ObservedRestartGeneration=2`. Returns `Requeue: true`.

**Edge cases:**
- `Spec.RestartGeneration=0`, `Status.ObservedRestartGeneration=0` (initial state). Helper returns false (not >). ✅
- `Spec.RestartGeneration=10`, `Status.ObservedRestartGeneration=5` (multi-bump catch-up). Helper handles, jumps observed to 10. RestartCount increments by 1 (not 5) — bumps are coalesced. ✅
- `Spec.RestartGeneration < Status.ObservedRestartGeneration` (illegal — operator wrote a smaller value). Helper returns false (not >). The illegal value persists until the next valid bump. Acceptable.
- Update conflict on `Status().Update`. Helper returns the error; caller propagates; controller-runtime requeues; on retry, the bump may have been already observed by another reconcile path. Idempotent.

**Negative tests:**
- `nil workspace`: panics. Caller error; documented.

### Integration tests (envtest)

**End-to-end Change A from Failed:**
- Create workspace, induce failure → Phase=Failed.
- `kubectl patch workspace … --type=merge -p '{"spec":{"restartGeneration":1}}'`.
- Assert: workspace transitions Failed → Pending → Creating → Active. RestartCount=1. ObservedRestartGeneration=1.

**Bump during backoff:**
- Create workspace, induce 3 transient failures → Phase=Creating with NextRetryAt=now+5min.
- `kubectl patch workspace … --type=merge -p '{"spec":{"restartGeneration":1}}'` (the operator wants to retry NOW, not in 5 min).
- Assert: within 1 reconcile cycle, Phase is still Creating but NextRetryAt is nil and pod is being recreated. Operator's wait time: < 30s, not 5min.

**Bump from Active:**
- Workspace healthy. Bump RestartGeneration. Assert: Phase=Pending → Creating → Active with new pod (RestartCount incremented).

**Bump applied via API endpoint:**
- Use the `POST /api/v1/workspaces/:id/restart` endpoint (Change A's API surface, shipped in worklog 0100).
- Assert: same outcome as direct kubectl bump.

### E2E tests

- Real cluster, frontend "Retry" button (when shipped) calls the restart API. Workspace recovers. User-visible time-to-recovery: < 1 minute (vs. potentially hours during long backoff windows).

### Security tests

- **Authz:** the API endpoint already enforces ownership (worklog 0100). Direct CRD bumps require RBAC permission to update the workspace spec, which is restricted to the controller's service account and the API service.
- **No DoS via bump-spam:** an attacker bumping RestartGeneration repeatedly causes only one full restart cycle; the helper coalesces (RestartCount increments by 1 per reconcile, not per bump). The pod-restart cost is bounded by the controller's reconcile rate.
- **Status update conflict ≠ silent failure:** if the helper's Update fails with Conflict, controller-runtime requeues. The operator's bump is eventually observed. Test: simulate persistent conflict; assert eventual consistency.

### Performance tests

- **Bump-to-recovery latency:** measure time from `kubectl patch` to `Phase=Active` with a fresh pod. Target: < 60s on a healthy cluster.
- **Helper overhead:** the helper adds two field reads (RestartGeneration comparison) on every reconcile of every workspace. Bench: < 100ns per call. Negligible.

### Robustness tests

- **Concurrent bump from API + kubectl:** two writers (frontend's retry button and an operator's kubectl patch) bump the field nearly simultaneously. K8s's optimistic concurrency ensures one wins; the second gets Conflict and either retries (the API service does this) or surfaces the error (kubectl reports the conflict). The controller observes a single bump (the resourceVersion advances once).
- **Bump during a status update:** controller is mid-`Status().Update` when the spec write lands. Spec writes don't conflict with status writes (different subresources). The next reconcile reads the bumped spec and observed Generation = the old value, → bump-handler fires. ✅
- **Controller restart after bump but before observation:** kubectl bumps spec; controller pod dies before reconcile picks it up. Replacement controller starts, reconciles, sees `Spec.RestartGeneration > Status.ObservedRestartGeneration`, fires the helper. ✅

### Negative tests

- **Bump on a Terminated workspace:** workspace is being deleted (DeletionTimestamp set). The reconcile early-returns at the deletion path before any phase handler. Bump is silently ignored — workspace will be deleted regardless. Acceptable.
- **Bump on a Suspended workspace:** the Suspended handler doesn't call `honorRestartGenerationBump`. The bump is observed only when the workspace transitions out of Suspended (e.g., user resumes it). At that point, the bump fires through handleActive's call. **Acceptable** but document that resuming + bumping is one effective restart, not two.

## Risk Assessment

- **Low-medium risk.** The helper is small. The three call sites are well-defined. The biggest risk is the order-of-checks bug: if `shouldSkipForBackoff` runs before `honorRestartGenerationBump`, the operator's bump is masked by the backoff window. The story's acceptance criterion explicitly mandates the correct order; tests verify it.
- **Mitigation:** integration test specifically asserts that bumping during backoff bypasses `shouldSkipForBackoff`.

## Rollout

Single commit. Deploy via Helm. Existing workspaces in Failed phase recover via the existing Change A path (no behavior change). Workspaces newly entering backoff (post-US-21.2 deploy) become recoverable via bump (new behavior in this story).
