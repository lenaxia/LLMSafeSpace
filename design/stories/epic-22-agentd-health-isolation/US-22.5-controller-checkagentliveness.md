# US-22.5: Controller's `checkAgentLiveness` Polls `/v1/healthz`

**Epic:** 22 — agentd Health Endpoint Redesign
**Priority:** Critical (closes the controller-driven termination path that's been killing workspaces)
**Depends on:** US-22.1 (`/v1/healthz` is now process-only)
**Blocks:** US-22.6 (deep-status separation), US-22.7 (kubelet behavior verification)
**Layer:** Controller

## User Story

As the workspace controller, I need to differentiate "is the agentd pod alive?" (a fast, frequent check) from "what sessions does the agent know about?" (a slow, infrequent enrichment). Today's `checkAgentHealth` polls `/v1/statusz` every 15 seconds with a 5-second timeout — but `/v1/statusz` is the expensive deep-introspection endpoint that makes 4 opencode calls. Migrating the liveness check to `/v1/healthz` (which after US-22.1 makes zero opencode calls) eliminates the false-positive failures that today drive workspaces to terminal Failed.

## Acceptance Criteria

- [ ] A new function `checkAgentLiveness(ctx, podIP)` is added to `controller/internal/workspace/controller.go`. It:
  - Builds URL `http://<podIP>:<agentdPort>/v1/healthz`.
  - GETs with a 5-second context deadline (existing `healthHTTPClient.Timeout`).
  - Decodes the JSON response.
  - Returns `(healthy bool, version string, err error)`.
- [ ] The existing `checkAgentHealth` (controller.go:990-1030 area) is renamed to `checkAgentDeepStatus` and its endpoint URL is changed to `/v1/statusz`. **Behavior change:** failures of this function do NOT trigger pod restart — they only cause the deep-status fields to remain stale until the next successful poll. (Pod-restart logic moves to `checkAgentLiveness`'s caller; see US-22.6.)
- [ ] `handleActive` calls `checkAgentLiveness` on a 15-second cadence (using existing `healthCheckInterval`) and `checkAgentDeepStatus` on a longer cadence (proposed: 60 seconds; new constant `deepStatusInterval`).
- [ ] `ConsecutiveHealthFailures` field tracks ONLY `checkAgentLiveness` failures. Failures of `checkAgentDeepStatus` are tracked separately via a new `ConsecutiveDeepStatusFailures` field (or aren't tracked at all if simpler — only `checkAgentLiveness` drives pod restart).
- [ ] `checkAgentLiveness` failure threshold (3 consecutive) triggers pod-restart via existing logic, now flowing through Epic 21's `recoverFromTransientPodLoss` with `reason=FailureReasonTransientPodLoss`.
- [ ] All existing tests for `checkAgentHealth` are migrated to `checkAgentLiveness` semantically; new tests cover the deep-status path's degraded behavior on failure.

## Technical Details

### `checkAgentLiveness`

```go
// checkAgentLiveness probes /v1/healthz to determine whether the
// workspace-agentd process is alive. After US-22.1, /v1/healthz makes
// zero opencode calls and is guaranteed to respond in < 100ms p99
// regardless of opencode's state. Failures of this probe indicate a
// dead agentd process (or network failure), which DOES warrant pod
// restart via Epic 21's transient-recovery path.
func (r *WorkspaceReconciler) checkAgentLiveness(ctx context.Context, podIP string) (healthy bool, version string, err error) {
    endpoint := fmt.Sprintf("http://%s:%d/v1/healthz", podIP, agentd.AgentdPort)
    req, err := http.NewRequestWithContext(ctx, http.MethodGet, endpoint, nil)
    if err != nil {
        return false, "", err
    }
    resp, err := healthHTTPClient.Do(req)
    if err != nil {
        return false, "", err
    }
    defer resp.Body.Close()
    if resp.StatusCode != http.StatusOK {
        return false, "", fmt.Errorf("livez returned %d", resp.StatusCode)
    }
    var body struct {
        Healthy       bool   `json:"healthy"`
        Version       string `json:"version"`
        UptimeSeconds int    `json:"uptime_seconds"`
    }
    if err := json.NewDecoder(resp.Body).Decode(&body); err != nil {
        return false, "", err
    }
    return body.Healthy, body.Version, nil
}
```

### Cadence wiring in `handleActive`

```go
const (
    healthCheckInterval = 15 * time.Second  // unchanged
    deepStatusInterval  = 60 * time.Second  // new
)

// In handleActive, after backoff/restart-gen guards and pod-running checks:

now := nowFunc()

// Liveness probe — fast, frequent.
if shouldRunLivenessProbe(workspace, now) {
    healthy, version, err := r.checkAgentLiveness(ctx, workspace.Status.PodIP)
    workspace.Status.LastHealthCheckAt = &metav1.Time{Time: now}
    if err != nil || !healthy {
        workspace.Status.ConsecutiveHealthFailures++
        if workspace.Status.ConsecutiveHealthFailures >= int32(healthCheckFailureThreshold) {
            // Existing pod-restart path. Now flows through Epic 21's
            // recoverFromTransientPodLoss with TransientPodLoss reason.
            return r.recoverFromTransientPodLoss(ctx, workspace,
                v1.FailureReasonTransientPodLoss,
                fmt.Sprintf("agent unreachable beyond threshold: %v", err))
        }
        return ctrl.Result{RequeueAfter: requeueActive}, r.Status().Update(ctx, workspace)
    }
    workspace.Status.ConsecutiveHealthFailures = 0
    workspace.Status.AgentVersion = version  // populate from livez response
}

// Deep status — slow, infrequent.
if shouldRunDeepStatus(workspace, now) {
    deepResult, err := r.checkAgentDeepStatus(ctx, workspace.Status.PodIP)
    workspace.Status.LastDeepStatusAt = &metav1.Time{Time: now}
    if err == nil {
        // Populate sessions, disk usage, etc. from deepResult.
        workspace.Status.Sessions = deepResult.Sessions
        workspace.Status.DiskUsedBytes = deepResult.DiskUsedBytes
        workspace.Status.DiskTotalBytes = deepResult.DiskTotalBytes
    }
    // Failure: leave fields stale; do NOT increment ConsecutiveHealthFailures.
}

// ... rest of handleActive unchanged ...
```

`shouldRunLivenessProbe` returns true if `LastHealthCheckAt` was more than `healthCheckInterval` ago (or nil). Same pattern for deep status.

### Why two cadences

- **Liveness (15s):** detects dead agentd processes. Fast probe (< 100ms p99 after US-22.1). Failures matter — drive pod restart.
- **Deep status (60s):** populates session list, disk usage. Expensive probe (multiple opencode calls). Failures are tolerable — fields just stay stale.

The 60s cadence reduces opencode load by 4× compared to the old 15s `/v1/statusz` cadence.

### `LastDeepStatusAt` is a new status field

Add to `WorkspaceStatus` in this story (or as part of US-22.6 if cleaner):

```go
LastDeepStatusAt *metav1.Time `json:"lastDeepStatusAt,omitempty"`
```

Used by `shouldRunDeepStatus` to decide when to next poll.

## Test Plan

### Unit tests

**`checkAgentLiveness` happy path:**
- Mock HTTP server returns `{"healthy": true, "version": "v1.0", "uptime_seconds": 100}`. Invoke. Assert `(true, "v1.0", nil)`.

**`checkAgentLiveness` 5xx:**
- Mock returns 503. Invoke. Assert err contains "livez returned 503".

**`checkAgentLiveness` timeout:**
- Mock takes 6s; `healthHTTPClient.Timeout=5s`. Invoke. Assert err is a context-deadline-exceeded.

**`checkAgentLiveness` connection refused:**
- No server listening. Invoke. Assert err is a network-level error.

**`checkAgentLiveness` invalid JSON:**
- Mock returns 200 with body `not json`. Invoke. Assert err mentions JSON decode.

**Liveness probe cadence:**
- Workspace with `LastHealthCheckAt = now - 5s`. `shouldRunLivenessProbe` returns false (within 15s window).
- `LastHealthCheckAt = now - 16s`. Returns true.
- `LastHealthCheckAt = nil` (never run). Returns true.

**Deep-status cadence:**
- Workspace with `LastDeepStatusAt = now - 30s`. `shouldRunDeepStatus` returns false (within 60s window).
- `LastDeepStatusAt = now - 61s`. Returns true.

**Failure threshold flow:**
- Liveness fails 3 times consecutively. ConsecutiveHealthFailures: 1, 2, 3. On the 3rd: `recoverFromTransientPodLoss` is invoked with `reason=TransientPodLoss`.
- Liveness succeeds on attempt 2 (after 1 failure). ConsecutiveHealthFailures resets to 0.

**Deep-status failure does NOT increment liveness counter:**
- Liveness succeeds. Deep status fails. Assert: ConsecutiveHealthFailures unchanged. Assert: Sessions/DiskUsedBytes fields unchanged (stale). Assert: no pod-restart triggered.

**Deep-status success populates fields:**
- Mock returns sessions list and disk usage. Invoke. Assert: workspace.Status.Sessions == expected; workspace.Status.DiskUsedBytes == expected.

### Integration tests (envtest)

**Two-cadence behavior:**
- Boot workspace with mock agentd. Mock /v1/healthz fast (10ms), /v1/statusz slow (5s). Run controller for 5 minutes. Assert: liveness probes run every 15s (= 20 probes); deep-status runs every 60s (= 5 probes). No false-failure restarts.

**Slow opencode does not trigger restart:**
- Mock /v1/statusz to take 30s consistently. /v1/healthz responds in 10ms.
- Run controller for 10 minutes. Assert: ConsecutiveHealthFailures stays 0. No pod restarts. Sessions list goes stale (LastDeepStatusAt grows older) but workspace remains Active.

**Genuine agentd death triggers restart:**
- Kill agentd mock. /v1/healthz starts failing (connection refused). Within 3 × 15s = 45s, ConsecutiveHealthFailures hits 3. Pod is deleted; Epic 21's recoverFromTransientPodLoss schedules retry.

### E2E tests

- Real cluster, real workspace, induce SSE load. Watch controller logs: liveness probes succeed in <100ms; deep-status probes occasionally slow but never trigger restart. Workspace stays Active throughout the SSE load.

### Security tests

- **No new auth:** the new endpoint URL is the existing `/v1/healthz`; same NetworkPolicy applies.
- **Response field validation:** if agentd returns malformed JSON or unexpected fields, decode falls through gracefully (extra fields ignored, missing fields default to zero values).

### Performance tests

- **Liveness probe latency:** target p99 < 100ms (matches `/v1/healthz` SLO from US-22.1).
- **Deep-status probe latency:** unchanged from existing `/v1/statusz` characteristics.
- **Total opencode load reduction:** with 60s deep-status cadence vs 15s old-statusz cadence: 4× reduction in opencode `/global/health` and `/provider`/`/config/providers`/`/session` calls per workspace.

### Robustness tests

- **Network partition between controller and workspace pod:** liveness probes time out. ConsecutiveHealthFailures grows. Pod-restart triggered. Once partition heals, new pod's liveness succeeds.
- **Clock skew between controller and pod:** `LastHealthCheckAt` is the controller's local clock; cadence decisions use the same. Skew doesn't affect correctness.
- **Controller restart:** new controller observes `LastHealthCheckAt` from CRD; cadence resumes correctly.

### Negative tests

- `checkAgentLiveness(ctx, "")` (empty podIP): produces malformed URL; HTTP request fails. Acceptable. Caller should validate podIP is non-empty before invoking.

## Dependencies

- US-22.1 must be deployed (otherwise `/v1/healthz` still calls opencode, defeating the point of this story).

## Risk Assessment

- **Medium-high risk.** Splitting one probe into two introduces complexity and changes the failure semantics. A bug here could either (a) miss real failures (liveness too lenient → workspace stays Active when broken) or (b) trigger restarts on stale-but-OK pods (liveness too strict).
- **Mitigation:** integration tests cover the SSE-load scenario (which is the original incident); chaos test verifies genuine agentd death is still caught.

## Rollout

Single commit. Deploy. Existing workspaces immediately benefit on next reconcile — liveness probes are fast, deep-status is slower. Old `LastHealthCheckAt` semantics are preserved (still tracks liveness probes); new `LastDeepStatusAt` is unset until the first deep-status probe runs.
