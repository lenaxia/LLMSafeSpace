# US-22.2: Eager-Refresh `IsHealthy` Cache

**Epic:** 22 — agentd Health Endpoint Redesign
**Priority:** Critical
**Depends on:** US-22.1 (healthz refactored — establishes the "no opencode in liveness" precedent)
**Blocks:** US-22.3 (readyz consumes the cache)
**Layer:** workspace-agentd

## User Story

As `/v1/readyz` (next story), when I'm asked "is the workspace ready to serve traffic?", I need a recently-fresh answer to "did opencode's `/global/health` recently respond healthy?" without making a synchronous opencode call on the request path. A background goroutine refreshing the answer every 5 seconds, with single-writer ownership of the cache, gives me a near-fresh value at zero per-request cost.

## Acceptance Criteria

- [ ] A new package-level type `healthzCache` holds:
  - `healthy bool` (most recent observation)
  - `version string`
  - `lastRefreshedAt time.Time`
  - `consecutiveFailures int`
  - `lastError string` (for diagnostics)
- [ ] The cache is read by `/v1/readyz` (US-22.3) and ONLY ever written by the refresher goroutine.
- [ ] A goroutine `refreshIsHealthyLoop(ctx, client, cache, log)` runs from agentd boot until process exit:
  - Every `readinessRefreshInterval` (proposed: 5 seconds), call `client.IsHealthy(refreshCtx)` with `refreshCtx` carrying a 4-second deadline (less than the interval to avoid pile-up).
  - On success: update `healthy=true`, `version=…`, `lastRefreshedAt=now`, `consecutiveFailures=0`, `lastError=""`.
  - On failure: increment `consecutiveFailures`, set `lastError=err.String()`, leave `healthy` and `version` unchanged. After `consecutiveFailures >= readinessFailureThreshold` (proposed: 3), set `healthy=false`.
- [ ] Cache writes are atomic (single goroutine; no mutex needed, but use `atomic.Pointer[healthzCache]` for safe reads from `/v1/readyz`).
- [ ] On first invocation (cache uninitialized), `healthy=false` until the first successful refresh. `/v1/readyz` returns 503 in this window.
- [ ] When agentd shuts down, the goroutine exits cleanly (context cancellation).
- [ ] A read accessor `cache.Snapshot() healthzCacheSnapshot` returns a stable copy for `/v1/readyz` to inspect.

## Technical Details

### Cache type

```go
package main // or a sub-package, depending on layout

import (
    "sync/atomic"
    "time"
)

// healthzCache is the in-process cache of opencode's /global/health result,
// refreshed by a single background goroutine. Reads are lock-free via
// atomic pointer; writes are by the refresher goroutine only.
type healthzCache struct {
    snapshot atomic.Pointer[healthzCacheSnapshot]
}

type healthzCacheSnapshot struct {
    Healthy             bool
    Version             string
    LastRefreshedAt     time.Time
    ConsecutiveFailures int
    LastError           string
    Initialized         bool // false until the first successful refresh
}

func newHealthzCache() *healthzCache {
    c := &healthzCache{}
    c.snapshot.Store(&healthzCacheSnapshot{Healthy: false, Initialized: false})
    return c
}

func (c *healthzCache) Snapshot() healthzCacheSnapshot {
    return *c.snapshot.Load()
}
```

### Refresher loop

```go
const (
    readinessRefreshInterval  = 5 * time.Second
    readinessRefreshTimeout   = 4 * time.Second
    readinessFailureThreshold = 3
)

func refreshIsHealthyLoop(ctx context.Context, client *OpenCodeClient, cache *healthzCache, log *zap.Logger) {
    tick := time.NewTicker(readinessRefreshInterval)
    defer tick.Stop()

    // Run an immediate first refresh on boot so /v1/readyz has a meaningful
    // answer faster than waiting for the first tick.
    refreshOnce(ctx, client, cache, log)

    for {
        select {
        case <-ctx.Done():
            return
        case <-tick.C:
            refreshOnce(ctx, client, cache, log)
        }
    }
}

func refreshOnce(ctx context.Context, client *OpenCodeClient, cache *healthzCache, log *zap.Logger) {
    refreshCtx, cancel := context.WithTimeout(ctx, readinessRefreshTimeout)
    defer cancel()

    prev := cache.Snapshot()
    healthy, version, err := client.IsHealthy(refreshCtx)
    next := healthzCacheSnapshot{
        Initialized:     true,
        LastRefreshedAt: time.Now(),
        Version:         prev.Version, // preserve old version on failure
        Healthy:         prev.Healthy, // preserve old healthy until threshold flips
    }
    if err != nil {
        next.ConsecutiveFailures = prev.ConsecutiveFailures + 1
        next.LastError = err.Error()
        if next.ConsecutiveFailures >= readinessFailureThreshold {
            next.Healthy = false
        }
        log.Warn("readyz refresh failed",
            zap.Int("consecutiveFailures", next.ConsecutiveFailures),
            zap.Error(err))
    } else {
        next.Healthy = healthy
        next.Version = version
        next.ConsecutiveFailures = 0
        next.LastError = ""
    }
    cache.snapshot.Store(&next)
}
```

### Lifecycle wiring

In `cmd/workspace-agentd/main.go`'s startup sequence:

```go
healthCache := newHealthzCache()
go refreshIsHealthyLoop(rootCtx, client, healthCache, log)

// healthCache is then passed to the /v1/readyz handler (US-22.3).
```

`rootCtx` is the agentd's process context; cancelled on SIGTERM/SIGINT. The goroutine exits cleanly.

### Why eager refresh, not lazy

Lazy (refresh on first request after TTL expiry) means:
- The first user gets a slow response (the refresh makes the opencode call inline).
- A burst of concurrent requests during a refresh either thunder-herds opencode (without singleflight) or queue up behind one slow refresh.

Eager:
- Constant load on opencode (1 req every 5s) regardless of incoming `/v1/readyz` traffic.
- Per-request cost is a single atomic load.
- Decouples user-visible `/v1/readyz` latency from opencode's response time entirely.

The cost is one persistent goroutine and one HTTP call every 5 seconds. Negligible.

## Test Plan

### Unit tests for `healthzCache`

**Initial state:**
- New cache returns `Snapshot{Initialized: false, Healthy: false}`.

**Single successful refresh:**
- Mock client `IsHealthy` returns `(true, "v1.0", nil)`. Run `refreshOnce`. Snapshot has `Initialized: true, Healthy: true, Version: "v1.0", ConsecutiveFailures: 0, LastError: ""`.

**Single failed refresh:**
- Mock returns error. Run. Snapshot has `Initialized: true, Healthy: false (still — was uninitialized), ConsecutiveFailures: 1, LastError: <err>`. Note: on the very first refresh, `Healthy` was already false (uninitialized), so it stays false.

**Failure threshold preserves prior healthy:**
- Sequence: succeed → succeed → fail → fail. Final snapshot: `Healthy: true (still), ConsecutiveFailures: 2`. `Healthy` does NOT flip to false until `ConsecutiveFailures >= 3`.

**Threshold flip:**
- Sequence: 4 consecutive failures. After the 3rd: `Healthy: false`. ConsecutiveFailures keeps incrementing (4, 5, …) until a success resets.

**Recovery:**
- After a flip-to-unhealthy, a successful refresh sets `Healthy: true, ConsecutiveFailures: 0`. Single recovery is enough.

**Atomic reads:**
- Concurrent `Snapshot()` calls during writes. Run with `-race`. Assert no data race; readers see either the old or new snapshot, never a torn read.

**Edge cases:**
- `IsHealthy` returns `(false, "v1.0", nil)` (opencode itself reports unhealthy). Snapshot has `Healthy: false` immediately, `ConsecutiveFailures: 0` (the refresh succeeded; opencode just said "not healthy"). Different from a network failure.
- `IsHealthy` blocks for 5+ seconds (longer than `readinessRefreshTimeout`). The refresh is cancelled by ctx; treated as a failure.

### Unit tests for `refreshIsHealthyLoop`

**Goroutine lifecycle:**
- Start with cancelable ctx. Verify it runs at least one refresh. Cancel ctx. Verify goroutine exits within 100ms.

**Ticker cadence:**
- Use a fake clock. Advance by 5s. Assert one refresh happened. Advance another 5s. Two total refreshes. Etc.

**Ticker doesn't pile up:**
- Mock IsHealthy to take 6 seconds (slower than the 5s tick). The ticker fires at t=5s while the previous refresh is still running. Assert: no overlap, only one refresh in flight at any time. (Use a single-fire ticker pattern, or document that overlap is acceptable since each refresh has its own ctx.)

  Decision: tick.C buffers one tick; if a refresh takes > 5s, the next refresh fires immediately when the current one returns. Acceptable. Document.

**Initial refresh on boot:**
- Verify `refreshOnce` is called once before entering the ticker loop. Cache is initialized within < 5s of boot (not delayed by the first ticker interval).

### Integration tests (in-process, real http.Server with mock opencode)

**Cache reflects opencode state:**
- Start agentd with mock opencode that returns healthy. Wait > 5s. Snapshot shows healthy.
- Switch mock to unhealthy. Wait > 15s (3 × 5s tick). Snapshot shows unhealthy.
- Switch back to healthy. Wait > 5s. Snapshot shows healthy.

**Goroutine survives opencode failures:**
- Mock returns errors continuously. Run for 60 seconds. Goroutine doesn't crash; cache shows ConsecutiveFailures growing.

**Concurrent reads during writes:**
- 1000 concurrent goroutines call `cache.Snapshot()` while the refresher writes. Run with `-race` for 30 seconds. No race detected, no panic.

### Security tests

- **No secrets in cache:** `Version` is build version, not a credential. `LastError` is opencode's error message — could contain pod IP or the upstream URL, which are non-sensitive within the cluster. Reviewed; no PII or secret material.
- **Single-writer invariant:** if a future contributor adds a second writer, race detector flags it. Test with a deliberately-broken second-writer goroutine; assert race detector catches it.

### Performance tests

- **Snapshot overhead:** benchmark `cache.Snapshot()`. Target: < 50ns (single atomic pointer load).
- **Refresh overhead:** benchmark `refreshOnce` with mock opencode returning instantly. Target: < 1ms (allocation of next snapshot, atomic store).
- **Memory:** the cache is one struct + one `atomic.Pointer`. Constant memory regardless of refresh count.

### Robustness tests

- **opencode mock panics during refresh:** the panic is contained within the goroutine's `IsHealthy` call (assuming `client.IsHealthy` doesn't propagate panics from the network layer — if it did, the goroutine would die and the cache would freeze). Mitigation: wrap `refreshOnce` body in a `defer recover()` if needed.

  Decision: add the defer-recover. Worst case is logging the panic and treating as a failure for that tick. Verified by panic-injection test.

- **Clock skew:** `LastRefreshedAt` uses `time.Now()` which is the monotonic-and-wall-clock pair. Skew on wall-clock doesn't affect the refresh interval (timer-driven) or the cache logic. ✅
- **Process under memory pressure:** the goroutine's allocation per refresh (one snapshot) is small. No pressure-related failures expected.

### Negative tests

- `refreshIsHealthyLoop` invoked with nil client: panics. Caller error.
- Multiple `refreshIsHealthyLoop` goroutines started concurrently (programmer error): each writes the cache; race conditions ensue. Single-writer invariant violated. Mitigation: document; no runtime guard.

## Dependencies

- US-22.1 must be merged (establishes the "agentd doesn't call opencode for liveness" pattern).

## Risk Assessment

- **Medium risk.** Adds a long-lived goroutine and an in-process cache. Bugs in the goroutine could leak (memory) or freeze (cache stale forever). Bugs in the snapshot type could cause race conditions.
- **Mitigation:** explicit single-writer invariant; race-detector tests; goroutine lifecycle tested via ctx cancellation.

## Rollout

Single commit. Deploy. Goroutine starts at agentd boot. First refresh completes within 5s; cache populated. No user-visible behavior change yet — US-22.3 wires the cache into `/v1/readyz`.
