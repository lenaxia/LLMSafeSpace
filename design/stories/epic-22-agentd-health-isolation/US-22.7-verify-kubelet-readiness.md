# US-22.7: Verify Kubelet Readiness Probe Behavior

**Epic:** 22 тАФ agentd Health Endpoint Redesign
**Priority:** High (validation, no production code change)
**Depends on:** US-22.3 (`/v1/readyz` reads from cache)
**Blocks:** none
**Layer:** Test infrastructure (envtest) + (possibly) pod spec tuning

## User Story

As a platform engineer, I need to confirm that the kubelet's readiness probe still correctly marks pods NotReady when opencode is genuinely dead and Ready when opencode is healthy, given the new cached `/v1/readyz` semantics from US-22.3. The cache introduces a delay between opencode-state-change and `/v1/readyz`-response-change; I need to validate that the existing `PeriodSeconds=15, FailureThreshold=5` probe configuration plays correctly with the new 5s-cache-refresh + 3-failure-flip threshold.

## Acceptance Criteria

- [ ] An envtest scenario `TestReadyzKubelet_GenuineOpencodeDeath` boots a workspace pod with the new agentd image. The mock opencode is healthy initially; after 30 seconds, mock kills its `/global/health` endpoint (returns 503 forever).
- [ ] Assert: pod transitions to `Ready=False` within 75-90 seconds of the death (per the calculation: 15s for cache to flip ready=false + up to 5 ├Ч 15s of failed kubelet probes = 75-90s).
- [ ] An envtest scenario `TestReadyzKubelet_SlowOpencodeDoesNotFlap` boots a pod, mocks opencode to take 30s for `/global/health` calls.
- [ ] Assert: pod stays Ready continuously for тЙе 5 minutes. No flapping. (The cache refresh times out per refresh, but the failure-threshold logic in US-22.2 keeps `Healthy: false` only after 3 consecutive failures; even then, `/v1/readyz` returns 503 only briefly because the next refresh succeeds when opencode-mock resumes.) Specifically tests the boundary: a single slow refresh does NOT flip ready=false.
- [ ] An envtest scenario `TestReadyzKubelet_OpencodeRecoveryRestoresReady` exercises the recovery path: after 90 seconds NotReady, the mock restores. Pod transitions back to Ready within 30 seconds.
- [ ] If observed behavior diverges from the expected timing, US-22.7 either (a) tunes the pod spec's probe parameters or (b) tunes US-22.2's cache thresholds. The story is complete only when the observed behavior matches expectations.
- [ ] No production code change unless test reveals a problem.

### Test scaffolding

```go
func TestReadyzKubelet_GenuineOpencodeDeath(t *testing.T) {
    // Boot envtest with workspace pod, agentd image, mock opencode.
    fixture := newEnvtestFixture(t)
    defer fixture.Teardown()

    // Mock opencode: healthy.
    fixture.OpencodeMock.SetHealthy(true)

    // Wait for pod Ready.
    fixture.AwaitPodReady(t, fixture.Pod, 30*time.Second)

    // Kill opencode.
    deathTime := time.Now()
    fixture.OpencodeMock.SetHealthy(false)

    // Assert pod transitions to NotReady within 90s.
    fixture.AwaitPodNotReady(t, fixture.Pod, 90*time.Second)
    elapsed := time.Since(deathTime)
    if elapsed < 60*time.Second || elapsed > 100*time.Second {
        t.Errorf("expected NotReady within 60-100s, got %v", elapsed)
    }
}
```

`AwaitPodReady` and `AwaitPodNotReady` poll `Pod.Status.Conditions[type=Ready]` via the K8s client until the condition matches or the timeout fires.

### Possible tuning levers (if tests reveal issues)

1. **Cache refresh interval** тАФ tighten from 5s to 3s if 90s ceiling is too long.
2. **Cache failure threshold** тАФ tighten from 3 to 2 if NotReady should flip faster.
3. **Kubelet `FailureThreshold`** тАФ tighten from 5 to 3 for faster NotReady detection (cost: more sensitive to single slow probes).
4. **Kubelet `PeriodSeconds`** тАФ tighten from 15s to 10s for finer granularity.
5. **Kubelet `TimeoutSeconds`** тАФ already 3s; could go lower since `/v1/readyz` is now < 200ms.

If any tuning is needed, the story commits update the relevant constants/spec along with the test.

### Performance tests

- Time-to-NotReady on opencode death: target 60-90s.
- Time-to-Ready on opencode recovery: target 15-30s.

### Robustness tests

- **Probe phase alignment:** rerun the test 10 times with random opencode-death moments. Assert: time-to-NotReady is in [60, 100] seconds across all runs (no outliers due to phase alignment).
- **Concurrent traffic during failure:** during the death period, send 1000 requests to the pod's service. Assert: kubelet routes all to the failing pod until NotReady, then routes elsewhere (or fails if no replicas). This is K8s default behavior; we just verify our timing is sensible.
