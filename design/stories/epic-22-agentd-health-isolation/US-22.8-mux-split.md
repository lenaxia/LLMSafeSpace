# US-22.8: Split Mux into Admin Port + User Port

**Epic:** 22 — agentd Health Endpoint Redesign
**Priority:** Medium (in-scope per audit pass 3 — promoted from stretch)
**Depends on:** US-22.1 (process-only `/v1/healthz`), US-22.3 (cached `/v1/readyz`)
**Blocks:** none
**Layer:** workspace-agentd + controller pod-spec

## User Story

As a platform engineer, even after US-22.1–22.7 cap the per-request work of `/v1/healthz` and `/v1/readyz`, I need defense-in-depth at the listener layer: heavy SSE proxy traffic on the same `http.Server` shares an accept queue and goroutine pool with admin/probe traffic. Under bursts (e.g., 1000 concurrent SSE streams or a denial-of-service attempt against the proxy), connection acceptance latency on the shared listener can rise into seconds even when the handler itself is fast. Splitting agentd into two `http.Server` instances on different ports — admin port for healthz/readyz/statusz, user port for proxy/upgrade/reload-secrets — gives the kubelet probes their own accept loop, independent of user-traffic load.

## Acceptance Criteria

- [ ] A new constant `AgentdAdminPort` (proposed: 4098) is defined in `pkg/agentd/types.go` alongside the existing `AgentdPort` (4097).
- [ ] `cmd/workspace-agentd/main.go` startup creates two `http.ServeMux` instances:
  - `userMux` bound to port `AgentdPort` (4097): proxy, upgrade, reload-secrets.
  - `adminMux` bound to port `AgentdAdminPort` (4098): healthz, readyz, statusz.
- [ ] Two `http.Server` goroutines; both started concurrently; both shut down via the same root-context cancellation with a shared 10s graceful-shutdown deadline.
- [ ] `controller/internal/workspace/controller.go` `buildPod` updates pod spec to:
  - Expose both `containerPort: 4097` (name: `agentd`) and `containerPort: 4098` (name: `agentd-admin`).
  - Point `livenessProbe.httpGet.port` at `4098`.
  - Point `readinessProbe.httpGet.port` at `4098`.
- [ ] Controller's `checkAgentHealth` (and post-US-22.5 `checkAgentLiveness`) updates URL:
  - `checkAgentLiveness` (cheap, frequent) targets `http://%s:4098/v1/healthz`.
  - `checkAgentHealth` deep-status poll targets `http://%s:4098/v1/statusz` (statusz lives on admin port — see Rationale).
- [ ] API service's proxy continues to target `AgentdPort` (4097) for proxy/upgrade traffic.
- [ ] NetworkPolicy ingress on workspace pods is updated to permit traffic on both ports from the API service namespace and the kubelet (node-IP traffic).
- [ ] Existing tests pass; new tests verify the two-listener split (see Test Plan).

## Technical Details

### Why two ports

Inside a single `http.Server`, all incoming connections share:
- The TCP `Listen` queue (controlled by the kernel's `SOMAXCONN` and the server's `ListenConfig.Backlog`).
- The accept loop goroutine.
- The connection-handling goroutine pool (one goroutine per connection).

Under enough concurrent connections, the accept loop itself can fall behind, queueing new connections until the kernel's listen backlog overflows. At that point, the kubelet's probe — which initiates a fresh connection per probe — gets the same accept-queue penalty as the heaviest user request. Even an O(1) handler can't compensate for accept-layer delay.

Two listeners on two ports = two independent accept queues + two independent goroutine pools. A burst on the user port consumes only the user listener's resources.

### Why statusz goes on the admin port

`/v1/statusz` is consumed by the controller's deep-status poll (US-22.6). The controller is a privileged caller, not a user-facing path. Putting statusz on the admin port:
- Aligns "admin/probe operations" together.
- Means user-visible API misuse can't slow the controller's view of the workspace.
- Costs nothing — statusz already lives behind a mutex and is rate-limited by the controller's 60s deep-poll cadence.

The user port is exclusively user-facing traffic (chat proxy, terminal upgrade, secrets reload triggered by ConfigMap update).

### Implementation sketch

```go
const (
    AgentdPort      = 4097
    AgentdAdminPort = 4098
)

func main() {
    // ... existing setup (logger, opencode client, sse tracker, cache) ...

    userMux := http.NewServeMux()
    userMux.HandleFunc("/v1/proxy/", proxyHandler)
    userMux.HandleFunc("/upgrade", upgradeHandler)
    userMux.HandleFunc("/v1/reload-secrets", reloadSecretsHandler(...))

    adminMux := http.NewServeMux()
    adminMux.HandleFunc("/v1/healthz", healthzHandler)
    adminMux.HandleFunc("/v1/readyz", readyzHandler)
    adminMux.HandleFunc("/v1/statusz", statuszHandler)

    userSrv := &http.Server{
        Addr:    fmt.Sprintf(":%d", AgentdPort),
        Handler: userMux,
    }
    adminSrv := &http.Server{
        Addr:    fmt.Sprintf(":%d", AgentdAdminPort),
        Handler: adminMux,
    }

    rootCtx, cancel := context.WithCancel(context.Background())

    var wg sync.WaitGroup
    wg.Add(2)
    go func() {
        defer wg.Done()
        if err := userSrv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Error("user server died", zap.Error(err))
            cancel()
        }
    }()
    go func() {
        defer wg.Done()
        if err := adminSrv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Error("admin server died", zap.Error(err))
            cancel()
        }
    }()

    <-rootCtx.Done()
    shutdownCtx, shutdownCancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer shutdownCancel()
    _ = userSrv.Shutdown(shutdownCtx)
    _ = adminSrv.Shutdown(shutdownCtx)
    wg.Wait()
}
```

### NetworkPolicy update

The workspace pod's NetworkPolicy currently allows ingress on `AgentdPort` from the API service namespace + kubelet node IPs. Add the admin port:

```yaml
ingress:
  - from:
      - namespaceSelector:
          matchLabels:
            kubernetes.io/metadata.name: <api-namespace>
    ports:
      - protocol: TCP
        port: 4097
      - protocol: TCP
        port: 4098  # NEW
  - from:
      # kubelet probe traffic from node IPs (existing)
    ports:
      - protocol: TCP
        port: 4098  # kubelet probes admin port now
```

## Test Plan

### Unit tests

**Two listeners coexist:**
- Boot agentd in a test fixture. Send GET to `:4097/v1/proxy/health` (or any user-port path). Assert: handled by user mux.
- Send GET to `:4098/v1/healthz`. Assert: handled by admin mux.
- Send GET to `:4097/v1/healthz`. Assert: 404 (not registered on user mux).
- Send GET to `:4098/v1/proxy/...`. Assert: 404.
- Send GET to `:4098/v1/statusz`. Assert: handled by admin mux.

**Shutdown is clean:**
- Cancel root ctx. Both servers shut down within the 10s deadline. No leaked goroutines (verified by `runtime.NumGoroutine()` before/after).

**One server's panic does not affect the other:**
- Inject a panic in a user-mux handler. Assert: user server recovers (Go's net/http per-request recover) and admin server continues responding.
- More pessimistically: kill `userSrv` goroutine externally. Assert: admin server keeps running until the shared cancel propagates.

### Integration tests

**Burst on user port doesn't affect admin:**
- Open 1000 concurrent SSE connections to `:4097/upgrade` (mock SSE source returns slowly). While the burst is in flight, send 100 GETs to `:4098/v1/healthz`. Assert: all 100 admin requests return < 100ms p99.
- Compare to baseline single-mux setup: same workload, single port — the admin request p99 should be observably higher (this proves the split is doing useful work).

**Statusz on admin port performs as expected:**
- Send 10 concurrent GETs to `:4098/v1/statusz`. Assert: serialize through `cache.mu` (existing behavior preserved); response shape unchanged.

**Admin port honors NetworkPolicy:**
- In envtest with a NetworkPolicy that excludes the admin port (deliberate misconfiguration): kubelet probes fail; pod fails to become Ready. Caught by integration test.
- With correct NetworkPolicy: probes succeed; pod becomes Ready.

### E2E tests

- Real cluster: deploy new agentd image with two ports. Verify pod becomes Ready (kubelet probes :4098). Verify proxy traffic flows on :4097.
- Run a Helm chart upgrade from old (single-port) to new (two-port). Existing workspace pods cycle one at a time; each new pod gets the new spec. Assert: no workspace becomes Failed during the rollout.
- Send chat traffic during the rollout. Assert: no user-visible errors.

### Chaos tests

**SSE-burst probe-starvation:**
- 5000 concurrent SSE clients hitting user port. Continuously poll admin port. Assert: admin port p99 < 100ms, no kubelet liveness failures.

**Admin port DoS:**
- 5000 concurrent admin-port requests. Assert: admin port handles them; user port unaffected (admin handlers are O(1) so no real risk).

**Cross-port saturation:**
- Saturate both ports simultaneously. Assert: each independently degrades; no cross-pollination of latency.

### Performance tests

- p99 latency at `:4098/v1/healthz` < 100ms even under user-port saturation. Acceptance same as US-22.1's SLO.
- p99 latency at `:4098/v1/readyz` < 200ms even under user-port saturation. Acceptance same as US-22.3's SLO.
- No measurable per-request overhead from the split (each port is independent; routing decision is OS-level).

### Security tests

- **NetworkPolicy completeness:** the admin port must be in the ingress allow-list. Test asserts that a pod created without the policy update fails kubelet probes and never becomes Ready.
- **No new authn:** admin port is unauthenticated, same as user port for probe paths. ACL is via NetworkPolicy.
- **Port-spoofing:** an attacker on the cluster network with workspace-namespace access can hit the admin port. Mitigation: NetworkPolicy restricts to API namespace + kubelet IPs (existing posture; same exposure surface as before because statusz was already accessible at the user port pre-split).
- **No information disclosure:** the new port doesn't expose new endpoints; it relocates existing endpoints. Same data, different port number.

### Robustness tests

- **Port already bound (4098 in use):** agentd fails to start. Container exits non-zero; kubelet restarts; eventually CrashLoopBackOff if persistent. Operator-visible. Document.
- **kubelet's probe traffic fluctuates:** the admin port is provisioned for thousands of probes/s; kubelet's actual rate is one per `PeriodSeconds=15` per probe type per pod. No pressure.
- **Mixed-version cluster during rollout:** old workspace pods (single-mux) coexist with new (two-mux). Each pod is internally consistent; cross-pod calls go through the K8s service / direct PodIP — IP is the same, port is the only thing that changes per-pod. Controller's pod-IP-based probes work because the probe-port config lives on the per-pod spec.

### Negative tests

- A handler is registered on the wrong mux at startup (e.g., `/v1/healthz` accidentally on userMux). Caught at code review; runtime check (`adminMux` should have exactly the three admin handlers, `userMux` should have exactly three user handlers) added as a startup self-check + unit test.

## Risk Assessment

- **Medium risk.** Two-server setup is more complex; pod spec change touches networking; shutdown ordering matters; NetworkPolicy must be updated correctly. A NetworkPolicy miss makes pods fail-to-Ready immediately, which is loud and easy to detect but disruptive.
- **Mitigation:** envtest catches NetworkPolicy issues before production. Helm chart change is reviewed alongside code change. Rollout one workspace at a time (existing Epic 18 drain mechanism).

## Rollout

Single Helm release. The chart upgrade rolls workspace pods one at a time. New agentd images bind both ports; new pod spec exposes both; new NetworkPolicy allows both; new probe config targets the admin port. Old pods continue running on single-port until they cycle.

During the rollout window, expect ~30s of workspace unavailability per pod cycle (existing behavior, unchanged). No user-visible regressions.

After landing, observe:
- Kubelet probe latency on admin port (should be < 100ms).
- User-port latency under SSE load (unchanged from pre-split, since this story doesn't change handler logic).
- Controller's `checkAgentLiveness` success rate (should be 100% modulo genuine pod death).
