# US-22.1: Refactor `/v1/healthz` to Process-Only Liveness

**Epic:** 22 — agentd Health Endpoint Redesign
**Priority:** Critical (hot-patch — kubelet liveness probe currently restarts pods under SSE load)
**Depends on:** none
**Blocks:** US-22.2 (cache lives next to it), US-22.3 (readyz refactor reuses pattern), US-22.5 (controller probe migrates here)
**Layer:** workspace-agentd (cmd/workspace-agentd)

## User Story

As the kubelet, when I probe `/v1/healthz` to determine whether the workspace-agentd process is alive, I need a definitive answer in well under my 5-second `TimeoutSeconds`, even when opencode (the in-pod agent process workspace-agentd shares the pod with) is heavily loaded by SSE traffic. The current `/v1/healthz` makes an HTTP call to opencode's `/global/health` on every probe; if opencode is busy, the probe times out and after `FailureThreshold=6` failures (~3 minutes) the kubelet kills the pod, taking down a workspace that wasn't actually broken.

## Acceptance Criteria

- [ ] `/v1/healthz` handler in `cmd/workspace-agentd/main.go:372-388` is rewritten to make ZERO HTTP calls to opencode. The handler answers exclusively from in-process state (build version + uptime).
- [ ] Response body shape unchanged from existing `agentd.HealthzResponse`:
  ```json
  {"healthy": true, "version": "<workspace-agentd build version>", "uptime_seconds": 42}
  ```
  `healthy` is always `true` when the handler executes (a dead process can't respond, by definition). Field exists for forward-compat.
- [ ] Status code: always 200.
- [ ] Latency p99 < 100ms verified by benchmark, even with 1000 concurrent requests and opencode mocked to 30s response time.
- [ ] Mock-call assertions in tests: `client.IsHealthy`, `client.ConnectedProviders`, `client.ConfiguredProviderCount`, `client.ListSessions` are NEVER invoked from the `/v1/healthz` request path.
- [ ] No regression to other endpoints (`/v1/readyz`, `/v1/statusz`, `/v1/reload-secrets`, proxy/upgrade endpoints) — they share the same mux and stay on their existing implementation paths.
- [ ] Code comment explicitly documents the contract: "Process-only liveness. Never call opencode."

## Technical Details

### Rewritten handler

```go
mux.HandleFunc("/v1/healthz", func(w http.ResponseWriter, r *http.Request) {
    // Process-only liveness. Never call opencode. If this handler executes,
    // the workspace-agentd process is alive and able to respond to HTTP.
    // That's the entire signal kubelet's liveness probe needs.
    //
    // Earlier versions of this handler called client.IsHealthy(), which
    // hit opencode's /global/health. That conflated agentd's liveness with
    // opencode's responsiveness, causing kubelet to kill pods under SSE
    // load when opencode was merely slow. See worklog 0096.
    w.Header().Set("Content-Type", "application/json")
    _ = json.NewEncoder(w).Encode(agentd.HealthzResponse{
        Healthy:       true,
        Version:       buildVersion,
        UptimeSeconds: int(time.Since(startedAt).Seconds()),
    })
})
```

### Why no opencode dependency

`/v1/healthz` is the kubelet **liveness** probe — its purpose is "is this container's main process responsive?" not "is everything inside the pod functional?" The right signal is "the HTTP server is accepting connections and the request handler ran to completion." Anything more is conflation.

If a hard requirement emerges to detect agentd-internal deadlocks (e.g., the SSE tracker goroutine has died but the HTTP server is still serving), that's a separate self-check (see Open Question 4 in the epic README), implementable as in-process state without calling out to opencode.

### Files modified

- `cmd/workspace-agentd/main.go` — rewrite `/v1/healthz` handler (~17 lines).
- `cmd/workspace-agentd/main_test.go` (or equivalent) — new tests.

### Files NOT modified

- `pkg/agentd/types.go` — `HealthzResponse` struct unchanged.
- `cmd/workspace-agentd/main.go` other handlers — unchanged.
- The pod spec at `controller/internal/workspace/controller.go:687-695` — kubelet liveness still points at `/v1/healthz`. This story changes what `/v1/healthz` does, not which endpoint kubelet uses.

## Test Plan

### Unit tests (in-process httptest)

**Happy path:**
- Spin up the handler. Send GET. Assert: 200, body has `healthy: true`, `version` matches `buildVersion`, `uptime_seconds` is a non-negative integer.

**No opencode calls (mock-call assertion):**
- Pass a mock `OpenCodeClient` to the handler that fails the test if any of its methods are invoked. Send GET. Assert: handler returns 200 without invoking any client method.

**Concurrent requests:**
- Send 1000 parallel GETs. All return 200. No race detected (run with `-race`).

**Edge cases:**
- Send GET with a body (illegal but possible). Assert: handler ignores the body and returns 200.
- Send POST/PUT/DELETE. Assert: existing `mux.HandleFunc` ignores method, returns 200 for all. (Acceptable for a liveness probe; if stricter behavior wanted, add method check separately. Not in scope for this story.)
- Header `Accept: text/html`. Assert: response is still `application/json` (we don't content-negotiate).

**Unhappy paths:**
- The handler can never return 5xx by design (no internal calls fail). If `json.NewEncoder.Encode` fails (e.g., io error writing to a closed connection), the error is swallowed — the kubelet sees a partial/incomplete response and treats it as a probe failure, which is correct.

### Integration tests (in-process, real http.Server)

**Latency under opencode load:**
- Configure a fake opencode that takes 30s to respond to ANY request (including `/global/health`).
- Send 100 parallel GETs to `/v1/healthz`. Assert: all 100 return in < 100ms (median; p99 < 200ms).
- Assert: opencode receives ZERO requests during this run.

**Coexistence with other endpoints:**
- Concurrent traffic on `/v1/healthz` (1000 req/s) AND `/v1/statusz` (1 req/s, slow). Assert: `/v1/healthz` p99 stays < 100ms; `/v1/statusz` performance is unchanged from baseline.
- Concurrent traffic on `/v1/healthz` AND `/upgrade` (SSE long-lived connection). Assert: `/v1/healthz` p99 stays < 100ms; SSE delivers normally.

### E2E tests (envtest with real workspace pod, fake opencode)

- Boot a workspace pod with the new agentd image. Throttle the in-pod opencode mock to 30s response time.
- Observe kubelet behavior over 10 minutes: assert pod stays Running, no restarts.
- Compare to baseline (pre-US-22.1 image): pod restarts within ~3 minutes due to liveness failures.

### Security tests

- **No sensitive data in response:** `version` field is the build version (e.g., `sha-abc1234`), not a workspace-specific secret. `uptime_seconds` is a non-sensitive integer. Reviewed and confirmed.
- **No authentication required:** the existing handler is unauthenticated (kubelet probes don't carry auth). Story 1 preserves this. Risk: unauthenticated callers can determine the pod is alive and the agentd version. Acceptable — version is already exposed via `/v1/statusz` (with auth, but agentd's pod is reachable only by API service / controller / kubelet on the cluster network).
- **No path-traversal or injection vector:** the handler reads no input. Body, query string, headers (other than `Content-Type` we set) are ignored.
- **Resource exhaustion:** an unauthenticated attacker on the cluster network could spam `/v1/healthz` to consume CPU. Mitigation: 100ms-per-request handler is cheap; even 100k req/s is < 50% of one CPU core. Operational mitigation: NetworkPolicy already restricts ingress to the pod's port (existing).

### Performance tests

- **Single-request latency:** benchmark in-process. Target: p50 < 1ms, p99 < 100ms.
- **Concurrent throughput:** benchmark with 1000 concurrent goroutines hitting the endpoint. Target: ≥ 50,000 req/s sustained.
- **No allocations under hot path beyond `json.NewEncoder.Encode`:** measure with `go test -benchmem`. Acceptable: ≤ 100 allocations/op (the encode is unavoidable; we're not worse than baseline).
- **No goroutine leak:** run 10,000 sequential GETs; assert `runtime.NumGoroutine()` is stable before and after.

### Robustness tests

- **Process under memory pressure:** simulated by allocating 90% of pod memory limit. Assert `/v1/healthz` still returns within 100ms (Go runtime can usually still serve simple HTTP under high heap pressure).
- **Disk full:** the handler writes nothing to disk. No-op test.
- **Network partition between pod and kubelet:** kubelet's TCP probe times out, kubelet treats as failure, eventually restarts pod. The handler itself behaves correctly — failure mode is on the network path, not our code.
- **opencode panicking inside the same pod:** the agentd handler doesn't depend on opencode. `/v1/healthz` continues to return 200. Verified by chaos test: SIGKILL the in-pod opencode mock; agentd's `/v1/healthz` keeps responding until kubelet restarts the pod for OTHER reasons (e.g., container exit if opencode is the main container — irrelevant for agentd's liveness).

### Negative tests

- Handler invoked with a closed `http.ResponseWriter` (caller cancelled). The `json.NewEncoder.Encode` returns an error; we swallow it. Acceptable; kubelet/test fixture sees an EOF.

## Dependencies

- None.

## Risk Assessment

- **Low risk.** Removes external dependency from a hot-path handler. No new code paths; the change is a removal. Worst-case bug: returning the wrong shape would break kubelet's probe parsing — but the `agentd.HealthzResponse` struct is unchanged, so this is structurally impossible.
- **Mitigation:** in-process tests verify exact response shape. envtest verifies kubelet behavior end-to-end.

## Rollout

Single commit. Deploy via Helm. The new agentd image rolls out workspace-by-workspace as pods cycle (controlled by Epic 18's drain mechanism, or natural pod restart). No stale-cache concerns; the change is purely server-side.

Workspaces with old agentd image continue to use the old `/v1/healthz` (still calls opencode). Once their pods cycle to the new image, behavior switches. No mixed-version concerns within a single pod.
