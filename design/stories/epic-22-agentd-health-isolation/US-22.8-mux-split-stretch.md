# US-22.8: (Stretch) Split Mux into Admin Port + User Port

**Epic:** 22 — agentd Health Endpoint Redesign
**Priority:** Low (stretch goal — only ship if metrics show it's needed)
**Depends on:** US-22.1, US-22.2, US-22.3 (the cached/process-only redesign should already eliminate the bottleneck)
**Blocks:** none
**Layer:** workspace-agentd

## User Story

As a platform engineer, IF metrics after US-22.1–22.7 show that `/v1/healthz` or `/v1/readyz` is still occasionally slow under heavy `/upgrade` SSE traffic (suggesting a contention issue at the Go `http.Server` level — accept queue depth, listener-level backpressure, or HTTP/1.1 connection multiplexing), I need to split the agentd mux into two listeners on different ports: one for admin/probe endpoints (cheap, fast), one for proxy/SSE endpoints (heavy, long-lived). Two separate `http.Server` instances bind to the two ports; a single attacker or burst on the heavy port cannot starve the admin port.

## Acceptance Criteria

**This story is conditional.** It ships ONLY if the following triggering conditions are met after US-22.1–22.7 have been deployed:

- [ ] Production metrics over a 7-day observation window show `/v1/healthz` or `/v1/readyz` p99 latency exceeding 500ms during periods of heavy SSE proxy traffic. (Acceptance metric for US-22.1: `/v1/healthz` p99 < 100ms; for US-22.3: `/v1/readyz` p99 < 200ms. Sustained breach for 5+ minutes triggers consideration.)
- [ ] OR an envtest reproduction shows that 1000 concurrent `/upgrade` SSE clients degrade `/v1/healthz` latency on the same agentd to > 500ms.

**If the trigger condition is met, the story ships with:**

- [ ] A new constant `agentdAdminPort` (proposed: 4098) — distinct from `agentd.AgentdPort` (4097).
- [ ] Agentd's startup creates two muxes:
  - `userMux` on port `AgentdPort` — proxy, upgrade, secrets-reload, statusz handlers.
  - `adminMux` on port `agentdAdminPort` — healthz, readyz handlers.
- [ ] Two `http.Server` instances; both started in goroutines; both shut down via the same root context cancellation.
- [ ] Pod spec is updated to expose the new admin port.
- [ ] Kubelet liveness/readiness probes are reconfigured to point at the new admin port.
- [ ] Controller's `checkAgentLiveness` (US-22.5) is updated to use the new admin port.
- [ ] `/v1/statusz` stays on the user port (controller's deep-status poll continues to use `AgentdPort`). This intentionally keeps statusz contended; statusz is supposed to be expensive.

**If the trigger condition is NOT met after the observation window:**

- [ ] Story is **closed without shipping.** A follow-up worklog documents the observed metrics and the decision not to split.

## Technical Details

### Why two ports?

Inside a single `http.Server`, all incoming connections share:
- The TCP `Listen` queue (controlled by the kernel's `SOMAXCONN` and the server's `Listener.Backlog`).
- The accept loop goroutine.
- The connection-handling goroutine pool (one goroutine per connection, but the runtime scheduler can starve admin connections behind heavy ones if CPU is saturated).

Under enough load, even a fast handler can't get a goroutine scheduled in time. Splitting into two listeners on different ports gives the admin endpoints their own accept loop, independent of user-traffic load.

### Why we may NOT need this

After US-22.1 (process-only `/v1/healthz`) and US-22.2/22.3 (cached `/v1/readyz`), the per-request cost of admin endpoints is < 10ms. The admin endpoints don't share `cache.mu` (the legacy `cachedState` mutex) anymore. The bottleneck that drove the original design — `cachedState` mutex held across opencode round-trips — is gone.

We expect the redesign to be sufficient. This story exists in case it isn't.

### Implementation sketch (if shipped)

```go
const (
    AgentdPort      = 4097
    AgentdAdminPort = 4098
)

func main() {
    // ... existing setup ...

    userMux := http.NewServeMux()
    userMux.HandleFunc("/v1/proxy/", proxyHandler)
    userMux.HandleFunc("/upgrade", upgradeHandler)
    userMux.HandleFunc("/v1/reload-secrets", reloadSecretsHandler)
    userMux.HandleFunc("/v1/statusz", statuszHandler)

    adminMux := http.NewServeMux()
    adminMux.HandleFunc("/v1/healthz", healthzHandler)
    adminMux.HandleFunc("/v1/readyz", readyzHandler)

    userSrv := &http.Server{Addr: fmt.Sprintf(":%d", AgentdPort), Handler: userMux}
    adminSrv := &http.Server{Addr: fmt.Sprintf(":%d", AgentdAdminPort), Handler: adminMux}

    go func() {
        if err := userSrv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Fatal("user server died", zap.Error(err))
        }
    }()
    go func() {
        if err := adminSrv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Fatal("admin server died", zap.Error(err))
        }
    }()

    // ... shutdown handler invokes both servers' Shutdown ...
}
```

## Test Plan

### Trigger validation tests (envtest)

Before deciding to ship, run:

- **Slow-opencode + heavy-SSE benchmark:** boot pod with new agentd. Mock opencode to take 30s for `/global/health`. Open 1000 concurrent `/upgrade` SSE connections. Send 100 GETs to `/v1/healthz` and `/v1/readyz`. Measure p99 latency.
- **Acceptance for NOT shipping US-22.8:** p99 < 200ms on both endpoints. If so, close story.
- **Acceptance for shipping US-22.8:** p99 > 500ms on either endpoint. If so, proceed with implementation.

### Unit tests (if shipped)

**Two listeners coexist:**
- Boot agentd. Send GET to `:4097/v1/proxy/...`. Assert: handled.
- Send GET to `:4098/v1/healthz`. Assert: handled.
- Send GET to `:4097/v1/healthz`. Assert: 404 (not registered on user mux).
- Send GET to `:4098/v1/proxy/...`. Assert: 404.

**Shutdown is clean:**
- Cancel root ctx. Both servers shut down. No leaked goroutines.

### Integration tests (if shipped)

**Burst on user port doesn't affect admin:**
- 1000 concurrent SSE clients on `:4097/upgrade`. Send 100 GETs to `:4098/v1/healthz`. Assert: all 100 admin requests return < 100ms p99.

**Admin port respects same NetworkPolicy:**
- Verify the new port is included in the workspace pod's NetworkPolicy ingress allow-list.

### E2E tests (if shipped)

- Real cluster: deploy new agentd image with two ports. Verify pod becomes Ready (kubelet probes :4098). Verify proxy traffic flows on :4097.

### Performance tests (if shipped)

- Confirm the trigger condition is resolved: post-deploy, p99 latencies meet the original SLOs (US-22.1's < 100ms, US-22.3's < 200ms) under realistic SSE load.

### Security tests (if shipped)

- **NetworkPolicy must be updated** to permit the new port. Default-deny ingress means a missed update would silently break the kubelet probes (causing immediate pod restart loop). Test in envtest.
- **No new authn:** admin port is unauthenticated, same as user port for the probe paths. Within-cluster ACL via NetworkPolicy.

### Robustness tests (if shipped)

- One server dies (e.g., panic in handler): does the other still respond? Each server has its own accept loop and goroutine pool; failure of one should not affect the other. Verified by panic-injection test.
- Port conflict on startup (4098 already bound): agentd fails to start. Caller (kubelet) restarts the container. Acceptable; document.

### Negative tests

- N/A.

## Dependencies

- US-22.1, US-22.2, US-22.3 must be deployed first.
- Trigger condition must be met (or anticipated based on benchmark).

## Risk Assessment

- **Medium-high risk** if shipped: two-server setup is more complex; pod spec change touches networking; shutdown ordering matters; NetworkPolicy must be updated correctly.
- **Mitigation:** trigger condition gating ensures we don't ship the complexity unless metrics show it's needed.

## Rollout (if shipped)

Single commit. Deploy via Helm. Pod spec changes cause workspace pods to roll. New agentd images bind both ports; new pod spec exposes both; kubelet probes the new port. No mixed-version concerns within a single pod (the pod and its agentd binary version always match).

Old workspace pods continue to use the single-port setup until they cycle. During the rollout window, kubelet probes pointed at the old port still work.
