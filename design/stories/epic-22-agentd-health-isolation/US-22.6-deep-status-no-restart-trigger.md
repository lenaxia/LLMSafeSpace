# US-22.6: Deep-Status Failures Don't Increment `ConsecutiveHealthFailures`

**Epic:** 22 — agentd Health Endpoint Redesign
**Priority:** High
**Depends on:** US-22.5 (two-cadence wiring)
**Blocks:** none
**Layer:** Controller

## User Story

As the workspace controller, when `checkAgentDeepStatus` (the slow `/v1/statusz` poll) fails — because opencode is busy or the network blipped — I must NOT increment `ConsecutiveHealthFailures` and trigger a pod restart. Deep-status failures are observability gaps, not pod-health signals; the pod's liveness is independently verified by `checkAgentLiveness`. Coupling the two is what produced the worklog 0100 incident.

## Acceptance Criteria

- [ ] `checkAgentDeepStatus` failures (any error: timeout, connection failure, decode error, 5xx response) result in:
  - `LastDeepStatusAt` is updated to the current time (so the next attempt waits the full `deepStatusInterval`).
  - A separate counter `ConsecutiveDeepStatusFailures` is incremented (for observability and metrics).
  - Sessions / DiskUsedBytes / DiskTotalBytes / AgentVersion fields are LEFT UNCHANGED (preserving the last-known values).
  - `ConsecutiveHealthFailures` is NOT touched.
  - No pod-restart logic fires.
- [ ] On `checkAgentDeepStatus` success: `ConsecutiveDeepStatusFailures` resets to 0; fields populated from the response.
- [ ] After N consecutive deep-status failures (proposed: 10, exposing as `MaxConsecutiveDeepStatusFailures` constant), an info-level log is emitted ("deep-status has been failing for N attempts; sessions/disk fields are stale; pod liveness is independently verified"). No pod-restart action.
- [ ] Metric `WorkspaceDeepStatusFailuresTotal{reason}` (counter) increments per failure (reason can be "timeout", "5xx", "decode_error", "connection_refused", or "other").
- [ ] No regression in deep-status's success path — fields populate correctly.

## Technical Details

### Behavior matrix

| Liveness result | Deep-status result | Action |
|---|---|---|
| OK | OK | Update fields; reset both counters. |
| OK | FAIL | Update LastDeepStatusAt; increment DeepStatusFailures; preserve fields; log if N reached. |
| FAIL | OK | Increment ConsecutiveHealthFailures; if threshold → pod restart. (Deep-status result is discarded — pod is being restarted.) |
| FAIL | FAIL | Same as Liveness FAIL row. |

The two probes are evaluated independently each reconcile.

### Code

```go
// In handleActive's deep-status branch (US-22.5 section above):

if shouldRunDeepStatus(workspace, now) {
    deepResult, err := r.checkAgentDeepStatus(ctx, workspace.Status.PodIP)
    workspace.Status.LastDeepStatusAt = &metav1.Time{Time: now}

    if err != nil {
        workspace.Status.ConsecutiveDeepStatusFailures++
        metrics.WorkspaceDeepStatusFailuresTotal.WithLabelValues(classifyError(err)).Inc()

        if workspace.Status.ConsecutiveDeepStatusFailures >= int32(MaxConsecutiveDeepStatusFailures) {
            log.FromContext(ctx).Info(
                "deep-status has been failing repeatedly; status fields stale, pod liveness OK",
                "workspaceID", workspace.Name,
                "consecutive", workspace.Status.ConsecutiveDeepStatusFailures,
                "lastError", err.Error(),
            )
        }
        // Do NOT touch ConsecutiveHealthFailures.
        // Do NOT modify Sessions/Disk* fields.
    } else {
        workspace.Status.ConsecutiveDeepStatusFailures = 0
        workspace.Status.Sessions = deepResult.Sessions
        workspace.Status.DiskUsedBytes = deepResult.DiskUsedBytes
        workspace.Status.DiskTotalBytes = deepResult.DiskTotalBytes
        if deepResult.AgentVersion != "" {
            workspace.Status.AgentVersion = deepResult.AgentVersion
        }
    }
}

// classifyError categorizes a deep-status error for the Prometheus label.
func classifyError(err error) string {
    if errors.Is(err, context.DeadlineExceeded) {
        return "timeout"
    }
    if strings.Contains(err.Error(), "connect: connection refused") {
        return "connection_refused"
    }
    if strings.Contains(err.Error(), "statusz returned ") {
        return "5xx"
    }
    if strings.Contains(err.Error(), "json") {
        return "decode_error"
    }
    return "other"
}
```

### New status field

```go
ConsecutiveDeepStatusFailures int32 `json:"consecutiveDeepStatusFailures,omitempty"`
```

Added to `WorkspaceStatus` in this story.

### New constant

```go
const MaxConsecutiveDeepStatusFailures = 10
```

The 10-failure threshold (vs liveness's 3) reflects the different cost calculus: a deep-status failure costs you stale observability; a liveness failure costs you a hung pod. We tolerate 10 deep-status failures (10 minutes of staleness at 60s cadence) before logging.

## Test Plan

### Unit tests

**Deep-status success:**
- Mock returns valid response. Invoke. Assert: ConsecutiveDeepStatusFailures=0, Sessions populated, ConsecutiveHealthFailures untouched.

**Deep-status timeout:**
- Mock takes longer than ctx deadline. Invoke. Assert: ConsecutiveDeepStatusFailures=1. Sessions unchanged. ConsecutiveHealthFailures unchanged. Metric labeled "timeout" incremented.

**Deep-status 503:**
- Mock returns 503. Invoke. Assert: ConsecutiveDeepStatusFailures=1. Metric labeled "5xx". Sessions unchanged.

**Deep-status connection refused:**
- No server. Invoke. Assert: counter increments. Metric labeled "connection_refused".

**Deep-status decode error:**
- Mock returns 200 with non-JSON body. Invoke. Assert: counter increments. Metric labeled "decode_error".

**Liveness OK + deep-status FAIL doesn't trigger restart:**
- Set up handleActive with liveness mock returning OK and deep-status mock returning err.
- Run 100 reconciles. Assert: ConsecutiveDeepStatusFailures=100; ConsecutiveHealthFailures=0; no pod restart attempted.

**Liveness FAIL takes precedence:**
- Liveness fails 3 times in a row. The 3rd reconcile triggers `recoverFromTransientPodLoss`. Assert: deep-status's result on those reconciles is irrelevant (we don't even reach the deep-status branch since liveness failure short-circuits to recovery).

**Threshold log emission:**
- 10 consecutive deep-status failures. Assert: log line with "deep-status has been failing repeatedly" and the consecutive count. No further action.

**Recovery:**
- After 10 failures, mock starts succeeding. ConsecutiveDeepStatusFailures resets to 0. Fields populated.

### Integration tests (envtest)

- Real-K8s scenario: agentd `/v1/healthz` mocked fast and OK; `/v1/statusz` mocked to fail consistently. Run 5 minutes. Assert: workspace stays Active throughout (no restart). Sessions field stays at whatever was last successfully populated (or empty if never populated).

### E2E tests

- Real cluster: induce SSE load on a workspace; observe `/v1/statusz` slow under load (timeouts), `/v1/healthz` fast. Assert: workspace remains Active; deep-status metric shows non-zero failures; no pod restart.

### Security tests

- N/A — no new attack surface.

### Performance tests

- Failure path is one counter increment, one log line (rare), one metric increment. < 1ms.
- Deep-status's failure case is faster than its success case (we don't decode the response body).

### Robustness tests

- **Liveness and deep-status succeed at different times:** the two probes are independent. Test with liveness probe failing on reconcile N, succeeding on N+1, while deep-status succeeds on N, fails on N+1. Counters track independently. No coupling.
- **Counter overflow:** `int32` max is ~2 billion. At 10 failures/minute, overflow takes 380 years. Not a concern.

### Negative tests

- Deep-status response with extra unknown fields: ignored (Go's encoding/json default).
- Deep-status response with missing required fields: those fields stay at zero values; the populated fields update. Acceptable; document.

## Dependencies

- US-22.5 (two-cadence wiring exists).

## Risk Assessment

- **Low-medium risk.** Decoupling failure semantics is a behavior change, but in the safer direction (stops false-positive restarts). Worst case: deep-status genuine failures are silently tolerated longer than ideal. Mitigation: the threshold log surfaces this in operator logs; metrics expose it for alerting.

## Rollout

Single commit. Deploy. Behavior change takes effect on next reconcile per workspace. Existing workspaces with non-zero ConsecutiveHealthFailures (from old combined counter) are unaffected — that field's semantics are unchanged; only the deep-status path's contribution to it stops.
