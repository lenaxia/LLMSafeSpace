# US-22.3: Refactor `/v1/readyz` to Read from Cache

**Epic:** 22 â€” agentd Health Endpoint Redesign
**Priority:** Critical
**Depends on:** US-22.2 (cache and refresher exist)
**Blocks:** US-22.7 (kubelet-readiness-probe verification)
**Layer:** workspace-agentd

## User Story

As the kubelet's readiness probe, when I probe `/v1/readyz` to determine whether to route service traffic to this pod, I need a fast, fresh-enough answer that doesn't depend on opencode's response time. Today's `/v1/readyz` calls opencode 4 times under a mutex; under SSE load, the probe times out and the kubelet stops sending traffic to a workspace that's actually fine. By reading the cached `IsHealthy` snapshot from US-22.2, I get an answer in < 200ms p99 even when opencode is slow.

## Acceptance Criteria

- [ ] `/v1/readyz` handler in `cmd/workspace-agentd/main.go:390-403` is rewritten to:
  - Read `cache.Snapshot()` (US-22.2's cache).
  - If `!snapshot.Initialized`: return 503 with body `{"ready": false, "reason": "uninitialized"}`. Cache hasn't completed first refresh yet.
  - Else, also fetch a non-cached signal: at least one provider connected. This part keeps a small upstream call but uses the existing `cachedState` path (which already caches providers/sessions for `connectedCacheTTL=15s`). No new cache; reuse.
  - Combine: `ready := snapshot.Healthy && len(connected) > 0`.
  - Return 200 with body `{"ready": true, "providers_connected": [...], "agent_version": "..."}` if ready.
  - Return 503 with body `{"ready": false, "reason": "<diagnostic>", "staleness_seconds": <int>}` if not ready.
- [ ] No call to `client.IsHealthy` in the `/v1/readyz` request path. Mock-call assertion enforces.
- [ ] Latency p99 < 200ms verified by benchmark with opencode throttled to 30s response time.
- [ ] Status code: 200 if ready, 503 otherwise. The kubelet readiness probe interprets 200 as Ready, anything else as NotReady.
- [ ] `staleness_seconds` field is `time.Since(snapshot.LastRefreshedAt).Seconds()` (rounded to int). Useful for diagnostics; non-zero indicates the cache is stale.

## Technical Details

### Rewritten handler

```go
mux.HandleFunc("/v1/readyz", func(w http.ResponseWriter, r *http.Request) {
    w.Header().Set("Content-Type", "application/json")

    snapshot := healthCache.Snapshot()

    if !snapshot.Initialized {
        w.WriteHeader(http.StatusServiceUnavailable)
        _ = json.NewEncoder(w).Encode(map[string]interface{}{
            "ready":  false,
            "reason": "uninitialized â€” cache has not yet completed first refresh",
        })
        return
    }

    // We still want providers info in the response. Use the existing
    // cachedState path (15s TTL on providers/sessions). On cache hit this
    // is in-memory; on cache miss it makes opencode calls but only the
    // refresher goroutine should be primed to handle slow opencode (the
    // first request after a 15s window will be slow). For Story 22.7's
    // bench, this is the worst case; we're OK with it because /v1/readyz
    // is called every 15s anyway.
    connected, _, _ := cachedState(r.Context(), client, cache, sseTracker)

    staleness := int(time.Since(snapshot.LastRefreshedAt).Seconds())
    ready := snapshot.Healthy && len(connected) > 0

    if !ready {
        w.WriteHeader(http.StatusServiceUnavailable)
        _ = json.NewEncoder(w).Encode(map[string]interface{}{
            "ready":             false,
            "reason":            describeNotReady(snapshot, connected),
            "staleness_seconds": staleness,
        })
        return
    }

    _ = json.NewEncoder(w).Encode(map[string]interface{}{
        "ready":               true,
        "providers_connected": connected,
        "agent_version":       snapshot.Version,
        "staleness_seconds":   staleness,
    })
})

func describeNotReady(snap healthzCacheSnapshot, connected []string) string {
    if !snap.Healthy {
        if snap.LastError != "" {
            return fmt.Sprintf("opencode unhealthy: %s", snap.LastError)
        }
        return "opencode reports unhealthy"
    }
    if len(connected) == 0 {
        return "no providers connected"
    }
    return "unknown"
}
```

### Why we still call `cachedState`

`/v1/readyz` reports providers in its response so kubelet (and operators inspecting the endpoint manually) can see what's connected. The existing `cachedState` already caches this for 15s. Per-request cost on cache hit is one mutex lock + map read â€” well under our 200ms budget.

If a future iteration finds this still too slow under contention, US-22.8 (mux split) addresses it. For now, accept the existing semantics.

The CRITICAL change in this story is removing the inline `client.IsHealthy` call (which `cachedState` doesn't make â€” that was a separate call in the old handler). With US-22.2's cache, the IsHealthy answer comes from in-memory state.

## Test Plan

### Unit tests

**Happy path â€” ready:**
- Cache snapshot: `{Initialized: true, Healthy: true, Version: "v1.0", ...}`. cachedState returns `["opencode"]`.
- Send GET. Assert: 200, body has `ready: true, providers_connected: ["opencode"], agent_version: "v1.0", staleness_seconds: <small int>`.

**Uninitialized cache:**
- Cache snapshot: `{Initialized: false, ...}`. Send GET. Assert: 503, body has `ready: false, reason: "uninitialized..."`.

**opencode unhealthy:**
- Snapshot: `{Initialized: true, Healthy: false, LastError: "connection refused", ...}`. Send GET. Assert: 503, body has `ready: false, reason: "opencode unhealthy: connection refused"`.

**No providers:**
- Snapshot: healthy. cachedState returns `[]`. Send GET. Assert: 503, `reason: "no providers connected"`.

**No opencode call (mock-call assertion):**
- Pass mock client that fails the test if `IsHealthy` is invoked. Send GET. Assert: handler does NOT invoke `IsHealthy` (the cache provides the value).

**Edge cases:**
- Snapshot `LastRefreshedAt` in the future (clock skew). `staleness_seconds` could be negative; clamp to 0 for the response. Defensive.
- `cachedState` returns an error (opencode timing out and cache miss). The cache hit path doesn't error; cache miss path could. Handler should treat error as "no providers" â†’ 503. Document.

### Integration tests (in-process, real http.Server, mock opencode)

**Latency under opencode load:**
- Mock opencode takes 30s for any call. Refresher's first call has completed before this test starts, so cache is initialized. cachedState's TTL of 15s; after expiry, it would call opencode (slow). Test sequence:
  - Phase 1: warm cache. Send 1 GET. Assert: 200 in < 200ms.
  - Phase 2: 100 parallel GETs. All within cache TTL. Assert: all 200, all < 200ms.
- Note: this story does NOT eliminate the cachedState slow-cache-miss case. US-22.8 may address it; for now, document that p99 over a long run with opencode slow could spike when the 15s providers cache expires.

**Cache flip after refresh failures:**
- Refresher fails 3 times consecutively (test-controlled). Cache flips to `Healthy: false`. `/v1/readyz` returns 503.
- Refresher succeeds. Cache flips back. `/v1/readyz` returns 200.

**Concurrent requests:**
- 1000 parallel GETs. All return correctly; no data race detected with `-race`.

### E2E tests (envtest)

- Boot pod with new agentd. Mock opencode healthy. `kubectl get pod` shows Ready within ~10s (kubelet probes `/v1/readyz`).
- Throttle opencode to 30s. Wait 60s. Assert: pod stays Ready (cache flips after refresher's threshold, but kubelet's PeriodSeconds=15 + FailureThreshold=5 = ~75s grace).
- Wait 90s with opencode throttled. Assert: pod transitions NotReady. (Per US-22.7's calculation.)
- Restore opencode. Assert: pod returns to Ready within ~30s (one cache refresh + 1 successful kubelet probe).

### Security tests

- **No sensitive data in response:** `agent_version`, `providers_connected` (list of strings like "opencode") â€” no secrets. `LastError` is opencode's error string; could contain its URL but not credentials. Reviewed.
- **Same NetworkPolicy as `/v1/healthz`:** the endpoint is reachable on the cluster network from the kubelet and from within the namespace. Existing NetworkPolicy.

### Performance tests

- **Single-request latency (cache hit):** target p50 < 5ms, p99 < 50ms.
- **Single-request latency (cache miss on cachedState):** target p99 < 1s (one opencode call). Documented limitation; acceptable since cache miss is once per 15s.
- **Concurrent throughput:** 10,000 req/s sustained on cache hit.
- **Memory:** no per-request allocations beyond JSON encoding.

### Robustness tests

- **Refresher crashed (goroutine died â€” shouldn't happen but defensive):** cache freezes. Snapshot's `LastRefreshedAt` ages. `staleness_seconds` reflects this. The endpoint still returns the last-known state. **This is a graceful degradation:** kubelet sees the last-known answer rather than spurious 503s. After a long enough freeze, operators alert on staleness.

  Mitigation: emit a metric `agentd_readyz_cache_staleness_seconds` (gauge); operators alert if > 60s. (Add to Story 22.2 if simpler, or as a follow-up.)
- **opencode oscillating between healthy and unhealthy every 5s:** cache reflects the last refresh. `/v1/readyz` flips with the cache. kubelet probes every 15s; doesn't see every flip. Acceptable.

### Negative tests

- Non-GET method (POST, etc.): existing mux behavior â€” handler runs anyway. Not strict; acceptable for kubelet probes (which use GET).
- Invalid query string: ignored.

## Risk Assessment

- **Medium risk.** Changes a kubelet-driven endpoint that controls service traffic routing. A bug here could either stop traffic to healthy pods (false-NotReady) or route to broken pods (false-Ready).
- **Mitigation:** integration tests verify the cacheâ†”response mapping at unit level; envtest verifies kubelet behavior end-to-end.

## Rollout

Single commit. Deploy via Helm. Once new agentd image rolls out, kubelet starts seeing fast `/v1/readyz` responses. Old pods continue using old behavior until they cycle.
