# US-15.5: Final History Reconciliation on Idle

**Epic:** 15 - Streaming State Resilience & Mid-Stream Reconnect
**Priority:** High
**Depends on:** US-15.4

## User Story

As a user viewing a reconnected streaming session, when the agent finishes processing I need the chat to show the complete, authoritative message history (replacing any partial/streaming content), so that I see the final result without gaps or artifacts.

## Acceptance Criteria

- [ ] When `session.status: idle` arrives (for the current session), a final history fetch is triggered
- [ ] On successful history fetch, `sseStreamParts` is cleared (streaming content replaced by authoritative history)
- [ ] On successful history fetch, `isReconnectMode` is set to `false` (boundary detection gate disabled)
- [ ] On successful history fetch, `knownLivePartIds` is cleared
- [ ] The streaming indicator disappears (handled by US-15.2's idle → `serverBusy=false`)
- [ ] If the history fetch fails, the streaming parts remain visible (don't clear content on error) and a retry is scheduled
- [ ] The reconciliation does NOT cause a visible flicker — the transition from streaming parts to history should be a single React render (batch the state updates)
- [ ] This reconciliation also applies to the NORMAL send flow (not just reconnect) — after every idle, history is the source of truth

## Assumptions

| # | Assumption | Validation |
|---|-----------|-----------|
| A1 | After `session.status: idle`, `GET /session/:id/message` returns the complete final history including the just-finished assistant response | ✅ This is the existing behavior — `useChatStream.send()` already fetches history after idle |
| A2 | The history fetch after idle is fast enough that the user doesn't notice a gap | ✅ Typically <100ms for the proxied GET; messages are already in opencode's SQLite |
| A3 | `queryClient.invalidateQueries(["messages", workspaceId, sessionId])` triggers `useMessageHistory` to refetch | ✅ Standard react-query behavior |
| A4 | React batches state updates within the same event handler (clearing sseStreamParts + updating history in one render) | ✅ React 18+ automatic batching applies to all state updates within the same synchronous scope |

## Technical Details

### Current Idle Handling

The existing flow in `useChatStream.send()`:
```typescript
// Wait for idle SSE (or timeout)
await new Promise<void>((resolve) => { ... });

// Fetch final history
const history = await messagesApi.getHistory(workspaceId, capturedSessionId);
const lastAssistant = [...history].reverse().find((m) => m.role === "assistant");
onComplete(msg);  // Adds to localMessages
```

This works for the send flow but has issues:
1. It only runs inside `send()` — not on reconnect
2. It adds the last assistant message to `localMessages` instead of refreshing the react-query cache
3. `sseStreamParts` is cleared in `handleSend`'s `onComplete` callback, not atomically with the history update

### New Reconciliation Flow

Move the "idle → fetch → reconcile" logic out of `send()` and into the SSE event handler, so it works for both send and reconnect:

```typescript
// In ChatPage.tsx handleSSEEvent:
if (statusEvent.session_id === sessionId && statusEvent.status === "idle") {
  // 1. Notify send() promise (existing behavior)
  notifySessionIdle(statusEvent.session_id);
  
  // 2. Reconcile: fetch authoritative history and clear streaming state
  reconcileOnIdle();
}

async function reconcileOnIdle() {
  try {
    // Invalidate react-query cache → triggers refetch of useMessageHistory
    await queryClient.invalidateQueries({ queryKey: ["messages", workspaceId, sessionId] });
    
    // Clear streaming artifacts (batched with the query update)
    setSseStreamParts([]);
    isReconnectMode.current = false;
    knownLivePartIds.current.clear();
    sentTextRef.current = "";
    activePartTypeRef.current = null;
    currentThinkingIdxRef.current = -1;
    currentTextIdxRef.current = -1;
  } catch {
    // History fetch failed — keep streaming parts visible, retry on next opportunity
    // The react-query retry logic will handle this automatically
  }
}
```

### Eliminating `localMessages`

Currently, `localMessages` is used to show the user's sent message and the final assistant response before history refreshes. With the new flow:

- **User message echo:** Still needed — show immediately on send, before history updates
- **Assistant response:** No longer needed in `localMessages` — history fetch provides it

The `onComplete` callback in `send()` can be simplified or removed. The user message is added to `localMessages` on send; it's cleared when history refreshes (history will contain the real user message).

### Preventing Flicker

The concern: clearing `sseStreamParts` before history arrives causes a frame where neither streaming content nor history is visible.

Solution: `queryClient.invalidateQueries` returns a promise that resolves when the refetch completes. Clear streaming state AFTER the refetch:

```typescript
async function reconcileOnIdle() {
  // Wait for history to actually arrive in the cache
  await queryClient.refetchQueries({ queryKey: ["messages", workspaceId, sessionId] });
  
  // NOW clear streaming state — history is already rendered
  setSseStreamParts([]);
  // ... rest of cleanup
}
```
