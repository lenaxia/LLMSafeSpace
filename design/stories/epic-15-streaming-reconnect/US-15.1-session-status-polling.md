# US-15.1: Session Status Polling on Mount

**Epic:** 15 - Streaming State Resilience & Mid-Stream Reconnect
**Priority:** Critical
**Depends on:** None

## User Story

As a user who refreshes the page while the agent is processing, I need the chat UI to immediately show a "working" indicator, so that I know the agent hasn't crashed and my request is still being handled.

## Acceptance Criteria

- [ ] When `ChatPage` mounts with a valid `workspaceId` + `sessionId`, it checks the session's current status via the workspace status endpoint
- [ ] If the session status is `"busy"`, `streaming` is set to `true` before any SSE events arrive
- [ ] If the session status is `"idle"` (or session not found in status response), `streaming` remains `false`
- [ ] The status check does not block rendering â€” history and UI render immediately; the streaming indicator appears as soon as the status response arrives
- [ ] If the workspace is not `Active` (e.g., `Suspended`, `Pending`), no status check is made
- [ ] The status check is a single GET request, not a polling loop (SSE takes over after mount)
- [ ] No flickering: if the user just sent a message (local `streaming=true`), the status poll does not reset it to `false` and back

## Technical Details

### Current State

```typescript
// useChatStream.ts
const [streaming, setStreaming] = useState(false);
// streaming is ONLY set to true inside send(), and false in the finally{} block
```

The `streaming` state is entirely local to the send flow. There is no external input.

### Implementation

**Option A (Recommended): Derive from existing `useWorkspaceStatus` data**

`ChatPage.tsx` already calls `useWorkspaceStatus(workspaceId)` which returns `WorkspaceStatus`. This response includes `sessions: AgentSessionInfo[]`. We can derive the streaming state from this without any new API calls:

```typescript
// In ChatPage.tsx, after existing useWorkspaceStatus call:
const sessionStatus = status?.sessions?.find(s => s.id === sessionId);
const serverBusy = sessionStatus?.status === "busy";
```

Then pass `serverBusy` to `useChatStream` or use it directly to control the indicator.

**Key constraint:** The local `streaming` state (set by `send()`) must take priority over the server state during the send flow. Otherwise:
1. User clicks send â†’ `streaming=true` (local)
2. Status poll returns (stale) â†’ `streaming=false` (server says idle because the request hasn't hit opencode yet)
3. SSE busy arrives â†’ `streaming=true` (server)

This would cause a flicker. Solution: use a `localStreamingRef` that, when set, suppresses server-derived state for a brief window (until the first SSE `session.status: busy` confirms the server caught up).

### State Priority Logic

```typescript
const effectiveStreaming = localStreaming || (!localStreamingCooldown && serverBusy);
```

Where:
- `localStreaming`: set by `send()`, cleared by idle SSE or timeout
- `localStreamingCooldown`: true for 2s after `send()` returns, prevents stale server state from flickering
- `serverBusy`: derived from workspace status or SSE `session.status` events
