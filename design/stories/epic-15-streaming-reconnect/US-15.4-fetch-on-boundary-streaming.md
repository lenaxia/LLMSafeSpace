# US-15.4: Fetch-on-Boundary Live Streaming

**Epic:** 15 - Streaming State Resilience & Mid-Stream Reconnect
**Priority:** High
**Depends on:** US-15.3

## User Story

As a user who reloaded the page mid-stream, I need new conversation parts (tool calls, new text blocks) to render live after the static history, so that I can follow the agent's progress in real-time from the point I reconnected.

## Acceptance Criteria

- [ ] After history renders (US-15.3), incoming SSE `message.part.updated` events are checked against `historyPartIds`
- [ ] If `part.id` is already in `historyPartIds` ŌåÆ event is ignored (already rendered from history)
- [ ] If `part.id` is NOT in `historyPartIds` ŌåÆ this is a new part; render it live in `sseStreamParts`
- [ ] `message.part.delta` events for parts NOT in `historyPartIds` are rendered live (appended to the streaming part)
- [ ] `message.part.delta` events for parts IN `historyPartIds` are ignored (that part is already fully rendered from history)
- [ ] Tool call parts (`type: "tool"`) with new `callID` values not in history render live with their state transitions
- [ ] The transition from "static history" to "live streaming" is seamless ŌĆö no gap, no duplicate content
- [ ] If no new parts arrive (agent is still working on the same text block that's in history), the streaming indicator shows but no new content appears until the next boundary

## Technical Details

### Boundary Detection Logic

```typescript
function handleStreamEvent(event: OpenCodeEvent) {
  const payload = event.data as Record<string, unknown>;
  const props = payload?.properties as Record<string, unknown>;
  
  if (payload?.type === "message.part.updated") {
    const part = props?.part as Record<string, unknown>;
    const partId = part?.id as string;
    
    if (!partId || historyPartIds.has(partId)) {
      // Already rendered from history ŌĆö ignore
      // But update historyPartIds if this is a snapshot update to an existing part
      return;
    }
    
    // NEW PART ŌĆö start rendering live
    knownLivePartIds.current.add(partId);
    renderLivePart(part);
  }
  
  if (payload?.type === "message.part.delta") {
    const partId = props?.partID as string;
    
    if (!partId || historyPartIds.has(partId)) {
      // Delta for a history part ŌĆö ignore (it's already rendered in full from history)
      return;
    }
    
    if (knownLivePartIds.current.has(partId)) {
      // Delta for a live part ŌĆö append
      appendDelta(partId, props?.delta as string, props?.field as string);
    }
    // Orphan delta (no prior updated event) ŌĆö ignore safely
  }
}
```

### Key Insight: History Parts Are "Frozen"

Once a part ID is in `historyPartIds`, we never update it from SSE. The history fetch gave us the snapshot at that point in time. Even if the part grows (e.g., text is still being appended), we don't update the history-rendered version ŌĆö we wait for the NEXT part to start rendering live.

This means: if the user reloads mid-sentence in a text block, they see the text as it was at history-fetch time (static), and live streaming only begins when a NEW part starts (next tool call, next text block, etc.).

This is the intentional "fetch-on-boundary" tradeoff: simpler code, no merge/dedup within a single part, acceptable UX.

### Integration with Existing `parseStreamEvent`

The existing `parseStreamEvent` in `ChatPage.tsx` handles all the rendering logic (tool state, thinking blocks, text blocks, user echo stripping). The change is to add a **gate** at the top:

```typescript
const parseStreamEvent = useCallback((event: OpenCodeEvent, currentSessionId: string) => {
  // --- NEW: Boundary detection gate ---
  if (isReconnectMode.current) {
    const dominated = isDominatedByHistory(event);
    if (dominated) return; // Skip ŌĆö already in history
  }
  // --- END gate ---
  
  // ... existing parsing logic unchanged ...
}, [historyPartIds]);
```

Where `isReconnectMode` is `true` when the page loaded into a busy session (set in US-15.3, cleared in US-15.5 when idle reconciliation completes).

### State: `knownLivePartIds`

A `useRef<Set<string>>` that tracks part IDs we've started rendering live. This prevents re-rendering a part if we receive multiple `message.part.updated` snapshots for it (opencode sends snapshots on every update, not just the first).

```typescript
const knownLivePartIds = useRef<Set<string>>(new Set());

// Reset when session changes or idle reconciliation happens
useEffect(() => {
  knownLivePartIds.current.clear();
}, [sessionId]);
```
