# US-15.3: History Fetch on Busy Reconnect

**Epic:** 15 - Streaming State Resilience & Mid-Stream Reconnect
**Priority:** High
**Depends on:** US-15.1, US-15.2

## User Story

As a user who refreshes the page while the agent is mid-response, I need to see all the content the agent has produced so far (as static text), so that I have context for what's happening and don't think the agent is starting from scratch.

## Acceptance Criteria

- [ ] When the page mounts and the session is detected as `busy` (via status poll from US-15.1), a history fetch is triggered immediately
- [ ] The fetched history renders as static message bubbles (not streaming parts)
- [ ] If the history includes a partial assistant response (agent still working), it renders as a normal message bubble — no special "incomplete" styling
- [ ] The history fetch does NOT clear or interfere with the streaming indicator (dots still show)
- [ ] If the session is `idle` on mount, normal behavior applies (history loads via existing `useMessageHistory` hook, no special handling)
- [ ] The history fetch uses the same `messagesApi.getHistory()` endpoint as the existing flow
- [ ] Race condition handled: if the session goes idle BEFORE the history fetch completes, the fetch result is still used (it's now the final state)

## Technical Details

### Validation Required (Assumption A1)

Before implementing, we MUST validate whether opencode returns in-progress messages via the history endpoint. Test procedure:

```bash
# 1. Send a long prompt (e.g., "write a 2000 word essay about kubernetes")
curl -X POST http://pod:4096/session/$SID/prompt_async \
  -u opencode:$PW -H "Content-Type: application/json" \
  -d '{"parts":[{"type":"text","text":"Write a 2000 word essay about Kubernetes architecture"}]}'

# 2. Wait 3-5 seconds (agent is mid-stream)

# 3. Fetch history
curl http://pod:4096/session/$SID/message -u opencode:$PW

# 4. Check: is there an assistant message with partial text?
```

**If YES (partial messages returned):** Proceed with this story as designed.
**If NO (only completed messages returned):** The history on reconnect will show all messages UP TO the current response. The current response will only appear after idle. This is still acceptable UX — the user sees prior context and the streaming indicator, just not the partial response text.

### Implementation

The existing `useMessageHistory` hook already fetches history on mount when `workspaceId` and `sessionId` are defined. No new fetch logic is needed for the basic case.

The key change is: **when the session is busy on mount, we need to refetch history periodically or on-demand to capture the growing response.**

**Approach: Single fetch on mount + rely on US-15.4 for live updates**

```typescript
// In ChatPage.tsx:
// The existing useMessageHistory already fetches on mount.
// When session is busy, the history may be partial.
// US-15.4 (fetch-on-boundary) handles rendering new parts live.
// US-15.5 (reconciliation on idle) does the final authoritative fetch.

// No additional periodic polling needed — SSE events drive live updates.
```

The only new behavior in this story is ensuring that:
1. The history fetch happens even when `streaming=true` (currently it does — `useMessageHistory` is independent of streaming state)
2. The rendered history is treated as the "baseline" that US-15.4 builds upon

### Tracking the History Baseline

To support US-15.4's boundary detection, we need to know which `part.id` values are already rendered from history:

```typescript
// Derive the set of known part IDs from the history response
const historyPartIds = useMemo(() => {
  if (!history) return new Set<string>();
  const ids = new Set<string>();
  for (const msg of history) {
    for (const part of msg.parts) {
      if (part.id) ids.add(part.id);
    }
  }
  return ids;
}, [history]);
```

This set is passed to the SSE event handler in US-15.4 to determine whether an incoming `message.part.updated` event is "new" or "already rendered."
