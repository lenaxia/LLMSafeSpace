# US-15.2: SSE-Driven Streaming State

**Epic:** 15 - Streaming State Resilience & Mid-Stream Reconnect
**Priority:** Critical
**Depends on:** US-15.1

## User Story

As a user with the chat page open, I need the streaming indicator to reflect the real-time session state from the server (via SSE), so that the indicator is always accurate regardless of how I arrived at the page.

## Acceptance Criteria

- [ ] When a `session.status` SSE event arrives with `status: "busy"` for the current session, `effectiveStreaming` becomes `true`
- [ ] When a `session.status` SSE event arrives with `status: "idle"` for the current session, `effectiveStreaming` becomes `false`
- [ ] SSE-driven state transitions work independently of the `send()` flow — i.e., if another client triggers a prompt on the same session, this client shows the indicator
- [ ] The `notifySessionIdle` callback (used by `useChatStream` to resolve the send promise) is still called on idle — no regression to the existing send→wait→fetch flow
- [ ] If SSE disconnects and reconnects, the streaming state is re-derived from the next `session.status` event (no stale state persists across reconnects)
- [ ] `session.status` events for OTHER sessions (different `session_id`) are ignored

## Assumptions

| # | Assumption | Validation |
|---|-----------|-----------|
| A1 | The proxy emits `session.status` with `status: "busy"` when `prompt_async` is called | ✅ Verified: `proxy.go:841-846` publishes busy event |
| A2 | The proxy emits `session.status` with `status: "idle"` when opencode's SSE stream reports idle | ✅ Verified: `session_tracker.go` detects idle from opencode's `session.status` event |

| A5 | After SSE reconnect, if the session is busy, a `session.status: busy` event will eventually arrive (or the session will go idle) | ⚠️ Not guaranteed — if the session became busy BEFORE the reconnect and no new status change occurs, no event fires. This is why US-15.1 (status polling) is needed as the baseline. |

## Technical Details

### Current State

The `handleSSEEvent` callback in `ChatPage.tsx` already processes `session.status` events:

```typescript
if (event.type === "session.status") {
  const statusEvent = event as SessionStatusEvent;
  queryClient.invalidateQueries({ queryKey: ["sessions", workspaceId] });
  if (statusEvent.session_id === sessionId && statusEvent.status === "idle") {
    notifySessionIdle(statusEvent.session_id);
  }
}
```

It handles idle (to resolve the send promise) but does NOT set `streaming` state from busy events.

### Implementation

Extend the `session.status` handler to also drive the streaming indicator:

```typescript
if (event.type === "session.status") {
  const statusEvent = event as SessionStatusEvent;
  queryClient.invalidateQueries({ queryKey: ["sessions", workspaceId] });
  
  if (statusEvent.session_id === sessionId) {
    if (statusEvent.status === "idle") {
      notifySessionIdle(statusEvent.session_id);
      setServerBusy(false);  // NEW
    } else if (statusEvent.status === "busy") {
      setServerBusy(true);   // NEW
    }
  }
}
```

The `serverBusy` state feeds into the priority logic from US-15.1:
```typescript
const effectiveStreaming = localStreaming || serverBusy;
```

### Edge Case: SSE Reconnect While Busy

Scenario:
1. Session is busy, user is seeing streaming indicator
2. SSE connection drops (network blip)
3. `useEventStream` reconnects after backoff
4. No `session.status` event fires (session is still busy, no state change)

Result: `serverBusy` remains `true` from before the disconnect (React state persists across the reconnect since the component doesn't unmount). The indicator stays visible. ✅

Scenario:
1. Session is busy, SSE drops
2. Session goes idle WHILE SSE is disconnected
3. SSE reconnects — no idle event fires (it already fired and was missed)

Result: `serverBusy` is stale (`true`). Fix: on SSE reconnect, invalidate the workspace status query to re-poll:

```typescript
// In useEventStream, on successful reconnect:
onReconnect?.();

// In ChatPage:
const handleSSEReconnect = useCallback(() => {
  queryClient.invalidateQueries({ queryKey: ["workspace-status", workspaceId] });
}, [queryClient, workspaceId]);
```

This triggers `useWorkspaceStatus` to refetch, which updates `serverBusy` from the fresh status response.
