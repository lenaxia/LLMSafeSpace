# US-35.7: Tmpfs Credential Paths — Close the PVC-at-Rest Gap

**Status:** Ready to Implement
**Depends On:** Epic 35 (secretless credential injection — shipped)
**Estimated Effort:** ~6 hours

---

## Problem Being Solved

Epic 35 eliminated plaintext credentials in etcd (the `workspace-secrets-<id>` K8s Secret). But `materialize` still writes plaintext to PVC-backed paths. While a workspace is **Suspended** (pod deleted, PVC retained), stale plaintext from the previous boot persists on disk and is recoverable by any principal that can attach the PVC — including a cluster-admin, a Longhorn snapshot consumer, or an etcd backup restore.

Every credential-bearing file written by the running pod falls into one of three groups:

| Group | Files | PVC mount | Read by | Written at runtime? |
|---|---|---|---|---|
| **A — opencode direct** | `agent-config.json` | subPath `tmp` | opencode (via `OPENCODE_CONFIG`) | Yes — every reload |
| **B — agentd env forwarding** | `secrets-env` | subPath `tmp` | entrypoint `source` → opencode child env | Yes — every reload |
| **C — tool-consumed** | `.ssh/*`, `.git-credentials`, `.secrets/*` | subPath `home` | git/ssh/user tools via `$HOME` | Yes — every reload |
| **D — opencode auth store** | `auth.json` | subPath `workspace` (`XDG_DATA_HOME`) | opencode (on boot + dispose) | Yes — StageCredentials + relay injector |

Groups A–C are the gap documented in Epic 35's "Known Gap" section. Group D (auth.json) was identified post-merge: it holds **all provider API keys** (not just the Zen relay token as initially assumed), written via `PUT /auth/:providerID` (StageCredentials) and by the relay injector.

---

## Solution Overview

Redirect all credential-bearing files to a **RAM-backed tmpfs volume** (`sandbox-runtime`) that is wiped when the pod dies — regardless of how it dies (graceful SIGTERM, OOM SIGKILL, force-eviction).

Two mechanisms, by group:

**Direct redirect (Groups A + B):** change the path constants from PVC-backed `/tmp/*` to tmpfs-backed `/sandbox-runtime/*`. Both paths are env-var-indirected (`OPENCODE_CONFIG`, agentd path config) — opencode and the entrypoint follow env vars, not hardcoded locations.

**Symlink farm (Groups C + D):** the bytes must live at `$HOME`-relative or `XDG_DATA_HOME`-relative paths because that's where git/ssh/opencode resolve them. The init container creates symlinks from the PVC paths to tmpfs targets. Writes transparently follow the symlink to RAM. On pod death, the PVC retains only dangling symlink inodes — no plaintext bytes.

**Why not delete-on-suspend?** A pre-stop hook or controller-side delete is **SIGKILL-unsafe**: OOM kills, force-evictions, and node failures bypass graceful shutdown, leaving auth.json on the PVC. The tmpfs approach is unconditionally safe — RAM is wiped when the pod's cgroup is destroyed, regardless of the termination signal.

## Volume Layout (Post-US-35.7)

| Volume | Type | Mount | Mode | Contents |
|---|---|---|---|---|
| `sandbox-cfg` | emptyDir (Memory, 32Mi) | `/sandbox-cfg` (main: RO, init: RW) | input | `secrets.json`, `workspace-config.json`, `password` (from bootstrap — unchanged by this story) |
| `sandbox-runtime` | emptyDir (Memory, 96Mi) | `/sandbox-runtime` (main: RW, init: RW) | output | `agent-config.json`, `secrets-env`, `rt/ssh/*`, `rt/git-credentials`, `rt/secrets/*`, `rt/auth.json` |
| `workspace` | PVC | `/workspace`, `/home/sandbox`, `/tmp` (subPaths) | persistent | user data, session DB, **symlinks only** for credential paths |

**Memory budget:** 32Mi + 96Mi = 128Mi tmpfs. Counts against pod memory limit (cgroup). Pod memory request defaults to 512Mi; 128Mi tmpfs uses 25% of that. Acceptable — credential files are small (KB range); the 96Mi `sandbox-runtime` budget is 20x the realistic p100 to absorb model-list caching during `EnrichProviders`.

### US-35.7.2: Redirect Direct Paths — agent-config.json + secrets-env (agentd + entrypoint)

**Goal:** Move `AgentConfigPath` and `SecretsEnvPath` from PVC-backed `/tmp/*` to tmpfs-backed `/sandbox-runtime/*`.

**Files:**
- `pkg/agentd/types.go` — update path constants:
  ```go
  AgentConfigPath = "/sandbox-runtime/agent-config.json"
  SecretsEnvPath  = "/sandbox-runtime/secrets-env"
  ```
- `runtimes/base/tools/entrypoints/entrypoint-opencode.sh` — update `OPENCODE_CONFIG` and the `source` path:
  ```sh
  export OPENCODE_CONFIG=/sandbox-runtime/agent-config.json
  # ...
  if [[ -f /sandbox-runtime/secrets-env ]]; then
      source /sandbox-runtime/secrets-env
  fi
  ```
- Remove the vestigial `/sandbox-cfg/env` source block (nothing writes that file)

**Lifecycle impact: NONE.**
- Boot: `materialize` subcommand writes agent-config.json to the new path (follows `AgentConfigPath` constant). opencode reads it via `OPENCODE_CONFIG` env var.
- Reload: `AgentConfigWriter.Rebuild()` writes to the same path. `Materializer` writes secrets-env to the same path.
- Dispose: opencode re-reads agent-config.json from the path it resolved at startup (the env var). Works identically.
- `/v1/reload-secrets` receives secrets via HTTP body — never reads the file. No change.

**Acceptance criteria:**
- `AgentConfigPath` and `SecretsEnvPath` point to `/sandbox-runtime/*`
- `entrypoint-opencode.sh` references the new paths
- `OPENCODE_CONFIG` env var matches the new `AgentConfigPath`
- Vestigial `/sandbox-cfg/env` source block removed
- `materialize` boot path writes to tmpfs
- `/v1/reload-secrets` runtime path writes to tmpfs
- opencode boots and serves LLM calls with credentials from tmpfs
- `TestRunMaterializeCommand_*` tests pass with updated paths

### US-35.7.3: Symlink Farm for $HOME-Relative Paths (Controller + agentd)

**Goal:** Redirect SSH keys, git credentials, secret-files, and auth.json to tmpfs via symlinks. The PVC stores only symlink inodes; plaintext bytes live in RAM.

**Mechanism:**

The credential-setup init container creates symlinks BEFORE agentd/opencode start:

```sh
# Group C: $HOME-relative credential paths → tmpfs
mkdir -p /sandbox-runtime/rt/ssh /sandbox-runtime/rt/secrets
touch /sandbox-runtime/rt/git-credentials /sandbox-runtime/rt/auth.json
chmod 700 /sandbox-runtime/rt/ssh /sandbox-runtime/rt/secrets
chmod 600 /sandbox-runtime/rt/git-credentials /sandbox-runtime/rt/auth.json

ln -sfn /sandbox-runtime/rt/ssh          /home/sandbox/.ssh
ln -sfn /sandbox-runtime/rt/secrets      /home/sandbox/.secrets
ln -sfn /sandbox-runtime/rt/git-credentials /home/sandbox/.git-credentials

# Group D: auth.json → tmpfs
mkdir -p /workspace/.local/opencode
ln -sfn /sandbox-runtime/rt/auth.json /workspace/.local/opencode/auth.json
```

The `Materializer.reset()` already does `RemoveAll(SSHDir)` + `MkdirAll` on every cycle — it must be updated to recreate the symlinks after reset, or to operate on the tmpfs targets directly. The cleanest approach: update `DefaultPaths` so the Materializer writes directly to `/sandbox-runtime/rt/*`, and the symlinks (created once by the init container) point there. `reset()` then cleans the tmpfs targets, not the symlink farm.

**Files:**
- `controller/internal/workspace/pod_builder.go` — extend credential-setup init script to create the symlink farm
- `pkg/agentd/secrets/secrets.go` — update `DefaultPaths`:
  ```go
  SecretsBaseDir:  "/sandbox-runtime/rt/secrets",
  SSHDir:          "/sandbox-runtime/rt/ssh",
  GitCredsPath:    "/sandbox-runtime/rt/git-credentials",
  ```
- `cmd/workspace-agentd/main.go` — update `authJSONPath` to `/sandbox-runtime/rt/auth.json` (or leave resolution from symlink — opencode resolves via `XDG_DATA_HOME` which follows the symlink transparently)

**Why symlinks survive `reset()`:**

`Materializer.reset()` currently does:
```go
m.FS.RemoveAll(m.Paths.SecretsBaseDir)  // wipes /sandbox-runtime/rt/secrets
m.FS.MkdirAll(m.Paths.SecretsBaseDir)   // recreates it
m.FS.RemoveAll(m.Paths.SSHDir)          // wipes /sandbox-runtime/rt/ssh
m.FS.MkdirAll(m.Paths.SSHDir)           // recreates it
m.FS.Remove(m.Paths.GitCredsPath)       // removes file
m.FS.Remove(m.Paths.AgentConfigPath)    // removes file
m.FS.Remove(m.Paths.SecretsEnvPath)     // removes file
```

With paths pointing at tmpfs targets, `reset()` wipes and recreates the tmpfs directories. The PVC-side symlinks (`/home/sandbox/.ssh → /sandbox-runtime/rt/ssh`) are NOT touched by reset — they point at the tmpfs path, which `MkdirAll` recreates. The symlinks remain valid after reset.

**Lifecycle impact: NONE.**
- git/ssh read from `$HOME/.ssh` → follows symlink → reads tmpfs. No behavior change.
- opencode reads `XDG_DATA_HOME/opencode/auth.json` → follows symlink → reads tmpfs. No behavior change.
- On dispose: opencode re-reads auth.json through the symlink → reads tmpfs. No change.
- `StageCredentials` (PUT /auth/:providerID) writes through the symlink → lands in tmpfs.

**SIGKILL safety:**
- Pod dies → cgroup destroyed → tmpfs wiped → PVC retains only dangling symlinks.
- No pre-stop hook needed. No controller-side delete needed.
- Covers: graceful suspend, OOM kill, force-eviction, node failure.

**Acceptance criteria:**
- `/home/sandbox/.ssh`, `/home/sandbox/.secrets`, `/home/sandbox/.git-credentials` are symlinks to `/sandbox-runtime/rt/*`
- `/workspace/.local/opencode/auth.json` is a symlink to `/sandbox-runtime/rt/auth.json`
- `Materializer.reset()` operates on tmpfs targets, not the symlinks
- git clone with SSH key credential works
- git clone with HTTPS credential works
- opencode boots and LLM calls succeed (auth.json resolves through symlink)
- After SIGKILL + resume: no plaintext credential files on PVC (verify by mounting PVC to a debug pod and listing)

**Tests:**
- `TestSandboxPod_SymlinkFarm_*` — init script creates correct symlinks
- `TestMaterializer_ResetPreservesSymlinks` — reset() cleans tmpfs targets, not the symlink inodes
- Integration: SSH key credential → git operation succeeds
- Integration: auth.json through symlink → opencode LLM call succeeds

## Assumptions (To Validate)

1. **opencode follows symlinks for auth.json writes.** Linux `open(2)` transparently follows symlinks for `O_WRONLY|O_CREAT`. opencode's `PUT /auth/:providerID` handler writes via standard file I/O. **Validate:** cluster test — configure a provider, verify auth.json on tmpfs has the key.
2. **opencode follows symlinks for auth.json reads on dispose.** Same `open(2)` semantics. **Validate:** cluster test — dispose instance, verify provider still works.
3. **`Materializer.reset()` does not remove the symlink inodes.** With paths pointing at tmpfs targets, reset operates on `/sandbox-runtime/rt/*`, not on `/home/sandbox/.ssh` etc. The symlinks are created once by the init container and never touched by reset. **Validate:** unit test — reset, verify symlinks still resolve.
4. **git/ssh follow `$HOME/.ssh` symlinks.** Standard POSIX behavior — git and ssh resolve `$HOME` then follow the path, which hits the symlink. **Validate:** integration test — SSH key credential → git clone.
5. **`sandbox-runtime` 96Mi is sufficient.** Largest payload: agent-config.json with 50 providers + enriched model lists (~500KB). 96Mi is 200x headroom. **Validate:** monitor `sandbox-runtime` usage on a high-provider workspace; adjust if needed.
6. **auth.json is fully repopulated on resume.** At boot, `materialize` writes agent-config.json (has all provider keys in `options.apiKey`). The relay injector writes the relay entry to auth.json. opencode reads provider keys from agent-config.json at boot — auth.json is not needed until a `StageCredentials` reload or dispose. **Validate:** cluster test — resume a suspended workspace, verify LLM calls work before any manual reload.

## Non-Requirements (Explicitly Out of Scope)

| Item | Rationale |
|---|---|
| Per-workspace LLM proxy | Does not meaningfully hide keys from same-pod RCE (process memory, localhost interception). Complexity not justified for the marginal defense. |
| Redirect `XDG_DATA_HOME` to tmpfs | Would move opencode's session DB and cache to RAM — regressions session persistence. The auth.json symlink achieves the same goal without the regression. |
| Runtime RCE key-hiding | Fundamental constraint: the process that uses a key can always extract it from its own memory. Out of scope for at-rest zero-knowledge. |
| Encrypt auth.json at rest on PVC | Adds a per-pod decryption key (where does that live?), complexity for no net gain over tmpfs. |
| `/sandbox-cfg/env` | Vestigial — nothing writes it. Source block removed in US-35.7.2. |

## Adversarial Assessment

### What if opencode doesn't follow the auth.json symlink?

It will. Linux `open(2)` follows symlinks by default for all standard file operations (read, write, create). opencode is a Node.js process using `fs.readFile`/`fs.writeFile`, which call `open(2)`. There is no `O_NOFOLLOW` in opencode's auth file handling (it's not a security-sensitive path from opencode's perspective). **Validated assumption** — but the cluster test in US-35.7.3 confirms it empirically.

### What if the init container fails to create symlinks?

The credential-setup init container exits non-zero, the pod enters `CreateContainerError` or `Init:Error`, and the controller's recovery path handles it (backoff → retry). No silent degradation — the symlinks either exist or the pod doesn't boot. This is strictly better than the current design where a missing PVC subPath is a silent `subPath not found` error.

### What about PVC backup tools that follow symlinks?

Longhorn snapshots copy block-level PVC contents. A symlink inode copies as a symlink (a few bytes of path data), not the target. The tmpfs target is not on the PVC and is not snapshotted. Restoring a snapshot yields dangling symlinks — no plaintext. Tools that explicitly dereference symlinks during backup (e.g., `rsync -L`) would copy the tmpfs content, but only while the pod is running and the target exists — at which point the workspace is Active and the zero-knowledge claim ("only decrypted for active workspaces") holds.

### Does this change the `ReadOnlyRootFilesystem: true` invariant?

No. `ReadOnlyRootFilesystem` prevents writes to the container's overlay filesystem (the image layers). Tmpfs volumes are explicitly mounted writable paths, independent of the root filesystem. The existing G17 hardening is preserved.

### Does the symlink farm interfere with user-managed SSH keys?

`Materializer.reset()` already does `RemoveAll($HOME/.ssh)` on every materialize cycle — user-managed keys in `~/.ssh` are already not survivable across reloads. This is existing behavior, not a regression. Users who want persistent SSH keys should store them as `secret-file` credentials in the LLMSafeSpaces secrets store, not directly on the PVC.
