## Why Sessions Are Stuck

When opencode is killed mid-task (OOM or SIGTERM), the session remains with `status: "busy"` in SQLite. The API checks this status and returns "server busy; try again" to prevent concurrent LLM calls.

**Normal recovery path:** When the workspace restarts, `workspace-agentd` runs `abortStaleSessions()` which calls `POST /session/:id/abort` on each session to reset them to idle.

**Problem in this case:** The workspaces were deleted after the incidents, so:
- No pod exists to receive the abort request
- Session state persists in the API database
- UI continues to show "busy" error

### Option 2: Direct Database Update
**How it works:** Manually update the session status in the database.

**Location:** Sessions are stored in **opencode's SQLite database**, NOT in the API's PostgreSQL database. The SQLite file is located at `/home/workspace/.opencode/state.db` inside the workspace pod.

**Problem:** The workspace pods no longer exist, so we cannot access the SQLite files.

**Verdict:** ❌ Not feasible - SQLite files are gone with the pods

---

### Option 3: Wait for Session Expiry
**How it works:** Sessions may have a TTL in the API layer that automatically cleans them up after N days.

```bash
# Check if sessions have expiry logic

```

**Verdict:** ⚠️ Unknown - need to check if automatic cleanup exists

### Option 4: API-Level Session Abort (Best Option)
**How it works:** Add an admin endpoint to force-abort sessions without forwarding to opencode.

**Implementation:**
```go
// api/internal/handlers/admin.go
func (h *AdminHandler) ForceAbortSession(c *gin.Context) {
    sessionID := c.Param("sessionId")
    
    // Option A: Mark session as aborted in API database
    // This assumes session status is tracked in PostgreSQL
    
    // Option B: Delete the session record entirely
    // Sessions are transient - deleting is safe
    
    c.JSON(200, gin.H{"aborted": true})
}
```

**Endpoint:** `POST /api/v1/admin/sessions/:sessionId/force-abort`

**Verdict:** ✅ Best option - requires small code change

## Epic 44 Prevention

These stuck sessions are **exactly the problem Epic 44 solves:**

1. **US-44.1 (Terminal SSE Events):** Would have shown "Agent died due to OOM" or "Agent restarting" in the UI
2. **US-44.2 (Session-Aware Restart):** Would have prevented the unsafe restart in Incident B
3. **US-44.4 (OOM Detection):** Would have written recovery marker for automatic cleanup

After Epic 44 ships, stuck sessions will be impossible because:
- Sessions are aborted automatically on every restart (already implemented in `stale_sessions.go`)
- User sees clear error messages when agent dies
- Restarts only happen when sessions are idle (no data loss)
