# US-46.7: Define Service-Shaped Interfaces Where ≥2 Consumers Exist

**Epic:** 46 — Codebase Debt Audit & Abstraction Foundation
**Effort:** Medium (2d)
**ROI:** High
**Depends on:** US-46.4 (DomainError type — preferred first so interfaces can reference it)
**Status:** Done (2026-06-19, worklog 0373)

---

## Problem

The codebase has **30 `*Service` structs and zero shared `Service` interfaces**. Every consumer depends on the concrete struct. Two parallel mock trees (`mocks/` at root + `api/internal/mocks/`) exist because each consumer hand-writes its own mock — there is no contract, only convention.

This is not a violation in itself (Rule 4 forbids speculative abstractions), but several services have **multiple consumers that today duplicate mock infrastructure**:

- `database.Service` (1162 lines) is consumed by 8+ handlers, each defining its own minimal interface (`auditStore`, `invitationStore`, `policyStore`, `pendingOrgStore`, `orgStore`, `orgBindingAndAutoApplyStore`, etc.). These should be unified or at least co-located.
- `metering.Service` is consumed by handlers and middleware; a `metering.UsageReporter` interface already exists but is separate from the concrete service consumers.
- `settings.InstanceService` is consumed by 4 sites with no shared interface.

## Scope

**In:**
- For each service with ≥2 consumers, define a **consumer-shaped interface** (Interface Segregation Principle) co-located with the consumer or in a shared `interfaces` package.
- Specifically:
  - Extract a `WorkspaceMetadataStore` interface from the existing 8+ handler-local `*Store` interfaces that wrap `database.Service` workspace methods. One canonical interface in `api/internal/interfaces/`, satisfied by `*database.Service`.
  - Extract a `SettingsReader` interface (3–5 methods) for the methods `rate_limit.go`, `workspace_service.go`, and `metering/middleware.go` actually call on `*settings.InstanceService`.
  - Extract a `MeteringRecorder` interface (the `Record` + `RecordLifecycleEvent` methods) for handlers/middleware that only need to record events, not query usage.
  - Update consumers to depend on the interface, not the concrete struct.
  - Update `api/internal/mocks/` so there is **one** mock per interface (delete the duplicate handler-local mini-interfaces once the canonical one is in place).
- Keep interfaces **small (1–5 methods)** and **caller-shaped**, not implementation-shaped.

**Out:**
- A single top-level `Service` marker interface (Rule 4 forbids empty marker interfaces).
- Generic `Repository[T]` abstractions.
- Interfaces for single-consumer services (Tier C in the parent epic).
- Changing the concrete service structs themselves — only the consumers' view of them.
- Forcing every service to have an interface; only the high-consumer-count ones get one.

## Anti-Over-Abstraction Guard

Before extracting any interface, verify:
1. ≥2 real consumers exist today (grep for the method calls).
2. The interface is ≤5 methods (Interface Segregation).
3. The interface name describes the consumer's role, not the implementer (`SettingsReader`, not `SettingsService`).
4. At least one test benefits (the consumer can now use a fake instead of the full concrete struct).

If any criterion fails, do not extract — leave the concrete dependency. Tier C in the parent design doc lists candidates that are explicitly excluded.

## Acceptance Criteria

- [ ] `WorkspaceMetadataStore`, `SettingsReader`, `MeteringRecorder` interfaces defined.
- [ ] Each has ≥2 consumers migrated to depend on the interface.
- [ ] Handler-local mini-interfaces (`auditStore`, `invitationStore`, etc.) consolidated where they overlap with `WorkspaceMetadataStore`; remaining ones documented as to why they stay separate.
- [ ] `ls api/internal/mocks/` no longer has parallel mock trees for the same service.
- [ ] All existing tests pass; fakes replace concrete-struct dependencies in ≥3 test files.
- [ ] No interface exceeds 5 methods.
- [ ] Worklog entry listing each interface with its consumer count and method count.
