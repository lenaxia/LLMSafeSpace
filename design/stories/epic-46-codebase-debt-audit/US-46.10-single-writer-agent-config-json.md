# US-46.10: Single-Writer `agent-config.json`

**Epic:** 46 — Codebase Debt Audit & Abstraction Foundation
**Effort:** Large (4d)
**ROI:** High
**Depends on:** US-46.6 (split main.go — prerequisite for clean code organisation)
**Status:** Proposed

---

## Problem

`README-LLM.md:434-461` documents that `agent-config.json` has **four distinct writers**, none of which coordinate atomically:

| Writer | File | When |
|--------|------|------|
| `FlushProviders` | `pkg/agentd/secrets/secrets.go:623` | Boot materialize + every `/v1/reload-secrets` |
| `applyWorkspaceConfig` | `cmd/workspace-agentd/secrets.go:203` | Boot only |
| `startRelayInjector` goroutine | `cmd/workspace-agentd/relay_injector.go:423` | Once per pod lifetime at ~T+7s |
| `reloadSecretsHandler` re-merge | `cmd/workspace-agentd/secrets.go:362` | After every FlushProviders in reload handler |

The current design "works" because of three fragile coordination mechanisms:
1. Strict boot ordering (FlushProviders → applyWorkspaceConfig → relay injector)
2. `reloadMu sync.Mutex` in `reloadSecretsHandler` serialising concurrent reloads
3. opencode not hot-reloading the config file (so TOCTOU between FlushProviders and re-merge is benign)

The README itself (line 455) calls this out: *"The four-writer design is correct given the current boot sequence and `reloadMu` serialisation, but it is fragile. A future change that reorders the boot sequence or adds a new write path could reintroduce relay clobbering. The single-writer `WriteAgentConfig` design ... would eliminate this fragility."*

This story implements the single-writer design the README has been deferring.

## Scope

**In:**
- Introduce a single `AgentConfigWriter` type that owns all reads and writes of `agent-config.json`:
  ```go
  type AgentConfigWriter struct {
      mu       sync.Mutex
      path     string
      providers ProviderSource
      model    ModelSource
      relay    RelaySource
  }
  func (w *AgentConfigWriter) Rebuild(ctx context.Context) error // single atomic rewrite
  ```
- Route all four current writers through `Rebuild()`:
  - `FlushProviders` calls `w.Rebuild()` instead of writing the file directly.
  - `applyWorkspaceConfig` mutates `w.model` then calls `w.Rebuild()`.
  - `startRelayInjector` mutates `w.relay` then calls `w.Rebuild()`.
  - `reloadSecretsHandler` calls `w.Rebuild()` (no more manual re-merge — the writer always reflects current state).
- Remove the `atomic.Pointer[[]relayModel]` coordination between `relay_injector.go` and the reload handler (no longer needed — the writer holds the source-of-truth).
- Reduce `reloadMu` scope to protect only the writer's `Rebuild()` call (not the entire reload handler).
- Write to a temp file + atomic rename (`os.Rename`) so readers never see a partially-written file.
- Keep the `OPENCODE_CONFIG=/tmp/agent-config.json` env var path unchanged.

**Out:**
- Re-triggerable relay injector (README:457 — separate enhancement).
- Cross-replica model cache consistency (README:459 — Epic 45 US-30.11).
- Changing the on-disk JSON shape that opencode reads.
- Removing the `RelayInjected` signal flow (it stays; just the coordination simplifies).

## Risk Assessment

This is the **highest-risk story in Epic 46**. Per Rule 11 (Adversarial Self-Review), the PR must:
- Enumerate every current write path and prove each is routed through the new writer.
- Include regression tests for: boot sequence, credential reload, relay injection timing, concurrent reload + injection.
- Verify the stale window (README:502) is unchanged or improved.
- Run for ≥1 hour in a real workspace pod before merge (manual e2e verification per Rule 0).
