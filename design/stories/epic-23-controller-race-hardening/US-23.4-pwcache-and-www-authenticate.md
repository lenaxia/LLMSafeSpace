# US-23.4: pwCache Invalidation + WWW-Authenticate Stripping

**Epic:** 23 — Controller & API Race Hardening
**Priority:** Critical (hot-patch — produced a user-visible browser auth-prompt this session)
**Depends on:** US-23.1 (metrics already registered)
**Blocks:** none
**Layer:** API service (proxy)

## User Story

As a chat user, when my workspace recovers from a Failed phase via Change A's RestartGeneration bump, the new pod is created with a fresh password (because `cleanupFailedWorkspaceSecrets` deleted the old one and `ensurePasswordSecret` regenerated it). Today, the API service's `pwCache` keeps the OLD password until the workspace transitions to Suspending/Suspended/Terminating/Terminated — but never invalidates on transition to Active from a non-Active phase. So my browser request hits the API, the API forwards with the stale password, opencode returns 401 + `WWW-Authenticate: Basic`, and the API forwards the WWW-Authenticate header straight to my browser, which displays a "Sign in to access this site" prompt. This story fixes both the cache miss and the header passthrough.

## Acceptance Criteria

### Bug 1: pwCache invalidation on Active-from-non-Active transitions

- [ ] `api/internal/handlers/proxy.go` `ProxyHandler` gains a `priorPhase map[string]v1.WorkspacePhase` field (with mutex), tracking each workspace's last observed phase across reconciles.
- [ ] `onPhaseChange` (existing function around line 690-720) is rewritten to:
  1. Capture `prior := h.priorPhase[ws.Name]` BEFORE updating `priorPhase[ws.Name] = newPhase`.
  2. On `Suspending/Suspended/Terminating/Terminated/Failed`: invalidate caches (existing behavior plus `Failed` which is currently missing).
  3. On `Active` from a non-Active prior: invalidate `pwCache` (new).
  4. On `Active` from Active prior: just refresh `wsConfig` (existing partial behavior).
  5. On `Terminated`/`Terminating`: ALSO delete the `priorPhase` entry to prevent unbounded growth (intentionally NOT on `Failed` because Failed → Active is recoverable via Change A and the next Active branch needs `prior` to be correct).
- [ ] `APIProxyPasswordCacheInvalidationsTotal{trigger}` metric incremented on every invalidation, with `trigger` ∈ `{"failed", "suspending", "suspended", "terminating", "terminated", "active_from_non_active"}`.

### Bug 2: WWW-Authenticate stripping in proxy responses

- [ ] `proxy.go` defines a `forwardedResponseHeaders` allowlist of safe response headers:
  ```go
  var forwardedResponseHeaders = map[string]bool{
      "Content-Type":      true,
      "Content-Length":    true,
      "Cache-Control":     true,
      "Last-Modified":     true,
      "ETag":              true,
      "Vary":              true,
      "X-Accel-Buffering": true,  // SSE-related
  }
  ```
- [ ] In `doProxy` (existing function around line 410-470), the upstream-to-client header-copy loops at lines 451-454 and 462-466 are rewritten to filter through the allowlist. Headers NOT in the allowlist are dropped.
- [ ] If the upstream response is HTTP 401, the proxy:
  1. Increments `APIProxyUpstreamAuthFailuresTotal{workspace_phase=<current_phase>}`.
  2. Invalidates the workspace's `pwCache` (defensive — covers the case where `onPhaseChange`'s invalidation missed).
  3. Logs a Warn-level message identifying the workspace and the upstream path that returned 401.
  4. Returns a structured 502 response to the browser with body `{"error":"upstream authentication failed; please retry","workspaceID":"<id>"}` and NO `WWW-Authenticate` header.
- [ ] Browser never sees `WWW-Authenticate: Basic` from upstream. Verified by integration test.

### Rewritten `onPhaseChange`

```go
func (h *ProxyHandler) onPhaseChange(workspace *v1.Workspace) {
    phase := workspace.Status.Phase
    h.priorPhaseMu.Lock()
    prior := h.priorPhase[workspace.Name]
    h.priorPhase[workspace.Name] = phase
    h.priorPhaseMu.Unlock()

    // Publish phase change to browser SSE subscribers (existing behavior).
    if h.broker != nil {
        h.broker.Publish(workspace.Name, WorkspaceSSEEvent{
            Type:  "workspace.phase",
            Phase: string(phase),
        })
    }

    // Invalidating phases: on transition INTO any of these, drop all caches.
    if phase == phaseSuspending || phase == phaseSuspended ||
       phase == phaseTerminating || phase == phaseTerminated ||
       phase == v1.WorkspacePhaseFailed {
        h.invalidateCaches(workspace.Name)
        metrics.APIProxyPasswordCacheInvalidationsTotal.WithLabelValues(string(phase)).Inc()

        if h.sseTracker != nil {
            h.sseTracker.StopWatching(workspace.Name)
        }

        // Cleanup priorPhase for genuinely-terminal phases. Failed is
        // intentionally NOT cleaned up because Failed → Active is
        // recoverable via Change A; the Active branch needs `prior` set.
        if phase == phaseTerminated || phase == phaseTerminating {
            h.priorPhaseMu.Lock()
            delete(h.priorPhase, workspace.Name)
            h.priorPhaseMu.Unlock()
        }
        return
    }

    // Transition INTO Active.
    if phase == phaseActive {
        if prior != "" && prior != phaseActive {
            // Active from a non-Active prior phase. Password may have rotated
            // during the non-Active interval (e.g., ensurePasswordSecret
            // regenerated it after Failed→recovery). Invalidate pwCache.
            h.invalidateCaches(workspace.Name)
            metrics.APIProxyPasswordCacheInvalidationsTotal.WithLabelValues("active_from_non_active").Inc()
        } else {
            // Active→Active reconcile (no real transition). Just refresh wsConfig.
            h.wsConfigMu.Lock()
            delete(h.wsConfig, workspace.Name)
            h.wsConfigMu.Unlock()
        }
    }
}
```

## Test Plan

### Unit tests for `pwCache` invalidation

**Failed → invalidates:**
- Workspace transitions to `Failed`. Invoke `onPhaseChange`. Assert: `invalidateCaches` called. Metric `{trigger="failed"}` incremented.

**Active from Pending → invalidates:**
- `priorPhase[ws]=Pending`. Invoke with `phase=Active`. Assert: `invalidateCaches` called. Metric `{trigger="active_from_non_active"}` incremented.

**Active from Active → does NOT invalidate pwCache:**
- `priorPhase[ws]=Active`. Invoke with `phase=Active`. Assert: only `wsConfig` is invalidated; `pwCache` is preserved. Metric NOT incremented.

**First transition (priorPhase empty):**
- Fresh workspace, `priorPhase[ws]` not set. Invoke with `phase=Active`. Assert: treated as Active→Active (`prior == ""`); only wsConfig refreshed. Acceptable — no cached password to invalidate yet.

**Terminated cleans priorPhase:**
- Invoke with `phase=Terminated`. Assert: `priorPhase[ws]` is deleted from the map.

**Failed does NOT clean priorPhase:**
- Invoke with `phase=Failed`. Assert: `priorPhase[ws]` is set to Failed (not deleted). Then invoke with `phase=Pending`, then `phase=Creating`, then `phase=Active`. Assert: at the Active step, `prior=Creating != Active`, so invalidation fires correctly. (Verifies the recovery path works.)

- Assert: response is 502 with no `WWW-Authenticate` header. Body contains `{"error":"upstream authentication failed",...}`.

**Other auth headers stripped:**

**Allowlist-only forwarding for 200 responses:**

**SSE-related headers preserved:**
- Mock upstream returns 200 with `X-Accel-Buffering: no` (SSE flag). Assert: forwarded.
- `Cache-Control: no-cache`. Assert: forwarded.

**401 invalidates pwCache defensively:**
- Setup: `pwCache[ws]=password` is populated. Mock upstream returns 401.
- Invoke. Assert: `pwCache[ws]` is now empty.

**401 metric increment:**
- Mock upstream returns 401. Workspace's phase=Active. Assert: metric `{workspace_phase="Active"}` incremented.

### Integration tests

**Full recovery scenario:**
- Workspace flapped to Failed → recovered via Change A → password rotated → user makes API request.
- Pre-fix: API uses stale password → opencode 401 → browser auth prompt.
- Post-fix: API observes the Active-from-non-Active transition (via `onPhaseChange`) → invalidates `pwCache` → next request fetches fresh password → opencode authenticates → browser sees normal response.

**Concurrent password rotation and request:**
- Race condition: rotation happens while a request is in flight. The in-flight request fails with 401 (stale password); doProxy invalidates and returns 502. The user retries; the next request fetches fresh password. Bounded recovery.

**Header passthrough for non-401 responses:**
- Upstream returns 200, 404, 500. Verify each preserves intended headers (Content-Type, etc.) and drops everything else.

### E2E tests

- Real cluster: trigger the worklog 0102 scenario (workspace flap → Change A recovery → page refresh). Assert: no browser auth prompt. The original incident does not recur.

### Security tests

- **No credential leakage in 502 response:** the body has only `error`, `workspaceID`. No password, no token.
- **No information disclosure via header filtering:** the allowlist doesn't include `Server` or other potentially-fingerprinting headers — even if upstream sends them, browser doesn't see them.
- **Defense in depth:** the 401-handling invalidates `pwCache` even if `onPhaseChange` somehow missed. Two independent invalidation paths.
- **Cross-workspace isolation:** `pwCache` is keyed by workspace ID; invalidating one doesn't affect others.

### Performance tests

- Header-allowlist check is O(1) per header (map lookup). Per-response overhead < 1µs.
- `priorPhase` map lookup is O(1) with mutex; mutex is RW so concurrent reads are cheap. Scales with workspace count.
- 401 path adds one invalidation + one metric increment. < 100µs.

### Robustness tests

- **`priorPhase` map under churn:** if many workspaces are created/terminated rapidly, the map's growth is bounded by current-active-count + Failed-but-recoverable count. Cleanup on Terminated keeps it bounded.
- **Concurrent `onPhaseChange` calls for the same workspace:** mutex ensures atomic update of `priorPhase`. Last writer wins. Acceptable — phase transitions are serialized by the controller anyway.
- **Misordered phase events (rare):** if the API receives `Active` before the upstream `Pending`/`Creating` (e.g., out-of-order watch events), `priorPhase` reflects whatever was last-written. The cache invalidation logic still fires on ANY non-Active prior, including stale ones. Defensive.

### Negative tests

- Upstream returns 401 for a non-auth-related reason (e.g., expired Bearer token from somewhere — not applicable to opencode but defensive). Same handling: invalidate, return 502. User retries; fresh auth fetched. Recovery still works.
