# US-23.1: DeletionTimestamp Guard + Heavy Instrumentation

**Epic:** 23 — Controller & API Race Hardening
**Priority:** Critical (hot-patch — reproduced 3 times in one session: workspaces `c98963e7-…`, `6d36952e-…`, `8e80afc4-…`)
**Depends on:** none
**Blocks:** Epic 21 (the dying-pod misclassification poisons Change B's failure budget; this story must land first)
**Layer:** Controller + metrics

## User Story

As the workspace controller, when I observe a pod with `DeletionTimestamp != nil` (meaning kubelet is mid-termination — typically because *I* just deleted it as part of `checkAgentHealth`'s pod-restart logic), I must NOT interpret its briefly-observable `Status.Phase=Failed` as a "pod failed during creation" event. The verified incident from worklog 0100 was caused by exactly this misclassification: controller deletes pod → pod phase briefly Failed during termination → handleCreating treats it as terminal failure → workspace stuck. Adding a `pod.DeletionTimestamp` check before the existing PodFailed branch eliminates this entire failure mode.

The story also installs heavy Prometheus instrumentation — phase-transition counters, terminal-failure-by-reason counters, health-check-outcome counters, status-conflict counters, per-workspace failure gauges — so Epic 21's design decisions (backoff schedule, reset window, per-cause policy) can be validated against production data rather than guessed.

## Acceptance Criteria

### Bug fix: DeletionTimestamp guard

- [ ] A new helper `isPodTerminating(pod *corev1.Pod) bool` returns true iff `pod != nil && pod.DeletionTimestamp != nil`. Lives in `controller/internal/workspace/helpers.go` (new file) for reuse.
- [ ] In `handleCreating` at controller.go:246-249 (the existing `existingPod.Status.Phase == corev1.PodFailed` branch), insert a guard BEFORE the existing check:
  ```go
  if isPodTerminating(existingPod) {
      metrics.WorkspacePodTerminatingObservedTotal.WithLabelValues("handleCreating").Inc()
      return ctrl.Result{RequeueAfter: requeueCreating}, nil
  }
  ```
- [ ] In `handleActive` at controller.go:283 (the `if pod.Status.Phase != corev1.PodRunning` branch — NOT the NotFound branch at line 280, where the pod is genuinely gone and recovery is correct), insert the same guard BEFORE the existing path:
  ```go
  if isPodTerminating(pod) {
      metrics.WorkspacePodTerminatingObservedTotal.WithLabelValues("handleActive").Inc()
      workspace.Status.Phase = v1.WorkspacePhaseCreating
      workspace.Status.PodIP = ""
      workspace.Status.Endpoint = ""
      return ctrl.Result{RequeueAfter: requeueCreating}, r.Status().Update(ctx, workspace)
  }
  ```
  **Placement rationale:** the pod-NotFound path (line 280, `return r.recoverFromTransientPodLoss(ctx, workspace)`) means kubelet has already finished deletion — there's nothing to wait for, recreate is the right action. The guard belongs at the case where the pod is still observable but not Running (line 283). Adding the guard at line 280 would convert a legitimate transient pod loss (pod evicted, OOMKilled, deleted out-of-band) into a permanent stall.
- [ ] Behavior: a pod with DeletionTimestamp set is treated as "wait for kubelet to finish reaping; do not classify its current phase as a failure." The next reconcile (after kubelet completes deletion) sees no pod, falls into the existing `r.Get` NotFound branch, and proceeds with the normal recreate path.
- [ ] No regression: pods without DeletionTimestamp set continue to flow through the existing PodFailed / non-PodRunning branches as before.

- [ ] Nine new Prometheus metrics registered with the controller-manager metrics service:
  ```go

  WorkspaceTerminalFailuresTotal      // CounterVec{reason}
  WorkspaceHealthCheckTotal           // CounterVec{outcome}
  WorkspaceStatusUpdateConflictsTotal // CounterVec{site}

  WorkspaceSecondsSinceLastSuccess    // GaugeVec{workspace, namespace}

  APIProxyPasswordCacheInvalidationsTotal // CounterVec{trigger}  -- registered here, populated by US-23.4
  APIProxyUpstreamAuthFailuresTotal       // CounterVec{workspace_phase}  -- ditto
  ```
- [ ] Naming follows the existing `llmsafespace_*` convention from `controller/internal/metrics/metrics.go`.
- [ ] Counter increments wired at appropriate sites:
  - Phase transitions: every `r.Status().Update` that changes `Status.Phase` increments `WorkspacePhaseTransitionsTotal{from=oldPhase, to=newPhase, reason=…}`. The `reason` label is empty pre-Epic-21; populated with FailureReason values once Epic 21 ships. (Pre-Epic-21 buckets used: `"pod_failed_during_creation", "pvc_bind_timeout", "pod_build_failed", "pending_timeout", "transient_pod_loss"`.)
  - Terminal failures: every `Status.Phase = WorkspacePhaseFailed` write also increments `WorkspaceTerminalFailuresTotal{reason}`.
  - Health check: every `checkAgentHealth` (or post-Epic-22 `checkAgentLiveness`) probe increments with one of `"success", "timeout", "5xx", "decode_error", "agent_unhealthy"`.
  - Status update conflicts: every `r.Status().Update` that returns Conflict increments `WorkspaceStatusUpdateConflictsTotal{site=<calling_func>}`. Wired in this story; comprehensive coverage in US-23.2.
  - Backoff gauge: per-workspace `WorkspaceConsecutiveFailures{workspace, namespace}` updated after each reconcile. (Pre-Epic-21: gauge stays 0; populated post-Epic-21 from the new field.)
  - Last-success gauge: similar.
  - Pod terminating observed: the central counter for the bug being fixed. Two label values: `"handleCreating"`, `"handleActive"`.
- [ ] Metrics are scrapable at the controller's `/metrics` endpoint (existing endpoint; no new HTTP plumbing needed).

## Technical Details

### Helper

```go
// isPodTerminating reports whether the K8s pod is in the process of being
// deleted. A non-nil DeletionTimestamp means kubelet is running termination
// grace; the pod's Status.Phase during this window is unreliable for
// failure-classification purposes (e.g., a SIGKILLed container makes the
// pod briefly observable as Failed). Callers should treat such pods as
// "wait for reaping" rather than as genuine failures.
//
// Verified mechanism for the worklog 0100 incident: controller deleted the
// pod via checkAgentHealth's pod-restart logic; within ~1s the next reconcile
// observed the pod in K8s phase=Failed (containers terminated non-zero from
// SIGKILL); handleCreating's PodFailed branch wrote terminal Failed.
// This guard prevents that.
func isPodTerminating(pod *corev1.Pod) bool {
    return pod != nil && pod.DeletionTimestamp != nil
}
```

### Cardinality consideration

The two `*GaugeVec` metrics use `{workspace, namespace}` labels, producing one time series per workspace. At 1k workspaces this is 2k series (across both gauges) — well within Prometheus's comfortable range. Documented in the metric Help text. If the system scales to 100k workspaces, revisit (drop per-workspace labels, use histograms instead).

## Test Plan

### Unit tests

**`isPodTerminating` helper:**
- `nil` pod → false.
- Pod with `DeletionTimestamp = nil` → false.
- Pod with `DeletionTimestamp = &someTime` → true.

**handleCreating with DeletionTimestamp:**
- Setup: workspace `Phase=Creating`, pod exists with `DeletionTimestamp != nil` and `Status.Phase=Failed`.
- Invoke. Assert: returns `RequeueAfter=requeueCreating`. Phase remains `Creating`. No `Status.Update` for terminal-Failed. `WorkspacePodTerminatingObservedTotal{handler="handleCreating"}` incremented.

**handleCreating with PodFailed (regression — DeletionTimestamp nil):**
- Setup: pod exists with `DeletionTimestamp = nil`, `Status.Phase=Failed`.
- Invoke. Assert: terminal-Failed write fires (existing behavior). FailureReason = "PodFailedDuringCreation" (post-Epic-21) or message-based (pre-Epic-21).

**handleActive with DeletionTimestamp:**
- Setup: `Phase=Active`, pod `DeletionTimestamp != nil`.
- Invoke. Assert: Phase transitions to `Creating`, PodIP/Endpoint cleared. `WorkspacePodTerminatingObservedTotal{handler="handleActive"}` incremented. No `recoverFromTransientPodLoss` invocation (which would increment the failure counter).

**handleActive with non-Running and DeletionTimestamp nil (regression):**
- Setup: `Phase=Active`, pod `Status.Phase=Pending`, `DeletionTimestamp=nil`.
- Invoke. Assert: existing `recoverFromTransientPodLoss` flow fires (transient counter increments).

**Metric increments:**
- Each phase transition emits a `WorkspacePhaseTransitionsTotal` counter increment with correct `from/to/reason` labels.
- Each terminal-Failed transition emits `WorkspaceTerminalFailuresTotal{reason}`.
- Each health-check probe emits `WorkspaceHealthCheckTotal{outcome}` with correct outcome.

**Edge cases:**
- Pod with `DeletionTimestamp` set but `Status.Phase=Running` (mid-termination, not yet failed). Same guard fires — treated as terminating. Acceptable.
- Pod with `DeletionTimestamp` set in the FAR PAST (kubelet stuck). Reconcile keeps requeueing. K8s eventually force-deletes; then the next reconcile sees no pod and recreates. Recovery path works.
- DeletionTimestamp is non-nil but `DeletionGracePeriodSeconds=0` (force delete). Same handling — guard fires.

### Integration tests (envtest)

**Worklog 0100 scenario reproduction:**
- Boot envtest with workspace pod. Workspace becomes Active.
- Trigger `checkAgentHealth`'s pod-restart path (e.g., make agentd's `/v1/statusz` time out 3 times).
- The reconcile that observes the dying pod runs `handleCreating` (since US-22.5 isn't yet shipped, the reconcile may go through the old code path; this test fixture mocks the post-restart state).
- Assert: workspace phase is `Creating` (not `Failed`). The dying pod is observed with DeletionTimestamp, the guard fires, workspace stays alive.
- Wait for kubelet to GC the pod. Assert: new pod is created; workspace returns to `Active`.

**Metric scraping in envtest:**
- After the scenario above, scrape `/metrics`. Assert: `llmsafespace_workspace_pod_terminating_observed_total{handler="handleCreating"}` > 0.

### E2E tests

- Real cluster: induce SSE load → controller restarts pod → assert workspace stays Active. (This is exactly the scenario that broke 3 workspaces in this session.)

### Security tests

- **No new attack surface:** the helper reads `DeletionTimestamp` from the pod object; no input validation needed.
- **Metric labels are bounded:** `handler` label has 2 values; `reason` label has < 10 values. No cardinality explosion from user input.

### Performance tests

- Helper: < 10ns (single field read).
- Metric increment: < 100ns (Prometheus client lib cost).
- Per-workspace gauge: O(1) updates per reconcile per workspace. At 1k workspaces reconciling every 15s, gauge updates are ~67/s. Negligible.

### Robustness tests

- **Concurrent DeletionTimestamp observation:** multiple controllers (impossible due to leader election, but defensive) trying to handle the same dying pod. The guard is read-only on the pod side; no race.
- **Pod GC delay:** if kubelet takes > 60s to fully delete the pod (extreme edge case), the controller keeps requeueing every `requeueCreating` (5s). Acceptable; eventual GC unblocks. If GC never completes (kubelet broken), workspace stays in Creating-with-RequeueAfter forever. Acceptable; document. Operators can detect via `WorkspacePodTerminatingObservedTotal` rate (high rate persistent without progress = kubelet issue).
- **DeletionTimestamp set without an actual deletion request (test-only fabrication):** behavior is the same — guard fires, workspace requeues. Eventually if no actual deletion, the field would clear; reconcile observes no DeletionTimestamp and proceeds normally.

### Negative tests

- Calling `isPodTerminating` with a non-pod object (nil, struct mismatch): only `*corev1.Pod` is accepted. Caller responsibility.

## Dependencies

## Risk Assessment

- **Low risk for the bug fix** — adds a guard before existing branches, doesn't modify them. Worst case: a real PodFailed during creation is masked by a spurious DeletionTimestamp (would require kubelet-side bug). Acceptable.
- **Low risk for instrumentation** — purely additive. Worst case: cardinality issue at extreme scale (documented).
