# US-23.2: Retry-on-Conflict for `Status().Update`

**Epic:** 23 — Controller & API Race Hardening
**Priority:** High
**Depends on:** US-23.1 (the metric `WorkspaceStatusUpdateConflictsTotal` is registered there); **US-23.3 ships first** (single-writer migration eliminates the lost-update risk this story would otherwise carry — see "Sequencing" in epic README, refuting R4)
**Blocks:** none
**Layer:** Controller

## User Story

As the workspace controller, when my `r.Status().Update(ctx, workspace)` returns a Conflict error (after Story 3 lands, this is informer-cache-lag conflicts only — no longer cross-writer races on Phase or LastActivityAt), I need to re-fetch the latest workspace, re-apply my intended state mutation, and retry — rather than aborting the reconcile and dropping the change. The existing API service code already uses `retry.RetryOnConflict` (`api/internal/handlers/activity.go:117`); this story propagates the pattern to the controller's 21 status-update sites.

## Acceptance Criteria

- [ ] A new helper `updateStatusWithRetry(ctx, name, mutate)` lives in `controller/internal/workspace/controller.go`:
  ```go
  func (r *WorkspaceReconciler) updateStatusWithRetry(
      ctx context.Context,
      name types.NamespacedName,
      mutate func(*v1.Workspace),
  ) error {
      return retry.RetryOnConflict(retry.DefaultBackoff, func() error {
          var fresh v1.Workspace
          if err := r.Get(ctx, name, &fresh); err != nil {
              return err
          }
          mutate(&fresh)
          if err := r.Status().Update(ctx, &fresh); err != nil {
              if apierrors.IsConflict(err) {
                  metrics.WorkspaceStatusUpdateConflictsTotal.WithLabelValues(callerSiteName(ctx)).Inc()
              }
              return err  // retry.RetryOnConflict examines IsConflict
          }
          return nil
      })
  }
  ```
- [ ] All 21 `r.Status().Update(ctx, workspace)` call sites in `controller/internal/workspace/controller.go` are migrated to use `updateStatusWithRetry`. The mutation closure captures the *intended* state changes; the helper handles re-read + retry.
- [ ] **Mutation closure semantics are deterministic.** For mutations that depend on the current value (e.g., "increment ConsecutiveFailures"), the closure must read from the freshly-fetched workspace, not capture a local. Pattern:
  ```go
  // BAD (captures the stale local):
  next := workspace.Status.ConsecutiveFailures + 1
  return r.updateStatusWithRetry(ctx, name, func(ws *v1.Workspace) {
      ws.Status.ConsecutiveFailures = next  // wrong on conflict-retry
  })

  // GOOD (reads fresh):
  return r.updateStatusWithRetry(ctx, name, func(ws *v1.Workspace) {
      ws.Status.ConsecutiveFailures += 1
  })
  ```
- [ ] `retry.RetryOnConflict` uses `retry.DefaultBackoff` (5 attempts, exponential backoff totaling ~1s). Same pattern as existing `activity.go:117` for consistency.
- [ ] `WorkspaceStatusUpdateConflictsTotal{site}` metric increments per conflict observed. The `site` label is the calling function name (extracted via `runtime.Caller` or passed explicitly).
- [ ] Tests cover: happy path (no conflict, single Update), one-conflict-then-success, persistent-conflict-exhausts-retries.

## Technical Details

### Existing prior art

- `api/internal/handlers/activity.go:117` — uses `retry.RetryOnConflict(retry.DefaultBackoff, ...)` for `Status.LastActivityAt` writes. (After US-23.3, this writer migrates to a `metadata.annotations` Patch, but the retry pattern is the prior art for the controller helper.)
- `api/internal/services/workspace/workspace_service.go:1007` — uses `retry.RetryOnConflict(retry.DefaultRetry, ...)` for the rename flow.

This story adopts the same pattern in the controller. Not inventing a new helper — just propagating.

### `callerSiteName` for the metric label

```go
import "runtime"

// callerSiteName extracts the caller's function name for the metric's
// "site" label. Used to identify which controller code path is hitting
// conflicts most often.
//
// IMPORTANT: the skip-count must be verified at implementation time. The
// expected stack is: Caller(0)=callerSiteName, Caller(1)=closure-passed-to-RetryOnConflict,
// Caller(2)=retry.OnError internal, Caller(3)=retry.RetryOnConflict, Caller(4)=updateStatusWithRetry,
// Caller(5)=actual caller. The right skip is whatever resolves to the actual
// caller in the controller code; verify with a unit test that asserts the label
// matches the calling function's name. If runtime.Caller doesn't resolve
// cleanly through the closure stack, fall back to passing siteName explicitly
// as a parameter to updateStatusWithRetry.
func callerSiteName() string {
    pc, _, _, ok := runtime.Caller(4)  // VERIFY at implementation time
    if !ok {
        return "unknown"
    }
    fn := runtime.FuncForPC(pc)
    if fn == nil {
        return "unknown"
    }
    name := fn.Name()
    // Trim package prefix for readability: "...controller.handleActive" → "handleActive"
    if i := strings.LastIndex(name, "."); i >= 0 {
        name = name[i+1:]
    }
    return name
}
```

**Implementation requirement:** US-23.2 includes a unit test that invokes `updateStatusWithRetry` from a known function (e.g., `TestUpdateStatusWithRetry_CallerLabel`), forces a conflict, and asserts the metric label matches the test function's name. If the assertion fails, fix the skip count or switch to explicit `siteName string` parameter on the helper. **No `runtime.Caller`-based label ships without that verification test.**

### Migration strategy for the 21 call sites

For each existing `r.Status().Update(ctx, workspace)`:

1. Identify what fields the surrounding code mutated.
2. Refactor the mutations into a closure passed to `updateStatusWithRetry`.
3. Adjust the surrounding code so the post-Update logic uses the in-closure-modified copy (or re-fetches if needed).

Two patterns emerge:

**Pattern A: simple mutation, no post-Update read.**
```go
// Before:
workspace.Status.Phase = v1.WorkspacePhaseActive
if err := r.Status().Update(ctx, workspace); err != nil {
    return ctrl.Result{}, err
}
return ctrl.Result{}, nil

// After:
if err := r.updateStatusWithRetry(ctx, types.NamespacedName{Name: workspace.Name, Namespace: workspace.Namespace},
    func(ws *v1.Workspace) {
        ws.Status.Phase = v1.WorkspacePhaseActive
    }); err != nil {
    return ctrl.Result{}, err
}
return ctrl.Result{}, nil
```

**Pattern B: mutation depends on current value.**
```go
// Before:
workspace.Status.ConsecutiveFailures++
if err := r.Status().Update(ctx, workspace); err != nil { return ctrl.Result{}, err }

// After:
if err := r.updateStatusWithRetry(ctx, name, func(ws *v1.Workspace) {
    ws.Status.ConsecutiveFailures++  // applied to freshly-fetched ws
}); err != nil { return ctrl.Result{}, err }
```

The mutation closure runs with the freshly-fetched `*v1.Workspace`. On conflict, the helper re-fetches and re-runs the closure. The increment is applied to whatever the latest value is.

## Test Plan

### Unit tests

**Happy path — single Update, no conflict:**
- Mock `r.Get` returns workspace; `r.Status().Update` returns nil.
- Invoke `updateStatusWithRetry` with a closure that sets `Status.Phase = Active`.
- Assert: helper returns nil, mock `Update` called once, metric NOT incremented.

**One conflict, then success:**
- Mock first `Update` returns Conflict; second returns nil. Mock `Get` returns workspace each time.
- Invoke. Assert: helper returns nil, `Get` called twice, `Update` called twice, conflict metric incremented by 1.

**Persistent conflict (exhausts retries):**
- Mock `Update` always returns Conflict.
- Invoke. Assert: helper returns Conflict error after 5 attempts (retry.DefaultBackoff). Metric incremented 5 times.

**Non-conflict error short-circuits:**
- Mock `Update` returns Forbidden.
- Invoke. Assert: helper returns Forbidden after one attempt. No retry. Metric NOT incremented (only Conflict counts).

**Mutation closure runs with fresh data:**
- Mock first `Get` returns workspace with `ConsecutiveFailures=2`. First `Update` returns Conflict. Mock second `Get` returns workspace with `ConsecutiveFailures=5` (someone else incremented).
- Invoke with closure `ws.Status.ConsecutiveFailures++`.
- Assert: second `Update` is called with workspace where ConsecutiveFailures=6 (5+1, not 3=2+1). The increment correctly applies to fresh data.

**`Get` returns NotFound (workspace deleted mid-reconcile):**
- Mock `Get` returns NotFound.
- Invoke. Assert: helper returns NotFound. No `Update` attempt. (Workspace gone; nothing to update.)

### Integration tests (envtest)

**Real conflict from API service writer:**
- Run controller and a goroutine simulating the API service's `Status.LastActivityAt` write.
- Both write to the same workspace nearly simultaneously.
- Assert: at least one conflict observed (metric > 0). Both writes eventually succeed (the helper retries). Final state has BOTH the controller's mutation AND the API's LastActivityAt write applied.

**Conflict resolution under stress:**
- 100 status updates from the controller in rapid succession; concurrent goroutine writes LastActivityAt every 100ms.
- Assert: all 100 controller updates eventually succeed. Conflict count > 0.

### E2E tests

- Real cluster: run a workload that triggers frequent activity updates (chat traffic) while controller is doing heavy reconcile work (e.g., during a phase transition). Observe `WorkspaceStatusUpdateConflictsTotal` in `/metrics`. Counter > 0 confirms conflicts happen and are being resolved.

### Security tests

- **No data leaks in conflict-retry:** the helper re-fetches the workspace before re-applying the mutation. No stale data is written. No information disclosure.
- **No DoS via conflict-loop:** retry budget is 5 attempts (~1s). After exhaustion, the reconcile errors out and controller-runtime requeues. Bounded retry depth.

### Performance tests

- **Happy-path overhead:** `updateStatusWithRetry` adds one `r.Get` + closure invocation vs the old single `r.Status().Update`. Bench: ~5ms additional latency in happy path (the Get is the cost). Acceptable.
- **Conflict-resolution latency:** 5 retries × DefaultBackoff sums to ~1s worst case. Acceptable.
- **Metric increment cost:** < 100ns per conflict. Negligible.

### Robustness tests

- **Mutation idempotence:** the closure must produce the same final state when applied to the same input multiple times. Test-fixture pattern: invoke the closure twice on the same workspace; assert end state is identical to invoking once. (For commutative mutations like "set field X = value Y", trivially holds. For "increment", NOT idempotent — but the helper's re-fetch model makes that OK because each retry sees a different fresh state.)
- **Nested helper calls forbidden:** if a closure invokes `updateStatusWithRetry` recursively, deadlock or infinite retry. Document; tests assert.
- **Closure panics:** if the closure panics, the panic propagates out of the retry loop. Caller's deferred recover handles. Acceptable.

### Negative tests

- Closure that doesn't mutate anything: `Update` is still called (no-op write). Resourceversion advances; subsequent reads see the bump. Wasteful but not incorrect. Document.
- Closure that mutates fields outside `Status` (e.g., spec): `r.Status().Update` only persists status; spec changes are ignored. Caller error. Document.

## Risk Assessment

- **Medium risk.** Refactoring 21 call sites is mechanical but prone to subtle bugs (wrong field captured in closure; mutation depends on stale local; missed call site).
- **Mitigation:** unit tests assert mutation-closure correctness. Code review checks each migration. Integration test exercises real conflicts. The `callerSiteName` skip-count is verified by a dedicated test before the metric ships.
- **Sequencing risk eliminated:** the lost-update risk that previously made this story unsafe to ship before Story 3 (refuted R4 in epic README) is structurally eliminated by Story 3 landing first. After Story 3, conflicts come exclusively from informer-cache lag (controller's own re-reads vs etcd's latest), not from cross-writer races.

## Rollout

Single commit (or per-call-site PRs if you prefer smaller review surface). Deploy. Existing reconciles continue to work; conflicts that previously aborted now self-heal. Conflict metric becomes non-zero in production.
