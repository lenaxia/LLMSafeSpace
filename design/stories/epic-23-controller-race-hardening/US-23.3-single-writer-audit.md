# US-23.3: Single-Writer Migration (LastActivityAt + Phase)

**Epic:** 23 — Controller & API Race Hardening
**Priority:** High (structural — must land BEFORE US-23.2 to make retry-on-conflict safe; refer to epic README's Sequencing section and refuted hypothesis R4)
**Depends on:** US-23.1 (metrics already registered)
**Blocks:** US-23.2 (retry-on-conflict cannot ship safely until cross-writer races on Phase / LastActivityAt are eliminated)
**Layer:** API CRD types + API service + controller

## User Story

As a platform engineer, I need every `WorkspaceStatus` field to have exactly one writer so that the optimistic-concurrency conflict class is structurally eliminated, not just retried-around. Today, `LastActivityAt` has THREE writers (controller's `handleResuming`, API's ActivityTracker, API's Resume flow) and `Phase` has cross-writer behavior (controller for most transitions, API for Suspend/Resume). The "belt-and-suspenders" pattern at `workspace_service.go:443-448` and `controller.go:398-403` is the original authors anticipating the race and writing both sides defensively — that's tech debt this story eliminates.

By migrating `LastActivityAt` to a `metadata.annotations` Patch (separate optimistic-concurrency lane from `Status().Update`) with the **API service as sole writer**, and migrating Suspend/Resume to a spec-driven pattern where the **controller is sole writer of Phase**, we end up with every Status field having exactly one writer. After this story, US-23.2's retry-on-conflict can ship safely.

## Technical Details

### Annotation pattern

```go
const AnnotationLastActivityAt = "llmsafespace.dev/last-activity-at"

func GetLastActivityAt(ws *Workspace) (time.Time, bool) {
    if ws.Annotations == nil {
        return time.Time{}, false
    }
    s, ok := ws.Annotations[AnnotationLastActivityAt]
    if !ok || s == "" {
        return time.Time{}, false
    }
    t, err := time.Parse(time.RFC3339, s)
    if err != nil {
        return time.Time{}, false
    }
    return t, true
}

// PatchLastActivityAt updates the annotation via a strategic-merge Patch.
// Patches don't conflict with status updates (different field paths) so
// callers don't need retry-on-conflict for activity tracking.
func PatchLastActivityAt(ctx context.Context, client client.Client, ws *Workspace, t time.Time) error {
    patch := []byte(fmt.Sprintf(`{"metadata":{"annotations":{%q: %q}}}`,
        AnnotationLastActivityAt, t.UTC().Format(time.RFC3339)))
    return client.Patch(ctx, ws, client.RawPatch(types.StrategicMergePatchType, patch))
}
```

### Why annotations beat alternatives

| Option | Pros | Cons |
|---|---|---|
| **Annotation** (chosen) | No status-subresource conflicts; cheap Patch; standard K8s pattern; semantically "metadata about the object" | Stringly-typed; 64KB metadata limit (no concern for one timestamp) |
| Separate `WorkspaceActivity` CR | Strongest isolation | Overkill; ergonomic burden; new RBAC surface |
| `Spec.LastUserActivityAt` | Same conflict domain as `Spec.Suspend` writes | Mixes desired-state semantics ("here is what activity happened") with real desired state |
| Keep in Status | Status quo | Defeats single-writer goal |

### Suspend/Resume spec field

```go
// Suspend, when true, requests the controller suspend the workspace's pod.
// The controller transitions the workspace through Suspending → Suspended.
// Setting Suspend=false from Suspended triggers Resume → Resuming → Active.
//
// A user-driven Suspend/Resume action via the API service writes this field.
// The controller is the sole writer of Status.Phase.
//
// +kubebuilder:default=false
Suspend bool `json:"suspend,omitempty"`
```

### Controller-side handling

In `handleActive`:

```go
if workspace.Spec.Suspend {
    log.Info("Spec.Suspend=true; transitioning to Suspending", "workspaceID", workspace.Name)
    return r.updateStatusWithRetry(ctx, name, func(ws *v1.Workspace) {
        ws.Status.Phase = v1.WorkspacePhaseSuspending
    })
}
```

In `handleSuspended`:

```go
if !workspace.Spec.Suspend {
    log.Info("Spec.Suspend=false; transitioning to Resuming", "workspaceID", workspace.Name)
    return r.updateStatusWithRetry(ctx, name, func(ws *v1.Workspace) {
        ws.Status.Phase = v1.WorkspacePhaseResuming
    })
}
```

## Test Plan

### Unit tests for annotation helpers

**`GetLastActivityAt`:**
- Workspace with no annotations → `time.Time{}, false`.
- Workspace with valid RFC3339 annotation → parsed time, true.
- Workspace with malformed annotation → `time.Time{}, false`.
- Workspace with annotation but empty string → `time.Time{}, false`.

**`PatchLastActivityAt`:**
- Mock client: assert `Patch` is called with the right strategic-merge body.
- Patch failure: returns the error.
- Successful patch: subsequent `Get` shows the annotation set.

### Integration tests

**Activity tracker writes annotation, controller reads:**
- API service writes annotation. Controller's reconcile reads it. Idle-timeout logic uses the value.
- Concurrent: API service writes activity at high rate (every 100ms) while controller reconciles. No conflicts (different Patch paths). No retries needed.

**Suspend via spec:**
- API service calls SuspendWorkspace → spec.Suspend=true. Controller observes; phase transitions Active → Suspending → Suspended.
- API service calls ResumeWorkspace → spec.Suspend=false. Controller observes; phase transitions Suspended → Resuming → Active.

**API never writes status:**
- Run integration test for 5 minutes with mixed traffic. Inspect API service's k8s-client mock: assert zero `UpdateStatus` calls for Workspaces.

**Lint catches violations:**
- Inject a synthetic source file at `controller/.../bad.go` containing `Status.LastActivityAt =`. Run `make repolint`. Assert: lint fails with a message identifying the file.

### E2E tests

- Real cluster: user performs Suspend via the UI. Workspace transitions through Suspending → Suspended. Resume via UI. Transitions through Resuming → Active. Observe via `kubectl get workspace -w`.
- Activity tracking: send chat messages. Watch `kubectl get workspace -o jsonpath='{.metadata.annotations.llmsafespace\.dev/last-activity-at}'` — value advances.

### Security tests

- **Annotation is publicly readable from the workspace CRD by anyone with `get workspace` RBAC.** Same exposure as before (Status was also readable). No new disclosure.
- **Spec mutation requires `update workspace` RBAC.** Suspend/Resume already restricted to the API service and operators. No new authz surface.
- **Lint prevents accidental privilege boundary violation.** A future contributor can't write controller-only fields from API code without CI catching it.

### Performance tests

- **Patch latency:** ~10ms for a small annotation Patch via K8s API. Comparable to UpdateStatus.
- **No retries needed:** annotations and status are different conflict domains; the activity Patch and the controller's UpdateStatus don't conflict.

### Robustness tests

- **Patch on non-existent workspace:** returns NotFound. Caller (API service) handles gracefully (workspace deleted mid-flight).
- **Concurrent annotation writes:** `Patch` with strategic-merge merges values; if two patches arrive nearly simultaneously, both apply (LastActivityAt is overwritten with the latest). Acceptable.
- **Spec.Suspend toggling rapidly:** controller observes the eventual-consistent value. Phase transitions follow. If Suspend is toggled true→false→true in quick succession, the controller may transition through Suspending then back to Resuming then back. Eventually consistent with the latest spec value. Acceptable.

### Negative tests

- API service tries to `UpdateStatus` (regression of pre-migration behavior): lint catches at CI time. Compile is fine; lint is the gate.
- Controller tries to write the LastActivityAt annotation: lint catches.

## Risk Assessment

- **Medium-high risk.** Touches the API service's Suspend/Resume flow (user-facing); the controller's observation logic; the activity-tracking pipeline. Bug here could break user-facing Suspend/Resume.
- **Mitigation:** comprehensive integration tests; envtest end-to-end; staged rollout (could ship in two PRs — annotation migration first, Suspend migration second).
- **Why this story is safe to ship before US-23.2:** annotation patches and Status updates use independent optimistic-concurrency lanes (A12 in epic README). Until US-23.2 lands, the controller's existing one-shot `Status().Update` calls will continue to occasionally fail with Conflict — but those conflicts will be drastically rarer post-Story-3 because the cross-writer race on Phase / LastActivityAt is gone. Residual conflicts come from informer-cache lag (controller's own re-reads vs etcd's latest), which is exactly what US-23.2 then targets.

## Rollout

Two-PR sequence recommended:

1. **PR 1:** annotation migration. Activity tracker writes annotation; controller's idle-timeout reads from annotation; controller's `handleResuming` removes its `LastActivityAt` write. Old `Status.LastActivityAt` field retained transiently to avoid breaking existing CRD instances; populated only by a one-shot migration script (or simply deleted per A7 in Epic 21 README — no BC required).
2. **PR 2:** Spec.Suspend migration; remove old `LastActivityAt` field from Status entirely.

Alternatively, single PR with all changes — per Epic 23's "no BC concern" assumption, a clean cut-over is acceptable.

After landing: API service makes zero `UpdateStatus` calls on Workspaces. Conflict rate (US-23.1's metric) for Workspaces drops to near-zero. US-23.2 then ships to handle the residual informer-cache-lag conflicts.
