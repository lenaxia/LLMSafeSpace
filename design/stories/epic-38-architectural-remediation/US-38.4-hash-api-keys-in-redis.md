# US-38.4: Hash API Keys in Redis Cache Keys

## Summary

API keys are used directly (unhashed) in Redis key names for cache entries. If Redis is compromised via a dump file, `MONITOR` command, or network sniffing, the raw API keys are exposed. This story hashes API keys before using them in Redis key names, consistent with how JWT tokens are already handled at `auth.go:368`.

## Current Problem

### Location 1: `api/internal/services/auth/auth.go:134`

In `AuthenticateAPIKey`:

```go
cacheKey := fmt.Sprintf("apikey:%s", apiKey)
```

### Location 2: `api/internal/services/auth/auth.go:446`

In `validateAPIKey`:

```go
cacheKey := fmt.Sprintf("apikey:%s", apiKey)
```

Both use the raw API key string directly in the Redis key name. If an attacker gains read access to Redis (via `KEYS *`, `DEBUG SLEEP` + `MONITOR`, dump file exfiltration, or replication hijacking), they see every API key in plaintext.

### Contrast: JWT Caching Is Done Correctly

At `auth.go:368`, JWT tokens are hashed before caching:

```go
cacheKey := fmt.Sprintf("jwt:%s", pkgutil.HashString(token))
```

The same pattern should apply to API keys. Note the import alias is `pkgutil`, not `utilities` — this matches the existing convention in `auth.go`.

## Weaknesses

The following weaknesses are explicitly documented for awareness:

### 1. Two Code Paths Must Be Updated Atomically

There are exactly two cache-key construction sites in `auth.go`. Both must be updated in the same commit/deploy:

- **`AuthenticateAPIKey` at `auth.go:134`** — the cached fast path
- **`validateAPIKey` at `auth.go:446`** — the DB-backed slow path that also writes to cache

Updating only one of these would result in inconsistent cache behavior: one path writes hashed keys, the other writes raw keys, and neither can read the other's entries. Always update both atomically.

### 2. `DeleteAPIKey` Has No Cache Revocation

`DeleteAPIKey` removes the key from the database but does **not** invalidate the Redis cache entry. A deleted API key remains valid for up to 15 minutes (the cache TTL) after deletion. This is a **pre-existing gap not introduced by this story**, but it is worth documenting here because this story touches the same subsystem. A future story should add explicit cache invalidation on deletion (i.e., `redis.Del("apikey:" + pkgutil.HashString(key))` after the DB delete).

## Proposed Fix

### Change

Replace both occurrences of:

```go
cacheKey := fmt.Sprintf("apikey:%s", apiKey)
```

With:

```go
cacheKey := fmt.Sprintf("apikey:%s", pkgutil.HashString(apiKey))
```

### Pre-existing Dependency

The `pkgutil.HashString` function already exists at `pkg/utilities/hashing.go`. The import alias `pkgutil` is already used in `auth.go` (confirmed by the JWT caching at line 368). No new imports needed.

## Deployment Considerations

### Cache Invalidation

Changing the cache key format means all existing `apikey:*` entries in Redis become orphaned. They will never be read again because the new code hashes the key before lookup.

**This is safe and acceptable because:**

1. API key cache entries have a TTL (check the `Set` call — likely 5-15 minutes). Orphaned entries expire naturally.
2. No data is lost — the cache is purely a performance optimization. A cache miss triggers a fresh database lookup.
3. The old entries do not need to be cleaned up. They expire automatically.

**Optional (not required):** If you want to clean up immediately, run `redis-cli KEYS "apikey:*" | xargs redis-cli DEL` during deployment. But this is unnecessary and risks blocking Redis if there are many keys.

### Rolling Deploy Race Window

During a rolling deployment, old pods write cache entries with raw API keys (`apikey:<raw>`) while new pods look up cache entries with hashed keys (`apikey:<hash>`). Old-format entries written by old pods **will not be read** by new pods. This is safe (cache miss → DB lookup) but causes **elevated database load** for the duration of the rolling deploy as every API key validation hits the database instead of the cache. The elevated load window is bounded by the pod rollout time (typically 2-5 minutes) plus the cache TTL of the last old-format entry (5-15 minutes). Monitor DB connection count and query latency during the deploy window.

### No Database Changes

This is purely a cache-layer change. No database migration required.

### Rollback

If you need to roll back, the only effect is that cache entries written by the new code (hashed keys) are orphaned by the old code. Same situation as forward deployment — entries expire via TTL.

### Operational Key Lookup

To look up a specific API key's cache entry in Redis (e.g., for debugging), compute its hash first:

```bash
echo -n "lsp_..." | sha256sum | awk '{print $1}'
# Then: redis-cli GET "apikey:<hash>"
```

Replace `lsp_...` with the actual raw API key value.
