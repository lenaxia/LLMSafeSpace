# US-38.9 — Move Services Out of Handlers

| Field        | Value                                                        |
|--------------|--------------------------------------------------------------|
| Epic         | 38 — Architectural Remediation                               |
| Priority     | Medium                                                       |
| Depends On   | US-38.2 (decompose proxy handler), US-38.7, US-38.8         |
| Blocks       | None                                                         |
| Estimated    | 6 hours                                                      |
| Assignee     | TBD                                                          |

## Summary

Five service-type structs currently live in `api/internal/handlers/` but are not HTTP handlers — they are business-logic services (activity tracking, SSE management, CRD watching, event brokering). This story moves each one into a proper `api/internal/services/` sub-package, updating all imports and wiring along the way.

Moving services out of the handlers package enforces the single-responsibility principle: the `handlers` package should only contain HTTP request/response logic.

## General Move Procedure (applies to each type)

For each type being moved, follow this exact procedure:

### Step A — Create the target package directory

```bash
mkdir -p api/internal/services/<package_name>
```

### Step B — Read the source file completely

```bash
cat api/internal/handlers/<source_file>
```

Identify:
1. All exported types, interfaces, functions, and constants.
2. All unexported helper types and functions that support the exported ones.
3. All imports used.

### Step C — Create the new file

Write `api/internal/services/<package>/<file>.go` with:
1. `package <package_name>` (not `handlers`).
2. All the code from the source file.
3. Adjust any internal references that assumed the `handlers` package name.

### Step D — Move the corresponding test file

> **Test file inventory:** Run `ls api/internal/handlers/*_test.go` to inventory test files before starting. Confirm which test files exist: `activity_test.go`, `event_broker_test.go`, `event_broker_user_test.go`, `crd_watcher_test.go`, `session_tracker_test.go`.

Each source file has a corresponding `_test.go` file. Move it alongside the new implementation:

```bash
mv api/internal/handlers/<source_file>_test.go api/internal/services/<package>/<file>_test.go
```

Update the package declaration in the test file from `package handlers` to `package <package_name>` (or `package <package_name>_test` if tests use the external test pattern).

> **Test file access note:** The test files (e.g., `crd_watcher_test.go`) are in `package handlers` and may access unexported symbols. Before moving, grep each test file for unexported references. If found, either: (a) keep tests as `package <name>` (white-box), or (b) export the symbols. Failing to address this will cause a compile error in the new package where the unexported names are not visible.

### Step E — Check for handler-coupling and extract interfaces

If the type references other handler-package types (e.g., a handler struct or handler helper function):
1. Extract an interface for the dependency.
2. Define the interface in the services package or in `api/internal/interfaces/`.
3. The handlers package will implement the interface.

**WorkspaceWatcher specifically** depends on `*UserEventBroker` via two methods called on the `userBroker` field (see `crd_watcher.go`):
- `w.userBroker.RecordWorkspaceOwner(workspaceID, userID string)` — called in `seedResourceVersion` and `handleEvent`
- `w.userBroker.CleanupWorkspace(workspaceID string)` — called in `handleEvent` on deletion

Before moving WorkspaceWatcher, extract an interface for its broker dependency using exactly those two method signatures:

```go
// In api/internal/services/workspace/watcher.go (or a separate interfaces.go)
type WorkspaceOwnerTracker interface {
    RecordWorkspaceOwner(workspaceID, userID string)
    CleanupWorkspace(workspaceID string)
}
```

> **Interface placement:** Define `WorkspaceOwnerTracker` at the top of `api/internal/services/workspace/watcher.go`. Do NOT put it in `api/internal/interfaces/interfaces.go`.

WorkspaceWatcher's `SetUserBroker` method and the `userBroker` field should accept this interface instead of `*UserEventBroker` directly. This decouples the watcher from the concrete broker implementation.

> **Critical:** Do NOT use the interface name `EventBroker` — it does not match what `UserEventBroker` actually exposes. `UserEventBroker` has no `Publish` method. The two methods used by the watcher are `RecordWorkspaceOwner` and `CleanupWorkspace`, hence the name `WorkspaceOwnerTracker`.

### Step F — Delete the source file

```bash
rm api/internal/handlers/<source_file>
rm api/internal/handlers/<source_file>_test.go  # if not already moved
```

### Step G — Update all imports

Find every file that imports the moved type:
```bash
grep -rn 'handlers\.ActivityTracker\|handlers\.NewActivityTracker\|handlers\.SSETracker' --include='*.go' .
```

For each hit:
1. Change the import path from `api/internal/handlers` to `api/internal/services/<package>`.
2. If the file now imports `handlers` only for this type, remove the `handlers` import entirely.
3. If the file imports both `handlers` and the new services package, ensure there is no alias conflict.

### Step H — Skip: interfaces.go does not reference these types

`api/internal/interfaces/interfaces.go` does **not** reference any of these five types. Skip the interfaces.go update step entirely.

### Step I — Update wiring

Check `api/internal/app/app.go` for constructor calls:

```bash
grep -n 'NewActivityTracker\|NewSSETracker\|NewWorkspaceWatcher\|NewWorkspaceEventBroker\|NewUserEventBroker' api/internal/app/app.go
```

Update the import and any dependency injection.

### Step J — Build and test

```bash
go build ./...
go test ./api/... -v
```

## Pre-Move Step: Relocate WorkspaceSSEEvent to a shared types package

**Do this BEFORE moving any broker or watcher types.**

`WorkspaceSSEEvent` is defined in `api/internal/handlers/event_broker.go` and is used by both `event_broker_user.go` (which references it throughout `UserEventBroker`) and `crd_watcher.go` (indirectly via `UserEventBroker`). After the moves:

- The brokers will live in `api/internal/services/eventbroker/`
- The watcher will live in `api/internal/services/workspace/`

If `WorkspaceSSEEvent` stays inside `services/eventbroker/`, then `services/workspace/` must import `services/eventbroker/` just to get the event type — creating an unwanted hub dependency between the two new packages.

**Solution:** Move `WorkspaceSSEEvent` to a shared location first:

```bash
mkdir -p api/internal/types
```

Create `api/internal/types/sse_event.go`:
```go
package types

// WorkspaceSSEEvent is the canonical event type sent to browser SSE clients.
// (Copy the full type definition and its godoc comment from event_broker.go here.)
type WorkspaceSSEEvent struct { ... }
```

Then update all references (in `event_broker.go`, `event_broker_user.go`, `crd_watcher.go`, and any handler files) to import `api/internal/types` and use `types.WorkspaceSSEEvent`.

Run `go build ./...` and `go test ./api/... -v` before proceeding to the type moves.

## Execution Order

Move types in this order to satisfy dependencies:

1. **Pre-move:** Extract `WorkspaceSSEEvent` to `api/internal/types/sse_event.go`. Build and test before continuing.
2. **ActivityTracker** — fewest cross-dependencies.
3. **WorkspaceEventBroker** — move BEFORE WorkspaceWatcher. (Skip if US-38.8 already removed this type.)
4. **UserEventBroker** — move BEFORE WorkspaceWatcher. Items 3 and 4 share the target package; move together if both exist.
5. **SSETracker** — no dependency on the brokers; can be moved in parallel with Items 3–4 if preferred, but moving it after keeps the build green at each step.
6. **WorkspaceWatcher** — depends on `UserEventBroker` (now in `services/eventbroker`). Extract the `WorkspaceOwnerTracker` interface before moving.

> **Why brokers before watcher?** `WorkspaceWatcher.userBroker` is typed `*UserEventBroker`. If the watcher moves before the brokers, the watcher's new `services/workspace` package must import `handlers` for the `*UserEventBroker` type — which defeats the purpose of the move and likely introduces an import cycle (`handlers` → `services/workspace` → `handlers`). Moving the brokers first lets the watcher import `services/eventbroker` for the concrete type, and the `WorkspaceOwnerTracker` interface further decouples the dependency.

After each move:
```bash
go build ./...
go test ./api/... -v
git add -A && git commit -m "refactor: move <Type> to api/internal/services/<package>"
```
