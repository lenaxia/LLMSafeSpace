# US-38.6: Fix Controller Gauge Drift

## Summary

The workspace controller has a critical metrics bug: the `WorkspacesRunning` Prometheus gauge is incremented when entering the Active phase but never decremented when transitioning out. Every Active→Creating re-cycle adds +1 to the gauge without the corresponding -1, causing perpetual drift. Similarly, terminating Active workspaces never decrements the gauge. Additionally, `WorkspaceSafeModeActive` is a scalar gauge rather than per-workspace, preventing proper cleanup and causing cardinality issues. This story fixes all gauge leaks.

## Current Problem

### Gauge Leak 1: Active→Creating Transition (7 locations in phase_active.go)

In `controller/internal/workspace/phase_active.go`, the code transitions workspaces from Active back to Creating on multiple paths. None of these call `Dec()` on `WorkspacesRunning` before the transition:

- **Lines 32-41:** `RestartGeneration` bump triggers pod deletion and transitions Active→Creating.
- **Lines 46-54:** Password secret missing triggers pod recycle. Transitions Active→Creating.
- **Line 66:** Pod not found — calls `enterRecovery`, which sets phase to Creating (`recovery_policy.go:108`). Transitions Active→Creating.
- **Lines 73-79:** Pod has `DeletionTimestamp` set (controller-initiated termination) — transitions Active→Creating.
- **Lines 81-84:** Pod not Running — calls `enterRecovery`. Transitions Active→Creating.
- **Lines 95-101:** Architecture drift (node selector mismatch) triggers pod deletion and transitions Active→Creating.
- **Lines 104-109:** `CrashLoopBackOff` container status — calls `enterRecovery`. Transitions Active→Creating.

### Gauge Leak 2: Active→Creating Transition (2 locations in health.go)

`checkAgentHealth` is called from `handleActive` (line 156) and itself performs Active→Creating transitions in two paths, also without decrementing the gauge:

- **`health.go:113-122`:** Agent unreachable beyond `healthCheckFailureThreshold` — deletes pod and sets `ws.Status.Phase = v1.WorkspacePhaseCreating`.
- **`health.go:140-150`:** Agent unhealthy response beyond threshold — deletes pod and sets `ws.Status.Phase = v1.WorkspacePhaseCreating`.

**Total required Dec() locations: 9** (7 in `phase_active.go` + 2 in `health.go`).

### Gauge Leak 3: Creating→Active Always Incs

In `phase_creating.go:136`, the transition Creating→Active always calls `Inc()` on `WorkspacesRunning`. This is correct if every exit from Active calls `Dec()`. Since Active exits never Dec, each full cycle (Creating→Active→Creating→Active) adds +1 permanently.

### Gauge Leak 4: Terminating Active Workspace

In `phase_terminating.go:15-62`, when deleting a workspace that was Active, the gauge is never decremented. The `handleTerminating` function handles the full termination lifecycle but has no `Dec()` call.

### Gauge Leak 5: Failed→Active Self-Heal Missing Inc

In `recovery.go:74-88`, when a workspace self-heals from the legacy Failed phase back to Active (pod is Running/Ready), `WorkspacesRunning.Inc()` is never called. The gauge is not incremented for the re-entered Active state.

### Gauge Leak 6: WorkspaceSafeModeActive Is Scalar

In `metrics.go:48-50`, `WorkspaceSafeModeActive` is a scalar `prometheus.Gauge` (not a `GaugeVec`). This means:

- When workspace A enters safe mode, gauge = 1.
- When workspace B enters safe mode, gauge is still 1 (overwritten, not incremented to 2).
- When workspace A exits safe mode, gauge = 0, but workspace B is still in safe mode.
- When a workspace is deleted while in safe mode, the gauge is never cleaned up.

Furthermore, the else-branch in `metrics_wiring.go:77` calls `safeModeGauge.Set(0)`. After converting to a `GaugeVec`, this must become `DeleteLabelValues(workspaceID)`, NOT `Set(0)` with a label — `Set(0)` on a GaugeVec creates a spurious zero-valued label entry for every non-safe-mode workspace that enters recovery, causing label cardinality to grow unboundedly (one entry per workspace per recovery call, never cleaned up).

## Proposed Fix

### Fix 1: Add Dec() Before Every Active→Creating Transition (9 locations)

`WorkspacesRunning` is a `GaugeVec` with labels `["runtime", "security_level"]` (`metrics.go:21-24`). All `Dec()` calls must use `.WithLabelValues(runtime, secLevel)`, exactly matching the pattern from `phase_suspend.go:23-25`:

```go
runtime := workspace.Spec.Runtime
secLevel := string(workspace.Spec.SecurityLevel)
metrics.WorkspacesRunning.WithLabelValues(runtime, secLevel).Dec()
```

Because every transition path uses `r.Status().Update(ctx, workspace)` which is fallible, the Dec must be paired with a rollback Inc in the error path. The exact pattern, shown for one site, is:

```go
runtime := workspace.Spec.Runtime
secLevel := string(workspace.Spec.SecurityLevel)
metrics.WorkspacesRunning.WithLabelValues(runtime, secLevel).Dec()
workspace.Status.Phase = v1.WorkspacePhaseCreating
workspace.Status.PodIP = ""
workspace.Status.Endpoint = ""
// ... other field resets ...
if err := r.Status().Update(ctx, workspace); err != nil {
    metrics.WorkspacesRunning.WithLabelValues(runtime, secLevel).Inc() // rollback
    return ctrl.Result{}, err
}
return ctrl.Result{}, nil
```

This pattern must be applied individually at each of the 9 locations. All sites use `workspace.Spec.Runtime` and `string(workspace.Spec.SecurityLevel)` — the Spec fields, not Status, because the workspace is still in Active (its Spec has not changed). Do NOT use a hypothetical `w.Runtime` or `w.SecurityLevel` accessor — these fields do not exist.

#### Critical note on enterRecovery call sites

`enterRecovery` (`recovery_policy.go:84-124`) is **phase-agnostic**. It is called from both Active contexts (the 3 sites above) AND from Creating contexts:

- `phase_creating.go:90` — `buildPod` failure in Creating phase
- `phase_creating.go:159` — pod failed in Creating phase
- `phase_creating.go:171` — pod unschedulable in Creating phase

Adding `Dec()` **inside** `enterRecovery` would double-decrement for Creating-phase callers (who were never counted in the gauge). The `Dec()` must always be added in `phase_active.go` **before** calling `enterRecovery`, never inside `enterRecovery` itself.

For the rollback of enterRecovery sites: `enterRecovery` calls `r.Status().Update` internally and returns an error if it fails. The caller in `phase_active.go` returns the error from `enterRecovery` directly. Wrap each enterRecovery call site:

```go
runtime := workspace.Spec.Runtime
secLevel := string(workspace.Spec.SecurityLevel)
metrics.WorkspacesRunning.WithLabelValues(runtime, secLevel).Dec()
result, err := r.enterRecovery(ctx, workspace, class)
if err != nil {
    metrics.WorkspacesRunning.WithLabelValues(runtime, secLevel).Inc() // rollback
}
return result, err
```

For **line 66** specifically (pod not found), `class` is not in scope — use the complete concrete replacement:

```go
// phase_active.go line 66 replacement:
rt := workspace.Spec.Runtime
sl := string(workspace.Spec.SecurityLevel)
metrics.WorkspacesRunning.WithLabelValues(rt, sl).Dec()
result, err := r.enterRecovery(ctx, workspace, FailureClassInfrastructure)
if err != nil {
    metrics.WorkspacesRunning.WithLabelValues(rt, sl).Inc()
}
return result, err
```

For line 84 and 109, `class` is already in scope from `classifyFailure(obs)`, so the generic pattern above applies directly.

#### Active→Suspending paths are already handled (no change needed)

`phase_active.go:119-122` and `phase_active.go:143-144` transition Active→Suspending (timeout and idle auto-suspend). These do **not** need a `Dec()` here. The `handleSuspending` function (`phase_suspend.go:25`) already calls `metrics.WorkspacesRunning.WithLabelValues(runtime, secLevel).Dec()` as the first metric operation. Adding `Dec()` in `handleActive` for these paths would double-decrement.

### Fix 2: Add Dec() in handleTerminating

In `phase_terminating.go:15`, `handleTerminating` has no state tracking of which phase the workspace was in before deletion. There is no `previousState` field on the workspace object. The correct proxy is the status fields that are populated only when a workspace has been Active:

```go
func (r *WorkspaceReconciler) handleTerminating(ctx context.Context, workspace *v1.Workspace) (ctrl.Result, error) {
    uid := string(workspace.UID)
    name := podName(workspace.Name, uid)

    // Decrement the running gauge if the workspace was Active (i.e., had
    // a live pod assigned). PodIP is set in phase_creating.go:145 only
    // when the pod reaches Running/Ready and the workspace transitions to
    // Active; it is cleared on suspension and termination.
    if workspace.Status.PodIP != "" || workspace.Status.PodName != "" {
        runtime := workspace.Spec.Runtime
        secLevel := string(workspace.Spec.SecurityLevel)
        metrics.WorkspacesRunning.WithLabelValues(runtime, secLevel).Dec()
    }
    // ... rest of handleTerminating unchanged ...
}
```

**Edge case — Creating→Terminating:** A workspace in Creating phase also has `workspace.Status.PodName` set (`phase_creating.go:101`: `workspace.Status.PodName = pod.Name`). This means the proxy condition `PodName != ""` will also fire for a workspace that was terminated mid-creation, before it ever reached Active. This would spuriously decrement the gauge.

To avoid this, use only `PodIP` as the proxy: `PodIP` is set at `phase_creating.go:145` only when the pod reaches `PodRunning` with a non-empty IP and all containers are ready, which is also the exact moment the phase transitions to Active. A workspace in Creating phase with a pending or starting pod has `PodIP == ""`. Therefore, use only `PodIP`:

```go
if workspace.Status.PodIP != "" {
    runtime := workspace.Spec.Runtime
    secLevel := string(workspace.Spec.SecurityLevel)
    metrics.WorkspacesRunning.WithLabelValues(runtime, secLevel).Dec()
}
```

Also add `safeModeGauge` cleanup at the end of `handleTerminating` (after the GaugeVec conversion in Fix 4):

```go
metrics.WorkspaceSafeModeActive.DeleteLabelValues(string(workspace.UID))
```

### Fix 3: Add Inc() in recovery.go self-heal path

In `recovery.go:74-88`, the self-heal path (`handleFailed` finding a Running/Ready pod and transitioning to Active) is missing an `Inc()`. Add it immediately before the status field assignments:

```go
if ready {
    logger.Info("Workspace Failed but pod is Running/Ready; self-healing to Active")
    now := metav1.Now()
    runtime := workspace.Spec.Runtime
    secLevel := string(workspace.Spec.SecurityLevel)
    metrics.WorkspacesRunning.WithLabelValues(runtime, secLevel).Inc()
    workspace.Status.Phase = v1.WorkspacePhaseActive
    // ... rest unchanged ...
    return ctrl.Result{}, r.Status().Update(ctx, workspace)
}
```

No rollback is needed here: if `r.Status().Update` fails, the workspace remains Failed, not Active, and the gauge is overstated by 1. However, `SeedWorkspacesRunning` at startup will correct this on next controller restart (see Fix 6 note). If strict correctness is required, add the rollback Dec() in the error path mirroring the pattern from Fix 1.

#### Fix 4a: Update recordRecoveryMetricsInto signature

In `metrics_wiring.go:50-79`, the function takes `safeModeGauge prometheus.Gauge`. After the type change this must become `*prometheus.GaugeVec`, and the function must receive the workspace ID to use as a label. Update the signature and all usages:

```go
func recordRecoveryMetricsInto(
    ws *v1.Workspace,
    class FailureClass,
    attempts *prometheus.CounterVec,
    backoffHist *prometheus.HistogramVec,
    safeModeGauge *prometheus.GaugeVec,
    failedCtr *prometheus.CounterVec,
) {
    attempts.WithLabelValues(string(class)).Inc()

    if ws.Status.NextRetryAt != nil {
        backoff := time.Until(ws.Status.NextRetryAt.Time)
        if backoff < 0 {
            backoff = 0
        }
        backoffHist.WithLabelValues(string(class)).Observe(backoff.Seconds())
    }

    workspaceID := string(ws.UID)
    if ws.Status.SafeMode {
        safeModeGauge.WithLabelValues(workspaceID).Set(1)
        failedCtr.WithLabelValues(string(class)).Inc()
    } else {
        // Use DeleteLabelValues — NOT Set(0). Set(0) with a label creates
        // a persistent zero-valued time series for every non-safe-mode
        // workspace that passes through recovery, causing label cardinality
        // to grow without bound (one entry per workspace, never deleted).
        safeModeGauge.DeleteLabelValues(workspaceID)
    }
}
```

The `recordRecoveryMetrics` wrapper function at `metrics_wiring.go:82-90` does not need a signature change — it reads `ws.UID` itself to pass the workspace ID inside `recordRecoveryMetricsInto`.

### Fix 5: SeedWorkspacesRunning as startup safety net

`metrics.go:161-167` defines `SeedWorkspacesRunning`, which resets `WorkspacesRunning` to the actual count of Active workspaces after the informer cache syncs at controller startup. This is the ultimate recovery mechanism: even if any drift escapes all the above fixes (e.g., a controller crash between a Dec and its corresponding Update), the gauge is reset to ground truth on the next restart.

This function already exists and should be called correctly in the controller's `SetupWithManager` or startup path. Verify it is called after cache sync, not before.

## Failure Modes

| Failure | Consequence | Mitigation |
|---|---|---|
| Dec() added but Status().Update fails | Gauge goes negative by 1; workspace stays Active | Rollback Inc() in Update error path (required at all 9 sites) |
| Dec() added inside enterRecovery | Double-decrement for Creating-phase callers (phase_creating.go:90, 159, 171) | Never add Dec() inside enterRecovery; always add in phase_active.go before the call |
| safeModeGauge Set(0) else-branch not changed to DeleteLabelValues after GaugeVec conversion | Every non-safe-mode workspace recovery call creates a persistent zero-valued label; cardinality grows O(total workspaces × recovery events), never cleaned up | Use DeleteLabelValues in else-branch, not Set(0) |
| handleTerminating using PodName != "" as Active proxy | Workspaces terminated in Creating phase (PodName set at phase_creating.go:101) spuriously decrement gauge | Use PodIP != "" only — PodIP is set at creating.go:145, strictly after pod is Running+Ready = Active |
| health.go Dec() added but handleActive Update rollback missing | If checkAgentHealth decrements gauge and the Status().Update fails, gauge drifts -1 permanently | Add phaseBefore tracking in handleActive to detect the change and rollback |
| Controller crash between Dec and Status().Update | Gauge drifts -1 across the restart | SeedWorkspacesRunning at startup (metrics.go:161-167) resets to actual count |

> **Note on rollback scope:** The Status().Update failure rollback only rolls back the gauge. Other metric mutations (`accumulateActiveSeconds`, `setStorageBytes`) are additive and self-correct on the next reconcile — no additional rollback needed for them.
