| Field        | Value                                |
|--------------|--------------------------------------|
| Epic         | 38 — Architectural Remediation       |
| Priority     | High                                 |

| Blocks       | None                                 |
| Estimated    | 3 hours                              |
| Assignee     | TBD                                  |

## Summary

`cmd/workspace-agentd/main.go` has no graceful shutdown. Four specific problems:
1. **Three background goroutines** use `context.Background()`, never cancelled on shutdown — they leak: SSE tracker (line 773), fillGaps (line 776), refreshIsHealthyLoop (line 791).
2. `userSrv.ListenAndServe()` blocks `main()` at line 919 with no signal handling — `Ctrl+C` or `SIGTERM` kills the process mid-request.
3. `log.Fatal` at line 920 calls `os.Exit(1)` without cleanup — defers don't run, in-flight requests are dropped.
4. `relay_injector.go:246` uses `context.Background()` for the `/provider` fetch — should accept a context for cancellation.

This story adds proper signal handling, context propagation to all background goroutines, and ordered shutdown.

## Current Architecture

Read the main entry point:

```bash
cat cmd/workspace-agentd/main.go
```

Approximate current flow:

```
main()
  ├─ launch SSE tracker goroutine        (line 773, context.Background())
  ├─ launch fillGaps goroutine            (line 776, context.Background())
  ├─ launch refreshIsHealthyLoop goroutine (line 791, context.Background())
  ├─ setup admin server (port 8081)
  │   └─ adminSrv.ListenAndServe() in goroutine   (line 905–910)
  │       └─ log.Fatal on error                    ← kills process
  └─ userSrv.ListenAndServe()             (line 919) ← blocks forever
      └─ log.Fatal on error                        ← kills process
```

Key observation: the admin server **already runs in a goroutine** (line 905–910). Only `userSrv.ListenAndServe()` at line 919 blocks `main()`. The target architecture must reflect this correctly.

## Target Architecture

```
main()
  ├─ create rootCtx, cancelled by SIGTERM/SIGINT
  ├─ create bgCtx, cancelled before proc.stop()
  ├─ launch SSE tracker goroutine with bgCtx      (bgWg.Add)
  ├─ launch fillGaps goroutine with bgCtx         (bgWg.Add)
  ├─ launch refreshIsHealthyLoop goroutine with bgCtx (bgWg.Add)
  ├─ startRelayInjector(rootCtx, cfg)             (passes rootCtx)
  ├─ setup admin server (port 8081)
  │   └─ adminSrv.ListenAndServe() in goroutine (unchanged pattern)
  ├─ setup user server (port 8080)
  │   └─ userSrv.ListenAndServe() in goroutine (new)
  ├─ block on signal or server error
  └─ shutdown sequence:
      1. userSrv.Shutdown(shutdownCtx) ┐  CONCURRENT — both drain
      2. adminSrv.Shutdown(shutdownCtx)┘  within the same 25s budget
      3. bgCancel()                    — stop SSE tracker, fillGaps, refreshIsHealthyLoop
      4. bgWg.Wait() with 5s timeout   — bounded join of background goroutines
      5. proc.stop()                   — SIGTERM managed process, wait for exit
```

Shutdown order rationale:
- **User server first (concurrent with admin):** stop accepting new user requests immediately.
- **Admin server concurrent with user:** Kubernetes sends SIGTERM, then checks `/readyz` a few more times. Running both shutdowns concurrently ensures neither steals time from the other within the shared 25s budget.
- **Cancel background goroutines:** after servers are drained, no new data arrives, so goroutines can exit cleanly.
- **Stop managed process last:** the opencode process should outlive all HTTP draining so in-flight proxy requests can complete.

## Fix 1 — Background Goroutine Contexts

### Problem

Three background goroutines are launched with `context.Background()` and never cancelled:

| Goroutine | Line | Function |
|---|---|---|
| SSE tracker | 773 | `go sseTracker.subscribe(context.Background(), client)` |
| fillGaps | 776 | `go fillGaps(context.Background(), client, ...)` |
| refreshIsHealthyLoop | 791 | `go refreshIsHealthyLoop(context.Background(), client, ...)` |

### Current Code

```go
go sseTracker.subscribe(context.Background(), client)

go fillGaps(context.Background(), client, sseTracker, func() []agentd.SessionInfo {
    cache.mu.Lock()
    sessions := cache.sessions
    cache.mu.Unlock()
    return sessions
}, fillState)

go refreshIsHealthyLoop(context.Background(), client, healthCache, log, gr)
```

### Fix

Derive a background context from `rootCtx` and pass it to all three goroutines. Track them with a `sync.WaitGroup` so the shutdown sequence can do a bounded join:

```go
bgCtx, bgCancel := context.WithCancel(rootCtx)
// NOTE: do NOT add `defer bgCancel()` here — the explicit bgCancel() call
// in the shutdown sequence is the authoritative cancellation point. Adding
// a defer as well would be misleading about intent (even though Cancel is
// idempotent). See Fix 4.

var bgWg sync.WaitGroup

bgWg.Add(1)
go func() {
    defer bgWg.Done()
    sseTracker.subscribe(bgCtx, client)
}()

bgWg.Add(1)
go func() {
    defer bgWg.Done()
    fillGaps(bgCtx, client, sseTracker, func() []agentd.SessionInfo {
        cache.mu.Lock()
        sessions := cache.sessions
        cache.mu.Unlock()
        return sessions
    }, fillState)
}()

bgWg.Add(1)
go func() {
    defer bgWg.Done()
    refreshIsHealthyLoop(bgCtx, client, healthCache, log, gr)
}()
```

All three functions (`subscribe`, `fillGaps`, `refreshIsHealthyLoop`) must accept a `context.Context` as their first parameter. Verify their signatures accept context; if not, update them.

## Fix 2 — Signal Handling and Non-Blocking User Server

### Problem

`userSrv.ListenAndServe()` at line 919 blocks `main()` with no signal handling. The only way to stop is `Ctrl+C` or `SIGTERM`, which kills the process immediately — no request draining.

### Current Code

```go
go func() {
    log.Info("workspace-agentd admin server starting", zap.String("addr", agentd.AgentdAdminAddr))
    if err := adminSrv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
        log.Fatal("workspace-agentd admin server failed", zap.Error(err))
    }
}()

log.Info("workspace-agentd user server starting", zap.String("addr", listenAddr))
userSrv := &http.Server{
    Addr:              listenAddr,
    Handler:           userMux,
    ReadHeaderTimeout: 5 * time.Second,
}
if err := userSrv.ListenAndServe(); err != nil {
    log.Fatal("workspace-agentd user server failed", zap.Error(err))
}
```

### Fix

Move the user server into a goroutine (same pattern as admin server), add signal handling, and add a shutdown sequence. Replace `log.Fatal` with error-channel pattern:

```go
rootCtx, stop := signal.NotifyContext(context.Background(), syscall.SIGTERM, syscall.SIGINT)
defer stop()

bgCtx, bgCancel := context.WithCancel(rootCtx)
// No defer bgCancel() — see Fix 4.

// ... launch background goroutines with bgCtx (Fix 1) ...

// ... startRelayInjector(rootCtx, cfg) — see Fix 5 ...

adminSrv := &http.Server{
    Addr:              agentd.AgentdAdminAddr,
    Handler:           adminMux,
    ReadHeaderTimeout: 5 * time.Second,
}
adminErrCh := make(chan error, 1)
go func() {
    log.Info("workspace-agentd admin server starting", zap.String("addr", agentd.AgentdAdminAddr))
    if err := adminSrv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
        adminErrCh <- err
    }
}()

userSrv := &http.Server{
    Addr:              listenAddr,
    Handler:           userMux,
    ReadHeaderTimeout: 5 * time.Second,
}
userErrCh := make(chan error, 1)
go func() {
    log.Info("workspace-agentd user server starting", zap.String("addr", listenAddr))
    if err := userSrv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
        userErrCh <- err
    }
}()

select {
case err := <-adminErrCh:
    log.Error("admin server error", zap.Error(err))
case err := <-userErrCh:
    log.Error("user server error", zap.Error(err))
case <-rootCtx.Done():
    log.Info("shutdown signal received", zap.String("signal", rootCtx.Err().Error()))
}

// ---- Shutdown sequence ----
// 25s is the request-draining budget. proc.stop() adds up to 5s SIGKILL
// fallback, totalling ~30s. The pod manifest MUST set
// terminationGracePeriodSeconds: 35 (or more) to give a buffer.
// See Fix 6 (Deployment section).
shutdownCtx, shutdownCancel := context.WithTimeout(context.Background(), 25*time.Second)
defer shutdownCancel()

// BLOCKING FIX: run both server shutdowns CONCURRENTLY so each gets
// the full 25s budget — not the remainder after the other drains.
// See Fix 3 for the full rationale.
var serverWg sync.WaitGroup
serverWg.Add(2)
go func() {
    defer serverWg.Done()
    log.Info("shutting down user server", zap.String("addr", listenAddr))
    if err := userSrv.Shutdown(shutdownCtx); err != nil {
        log.Error("user server shutdown error", zap.Error(err))
    }
}()
go func() {
    defer serverWg.Done()
    log.Info("shutting down admin server", zap.String("addr", agentd.AgentdAdminAddr))
    if err := adminSrv.Shutdown(shutdownCtx); err != nil && err != http.ErrServerClosed {
        log.Error("admin server shutdown error", zap.Error(err))
    }
}()
serverWg.Wait()

log.Info("cancelling background goroutines")
bgCancel() // authoritative cancellation — no defer bgCancel() anywhere

// Bounded join: wait up to 5s for background goroutines to exit.
waitDone := make(chan struct{})
go func() { bgWg.Wait(); close(waitDone) }()
select {
case <-waitDone:
    log.Info("background goroutines exited cleanly")
case <-time.After(5 * time.Second):
    log.Warn("background goroutines did not exit within 5s")
}

log.Info("stopping managed process")
if proc != nil {
    proc.stop()
}

log.Info("shutdown complete")
```

## Fix 3 — Concurrent Server Shutdowns (BLOCKING)

### Problem

The original story had sequential server shutdowns sharing a single `shutdownCtx`:

```go
// WRONG: sequential — admin server gets only (25s - user_drain_time)
if err := userSrv.Shutdown(shutdownCtx); err != nil { ... }
if err := adminSrv.Shutdown(shutdownCtx); err != nil { ... }
```

`shutdownCtx` has a 25-second deadline from the moment it is created. If user server draining takes 20 seconds (many in-flight requests), the admin server gets only 5 seconds. Under any moderate load — or a slow-draining SSE connection to the SSE tracker — the admin server may be force-killed before Kubernetes stops routing traffic to the pod, causing unnecessary 503s.

### Fix

Run both shutdowns concurrently using `sync.WaitGroup` (or `errgroup`). Both servers share the same 25s `shutdownCtx` deadline but run in parallel, so each gets the full budget:

```go
var serverWg sync.WaitGroup
serverWg.Add(2)
go func() {
    defer serverWg.Done()
    if err := userSrv.Shutdown(shutdownCtx); err != nil {
        log.Error("user server shutdown error", zap.Error(err))
    }
}()
go func() {
    defer serverWg.Done()
    if err := adminSrv.Shutdown(shutdownCtx); err != nil && err != http.ErrServerClosed {
        log.Error("admin server shutdown error", zap.Error(err))
    }
}()
serverWg.Wait()
```

This is a **blocking correctness fix**: without it, the shutdown budget is consumed asymmetrically and one server may be force-killed prematurely.

## Fix 4 — Remove Redundant `defer bgCancel()`

### Problem

The story previously showed:

```go
bgCtx, bgCancel := context.WithCancel(rootCtx)
defer bgCancel()   // ← redundant
```

And then later in the shutdown sequence:

```go
bgCancel()   // ← authoritative, explicit call
```

`bgCancel` is idempotent — calling it twice is harmless. But having **both** `defer bgCancel()` and an explicit `bgCancel()` is misleading: a reader naturally asks "which one actually cancels the goroutines, and when?" The answer is the explicit call at shutdown time (after servers have drained, before proc.stop), not the deferred call at function exit. The defer fires later (at main() return), but by that point shutdown is already complete. The defer adds no safety value here and obscures intent.

> **bgCtx/rootCtx relationship:** Because `bgCtx` is derived from `rootCtx`, background goroutines begin exiting when the signal fires (rootCtx cancelled), before `serverWg.Wait()` completes. The explicit `bgCancel()` and `bgWg.Wait()` are belt-and-suspenders — in practice goroutines may already be stopped by the time `bgCancel()` is reached. This is correct and safe.

### Fix

Remove `defer bgCancel()`. Only the explicit call in the shutdown sequence remains:

```go
bgCtx, bgCancel := context.WithCancel(rootCtx)
// No defer — bgCancel() is called explicitly in the shutdown sequence
// after server draining, before proc.stop(). This ordering is intentional.
```

## Fix 5 — Replace `log.Fatal` in Both Servers

### Problem

`log.Fatal` calls `os.Exit(1)`, which bypasses all `defer` statements, does not drain in-flight requests, and does not clean up goroutines or child processes.

There are two `log.Fatal` calls:
- Line 908: `log.Fatal("workspace-agentd admin server failed", ...)` (already in a goroutine)
- Line 920: `log.Fatal("workspace-agentd user server failed", ...)` (in main goroutine, blocks)

### Fix

This is addressed by Fix 2:
- Both servers run in goroutines that send errors to channels.
- The `select` block handles errors without calling `log.Fatal`.
- The shutdown sequence runs after the `select` exits.

## Fix 6 — Use Existing `proc.stop()` Instead of New `stopManagedProcess()`

### Key point

The existing `proc.stop()` method at `main.go:1192–1218` already:
1. Checks idempotency (second call returns immediately via `doneCh`).
2. Sends SIGTERM to the managed process.
3. Sets a 5-second SIGKILL timer as a fallback.
4. Waits for the supervisor goroutine to exit (`<-doneCh`).
5. Drains in-flight health-probe goroutines (`probeWg.Wait()`).

This is more robust than writing a new `stopManagedProcess()`. Use it directly:

```go
if proc != nil {
    proc.stop()
}
```

The `proc` variable may be `nil` in tests (when `supervise` is false). Guard with a nil check.

## Fix 7 — Relay Injector Goroutine Shutdown Scope

### Problem

`startRelayInjector` (main.go:804) launches a background goroutine with a 5-minute health-wait deadline. The goroutine is not joined anywhere and is not passed a cancellable context. If agentd receives SIGTERM during the health-wait loop, the relay injector continues running until its own deadline, or until the process is killed, whichever comes first.

### Fix

Pass `rootCtx` as the first argument to `startRelayInjector`:

```go
// In main():
startRelayInjector(rootCtx, relayInjectorConfig{ ... })

// Signature change:
func startRelayInjector(ctx context.Context, cfg relayInjectorConfig) {
    if cfg.RelayURL == "" {
        return
    }
    lg := log
    go func() {
        // Health-wait loop: check ctx.Done() on each iteration
        deadline := time.Now().Add(5 * time.Minute)
        for time.Now().Before(deadline) {
            select {
            case <-ctx.Done():
                lg.Info("relay injector: context cancelled during health wait, aborting")
                return
            default:
            }
            if cfg.HealthCheck() {
                break
            }
            select {
            case <-ctx.Done():
                lg.Info("relay injector: context cancelled during health wait, aborting")
                return
            case <-time.After(2 * time.Second):
            }
        }
        // ... rest of injection logic ...
    }()
}
```

> **Free-model fetch retry loop:** Also replace the `time.Sleep(5 * time.Second)` at `relay_injector.go:428` (inside the free-model fetch retry loop) with:
> ```go
> select {
> case <-ctx.Done():
>     return
> case <-time.After(5 * time.Second):
> }
> ```
> This ensures the model-fetch retries are also cancellable on shutdown, not just the health-wait loop.

The relay injector's HTTP calls (via `fetchFreeModels`) should also use a `ctx`-derived context. Thread `rootCtx` through: `startRelayInjector` → the goroutine → `fetchFreeModels`. See Fix 8.

**Important:** file-write operations (`os.WriteFile` for agent-config.json and auth.json) cannot be cancelled mid-write at the OS level. Once the write syscall is issued, it runs to completion regardless of context cancellation. This is acceptable — the writes are fast (kilobytes) and the context check before the write is sufficient to avoid starting a write during shutdown.

**One-shot flag note:** even if context is cancelled during the health-wait loop, the relay injector only runs once per pod lifetime (controlled by the `activeRelayModels` atomic pointer — nil means not yet run, non-nil means complete). Cancelling via context during shutdown means the relay injector will not run for this pod — which is acceptable since the pod is terminating anyway. The next pod will start fresh and the injector will run again.

## Fix 9 — WaitGroup for Background Goroutines

### Problem

The three background goroutines (SSE tracker, fillGaps, refreshIsHealthyLoop) are launched as fire-and-forget. After `bgCancel()`, there is no guarantee they have exited before `proc.stop()` runs. If they reference shared state (cache, healthCache) that is torn down after proc.stop(), there is a window for races.

### Fix

Track all three with `bgWg sync.WaitGroup` (declared before the goroutine launches, as shown in Fix 1), then do a bounded join in the shutdown sequence:

```go
log.Info("cancelling background goroutines")
bgCancel()

waitDone := make(chan struct{})
go func() { bgWg.Wait(); close(waitDone) }()
select {
case <-waitDone:
    log.Info("background goroutines exited cleanly")
case <-time.After(5 * time.Second):
    log.Warn("background goroutines did not exit within 5s")
}
```

The 5-second timeout is a safety valve. All three goroutines respect `ctx.Done()`:
- `subscribe` checks `ctx.Done()` at the top of its loop and in the reconnect backoff sleep.
- `fillGaps` selects on `ctx.Done()` in its ticker loop.
- `refreshIsHealthyLoop` selects on `ctx.Done()` in its ticker loop.

In practice all three should exit within milliseconds of `bgCancel()`.

## Fix 10 — Intentionally Out-of-Scope context.Background() Uses

The following `context.Background()` uses in this package are **intentionally out of scope** for this story. Do not change them; add this note to prevent future confusion:

| Location | Reason out of scope |
|---|---|
| `secrets.go:163` | Used in the `materialize` subcommand, which exits via `os.Exit` after a single synchronous run. It is not a server process and has no graceful shutdown concern. |
| `main.go:1235` (`healthProbeAfterRestart`) | Uses `context.Background()` deliberately: the probe is spawned after `restart()` is called from an HTTP handler whose request context may be cancelled before the new opencode child becomes ready. The probe's own 15s timeout and the `doneCh` early-abort are the correct lifetime controls here. Changing it to use the handler's request context would cause the probe to abort whenever the HTTP connection closed. |

## Deployment: terminationGracePeriodSeconds

The 25-second `shutdownCtx` budget plus `proc.stop()`'s 5-second SIGKILL fallback totals **~30 seconds** of graceful shutdown time. Kubernetes sends SIGTERM and then, after `terminationGracePeriodSeconds`, sends SIGKILL to the entire pod.

**The pod manifest MUST set `terminationGracePeriodSeconds: 35`** (or more) to give a buffer between the process's self-managed shutdown and the kubelet's forced kill. Without this, the kubelet may kill the pod before `proc.stop()` completes, dropping the `proc.probeWg.Wait()` and leaving the managed opencode child process without SIGTERM.

To find the existing setting in the Helm chart, run:

```bash
grep -rn 'terminationGracePeriodSeconds' charts/
```

Update both the template file (e.g., `charts/workspace-agentd/templates/deployment.yaml`) and the default in `values.yaml` if present. Add or update:

```yaml
spec:
  terminationGracePeriodSeconds: 35
```

This is an **explicit implementation step** in addition to a code change — see the Execution Steps section below.

## Reload-Secrets Mutex Consideration

`reloadMu` (`secrets.go:56`) is a process-wide mutex that serializes the `/v1/reload-secrets` handler. During shutdown:
1. If a reload is in-flight when SIGTERM arrives, the reload holds `reloadMu`.
2. The shutdown sequence does NOT acquire `reloadMu` — it cancels the server via `Shutdown()`.
3. `http.Server.Shutdown()` drains in-flight requests, so the reload handler will complete.
4. After the handler returns, the server is fully shut down.

This is safe — no explicit `reloadMu` interaction is needed during shutdown. The handler completes naturally as part of server draining.

## Full main.go Structure

The restructured `main.go` should follow this skeleton:

```go
package main

import (
    "context"
    "net/http"
    "os/signal"
    "sync"
    "syscall"
    "time"

    "go.uber.org/zap"

    "github.com/lenaxia/llmsafespace/pkg/agentd"
)

func main() {
    // ---- Phase 1: Setup ----
    // Parse flags, load config, create clients, create handlers
    // (existing setup code, unchanged)

    // ---- Phase 2: Create root context ----
    rootCtx, stop := signal.NotifyContext(context.Background(), syscall.SIGTERM, syscall.SIGINT)
    defer stop()

    // ---- Phase 3: Create background context ----
    // No defer bgCancel() — bgCancel() is called explicitly in the shutdown
    // sequence after server draining (Fix 4). Having both a defer and an
    // explicit call is idempotent but misleading about intent.
    bgCtx, bgCancel := context.WithCancel(rootCtx)

    // ---- Phase 4: Start background goroutines ----
    // Track with WaitGroup for bounded join at shutdown (Fix 9).
    var bgWg sync.WaitGroup

    bgWg.Add(1)
    go func() {
        defer bgWg.Done()
        sseTracker.subscribe(bgCtx, client)
    }()

    bgWg.Add(1)
    go func() {
        defer bgWg.Done()
        fillGaps(bgCtx, client, sseTracker, func() []agentd.SessionInfo {
            cache.mu.Lock()
            sessions := cache.sessions
            cache.mu.Unlock()
            return sessions
        }, fillState)
    }()

    bgWg.Add(1)
    go func() {
        defer bgWg.Done()
        refreshIsHealthyLoop(bgCtx, client, healthCache, log, gr)
    }()

    // ---- Phase 4b: Start relay injector (if configured) ----
    // Pass rootCtx so the health-wait loop can be aborted on shutdown (Fix 7).
    // startRelayInjector is a no-op if INFERENCE_RELAY_BASEURL is unset.
    if relayURL := os.Getenv("INFERENCE_RELAY_BASEURL"); relayURL != "" && proc != nil {
        // ... path/auth setup ...
        startRelayInjector(rootCtx, relayInjectorConfig{
            RelayURL:         relayURL,
            OpenCodeBaseURL:  getAgentAddr(),
            OpenCodePassword: password,
            AgentConfigPath:  envOrDefault("LLMSAFESPACE_AGENT_CONFIG_PATH", agentd.AgentConfigPath),
            AuthJSONPath:     authJSONPath,
            HealthCheck:      func() bool { snap := healthCache.Snapshot(); return snap.Initialized && snap.Healthy },
            KillOpenCode:     func() { proc.restart() },
        })
    }

    // ---- Phase 5: Start admin server (already in goroutine) ----
    adminSrv := &http.Server{Addr: agentd.AgentdAdminAddr, Handler: adminMux, ReadHeaderTimeout: 5 * time.Second}
    adminErrCh := make(chan error, 1)
    go func() {
        log.Info("workspace-agentd admin server starting", zap.String("addr", agentd.AgentdAdminAddr))
        if err := adminSrv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            adminErrCh <- err
        }
    }()

    // ---- Phase 6: Start user server (now in goroutine too) ----
    userSrv := &http.Server{Addr: listenAddr, Handler: userMux, ReadHeaderTimeout: 5 * time.Second}
    userErrCh := make(chan error, 1)
    go func() {
        log.Info("workspace-agentd user server starting", zap.String("addr", listenAddr))
        if err := userSrv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            userErrCh <- err
        }
    }()

    // ---- Phase 7: Wait for signal or error ----
    select {
    case err := <-adminErrCh:
        log.Error("admin server error", zap.Error(err))
    case err := <-userErrCh:
        log.Error("user server error", zap.Error(err))
    case <-rootCtx.Done():
        log.Info("shutdown signal received", zap.String("signal", rootCtx.Err().Error()))
    }

    // ---- Phase 8: Graceful shutdown ----
    // 25s for server draining + up to 5s for proc.stop() SIGKILL fallback = ~30s.
    // Pod manifest must have terminationGracePeriodSeconds: 35 (see Deployment section).
    shutdownCtx, shutdownCancel := context.WithTimeout(context.Background(), 25*time.Second)
    defer shutdownCancel()

    // Both server shutdowns run CONCURRENTLY — each gets the full 25s budget.
    // Sequential shutdown would give the second server only (25s - first_drain_time).
    var serverWg sync.WaitGroup
    serverWg.Add(2)
    go func() {
        defer serverWg.Done()
        log.Info("shutting down user server", zap.String("addr", listenAddr))
        if err := userSrv.Shutdown(shutdownCtx); err != nil {
            log.Error("user server shutdown error", zap.Error(err))
        }
    }()
    go func() {
        defer serverWg.Done()
        log.Info("shutting down admin server", zap.String("addr", agentd.AgentdAdminAddr))
        if err := adminSrv.Shutdown(shutdownCtx); err != nil && err != http.ErrServerClosed {
            log.Error("admin server shutdown error", zap.Error(err))
        }
    }()
    serverWg.Wait()

    log.Info("cancelling background goroutines")
    bgCancel() // explicit call — no defer bgCancel() above (Fix 4)

    // Bounded join: wait up to 5s for background goroutines to exit cleanly.
    waitDone := make(chan struct{})
    go func() { bgWg.Wait(); close(waitDone) }()
    select {
    case <-waitDone:
        log.Info("background goroutines exited cleanly")
    case <-time.After(5 * time.Second):
        log.Warn("background goroutines did not exit within 5s")
    }

    log.Info("stopping managed process")
    if proc != nil {
        proc.stop()
    }

    log.Info("shutdown complete")
}
```

func TestGracefulShutdown(t *testing.T) {
    server := &http.Server{Addr: ":0", Handler: http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {

    }

func TestBackgroundContextCancellation(t *testing.T) {
    bgCtx, bgCancel := context.WithCancel(context.Background())

    done := make(chan struct{})
    go func() {
        defer close(done)
        <-bgCtx.Done()
    }()

    bgCancel()

    select {
    case <-done:
    case <-time.After(2 * time.Second):
        t.Fatal("goroutine did not exit after context cancellation")
    }
}

func TestConcurrentServerShutdown(t *testing.T) {
    // Verify that two servers with the same shutdownCtx both drain within the
    // budget when run concurrently (not sequentially).
    slow := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        time.Sleep(1 * time.Second)
        w.WriteHeader(http.StatusOK)
    })
    srv1 := &http.Server{Handler: slow}
    srv2 := &http.Server{Handler: slow}

    l1, _ := net.Listen("tcp", ":0")
    l2, _ := net.Listen("tcp", ":0")
    go srv1.Serve(l1)
    go srv2.Serve(l2)

    // Both slow requests in flight.
    go http.Get("http://" + l1.Addr().String())
    go http.Get("http://" + l2.Addr().String())
    time.Sleep(100 * time.Millisecond) // let requests land

    shutdownCtx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
    defer cancel()

    start := time.Now()
    var wg sync.WaitGroup
    wg.Add(2)
    go func() { defer wg.Done(); srv1.Shutdown(shutdownCtx) }()
    go func() { defer wg.Done(); srv2.Shutdown(shutdownCtx) }()
    wg.Wait()

    // Both drains ran concurrently: total elapsed should be ~1s, not ~2s.
    if elapsed := time.Since(start); elapsed > 2*time.Second {
        t.Errorf("concurrent shutdown took %v, expected <2s", elapsed)
    }
}
