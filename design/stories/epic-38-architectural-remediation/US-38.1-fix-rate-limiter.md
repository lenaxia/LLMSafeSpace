# US-38.1: Fix Rate Limiter ŌĆö All Three Strategies Non-Functional

**Epic:** 38 ŌĆö Architectural Remediation
**Priority:** CRITICAL (rate limiting is the primary defense against brute-force and abuse; all three strategies are broken)
**Depends on:** None
**Blocks:** None
**Layer:** API ŌĆö services + middleware
**Estimated effort:** Large (3-4 days)

## Summary

The rate limiter is completely non-functional across all three strategies. The root cause is that the `CacheService` interface (a `Get`/`Set` string cache) cannot express the Redis-native atomic operations required for correct rate limiting (`INCRBY`, `ZADD`/`ZRANGEBYSCORE`, `EXPIRE`). This story introduces a new `RedisRateLimitBackend` interface that exposes those primitives, rewrites all three strategy implementations to use it, and fixes the in-memory token bucket's unbounded growth.

### Why it's broken ŌĆö three failure modes

#### Failure Mode 1: Sliding window (stub implementation)

**Lines 58-69 of `ratelimit.go`:**

```go
func (s *Service) AddToWindow(ctx context.Context, key string, timestamp int64, member string, expiration time.Duration) error {
    cacheKey := keyPrefix + key + ":w:" + member
    return s.cache.Set(ctx, cacheKey, "1", expiration)  // Wrong: stores a flat key, not a sorted set member
}

func (s *Service) RemoveFromWindow(ctx context.Context, key string, cutoff int64) error {
    return nil  // Does nothing
}

func (s *Service) CountInWindow(ctx context.Context, key string, min, max int64) (int, error) {
    return 0, nil  // Always returns 0
}
```

`CountInWindow` always returns 0, so the check at `rate_limit.go:249` (`count > limit`) is never true. Every request passes. The sliding window strategy is a no-op.

The underlying reason: `CacheService` has no `ZADD`, `ZRANGEBYSCORE`, or `ZREMRANGEBYSCORE`. The implementer stubbed these out because the interface didn't support sorted sets.

#### Failure Mode 2: Fixed window (non-atomic read-modify-write)

**Lines 44-56 of `ratelimit.go`:**

```go
func (s *Service) Increment(ctx context.Context, key string, value int64, expiration time.Duration) (int64, error) {
    cacheKey := keyPrefix + key
    currentStr, err := s.cache.Get(ctx, cacheKey)  // Step 1: GET
    current := int64(0)
    if err == nil && currentStr != "" {
        current, _ = strconv.ParseInt(currentStr, 10, 64)
    }
    current += value                                // Step 2: add in-process
    if err := s.cache.Set(ctx, cacheKey, strconv.FormatInt(current, 10), expiration); err != nil {  // Step 3: SET
        return 0, fmt.Errorf("ratelimit: increment failed: %w", err)
    }
    return current, nil
}
```

This is a non-atomic GET ŌåÆ increment ŌåÆ SET. Two concurrent requests can both read `count=99`, both increment to `100`, and both SET `100`. The true count should be `101`. The rate limiter systematically undercounts under concurrency, allowing more requests than the configured limit.

The correct implementation is a single Redis `INCRBY` command (atomic) followed by `EXPIRE`.

#### Failure Mode 3: In-memory token bucket (unbounded map, no eviction)

**Lines 22-23, 84-107 of `ratelimit.go`:**

```go
type Service struct {
    // ...
    localBuckets map[string]*bucket
    mu           sync.Mutex
}
```

Three problems:

1. **No eviction.** Every distinct key (`HashString(apiKey)` or `HashString(clientIP)`) creates a permanent entry in `localBuckets`. Over time, this grows without bound ŌĆö a memory leak proportional to the number of unique users/IPs.
2. **Per-process limits.** Each API replica has its own `localBuckets`. N replicas = N├Ś the effective rate limit. An attacker sending requests round-robin across 3 pods gets 3├Ś the configured burst.
3. **No TTL.** Even the `lastTime` field is used only for refill calculation; no entry is ever removed.

### Root cause

The `CacheService` interface (`interfaces.go:87-100`) is a simple string `Get`/`Set` cache. It was designed for general-purpose caching (session storage, API key lookups, etc.) and intentionally hides Redis-specific commands.

The `RateLimiterService` was built on top of `CacheService`, but rate limiting requires Redis-native atomic operations:
- `INCRBY` for atomic counters (fixed window)
- `ZADD` + `ZRANGEBYSCORE` + `ZREMRANGEBYSCORE` for sorted-set windows (sliding window)
- `EXPIRE` for TTL management

Because `CacheService` cannot express these operations, the rate limiter could not implement them correctly. The implementer either stubbed them out (sliding window) or emulated them with non-atomic GET/SET (fixed window).

### New interface: `RedisRateLimitBackend`

Location: `api/internal/interfaces/interfaces.go` (add after `CacheService`)

```go
type RedisRateLimitBackend interface {
    IncrBy(ctx context.Context, key string, value int64) (int64, error)
    Expire(ctx context.Context, key string, ttl time.Duration) error
    TTL(ctx context.Context, key string) (time.Duration, error)
    ZAdd(ctx context.Context, key string, members ...ZMember) error
    ZRemRangeByScore(ctx context.Context, key string, min, max string) error
    ZCard(ctx context.Context, key string) (int64, error)
    Del(ctx context.Context, keys ...string) error
}

type ZMember struct {
    Score  float64
    Member string
}
```

Design notes:
- `IncrBy` accepts a `value int64` parameter and returns the new value after increment (matches Redis `INCRBY` behavior). This is inherently atomic. The `value` is passed through from `RateLimiterService.Increment`'s `value` parameter, satisfying the interface contract.
- `ZAdd` accepts variadic `ZMember` so we can add a single entry per request.
- `ZRemRangeByScore` takes string min/max so callers can use `"-inf"` / `"+inf"` or numeric strings.
- `ZCard` returns the cardinality (count) of the sorted set ŌĆö used for `CountInWindow`.
- `Del` is included for cleanup / testing.
- No `ZRangeByScore` ŌĆö the middleware only needs the count, not the individual entries. If a future dashboard needs entries, it can be added then.

```go
    val, err := b.client.IncrBy(ctx, key, value).Result()
    if err != nil {
        return 0, fmt.Errorf("ratelimit: INCRBY %s: %w", key, err)
    }
    return val, nil
}

    if err := b.client.Expire(ctx, key, ttl).Err(); err != nil {
        return fmt.Errorf("ratelimit: EXPIRE %s: %w", key, err)

    return nil

    val, err := b.client.TTL(ctx, key).Result()
    if err != nil {
        return 0, fmt.Errorf("ratelimit: TTL %s: %w", key, err)
    }
    if val < 0 {
        return 0, nil
    }
    return val, nil
```

#### Fixed window: atomic INCRBY + EXPIRE

```go
func (s *Service) Increment(ctx context.Context, key string, value int64, expiration time.Duration) (int64, error) {
    if s.backend == nil {
        return 0, fmt.Errorf("ratelimit: no backend configured (use NewWithBackend)")
    }
    cacheKey := keyPrefix + key

    val, err := s.backend.IncrBy(ctx, cacheKey, value)
    if err != nil {
        return 0, fmt.Errorf("ratelimit: increment failed: %w", err)
    }

    // Set expiration only on first increment (val == 1) to avoid resetting
    // the window on every request.
    if val == 1 {
        if err := s.backend.Expire(ctx, cacheKey, expiration); err != nil {
            s.logger.Error("ratelimit: failed to set expiry on new counter", err, "key", cacheKey)
        }
    }

    return val, nil
}
```

This replaces the non-atomic GET + increment + SET with a single `INCRBY` ŌĆö an O(1) atomic operation in Redis. The `value` parameter from the interface is now passed through to `IncrBy`, fixing the previous inconsistency where `value` was silently ignored. The `EXPIRE` is set only when the counter is first created (`val == 1`), preventing every request from resetting the TTL.

#### Sliding window: ZADD + ZREMRANGEBYSCORE + ZCARD

```go
func (s *Service) AddToWindow(ctx context.Context, key string, timestamp int64, member string, expiration time.Duration) error {
    if s.backend == nil {
        return fmt.Errorf("ratelimit: no backend configured (use NewWithBackend)")
    }
    cacheKey := keyPrefix + key
    return s.backend.ZAdd(ctx, cacheKey, interfaces.ZMember{
        Score:  float64(timestamp),
        Member: member,
    })
}

func (s *Service) RemoveFromWindow(ctx context.Context, key string, cutoff int64) error {
    if s.backend == nil {
        return fmt.Errorf("ratelimit: no backend configured (use NewWithBackend)")
    }
    cacheKey := keyPrefix + key
    // The '(' prefix means 'strictly less than cutoff' ŌĆö entries at exactly cutoff are kept (window boundary is inclusive).
    return s.backend.ZRemRangeByScore(ctx, cacheKey, "-inf", fmt.Sprintf("(%d", cutoff))
}

func (s *Service) CountInWindow(ctx context.Context, key string, min, max int64) (int, error) {
    if s.backend == nil {
        return 0, fmt.Errorf("ratelimit: no backend configured (use NewWithBackend)")
    }
    cacheKey := keyPrefix + key
    count, err := s.backend.ZCard(ctx, cacheKey)
    if err != nil {
        return 0, err
    }
    return int(count), nil
}
```

**Note on `CountInWindow`**: The `min` and `max` parameters are unused by the implementation. The middleware calls `RemoveFromWindow` (which removes entries with score < cutoff) before `CountInWindow`. After removal, `ZCARD` returns the count of remaining entries, which is exactly the count of requests within the window. The `min`/`max` params exist on the interface only for signature compatibility with existing middleware callers. This is correct and avoids the overhead of `ZRANGEBYSCORE` + counting the results.

**Alternative considered**: Use `ZCOUNT min max` instead of `ZCARD` after `ZREMRANGEBYSCORE`. This would be redundant since `ZREMRANGEBYSCORE` already removed everything outside the window. `ZCARD` is O(1) vs `ZCOUNT` O(log(N)+M), so `ZCARD` is preferred after a prior `ZREMRANGEBYSCORE`.

The middleware must also set expiration on the sorted set key so it doesn't leak. The `AddToWindow` call in the middleware sets per-member TTL, but sorted sets don't have per-member expiry. Instead, we set a key-level `EXPIRE` in the middleware (or in `AddToWindow` itself). The approach: after `ZADD`, call `EXPIRE` with `2 * window`. This ensures the key is cleaned up even if traffic stops.

Updated `AddToWindow` to also set expiry:

```go
func (s *Service) AddToWindow(ctx context.Context, key string, timestamp int64, member string, expiration time.Duration) error {
    if s.backend == nil {
        return fmt.Errorf("ratelimit: no backend configured (use NewWithBackend)")
    }
    cacheKey := keyPrefix + key
    if err := s.backend.ZAdd(ctx, cacheKey, interfaces.ZMember{
        Score:  float64(timestamp),
        Member: member,
    }); err != nil {
        return err
    }
    // Refresh key TTL on every add so the sorted set is cleaned up when
    // traffic stops. Use 2x the configured window to avoid premature expiry
    // during active use.
    if err := s.backend.Expire(ctx, cacheKey, expiration*2); err != nil {
        s.logger.Error("ratelimit: failed to set sorted-set expiry", err, "key", cacheKey)
    }
    return nil
}
```

#### Token bucket: in-memory with eviction

The `Allow` method stays structurally the same (in-memory token bucket) but with these changes:

1. `lastAccess` tracking on every access.
2. `Start()` launches a background eviction goroutine.
3. `Stop()` cancels the goroutine.
4. Map size is bounded at 10,000 entries (configurable). When full, the oldest entries are evicted.

```go
const maxBucketEntries = 10_000

func (s *Service) Start() error {
    if s.maxWindow > 0 {
        ctx, cancel := context.WithCancel(context.Background())
        s.cancelEvict = cancel
        go s.evictLoop(ctx)
    }
    return nil
}

func (s *Service) Stop() error {
    if s.cancelEvict != nil {
        s.cancelEvict()
    }
    return nil
}

func (s *Service) evictLoop(ctx context.Context) {
    // Run eviction every maxWindow interval (at least every 5 minutes)
    interval := s.maxWindow * 2
    if interval < 5*time.Minute {
        interval = 5 * time.Minute
    }
    ticker := time.NewTicker(interval)
    defer ticker.Stop()

    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            s.evictStaleBuckets(time.Now())
        }
    }
}

func (s *Service) evictStaleBuckets(now time.Time) {
    s.mu.Lock()
    defer s.mu.Unlock()

    evictionAge := s.maxWindow * 2
    if evictionAge < 10*time.Minute {
        evictionAge = 10 * time.Minute
    }

    for k, b := range s.localBuckets {
        if now.Sub(b.lastAccess) > evictionAge {
            delete(s.localBuckets, k)
        }
    }
}

func (s *Service) Allow(key string, rate float64, burst int) bool {
    s.mu.Lock()
    defer s.mu.Unlock()

    now := time.Now()
    b, exists := s.localBuckets[key]
    if !exists {
        // Evict oldest entry if at capacity
        if len(s.localBuckets) >= maxBucketEntries {
            s.evictOneOldestLocked(now)
        }
        b = &bucket{tokens: float64(burst), lastTime: now, lastAccess: now}
        s.localBuckets[key] = b
    }

    b.lastAccess = now
    elapsed := now.Sub(b.lastTime).Seconds()
    b.tokens += elapsed * rate
    if b.tokens > float64(burst) {
        b.tokens = float64(burst)
    }
    b.lastTime = now

    if b.tokens < 1 {
        return false
    }
    b.tokens--
    return true
}

func (s *Service) evictOneOldestLocked(now time.Time) {
    var oldestKey string
    var oldestTime time.Time
    first := true
    for k, b := range s.localBuckets {
        if first || b.lastAccess.Before(oldestTime) {
            oldestKey = k
            oldestTime = b.lastAccess
            first = false
        }
    }
    if oldestKey != "" {
        delete(s.localBuckets, oldestKey)
    }
}
```

## Other Options Considered and Why Discarded

### Option 1: Add rate-limit methods to `CacheService`

Add `IncrBy`, `ZAdd`, `ZRemRangeByScore`, etc. to the existing `CacheService` interface.

**Discarded because:**
- Every consumer of `CacheService` (session storage, API key lookups, etc.) would need to implement or mock these methods.
- Violates Interface Segregation Principle ŌĆö general caches don't need sorted sets.
- Makes the interface harder to test ŌĆö every mock must implement 5+ extra methods.
- The mock (`MockCacheService`) is used in many test files beyond rate limiting.

### Option 2: Replace `CacheService` with a generic Redis interface

Create a `RedisService` that exposes all Redis commands and have both cache and rate limiter depend on it.

**Discarded because:**
- Massive blast radius ŌĆö every `CacheService` consumer (10+ files) must be rewritten.
- The abstraction is wrong: `CacheService` intentionally hides Redis-isms for session/object caching.
- This is a refactoring project in itself, not a focused fix.

### Option 3: Use Lua scripts for sliding window

Execute a Lua script via `EVALSHA` for the entire sliding-window check (add + remove + count) in a single Redis round-trip.

**Discarded because:**
- Correct but over-engineered for this fix. The current three-command approach (ZADD, ZREMRANGEBYSCORE, ZCARD) is already correct and has acceptable performance.
- Lua scripts are harder to debug, harder to test, and require script-cache management.
- Can be introduced as a performance optimization in a later story if benchmarks show the three-command approach is too slow.

### Option 4: Replace in-memory token bucket with Redis-backed token bucket

Use a Redis HASH or Lua script to implement the token bucket in Redis so it's shared across replicas.

**Discarded because:**
- The token bucket strategy is intended for per-process, low-latency checks (no Redis round-trip). The per-process limitation (N replicas = N├Ś limit) is documented and acceptable for the token bucket use case.
- If a shared rate limit is needed, the fixed_window or sliding_window strategies should be used instead (they are Redis-backed and shared).
- Adding Redis round-trips to `Allow()` would increase latency from ~50ns to ~1ms per check.

### Option 5: Use a third-party rate-limiting library (e.g., `golang.org/x/time/rate`, `github.com/go-redis/redis_rate`)

**Discarded because:**
- `golang.org/x/time/rate` is in-memory only ŌĆö same per-process limitation as current token bucket, and doesn't fix the fixed_window or sliding_window strategies.
- `github.com/go-redis/redis_rate` is a specific implementation, not an interface. It would couple the project to a specific Redis client library version and prevent mocking.
- The project already has its own rate limiter interface and middleware ŌĆö replacing it wholesale is a larger refactor than fixing the existing code.

## Weaknesses of the Selected Design and Mitigations

### Weakness 1: Per-process token bucket allows N├Ś limit across replicas

**Impact:** An attacker round-robining across 3 API pods gets 3├Ś the token-bucket burst.

**Mitigation:**
- Document in `RateLimitConfig` and project docs that `token_bucket` is a per-instance defense suitable for coarse-grained throttling, NOT a precise distributed rate limit.
- The default strategy is `token_bucket` (set in `DefaultRateLimitConfig`). Consider changing the default to `sliding_window` in a follow-up story so new deployments get a shared rate limit by default.
- Operators who need precise distributed limits should use `fixed_window` or `sliding_window`.

### Weakness 2: Three Redis commands per sliding-window request (ZADD + ZREMRANGEBYSCORE + ZCARD)

**Impact:** Higher Redis latency than a single Lua `EVALSHA`. At high throughput (thousands of req/s), this could become a bottleneck.

**Mitigation:**
- All three commands are O(log N) or O(1) on Redis side (sorted sets are skiplists).
- The three commands execute as separate round-trips. If latency becomes an issue, they can be pipelined (add `Pipeline()` to `RedisRateLimitBackend`) or replaced with a Lua script in a follow-up.
- For now, the correctness fix (from "does nothing" to "works correctly") far outweighs the latency concern.

### Weakness 3: `cache.Service.NewWithClient` shares the `*redis.Client` with the rate limiter

**Impact:** The `*redis.Client` is shared between cache and rate limiter. If one component misconfigures the client (e.g., sets a global timeout), it affects both.

**Mitigation:**
- The shared client is created at the wiring level with appropriate defaults for both consumers.
- Both cache and rate limiter use the same connection pool, which is efficient and avoids doubling connections.
- A future refactor could inject separate clients if isolation is needed.
- This approach is preferred over exposing `Client()` because it doesn't allow arbitrary packages to bypass the cache abstraction.

### Weakness 4: `evictOneOldestLocked` is O(N) on the map

**Impact:** When the map is full (10,000 entries), each new key triggers a linear scan to find the oldest entry. This is ~10,000 comparisons under the mutex lock.

**Mitigation:**
- The map size cap (10,000) is small enough that a linear scan takes < 100╬╝s.
- Eviction only triggers when the map is full. In steady state, the background goroutine handles cleanup.
- A future optimization could use a min-heap or LRU structure, but this is unnecessary at the current scale.

### Weakness 5: Sorted set TTL is set to `2 * window` which may be too aggressive for long windows

**Impact:** If the window is 1 hour, the sorted set key expires after 2 hours. If there's a 1-hour gap in traffic, the key might expire and a new window starts, losing count history.

**Mitigation:**
- The `EXPIRE` is refreshed on every `AddToWindow` call. As long as there's any traffic within the 2├Ś window, the key persists.
- This is correct behavior: if there's no traffic for 2├Ś the window duration, the rate limit window SHOULD reset.
- The 2├Ś multiplier is a safety margin; it can be tuned if needed.

## Failure Modes

### Failure Mode A: Redis down

**Fixed window:** `Increment` returns error ŌåÆ middleware returns 500 ("Rate limit service unavailable"). Fail-open is NOT used ŌĆö when Redis is down, requests are rejected, not allowed through.

**Sliding window:** Same as fixed window ŌĆö `AddToWindow` returns error ŌåÆ middleware returns 500.

**Token bucket:** Unaffected. In-memory token bucket doesn't depend on Redis.

**Decision:** Fail-closed (reject requests when Redis is unavailable) is the correct security posture for a rate limiter. The current code already does this (middleware returns `errors.NewInternalError`), and the new code preserves this behavior.

### Failure Mode B: Redis latency spike

**Impact:** Fixed window and sliding window requests slow down proportionally to Redis response time.

**Mitigation:** All Redis calls should use the `context.Context` from the HTTP request. If the request context has a deadline (e.g., from Gin timeout middleware), the Redis call will time out and return error ŌåÆ 500. The token bucket strategy is unaffected.

### Failure Mode C: Eviction goroutine stopped (process shutting down)

**Impact:** The eviction goroutine stops via `context.Cancel` in `Stop()`. No memory leak ŌĆö the process is shutting down.

### Failure Mode D: Two API replicas use different `maxWindow` values

**Impact:** Each replica independently evicts its local token buckets based on its own `maxWindow`. Since token bucket is per-process anyway, this is harmless.

### Failure Mode E: Clock skew between API process and Redis

**Impact:** Sliding window uses `time.Now().UnixNano()` from the API process as the score in the sorted set. If the API process clock is skewed relative to the Redis server clock, the window boundaries may be slightly off.

**Mitigation:** The impact is bounded by the clock skew (typically < 1 second in a datacenter). This is acceptable for rate limiting. If clock skew is a concern, the Redis `TIME` command could be used instead of `time.Now()`, but this adds a round-trip.

### Failure Mode F: `INCRBY` succeeds but `EXPIRE` fails

**Impact:** The counter key has no TTL and persists forever in Redis, eventually counting to a very large number and permanently blocking the key.

**Mitigation:**
- The `EXPIRE` is set only on first increment (`val == 1`). If it fails, the error is logged.
- A future hardening: use a Lua script that atomically `INCRBY` + `EXPIRE` in one round-trip.
- Operational: Redis key monitoring should alert on keys without TTL (this is an operational best practice regardless).

```go
    errors := make(chan error, 100)

        go func() {
            count, err := svc.Increment(ctx, "concurrent:key", 1, 60*time.Second)
            results <- count
            errors <- err
        }()
    }

    var errCount int
    for i := 0; i < 100; i++ {

        count := <-results
        if err != nil {
            errCount++
        }
        if count > maxCount {
            maxCount = count
        }
    }

    assert.Equal(t, int64(100), maxCount)

    // Verify in miniredis
    val, err := mr.Get("rl:concurrent:key")
    assert.NoError(t, err)
    assert.Equal(t, "100", val)
```

### Step 9: Update service wiring in `services.go`

**File:** `api/internal/services/services.go`

This is the critical wiring step. The rate limiter's `Start()` and `Stop()` must be wired into the service lifecycle. Currently `services.go` skips the rate limiter entirely, meaning the eviction goroutine never starts and the in-memory token bucket map grows without bound.

**Replace** the rate limiter construction:

```go
// Before:
rateLimiter := ratelimit.NewWithCache(log, cacheSvc)

// After (using NewWithClient on cache.Service):
client := redis.NewClient(&redis.Options{
    Addr:     fmt.Sprintf("%s:%d", cfg.Redis.Host, cfg.Redis.Port),
    Password: cfg.Redis.Password,
    DB:       cfg.Redis.DB,
    PoolSize: cfg.Redis.PoolSize,
})
// Verify Redis connectivity at startup ŌĆö fail fast rather than silently
// accepting requests until the first rate-limit check.
if err := client.Ping(context.Background()).Err(); err != nil {
    return fmt.Errorf("redis health check failed: %w", err)
}
cacheSvc := cache.NewWithClient(client)
rateLimiter := ratelimit.NewWithRedisClient(log, client, time.Minute)
```

**Add** `Start()` call in the `Start()` method:

```go
func (s *Services) Start() error {
    // ... existing service start calls ...

    if err := s.RateLimiter.Start(); err != nil {
        return fmt.Errorf("failed to start rate limiter: %w", err)
    }
    return nil
}
```

**Add** `Stop()` call in the `Stop()` method:

```go
func (s *Services) Stop() error {
    // ... existing service stop calls ...

    if err := s.RateLimiter.Stop(); err != nil {
        fmt.Fprintf(os.Stderr, "failed to stop rate limiter: %v\n", err)
    }
    return nil
}
```

## Acceptance Criteria

- [ ] **Fixed window is atomic.** `Increment` uses Redis `INCRBY` (single atomic command). No GET+SET race condition. The `value` parameter is passed through to `IncrBy`. Verified by concurrent test: 100 goroutines incrementing the same key all get unique values, and the final value is exactly 100.
- [ ] **Sliding window works.** `AddToWindow` uses `ZADD` to store entries in a Redis sorted set. `RemoveFromWindow` uses `ZREMRANGEBYSCORE` to evict old entries. `CountInWindow` uses `ZCARD` to count remaining entries (note: `min`/`max` params are unused ŌĆö see interface comment). Verified by integration test: 3 entries added, 2 removed, count returns 1.
- [ ] **Token bucket map is bounded.** `localBuckets` has a maximum size of 10,000 entries. Old entries are evicted by a background goroutine and on-demand when the map is full. Verified by test filling 10,001 entries and asserting map size stays bounded.
- [ ] **Eviction goroutine lifecycle.** `Start()` launches the eviction goroutine. `Stop()` cancels it via context. `services.go` wires `Start()`/`Stop()` into the service lifecycle. No goroutine leak. Verified by `go test -race` and goroutine count check.
- [ ] **`CacheService` interface unchanged.** No methods added, removed, or modified on `CacheService`. All existing `CacheService` consumers compile and pass tests unchanged.
- [ ] **`RateLimiterService` interface unchanged.** No method signatures change. Interface comments added for `CountInWindow` (unused params) and `GetWindowEntries` (stub). Middleware compiles and passes all existing tests unchanged.
- [ ] **`NewWithCache` is deprecated but compiles.** Callers that haven't migrated yet get clear runtime errors ("no backend configured") instead of silently passing all requests.
- [ ] **Nil logger guarded.** `NewWithBackend` and `NewWithCache` both handle `nil` logger by creating a default fallback. No nil pointer panic from a nil logger.
- [ ] **No regression in middleware behavior.** All existing middleware tests (`rate_limit_test.go`, `rate_limit_settings_test.go`) pass without modification.
- [ ] **Fail-closed on Redis errors.** When Redis is unavailable, `Increment`, `AddToWindow`, `RemoveFromWindow`, `CountInWindow`, and `GetTTL` all return errors. The middleware translates these to HTTP 500 ("Rate limit service unavailable"). No request is allowed through when the rate limit cannot be checked.
- [ ] **Token bucket unaffected by Redis.** `Allow()` is pure in-memory. It does not call the backend. It works even when Redis is down.
- [ ] **TTL is real.** `GetTTL` returns the actual Redis TTL, not a hard-coded `60s`. Verified by integration test.
- [ ] **New `RedisRateLimitBackend` mock exists.** `MockRedisRateLimitBackend` in `api/internal/mocks/ratelimiter.go` implements the new interface for use in unit tests.
- [ ] **`GetWindowEntries` documented as stub.** Interface comment documents that it returns nil and is not used by middleware.
