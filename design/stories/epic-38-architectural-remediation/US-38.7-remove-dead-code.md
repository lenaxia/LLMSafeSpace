## Summary

The codebase carries eight categories of dead code — commented-out files, unused helpers, disabled feature branches, unregistered middleware, and phantom constants. Every unused line is a line someone might waste time reading, debugging, or refactoring. This story systematically removes all of it.

**Rule:** Before deleting any item, run the grep verification command listed below. If the grep returns **any** hit outside the file being deleted, **do not remove that item** — instead document the references in a comment on this story and skip it.

### Item 5 — `api/internal/handlers/proxy.go` (stripPatch permanently disabled)

**What:** The `stripPatch` boolean is hard-coded to `false` at its only assignment site
(`proxy.go:560`). The comment at lines 556–559 explains why re-enabling it is non-trivial
(opencode v1.15+ gzip-compresses JSON responses, which would break `json.Unmarshal` inside
`stripPatchParts`). The entire filter branch — including the three helper functions and their
call site — is permanently unreachable dead code.

**Grep verification:**
```bash
grep -rn 'stripPatch\|StripPatch\|strip_patch\|stripPatchParts\|filterOutPatch\|messageEnvelope' \
  --include='*.go' .
grep -rn 'maxNonStreamingResponseBytes' --include='*.go' .
grep -rn 'TestProxy_G1' --include='*.go' .
```

**Scope of changes (all must be done atomically in one commit):**

**(a) Remove `stripPatchParts`, `filterOutPatch`, and `messageEnvelope`** (proxy.go:771–846).
These three declarations form a single logical unit; removing any subset leaves orphan references.
The doc-comment for `stripPatchParts` begins at line 771, not 777 — include it in the removal.

**(b) Remove the `shouldFilter` branch inside `doProxy`** (proxy.go:655–679).
Specifically, remove the block:
```go
// Determine whether to filter the response. Filtering only applies when ...
contentType := resp.Header.Get("Content-Type")
isJSON := strings.Contains(contentType, "application/json")
shouldFilter := stripPatch && isJSON && resp.StatusCode >= 200 && resp.StatusCode < 300

if shouldFilter {
    // ... entire buffered-read / stripPatchParts block ...
    return nil
}
```
This is the call site at proxy.go:671 (`filtered, filterErr := stripPatchParts(raw)`) — leaving
it in place while removing `stripPatchParts` produces a compile error. Both must go together.

**(c) Remove the `stripPatch bool` parameter from `doProxy`'s signature** (proxy.go:609).
The function signature becomes:
```go
func (h *ProxyHandler) doProxy(c *gin.Context, podIP, targetPath, password string, body []byte) error
```

**(d) Update both callers of `doProxy` to remove the `stripPatch` argument.**
- `proxy.go:561`: `h.doProxy(c, podIP, targetPath, password, bodyBytes, stripPatch)` → remove `, stripPatch`
- `proxy.go:571`: `h.doProxy(c, freshWS.Status.PodIP, targetPath, password, bodyBytes, stripPatch)` → remove `, stripPatch`
- Also remove the `stripPatch := false` declaration at proxy.go:560 (now unused).

**(e) Remove both G1 tests** from `api/internal/handlers/proxy_test.go`:
- `TestProxy_G1_OversizedFilteredResponse_Returns502` (line 1576)
- `TestProxy_G1_WithinLimit_PassesThrough` (line 1635)
These tests exercise the `shouldFilter` path exclusively. Leaving them after removing the
production code results in test compilation errors (references to removed constants/functions).

**(f) Evaluate `maxNonStreamingResponseBytes` constant (proxy.go:40) for removal.**
This constant was introduced solely for the `shouldFilter` path (Epic 25 G1, as noted in its
comment). After removing the `shouldFilter` block, run:
```bash
grep -rn 'maxNonStreamingResponseBytes' --include='*.go' .
```
If the only remaining hit is the declaration itself, remove the constant. If it is referenced
elsewhere, leave it and note the reference.

**(g) Update stale doc-comments.** After removing the `stripPatch bool` parameter, two comments
in `proxy.go` still reference the old parameter and must be updated in the same commit:
- **`proxy.go:604–608`** — the doc-comment for `doProxy` mentions the `stripPatch` parameter
  by name. Remove all `stripPatch`-related sentences from that comment.
- **`proxy.go:458–459`** — the comment above `proxyToWorkspace` references patch filtering.
  Remove the patch-filtering mention from that comment.

**Action:**
1. Apply changes (a) through (g) together — they form one logical unit and cannot be split
   without intermediate compile failures.
2. Run `grep -rn 'maxNonStreamingResponseBytes' --include='*.go' .` and conditionally apply (f).
3. Run `go build ./...` and `go test ./api/internal/handlers/...`.

**Commit message:** `refactor: remove permanently disabled stripPatchParts code from proxy handler`

Include in the commit message: `NOTE: Re-enabling this feature requires stripping Accept-Encoding
from upstream requests because opencode 1.15+ compresses JSON responses via gzip/deflate (see
worklog 0070).`

> **Follow-up: SecurityMiddleware OPTIONS preflight bug**
>
> `cors.go` is the only reference implementation in this codebase for correct CORS preflight
> handling. Its removal surfaces a pre-existing bug: `security.go:114–117` short-circuits
> OPTIONS requests with `c.Next()` **without** setting any CORS headers first:
>
> ```go
> // Skip security checks for OPTIONS requests
> if c.Request.Method == "OPTIONS" {
>     c.Next()
>     return
> }
> ```
>
> Because `RequestIDMiddleware` and `CORSMiddleware` are never registered, no middleware in the
> chain sets `Access-Control-Allow-Origin` or the other preflight response headers before
> `c.Next()` is called. This means cross-origin preflight requests receive no CORS headers and
> browsers will block the actual request.
>
> **This is a pre-existing issue and does not block the current story**, but it must be tracked.
> File a follow-up task: fix `SecurityMiddleware` OPTIONS handling so it sets the required CORS
> response headers (`Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`,
> `Access-Control-Allow-Headers`, `Access-Control-Max-Age`) before calling `c.Next()`, or
> register a proper CORS middleware in the router chain.

## Failure Modes

The following failure scenarios have HIGH probability of occurring if this story is executed from
an earlier draft or without careful reading of the per-item notes above.

### HIGH — Executing old Item 4 description and deleting live controller code

**Risk:** An executor working from a cached or earlier version of this story may follow the
original Item 4 instructions (`remove utils.go:19–59`) and delete `SetCondition`, `FindCondition`,
`IsConditionTrue`, `AddFinalizer`, `RemoveFinalizer`, `IsPodReady`, and `GenerateRandomString`.
This would immediately break `phase_pending.go`, `phase_terminating.go`, and `secrets.go` with
`undefined: common.AddFinalizer` / `common.RemoveFinalizer` / `common.GenerateRandomString`
compile errors.

**Mitigation:** Item 4 in this story is now explicitly marked **"EVALUATED: NOT DEAD CODE —
DO NOT REMOVE"** with the grep evidence inline. There is no removal action. Any executor who
reads Item 4 to completion will find zero work to do and move on.

### HIGH — Removing `stripPatchParts` (proxy.go:777) without removing its call site at proxy.go:671

**Risk:** An executor may remove only the function definitions (`stripPatchParts`,
`filterOutPatch`, `messageEnvelope` at lines 777–846) without also removing the `shouldFilter`
block that calls `stripPatchParts(raw)` at line 671. The result is an immediate compile error:
`undefined: stripPatchParts`.

Conversely, removing only the `shouldFilter` block while leaving the now-unreferenced function
definitions causes `go vet` to report unused unexported functions (if they are not exported) or
silently leaves dead code in place.

**Mitigation:** Item 5 explicitly lists all seven sub-tasks (a)–(g) as an **atomic unit** with a
note that they cannot be split without intermediate compile failures. The commit must include
all changes together.
