# US-38.3: Replace HKDF-SHA256 with Argon2id for Password-Based Key Derivation

## Summary

The function `DeriveKEK` in `pkg/secrets/crypto.go:30-37` uses HKDF-SHA256 for all key derivation, including derivation from low-entropy user passwords. HKDF is designed to expand uniform-entropy key material, not to derive keys from low-entropy passwords. An attacker with a GPU can compute billions of HKDF-SHA256 derivations per second, making brute-force attacks on user passwords trivial. This story splits `DeriveKEK` into two functions — one for passwords (Argon2id) and one for high-entropy keys (HKDF-SHA256) — and migrates all 17 call sites to the correct function.

## Current Problem

### The Code

`pkg/secrets/crypto.go` lines 30-37:

```go
func DeriveKEK(password []byte, salt []byte, info string) ([]byte, error) {
    reader := hkdf.New(sha256.New, password, salt, []byte(info))
    kek := make([]byte, 32)
    if _, err := io.ReadFull(reader, kek); err != nil {
        return nil, fmt.Errorf("HKDF key derivation failed: %w", err)
    }
    return kek, nil
}
```

### Why This Is Wrong

1. **HKDF is not a password hash.** RFC 5869 states HKDF assumes the input keying material has high entropy. User passwords have ~20-40 bits of entropy.
2. **No computational hardness.** HKDF-SHA256 is a single SHA-256 call. A modern GPU computes ~2 billion SHA-256 hashes per second. An 8-character password is cracked in under a minute.
3. **No memory hardness.** Argon2id requires tunable memory (e.g., 64 MB), making GPU/ASIC attacks exponentially more expensive.
4. **No iteration count.** There is no parameter to slow down derivation.
5. **Wrong abstraction for high-entropy inputs.** Callers that derive KEKs from high-entropy key material (recovery keys, API keys, server keys) are forced onto the same path as password callers. These callers should stay on HKDF — it is the correct KDF for high-entropy input — and should not pay the ~50-100ms Argon2id latency penalty.
6. **`info` parameter is meaningless for Argon2id.** Argon2id has no info/context field. Passing `info` into a function that may use Argon2id creates a false sense of domain separation.

## Proposed Design

### Split DeriveKEK Into Two Functions

The root cause of several issues is that a single function serves two fundamentally different use cases. We split it:

```go
// DeriveKEKFromPassword derives a KEK from a low-entropy password using Argon2id.
// Use this for user passwords and operator passphrases.
// The info parameter is NOT used — Argon2id has no info field. Domain separation
// is achieved via unique salts (each derivation must use a distinct random salt).
func DeriveKEKFromPassword(password, salt []byte) ([]byte, error)

// DeriveKEKFromKey derives a KEK from high-entropy key material using HKDF-SHA256.
// Use this for recovery keys, API keys, server keys, and any cryptographic key material.
// The info parameter provides domain separation per HKDF spec (RFC 5869).
func DeriveKEKFromKey(keyMaterial, salt []byte, info string) ([]byte, error)
```

### Why Two Functions (Not One Versioned Function)

| Concern | Single versioned function | Two separate functions |
|---|---|---|
| `info` domain separation | Lost for Argon2id path (silently ignored) | Correct: HKDF path uses `info`, Argon2id path has no `info` param |
| High-entropy callers | Pay ~50-100ms Argon2id latency unnecessarily | Stay on HKDF (microseconds) |
| Type safety | Caller can pass password to HKDF or key to Argon2id | Function name encodes intent; wrong usage is obvious in review |
| Backward compat | One version switch handles both | Password path versions independently; key path is unversioned (HKDF forever) |

### Versioning: Password Path Only

Only `DeriveKEKFromPassword` needs a versioned migration path, because only password-derived KEKs are changing from HKDF to Argon2id. High-entropy callers (recovery keys, API keys, server keys) stay on HKDF permanently — HKDF is the correct KDF for high-entropy input — and need no versioning.

```go
const (
    KDFVersionHKDF     = 0
    KDFVersionArgon2id = 1
    KDFCurrentVersion  = KDFVersionArgon2id

    argon2Time    = 3
    argon2Memory  = 64 * 1024 // 64 MiB (65536 KiB) — argon2 takes memory in KiB
    argon2Threads = 4
    argon2KeyLen  = 32
)
```

#### Implementation in `pkg/secrets/crypto.go`

```go
func DeriveKEKFromPassword(password, salt []byte) ([]byte, error) {
    if len(salt) != 32 {
        return nil, fmt.Errorf("argon2id requires 32-byte salt, got %d bytes", len(salt))
    }
    return argon2.IDKey(password, salt, argon2Time, argon2Memory, argon2Threads, argon2KeyLen), nil
}

// DeriveKEKFromPasswordV0 is the legacy HKDF path for decrypting data encrypted
// with HKDF-derived KEKs. Used ONLY during migration to decrypt existing v0 records.
func DeriveKEKFromPasswordV0(password, salt []byte, info string) ([]byte, error) {
    reader := hkdf.New(sha256.New, password, salt, []byte(info))
    kek := make([]byte, 32)
    if _, err := io.ReadFull(reader, kek); err != nil {
        return nil, fmt.Errorf("HKDF key derivation failed: %w", err)
    }
    return kek, nil
}

func DeriveKEKFromKey(keyMaterial, salt []byte, info string) ([]byte, error) {
    reader := hkdf.New(sha256.New, keyMaterial, salt, []byte(info))
    kek := make([]byte, 32)
    if _, err := io.ReadFull(reader, kek); err != nil {
        return nil, fmt.Errorf("HKDF key derivation failed: %w", err)
    }
    return kek, nil
}
```

**Important:** `DeriveKEKFromKey` replaces `DeriveKEK` for all high-entropy callers with identical HKDF internals and the same function signature. Output is byte-for-byte identical. No migration or versioning is needed for these callers.

### Database Schema Changes

#### Table: `user_keys`

Add column for password-derived KEK version only:

```sql

```

#### Sealed Root Key: File Format (Not a Database Table)

The sealed root key is stored as a **file on disk**, not in a database table. The file format gains a version byte prefix:

```
Byte layout:
[0]     : kdf_version (uint8, 0x00 = HKDF, 0x01 = Argon2id)
[1..32] : salt (32 bytes)
[33..N] : encrypted root key (variable length)
```

**Reading:**

```go
// See LoadSealedRootKey implementation in the Backward compatibility section above.
// Old format = 92 bytes (32 salt + 60 ciphertext). New format = 93 bytes (1 version + 32 salt + 60 ciphertext).
```

**Writing:**

```go
func SaveSealedRootKey(path string, version uint8, salt, ciphertext []byte) error {
    buf := make([]byte, 0, 1+len(salt)+len(ciphertext))
    buf = append(buf, version)
    buf = append(buf, salt...)
    buf = append(buf, ciphertext...)
    return os.WriteFile(path, buf, 0600)
}
```

**Backward compatibility for unversioned files:** Detect the format by byte length, not by inspecting the version byte:

- **Legacy (v0) format:** exactly 92 bytes (32-byte salt + 60-byte ciphertext, no version prefix)
- **Versioned format:** 93+ bytes (1-byte version + 32-byte salt + 60-byte ciphertext)

Detection logic:

```go
func LoadSealedRootKey(path string) (version uint8, salt []byte, ciphertext []byte, err error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return 0, nil, nil, fmt.Errorf("read sealed root key: %w", err)
    }
    if len(data) == 92 {
        // Legacy v0 format: no version byte prefix
        return 0, data[0:32], data[32:], nil
    }
    if len(data) >= 33 {
        // Versioned format: first byte is version
        return data[0], data[1:33], data[33:], nil
    }
    return 0, nil, nil, fmt.Errorf("sealed root key file too short: %d bytes", len(data))
}
```

This avoids the fragile heuristic of checking whether `data[0]` is a "valid version byte": a legacy file whose first byte of salt happens to be `0x00` or `0x01` would be misread as versioned. Length is unambiguous.

### Login-Time Upgrade Path

When a user logs in and their KEK record has `kdf_version = 0`, the system upgrades them to Argon2id within the same login request. This avoids waiting for password changes to migrate users.

**Interface change — add to `KeyStore` interface:**

```go
// UpgradePasswordKDF atomically upgrades a user's KEK from HKDF (v0) to Argon2id (v1).
// Returns upgraded=true if the row was updated; upgraded=false if another concurrent
// login already upgraded it (conditional UPDATE matched no rows).
UpgradePasswordKDF(ctx context.Context, userID string, newWrappedDEK, newSalt []byte) (upgraded bool, err error)
```

**Implementation in `PgKeyStore`:**

```go
func (s *PgKeyStore) UpgradePasswordKDF(ctx context.Context, userID string, newWrappedDEK, newSalt []byte) (bool, error) {
    result, err := s.db.ExecContext(ctx,
        `UPDATE user_keys SET wrapped_dek=$1, salt=$2, kdf_version=1
         WHERE user_id=$3 AND kdf_version=0`,
        newWrappedDEK, newSalt, userID,
    )
    if err != nil {
        return false, fmt.Errorf("upgrade kdf: %w", err)
    }
    rows, err := result.RowsAffected()
    return rows > 0, err
}
```

The conditional `WHERE kdf_version=0` is the concurrency guard: only one of two simultaneous login requests will match and upgrade; the other proceeds with v1 on its next login.

**Location:** The upgrade runs inside `UnlockDEK` (in `key_service.go`), not in `auth.go`'s `Login`. `UnlockDEK` is the single function that performs KEK derivation and DEK decryption — it already has the password, salt, and KDF version. Performing the upgrade here keeps all KDF logic co-located.

**Flow (performed inside `UnlockDEK`, after successful decryption of a v0 record):**

```
1. User authenticates with password.
2. Load user_keys row; read kdf_version and salt.
3. If kdf_version == 0:
   a. Derive KEK via HKDF (v0): kek_v0 = DeriveKEKFromPasswordV0(password, salt, info)
   b. Decrypt wrapped DEK: dek = Unwrap(kek_v0, wrapped_dek)
   c. Generate new 32-byte salt.
   d. Derive KEK via Argon2id (v1): kek_v1 = DeriveKEKFromPassword(password, newSalt)
   e. Re-encrypt DEK: new_wrapped_dek = Wrap(kek_v1, dek)
   f. keyStore.UpgradePasswordKDF(ctx, userID, new_wrapped_dek, newSalt)
      (conditional UPDATE prevents race with concurrent logins)
   g. Zero kek_v0, kek_v1, dek from memory.
4. Proceed with normal login (session token, etc.).
```

**Key details:**

- The upgrade runs in the same transaction as the login to prevent partial states.
- The conditional `WHERE kdf_version = 0` handles the race where two login requests arrive simultaneously — only one will match and upgrade; the other proceeds with v1 on retry.
- If the upgrade fails (DB error, etc.), the login still succeeds using the v0 KEK. The upgrade can be retried on next login. Log the failure for monitoring.
- This means active users are migrated on their very next login, not just on password change. Dormant users who never log in again remain on v0.

### Migration Path

#### Phase 1: Deploy split functions and schema changes (this story)

1. Deploy `DeriveKEKFromPassword`, `DeriveKEKFromPasswordV0`, and `DeriveKEKFromKey` in `pkg/secrets/crypto.go`.
2. Deploy the database migration to add `kdf_version SMALLINT NOT NULL DEFAULT 0` to `user_keys`.
3. Deploy the sealed root key file format reader that handles both versioned (prefix byte) and legacy (no prefix byte) files.
4. Update all 17 call sites to use the correct function.
5. Existing rows have `kdf_version = 0`; existing sealed root key files have no version byte. All decrypt correctly.
6. New password-derived KEKs use Argon2id and write `kdf_version = 1`.
7. Login-time upgrade migrates existing users from v0 to v1 on their next login.

**Result:** Zero downtime, zero data loss. Users continue to log in. Active users are silently upgraded to Argon2id on their first login after deployment.

#### Phase 2: Monitor and clean up (separate story)

1. Monitor the `kdf_version` distribution in `user_keys`. When all active users are on v1, the HKDF password path (`DeriveKEKFromPasswordV0`) can be deprecated.
2. Dormant users remain on v0 indefinitely — their data is still accessible.
3. After a suitable grace period (e.g., 90 days), consider forcing a password reset for remaining v0 users.

#### Phase 3 (optional future): Remove HKDF password path

Once all users are on v1 (or dormant accounts are handled), remove `DeriveKEKFromPasswordV0` and the v0 dispatch branch. The HKDF function remains available for `DeriveKEKFromKey` — it is never removed.

## Weaknesses and Trade-offs

1. **Two functions instead of one.** Callers must choose the correct function. This is a feature, not a bug — the function name encodes the intent, making misuse obvious in code review.

2. **HKDF password path remains for migration.** `DeriveKEKFromPasswordV0` cannot be removed until all `user_keys` rows have `kdf_version >= 1`. The login-time upgrade accelerates this, but dormant users may remain on v0 indefinitely.

3. **Argon2id is slower.** Each derivation takes ~50-100ms (vs. microseconds for HKDF). This affects login latency for the 6 password-path call sites in `key_service.go`. The 9 high-entropy callers (key_service.go:147, 336, 376, 494, 570; auth.go:510, 532, 838; secrets_adapters.go:521) stay on HKDF and are unaffected. Parameters (time=3, memory=64 MiB, threads=4) are tuned per OWASP recommendations.

4. **`info` domain separation is lost for Argon2id.** Argon2id has no info parameter. Domain separation for password-derived KEKs relies on unique salts. This is safe because each KEK derivation uses a cryptographically random 32-byte salt. High-entropy callers retain `info` via `DeriveKEKFromKey`.

5. **Login-time upgrade adds latency for v0 users.** The first login after deployment performs both HKDF and Argon2id derivations plus a DEK re-wrap. This is a one-time cost per user (~100-200ms additional latency on first login only).

## Failure Modes

| Failure | Impact | Mitigation |
|---|---|---|
| Wrong `kdf_version` in DB | Decryption failure for that user | Default 0 ensures existing data works. Test thoroughly. |
| Argon2id parameters too aggressive | Login takes >1s, timeout | Benchmark on target hardware. Make parameters configurable via env var. |
| Sealed root key file has no version byte | Read failure | Detect legacy format by file size and assume v0. |
| Login upgrade fails mid-transaction | User stuck on v0 | Run upgrade in same transaction as login. If it fails, login still succeeds with v0; retry next time. |
| Concurrent login upgrades race | Double re-wrap | Conditional `WHERE kdf_version = 0` — only one wins. Loser retries with v1. |
| `argon2` package not in `go.mod` | Build failure | `go get golang.org/x/crypto` |
| Salt length mismatch | Argon2id returns error | Validate salt length in `DeriveKEKFromPassword`. Existing salts are 32 bytes. |
| Caller uses wrong function | Security bug | Function names are self-documenting. Code review catches `DeriveKEKFromKey` used with a password. |

## Acceptance Criteria

1. `DeriveKEKFromPassword(password, salt)` uses Argon2id and produces 32-byte keys.
2. `DeriveKEKFromKey(keyMaterial, salt, info)` uses HKDF-SHA256 and produces 32-byte keys identical to the old `DeriveKEK`.
3. `DeriveKEKFromPasswordV0(password, salt, info)` produces output identical to the old `DeriveKEK` (backward compat).
4. The old `DeriveKEK` function no longer exists in the codebase.
5. All 17 call sites across `key_service.go`, `auth.go`, `secrets_adapters.go`, and `root_key.go` use the correct new function.
6. Password-path callers (6 sites in `key_service.go` at lines 125, 187, 260, 281, 353, 453; and 2 in `root_key.go`) dispatch on `kdf_version` for decryption and write `kdf_version = 1` for new encryption.
7. High-entropy callers (9 sites: `key_service.go` lines 147, 336, 376, 494, 570; `auth.go` lines 510, 532, 838; `secrets_adapters.go` line 521) use `DeriveKEKFromKey` with no versioning.
8. `user_keys` table has `kdf_version` column. No other tables have this column.
9. Sealed root key file format includes version byte prefix.
10. Login-time upgrade migrates v0 users to v1 on their first login after deployment.
11. The `kdf_version` column and file version byte are NOT exposed in any API response.
12. All unit and integration tests pass.
13. Argon2id parameters match OWASP recommendations (time=3, memory=64 MiB / 65536 KiB, threads=4).
14. No call to `hkdf.New` remains outside of `DeriveKEKFromPasswordV0` and `DeriveKEKFromKey`.
