# S18.11: Decouple readyz from Provider Connectivity

**Epic:** 18 ‚Äî Hot Migration & RWX Storage
**Status:** Planning
**Priority:** High
**Depends on:** S18.10 (benchmark baseline ‚Äî validates this is the dominant gate)
**Estimated Effort:** 3 points

---

## Objective

`/v1/readyz` currently gates on `len(connected) > 0` ‚Äî the workspace pod is not
considered Ready until the opencode `/provider` endpoint returns a non-empty connected
list. This requires a successful round-trip to an upstream LLM provider API (Anthropic,
OpenAI, etc.) before Kubernetes marks the pod ready and before the controller transitions
the workspace to Active.

A slow, rate-limited, or momentarily unavailable provider API directly adds seconds to
minutes to every resume. This is not the right place for that check: provider
connectivity is a _user-facing health signal_, not a _pod availability signal_.

This story separates the two concerns:

- **readyz** (`/v1/readyz`): "Is this pod able to serve requests?" ‚Üí requires only that
  agentd and opencode are up and serving HTTP.
- **ProviderReady condition** (`WorkspaceCondition`): "Is the LLM provider connected?" ‚Üí
  reported asynchronously on the workspace CRD; frontend and API can surface it without
  blocking pod readiness.

## Design

### Change 1 ‚Äî readyz no longer requires provider connectivity

**Before** (`cmd/workspace-agentd/main.go` readyzHandler):
```go
connected, configured, _ := cachedState(r.Context(), client, cache, sseTracker)
ready := snap.Initialized && snap.Healthy && len(connected) > 0
```

**After:**
```go
ready := snap.Initialized && snap.Healthy
```

`connected` and `configured` are still fetched for the response body (for observability
and for the frontend to display), but they no longer gate readyz. A pod whose opencode
process is running and healthy is Ready, period.

The readyz response body remains unchanged so callers that inspect it get the same
information:
```json
{
  "ready": true,
  "providers_connected": [],      ‚Üê may still be empty at Ready time
  "providers_configured": 1,
  "agent_version": "...",
  "agent_type": "opencode"
}
```

### Change 2 ‚Äî Add `WorkspaceConditionProviderReady` condition

Add to `pkg/apis/llmsafespace/v1/workspace_types.go`:

```go
WorkspaceConditionProviderReady WorkspaceConditionType = "ProviderReady"
```

The controller sets this condition from the `/v1/statusz` deep-poll result
(`checkAgentHealth` in `health.go`), which already fetches `connected` from agentd. No
new agentd calls needed.

```go
// In checkAgentHealth, after fetching statusz:
providerStatus := metav1.ConditionFalse
providerReason := "NoProvidersConnected"
providerMsg := "opencode reports no connected LLM providers"
if len(status.Connected) > 0 {
    providerStatus = metav1.ConditionTrue
    providerReason = "ProvidersConnected"
    providerMsg = fmt.Sprintf("%d provider(s) connected: %s",
        len(status.Connected), strings.Join(status.Connected, ", "))
}
setCondition(ws, WorkspaceConditionProviderReady, providerStatus, providerReason, providerMsg)
```

`checkAgentHealth` runs every 15s in Active phase ‚Äî provider readiness is observable
within 15s of provider connectivity changing. This is acceptable; it does not need to be
real-time.

### Change 3 ‚Äî Expose ProviderReady on API status endpoint

`GET /workspaces/:id/status` already returns `conditions`. No handler change needed ‚Äî
the condition appears automatically in the conditions array once set.

### Change 4 ‚Äî Frontend: show ProviderReady warning banner

When `phase == Active` but `conditions.ProviderReady.status == False`:
- Show a non-blocking amber banner: "LLM provider not connected. Check credentials."
- Input is enabled (user can still type; they'll get an agent-level error on send).
- Banner auto-clears when condition flips to True (via status poll or SSE).

This is a frontend change scoped to the workspace view. The amber banner already exists
in the codebase as a component (credential-reload banner from Epic 27a) ‚Äî reuse or
adapt it.

### What does NOT change

- `/v1/statusz` ‚Äî still reports `connected`, `ready` (which includes provider check),
  `healthy`. The controller continues reading it for deep health purposes. `ready` in
  statusz is the **agent**'s definition of ready (healthy + providers connected); this is
  distinct from the **pod**'s definition of ready (serving HTTP). This distinction is now
  explicit.
- `/v1/healthz` ‚Äî unchanged (agentd process liveness only).
- `checkAgentHealth` ‚Äî unchanged (still uses statusz; will now additionally set
  ProviderReady condition).
- Credential injection ‚Äî unchanged. The credential path is independent of this story.

## Expected Latency Impact

Before this change, the dominant gate is:
```
Pod Running ‚Üí readyz 200 = opencode startup + provider API round-trip
                         ‚âà 8‚Äì15s            + 0‚Äì120s
                         = 8‚Äì135s total
```

After this change:
```
Pod Running ‚Üí readyz 200 = opencode startup only
                         ‚âà 8‚Äì15s
```

Expected p99 resume: **15‚Äì25s** (down from 90‚Äì140s). The remaining latency is:
- 10s `InitialDelaySeconds` hard floor
- 5s agentd healthz cache first refresh
- 5s `requeueCreating` controller reconcile

These are addressable in subsequent stories (see S18.12).

## Risks

| Risk | Mitigation |
|------|-----------|
| Proxy routes traffic to a pod whose opencode has no configured provider ‚Üí user gets an error | This already happens today when credentials are wrong. The error is well-handled by opencode and surfaced to the user. The ProviderReady condition gives the frontend a proactive signal. |
| Readyz no longer reflects LLM connectivity ‚Üí monitoring gaps | `ProviderReady` condition fills the gap. `statusz.ready` still carries the full signal for dashboards that need it. |
| checkAgentHealth 15s poll means ProviderReady lags | Acceptable. Provider connectivity is not a time-critical signal for routing; it is a user-facing informational signal. |
