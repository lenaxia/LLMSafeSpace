# US-55.2: Dual-Publish Input Events to the User Stream

**Epic:** 55 - Pending-Input State as a First-Class Session-Activity Facet
**Priority:** Critical
**Phase:** 2 (backend, additive; enables cross-workspace freshness)
**Depends on:** None (backend-independent; US-55.1 consumes the events on the frontend but is not blocked by this story shipping)

## User Story

As a user viewing workspace A, I need the sidebar to surface a pending question/permission that pops up on workspace B — so that actionable input on any of my workspaces is visible without switching.

## Context (Why This Story Exists)

`session.status` and `agent_died` already dual-publish to both the workspace stream (active-view consumers) and the user stream (cross-workspace, replay-buffered consumers) — `proxy_events.go:116-172` and `proxy_events.go:210-231` respectively. The `agent_died` site carries an explicit comment (lines 217-222) explaining why: `PublishToWorkspace` has no replay buffer, so a reconnecting user misses the event.

`agent.question`/`agent.permission`/`*.resolved` are mis-classified: they are low-frequency **control events** (like `session.status`) but they publish to the workspace stream only (`proxy_events.go:245-301`, `emitNormalizedInputEvent`). The token stream (`opencode.event`, `proxy_events.go:190`) is correctly workspace-only because it is high-volume and would evict the 128-entry replay ring — but input events are human-paced (one per prompt) and pose no such threat. They belong with `session.status`.

This story corrects the classification so input events reach the user stream. It is the **freshness mechanism** (delivery of post-connect events); US-55.3 is the **anti-entropy mechanism** (catch-up on missed events). Both are required (Epic D4).

## Acceptance Criteria

- [ ] `agent.question`, `agent.question.resolved`, `agent.permission`, `agent.permission.resolved` are published to the user stream (via `PublishToUser`) in addition to the existing `publishWorkspaceEvent` call.
- [ ] The user-stream event includes `WorkspaceID` (the workspace-scoped event does not — implicit for its subscribers; required for the frontend to route on the user stream).
- [ ] The auto-approve guard (`proxy_events.go:276`) is honored: auto-approved permissions are published to **neither** stream (existing behavior unchanged).
- [ ] If `WorkspaceOwner(workspaceID)` returns `""` (owner unrecorded, e.g. right after API restart), the user-stream publish is silently skipped; the workspace-stream publish still fires (existing behavior unchanged — R2).
- [ ] If `userBroker` is nil, no panic; workspace-stream publish still fires.
- [ ] No regression to workspace-stream delivery (in-chat prompt cards in `ChatPage` still work).
- [ ] The replay buffer is not threatened: input events are low-frequency and do not evict `session.status`/`agent_died` from the 128-entry ring under normal operation.

## Implementation

### Option A (preferred): dual-publish at the `publishWorkspaceEvent` call sites

There are 6 publish sites for input events — 4 in `emitNormalizedInputEvent` (`proxy_events.go:252,262,282,293`) and 2 in `emitPendingInputRequests` (`proxy_input.go:117,130`). Rather than hand-write the user-stream publish at each, add a small private helper that **mirrors the existing `publishWorkspaceEvent` shape but also publishes to the user stream**:

```go
// publishWorkspaceAndUserEvent delivers an input/control event to both the
// workspace stream (active-view consumers) and the user stream (cross-workspace,
// replay-buffered). Use for low-frequency events that affect global UI state
// (e.g. agent.question/permission). The user-stream copy carries WorkspaceID
// so the frontend can route it. If the workspace owner is unrecorded
// (WorkspaceOwner returns ""), the user-stream publish is skipped silently —
// the workspace-stream publish still fires.
func (h *ProxyHandler) publishWorkspaceAndUserEvent(workspaceID string, evt apitypes.WorkspaceSSEEvent) {
    if h.userBroker == nil {
        return
    }
    h.userBroker.PublishToWorkspace(workspaceID, evt)
    if userID := h.userBroker.WorkspaceOwner(workspaceID); userID != "" {
        evt.WorkspaceID = workspaceID
        h.userBroker.PublishToUser(userID, evt)
    }
}
```

Then replace the 6 `publishWorkspaceEvent` calls for input events with `publishWorkspaceAndUserEvent`.

> Note on Epic D7 ("no `publishControlEvent` helper"): this helper is named honestly for what it does — **publish to both sinks** — not for a false "control = dual" taxonomy. It is scoped to input events (a known, enumerated set), not a general-purpose control-event router. `workspace.phase` stays user-only (`proxy_events.go:32`) and is not routed through this helper. The per-sink decision remains explicit at each call site.

### Option B: inline at each site

If a helper is unwanted even scoped, inline the user-stream publish at each of the 6 sites, mirroring the `onSessionIdle`/`onSessionActive`/`onAgentDied` precedent (which currently hand-write the dual-publish). This duplicates the owner-lookup + `WorkspaceID` set logic 6×, matching the existing duplication. Rejected as the default due to the duplication, but acceptable if the reviewer prefers it.

### Event envelope — resolved (D10)

The user-stream event must expose `session_id` and `request_id` as **top-level fields** on `WorkspaceSSEEvent` so the provider can route without parsing `Data` (which would couple the state layer to the agent's event shape — exactly what the design argues against).

**D10: Add `RequestID string` to `WorkspaceSSEEvent`.** `SessionID` already exists top-level (`sse_event.go:11`). `RequestID` is a 1-line struct addition that makes control events uniform:

```go
// api/internal/types/sse_event.go
type WorkspaceSSEEvent struct {
    EventID     uint64      `json:"event_id,omitempty"`
    WorkspaceID string      `json:"workspace_id,omitempty"`
    Type        string      `json:"type"`
    Phase       string      `json:"phase,omitempty"`
    SessionID   string      `json:"session_id,omitempty"`
    RequestID   string      `json:"request_id,omitempty"`  // NEW — D10
    Status      string      `json:"status,omitempty"`
    EventType   string      `json:"event_type,omitempty"`
    Data        interface{} `json:"data,omitempty"`
}
```

Set `RequestID` and `SessionID` at each input-event publish site in `emitNormalizedInputEvent` and `emitPendingInputRequests`:

```go
// For asked events: req.ID is the request ID, req.SessionID is the session.
h.publishWorkspaceAndUserEvent(workspaceID, apitypes.WorkspaceSSEEvent{
    Type:      "agent.question",
    SessionID: req.SessionID,   // D10
    RequestID: req.ID,          // D10
    Data:      req,
})

// For resolved events: resolution.ID is the request ID.
h.publishWorkspaceAndUserEvent(workspaceID, apitypes.WorkspaceSSEEvent{
    Type:      "agent.question.resolved",
    SessionID: resolution.SessionID,  // D10
    RequestID: resolution.ID,         // D10
    Data:      map[string]string{...},
})
```

The `Data` payload is unchanged (no frontend regression for workspace-stream consumers like `ChatPage`). The top-level fields are additive (`omitempty`).
