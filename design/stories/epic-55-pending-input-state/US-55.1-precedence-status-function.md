# US-55.1: `useSessionStatus` + `pending > busy` Precedence

**Epic:** 55 - Pending-Input State as a First-Class Session-Activity Facet
**Priority:** Critical
**Phase:** 1 (frontend-only; fixes the reported symptom with no sourcing change)
**Depends on:** None

## User Story

As a user, when a session has a pending question or permission request, I need the sidebar to reliably show the amber `?` instead of the blue spinner — even while the session is still busy — so that I am always alerted to actionable input.

## Context (Why This Story Exists)

The reported bug: the `?` only inconsistently replaces the spinner. Root cause is a render-cascade precedence error in the view (`Sidebar.tsx:813-821`): `isBusy` is evaluated before `showPending`, and a session awaiting input is *by definition still busy* (opencode never emits `session.status=idle` mid-turn). So the spinner shadows the `?` whenever both are true — which is almost always.

The systematic fix relocates precedence out of the view into a single state-layer pure function (`useSessionStatus`), so the view becomes a `switch` that physically cannot re-introduce the shadowing.

This story ships safely **alone**: it changes only where precedence is computed, not where pending state is sourced. Cross-workspace visibility and anti-entropy are US-55.2/55.3.

## Acceptance Criteria

- [ ] A `SessionDisplayStatus` type is defined with members: `pending_input`, `busy`, `unread`, `idle`.
- [ ] A `useSessionStatus(sessionId)` hook resolves the status via a total-order precedence function: `pending_input > busy > unread > idle`.
- [ ] `pending_input` outranks `busy` (a waiting agent is busy because it is pending; pending is actionable).
- [ ] `Sidebar.tsx` `SessionTreeRow` consumes `useSessionStatus(s.id)` via a `switch` (or equivalent) and renders the `HelpCircle` for `pending_input`, `Loader2` for `busy`, `MessageSquareText` for `unread`, `MessageSquare` for `idle`.
- [ ] The `depth===0` gate is preserved as a **view-layer** concern: a subtask row whose status is `pending_input` still renders `busy` (spinner); only the parent shows the `?` via the existing `pendingIndicatorIds` bubble-up (`Sidebar.tsx:513-526`). (Epic D6.)
- [ ] The amber pulse class binds to `status === "pending_input" || status === "unread"` on the title span.
- [ ] No regression: the spinner shows for busy-only sessions; the `?` shows for pending sessions that are also busy (the case the existing tests missed — F7).
- [ ] The view contains **no core-precedence logic** — only a status→icon `switch`. Core precedence (`pending > busy > unread > idle`) lives in `resolveSessionStatus`, which the view cannot override via the `switch`. The depth-gating ternary (`depth===0` override for `pending_input`) is an explicit view-layer concern (D6) and is documented as such — a future edit could still change it, but cannot silently re-shadow the core precedence.

## Implementation

### 1. Define the status type and precedence function

In `SessionActivityProvider.tsx` (or a small adjacent `sessionStatus.ts` module if the provider should not own view-shaped enums — prefer colocating with the provider since the hooks already live there):

```ts
export type SessionDisplayStatus =
  | "pending_input" | "busy" | "unread" | "idle";

export function resolveSessionStatus(input: {
  isPendingInput: boolean;
  isBusy: boolean;
  isUnread: boolean;
}): SessionDisplayStatus {
  if (input.isPendingInput) return "pending_input";
  if (input.isBusy) return "busy";
  if (input.isUnread) return "unread";
  return "idle";
}
```

### 2. Expose `useSessionStatus`

```ts
export function useSessionStatus(sessionId: string): SessionDisplayStatus {
  const isBusy = useIsSessionBusy(sessionId);
  const isUnread = useIsSessionUnread(sessionId);
  const isPendingInput = useIsSessionPendingAction(sessionId);
  // dead/errored deferred — see A3 (no workspace-phase hook available).
  return resolveSessionStatus({ isBusy, isUnread, isPendingInput });
}
```

### 3. Consume in `Sidebar.tsx` `SessionTreeRow`

Replace the cascade at lines 813-821:

```tsx
// current (lines 734-738): predicates with manual precedence
const isBusy = useIsSessionBusy(s.id);
const isUnread = useIsSessionUnread(s.id);
const showPulse = isUnread && !isSelected && !isBusy && depth === 0;
const showPending = depth === 0 && pendingIndicatorIds.has(s.id);
```

with:

```tsx
const status = useSessionStatus(s.id);
const isSelected = s.id === selectedSessionId;
// depth gate is a VIEW concern (D6): subtasks render busy even if pending.
const rowStatus: SessionDisplayStatus =
  depth === 0 && pendingIndicatorIds.has(s.id) ? "pending_input"
  : status === "pending_input" ? "busy"   // pending subtask → spinner on its own row
  : status;
const showPulse = (rowStatus === "unread") && !isSelected && depth === 0;
```

and the icon block (lines 813-821):

```tsx
{(() => {
  switch (rowStatus) {
    case "pending_input": return <HelpCircle className="h-3.5 w-3.5 flex-shrink-0 animate-unread-pulse text-amber-500" />;
    case "busy":          return <Loader2 className="h-3.5 w-3.5 animate-spin text-blue-500 flex-shrink-0" />;
    case "unread":        return <MessageSquareText className="h-3.5 w-3.5 flex-shrink-0 animate-unread-pulse" />;
    default:              return <MessageSquare className="h-3.5 w-3.5 flex-shrink-0" />;
  }
})()}
```

The title span pulse binds to `rowStatus === "pending_input" || rowStatus === "unread"`.

> Note: the `depth===0` handling for `pending_input` preserves the bubble-up model — a subtask's pending status surfaces as `pending_input` on the **parent** via `pendingIndicatorIds`, and as `busy` on the subtask's own row. This is the existing intended UX (A4), made explicit rather than emergent.
