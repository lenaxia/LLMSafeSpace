# US-55.3: Snapshot Anti-Entropy + Provider Reconcile

**Epic:** 55 - Pending-Input State as a First-Class Session-Activity Facet
**Priority:** Critical
**Phase:** 3 (anti-entropy + ownership correction)
**Depends on:** US-55.2 (dual-publish — the snapshot events must reach the user stream for the provider to reconcile from them)

## User Story

As a user, when I reload the page (or my SSE connection drops and reconnects), I need the sidebar to show the `?` for any workspace that has a pending question/permission — including workspaces I am not currently viewing — so that the indicator survives cold start and is correct across all my workspaces.

## Context (Why This Story Exists)

Two gaps remain after US-55.1 (precedence) and US-55.2 (dual-publish):

1. **Cold start / reload.** US-55.2's live deltas only deliver events that fire *after* connect. A question that fired before connect and is outside the 128-entry replay window (cross-replica re-routing, long disconnect) is lost. `busy` solves this with `seedBusy` (REST seed, `SessionActivityProvider.tsx:64-94`); `unread` solves it with `reconcileUnread` (REST-authoritative re-read, lines 109-149). Pending has no equivalent.

2. **Ownership inversion.** The provider's `pendingActions` is populated by `ChatPage` (`ChatPage.tsx:688,699`), which is mounted only for the viewed workspace. Even with US-55.2, the provider is not the authoritative owner of the pending set — it's a passive receiver of view-pushed updates. **Additionally, the provider's `onEvent` has no handler for `agent.question`/`agent.permission`/`*.resolved` at all** (`SessionActivityProvider.tsx:172-279` handles only `session.status` and `workspace.phase`). Those events are consumed exclusively by `ChatPage` on the workspace stream. Without adding input-event handling to the provider's `onEvent`, US-55.2's dual-published events flow into the provider and are silently dropped.

This story closes both gaps: it makes the pod the authoritative source of truth (via the existing `emitPendingInputRequests` primitive), extends the existing `snapshotUserWorkspaces` fan-out to seed pending across all the user's Active workspaces on connect/reconnect, adds input-event handling to the provider's `onEvent` (the core wiring that makes the provider the authoritative owner), and uses a per-workspace marker event to commit snapshots without flicker.

This is the **anti-entropy mechanism** (catch-up); US-55.2 is the **freshness mechanism** (delivery). Both required (Epic D4).

## Acceptance Criteria

- [ ] On user-stream connect and reconnect, `snapshotUserWorkspaces` fans out `emitPendingInputRequests` for each Active workspace, so the pod's authoritative pending set is re-emitted as `agent.question`/`agent.permission` events to the user stream (via US-55.2's `publishWorkspaceAndUserEvent`).
- [ ] The fan-out is bounded: ≤10 active workspaces (scale constraint), each fetch with a timeout mirroring the existing `snapshotTimeout` budget (`stream_user_events.go:221-237`). A timed-out workspace is skipped gracefully (its pending stays stale until next reconnect — acceptable).
- [ ] The fan-out does not block the existing phase snapshot (run concurrently or after, with its own budget).
- [ ] **The provider's `onEvent` handles `agent.question`, `agent.permission`, `agent.question.resolved`, `agent.permission.resolved`**, calling `addPendingAction`/`removePendingAction` internally. This is the core wiring that makes the provider the authoritative owner — without it, dual-published events are silently dropped. (Finding 1.)
- [ ] The user-stream event exposes `session_id` and `request_id` as top-level fields on `WorkspaceSSEEvent` (D10 — US-55.2 adds the `RequestID` field). The provider reads these without parsing `Data`.
- [ ] **Per-workspace marker commit (D9).** `emitPendingInputRequests` emits an `agent.input.snapshot_complete` marker (with `WorkspaceID`) after publishing all question/permission events for a workspace — unconditionally, via `defer`, even on timeout/error. The provider stages snapshot events per-workspace and commits atomically on the marker, replacing `pendingActions` for that workspace's sessions only. This prevents the global flicker that a synchronous wipe would cause (Finding 2).
- [ ] **No global wipe on reconnect.** The provider does NOT wipe `pendingActions` on `onReconnect` (unlike `busySessions`, which reads its seed synchronously from the React Query cache). Pending's rebuild depends on N async pod fetches; a global wipe would blank all `?`s until the slowest fetch completes. Instead, the per-workspace marker replaces each workspace's entries independently as its snapshot completes.
- [ ] `ChatPage`'s `addPendingAction`/`removePendingAction` calls become **optimistic updates**: they update local state for responsiveness; the snapshot commit on reconnect authoritatively confirms/corrects.
- [ ] No regression: in-chat prompt cards (`ChatPage`'s `setPendingQuestions`/`setPendingPermissions`) work identically.

## Implementation

### Backend: extend `snapshotUserWorkspaces`

In `stream_user_events.go`, after the phase-snapshot loop (around line 259), add a pending fan-out for each Active workspace:

```go
// Pending-input anti-entropy (Epic 55): re-emit each Active workspace's
// authoritative pending set from the pod so a reconnecting client rebuilds
// pendingActions correctly (mirrors busy's seedBusy). Bounded by the ≤10
// active-workspace scale constraint; each fetch is timeout-guarded.
if h.dialect != nil {
    for _, wsID := range wsIDs {
        phase := ""
        if phases != nil {
            phase = phases[wsID]
        }
        if phase != phaseActive {
            continue
        }
        go func(id string) {
            defer func() { _ = recover() }() // never let a fetch panic the snapshot
            h.emitPendingInputRequests(id)
        }(wsID)
    }
}
```

> Concurrency note: `emitPendingInputRequests` already uses `publishWorkspaceEvent` internally (now `publishWorkspaceAndUserEvent` post-US-55.2), which is goroutine-safe. The fan-out may parallelize; for ≤10 workspaces this is fine. If a tighter deadline is wanted, wrap each in a `context.WithTimeout` (mirroring the existing k8s-list timeout at line 206).

### Backend: marker emission in `emitPendingInputRequests`

In `proxy_input.go`, add a `defer` at the top of `emitPendingInputRequests` that emits the snapshot-complete marker unconditionally (D9):

```go
func (h *ProxyHandler) emitPendingInputRequests(workspaceID string) {
    // D9: emit the snapshot-complete marker unconditionally on exit, even on
    // timeout/error. This lets the provider commit (or clear) the workspace's
    // pending set without hanging. On timeout, staging is empty → the provider
    // commits empty (pending cleared for this workspace). This is defensible:
    // the pod is unreachable, so we cannot confirm pending. The next reconnect
    // retries.
    defer func() {
        if h.userBroker == nil {
            return
        }
        if userID := h.userBroker.WorkspaceOwner(workspaceID); userID != "" {
            h.userBroker.PublishToUser(userID, apitypes.WorkspaceSSEEvent{
                Type:        "agent.input.snapshot_complete",
                WorkspaceID: workspaceID,
            })
        }
    }()

    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    // ... existing fetch + publish logic unchanged ...
}
```

The marker is published to the **user stream only** via `PublishToUser` (not `publishWorkspaceAndUserEvent`). The marker is for the provider's reconcile logic; `ChatPage` (workspace-stream consumer) does not need it. Using `PublishToUser` directly avoids unnecessary noise on the workspace stream and keeps the marker's scope honest (it is a provider signal, not an input event).

### Frontend: input-event handling in the provider's `onEvent` (Finding 1 — core wiring)

In `SessionActivityProvider.tsx`, extend `onEvent` (currently at lines 172-279) to handle the four input event types and the marker:

```ts
// After the existing session.status and workspace.phase handlers:

if (evt.type === "agent.question" || evt.type === "agent.permission") {
  const wsId = evt.workspace_id!;
  const sessionId = evt.session_id!;
  const requestId = evt.request_id!;  // D10: top-level field
  if (snapshotCommittedWs.has(wsId)) {
    // Post-snapshot (live event): apply directly.
    addPendingAction(wsId, sessionId, requestId);
  } else {
    // Pre-marker (snapshot or live-during-snapshot): stage.
    stagePendingAction(wsId, sessionId, requestId);
  }
}

if (evt.type === "agent.question.resolved" || evt.type === "agent.permission.resolved") {
  const wsId = evt.workspace_id!;
  const requestId = evt.request_id!;  // D10: top-level field
  // Remove from staging (so a stale snapshot doesn't re-commit it).
  unstagePendingAction(wsId, requestId);
  // Also remove from pendingActions (clears pre-reconnect stale value,
  // or the live value if already committed).
  removePendingAction(requestId);
}

if (evt.type === "agent.input.snapshot_complete") {
  const wsId = evt.workspace_id!;
  // Commit: replace pendingActions for this workspace's sessions with staging.
  commitStagedPending(wsId);  // sets pendingActions[sessions in staging[wsId]] = staging[wsId]
  snapshotCommittedWs.add(wsId);
}
```

Where `stagePendingAction`, `unstagePendingAction`, and `commitStagedPending` operate on a `stagingRef`:

```ts
// Per-workspace staging buffer. Keyed by workspaceId → Set<{sessionId, requestId}>.
// Accumulates snapshot events until the workspace's marker arrives.
const stagingRef = useRef(new Map<string, Set<{ sessionId: string; requestId: string }>>());

// Workspaces whose marker has been received since the last reconnect.
// Events for committed workspaces go live; events for uncommitted stage.
const snapshotCommittedWsRef = useRef(new Set<string>());
```

### Frontend: `onReconnect` (no global wipe — Finding 2)

In `onReconnect` (`SessionActivityProvider.tsx:168-171`), do NOT wipe `pendingActions`. Instead, reset the staging state:

```ts
onReconnect: () => {
  seededRef.current.clear();
  setBusySessions(new Map());     // busy: wipe (seedBusy rebuilds synchronously from cache)
  // pending: do NOT wipe. Per-workspace marker commits replace entries
  // incrementally as each workspace's snapshot completes. A global wipe
  // would blank all ?s until the slowest pod fetch finishes (Finding 2).
  stagingRef.current.clear();
  snapshotCommittedWsRef.current.clear();
}
```

### How the marker model handles each race

| Scenario | Flow | Result |
|---|---|---|
| Normal snapshot | question events → marker | staged, then committed; `?` shows |
| Live question during snapshot | question event (pre-marker) → stages → marker commits | correct — it IS pending |
| Live resolve during snapshot | resolve event → unstages + removes from pendingActions → marker commits without it | correct — resolved before commit |
| Live resolve after marker | resolve event (W committed) → removes from pendingActions directly | correct |
| Stale pre-reconnect entry cleared by snapshot | reconnect (no wipe) → snapshot returns empty → marker commits empty → pendingActions[W] replaced with empty | correct — ghost cleared without flicker |
| Pod timeout | `defer` marker fires with empty staging → pendingActions[W] = empty | defensible — pod unreachable; retries on next reconnect |
| Non-Active workspace | no fan-out → no marker → `workspace.phase` handler clears pending | correct (A8) |

`addPendingAction`/`removePendingAction` (called from `ChatPage`) stay as optimistic updates for responsiveness; the snapshot commit on reconnect confirms them. No `clearedRef`-style suppression is needed because the marker commit is authoritative-replace per-workspace.

### `ChatPage` changes

`ChatPage` keeps `setPendingQuestions`/`setPendingPermissions` (view-local prompt cards — unchanged). Its `addPendingAction`/`removePendingAction` calls remain (now optimistic). No other change.
