# US-16.8: Frontend Types + API Client

**Epic:** 16 - Agent Input Requests
**Priority:** Critical
**Depends on:** US-16.2a (API exists to call)
**Layer:** Frontend

## User Story

As a frontend developer, I need TypeScript types and API client methods for the question/permission system, so that I can build UI components with type safety.

## Acceptance Criteria

- [ ] `QuestionOption`, `QuestionInfo`, `QuestionRequest`, `PermissionRequest` types added to `api/types.ts`
- [ ] `AgentQuestionEvent`, `AgentQuestionResolvedEvent`, `AgentPermissionEvent`, `AgentPermissionResolvedEvent` SSE event types added
- [ ] `WorkspaceStreamEvent` union updated to include new event types
- [ ] New `api/input.ts` module with methods: `questionReply`, `questionReject`, `permissionReply`, `listQuestions`, `listPermissions`
- [ ] All API methods are typed (request body + response)
- [ ] API methods handle errors consistently with existing patterns (`ApiClientError`)

## Technical Details

### Types (`api/types.ts` additions)

```typescript
export interface QuestionOption {
  label: string;
  description: string;
}

export interface QuestionInfo {
  question: string;
  header: string;
  options: QuestionOption[];
  multiple?: boolean;
  // custom is absent from the event; treat absence as true
}

export interface QuestionRequest {
  id: string;
  session_id: string;
  questions: QuestionInfo[];
  tool?: { message_id: string; call_id: string };
}

export interface PermissionRequest {
  id: string;
  session_id: string;
  permission: string;
  patterns: string[];
  metadata?: Record<string, unknown>;
  always?: string[];
  tool?: { message_id: string; call_id: string };
}

export interface AgentQuestionEvent {
  type: "agent.question";
  data: QuestionRequest;
}

export interface AgentQuestionResolvedEvent {
  type: "agent.question.resolved";
  data: { request_id: string; session_id: string };
}

export interface AgentPermissionEvent {
  type: "agent.permission";
  data: PermissionRequest;
}

export interface AgentPermissionResolvedEvent {
  type: "agent.permission.resolved";
  data: { request_id: string; session_id: string; reply: string };
}
```

### API Client (`api/input.ts`)

```typescript
import { api } from "./client";
import type { QuestionRequest, PermissionRequest } from "./types";

export const inputApi = {
  listQuestions: (workspaceId: string) =>
    api.get<QuestionRequest[]>(`/workspaces/${workspaceId}/question`),

  questionReply: (workspaceId: string, requestId: string, answers: string[][]) =>
    api.post<boolean>(`/workspaces/${workspaceId}/question/${requestId}/reply`, { answers }),

  questionReject: (workspaceId: string, requestId: string) =>
    api.post<boolean>(`/workspaces/${workspaceId}/question/${requestId}/reject`, {}),

  listPermissions: (workspaceId: string) =>
    api.get<PermissionRequest[]>(`/workspaces/${workspaceId}/permission`),

  permissionReply: (workspaceId: string, requestId: string, reply: "once" | "always" | "reject", message?: string) =>
    api.post<boolean>(`/workspaces/${workspaceId}/permission/${requestId}/reply`, { reply, ...(message ? { message } : {}) }),
};
```

# US-16.11: ChatPage Integration

**Epic:** 16 - Agent Input Requests
**Priority:** Critical
**Depends on:** US-16.9, US-16.10, Epic 15
**Layer:** Frontend

## User Story

As a user chatting with the agent, I need question and permission prompts to appear and disappear at the right times, so that the interaction feels seamless and I'm never confused about what the agent is waiting for.

## Acceptance Criteria

- [ ] `agent.question` SSE event → `pendingQuestion` state set → `QuestionPrompt` renders
- [ ] `agent.question.resolved` SSE event → `pendingQuestion` cleared → prompt disappears
- [ ] `agent.permission` SSE event → `pendingPermission` state set → `PermissionPrompt` renders
- [ ] `agent.permission.resolved` SSE event → `pendingPermission` cleared → prompt disappears
- [ ] Only prompts for the CURRENT session are shown (filter by `session_id`)
- [ ] Deduplication: if a prompt with the same `id` is already showing, ignore the duplicate event
- [ ] Multiple prompts can be pending simultaneously (render both — question above permission)
- [ ] When session goes idle (`session.status: idle`), clear ALL pending prompts for that session
- [ ] When `session.error` fires for the session, clear ALL pending prompts
- [ ] On page mount (reconnect), pending prompts appear via `emitPendingInputRequests` (US-16.4)

## Technical Details

### State in ChatPage

```typescript
const [pendingQuestions, setPendingQuestions] = useState<QuestionRequest[]>([]);
const [pendingPermissions, setPendingPermissions] = useState<PermissionRequest[]>([]);
```

### Event Handling (in `handleSSEEvent`)

```typescript
if (event.type === "agent.question") {
  const req = event.data as QuestionRequest;
  if (req.session_id === sessionId) {
    setPendingQuestions(prev => {
      if (prev.some(q => q.id === req.id)) return prev; // dedup
      return [...prev, req];
    });
  }
}
if (event.type === "agent.question.resolved") {
  const { request_id } = event.data;
  setPendingQuestions(prev => prev.filter(q => q.id !== request_id));
}
if (event.type === "agent.permission") {
  const req = event.data as PermissionRequest;
  if (req.session_id === sessionId) {
    setPendingPermissions(prev => {
      if (prev.some(p => p.id === req.id)) return prev;
      return [...prev, req];
    });
  }
}
if (event.type === "agent.permission.resolved") {
  const { request_id } = event.data;
  setPendingPermissions(prev => prev.filter(p => p.id !== request_id));
}
```

### Clear on Idle/Error (US-16.12)

```typescript
if (event.type === "session.status" && event.session_id === sessionId && event.status === "idle") {
  setPendingQuestions([]);
  setPendingPermissions([]);
}
```

# US-16.12: Clear Prompts on Session Idle/Error

**Epic:** 16 - Agent Input Requests
**Priority:** High
**Depends on:** US-16.11
**Layer:** Frontend

## User Story

As a user, if the agent crashes or finishes unexpectedly while a question is pending, I need the stale prompt to disappear automatically, so that I'm not confused by a prompt that can no longer be answered.

## Acceptance Criteria

- [ ] When `session.status: idle` arrives for the current session, all pending questions and permissions are cleared
- [ ] When `session.error` arrives for the current session, all pending questions and permissions are cleared
- [ ] The clearing happens regardless of whether the user interacted with the prompt
- [ ] No API calls are made to reject/dismiss — the prompts are simply removed from UI state (opencode already cleaned up internally)

## Technical Details

This is implemented as part of US-16.11's event handling. Separated as a story for explicit testing of the edge cases.

### Why No API Call on Clear

When the session goes idle, it means the agent finished (either the question was answered by another client, or the agent errored/timed out). In all cases, opencode has already cleaned up the pending state internally (the deferred was rejected). Calling `reject` would return `200 true` (idempotent no-op) but serves no purpose.

### Edge Cases

1. **Pod restart**: opencode's in-memory pending map is lost. The session eventually errors or the SSETracker detects disconnect. The `session.status: idle` event fires when the pod comes back and the session is no longer busy.

2. **Agent timeout**: opencode has internal timeouts on tool execution. If the question tool times out, the agent errors, session goes idle, prompts clear.

3. **Another tab answers**: The `question.replied` event fires → `agent.question.resolved` → prompt clears. The idle event is redundant but harmless.
