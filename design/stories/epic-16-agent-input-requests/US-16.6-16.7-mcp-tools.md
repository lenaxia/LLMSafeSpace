# US-16.6: MCP Tools for Question/Permission Reply

**Epic:** 16 - Agent Input Requests
**Priority:** High
**Depends on:** US-16.0, US-16.2
**Layer:** Backend

## User Story

As an MCP client (external LLM agent), I need tools to reply to questions and permissions, so that I can programmatically interact with the agent's input request system.

## Acceptance Criteria

- [ ] `session_question_reply` tool: accepts `workspace_id`, `request_id`, `answers` (array of string arrays); calls the question reply API
- [ ] `session_question_reject` tool: accepts `workspace_id`, `request_id`; calls the question reject API
- [ ] `session_permission_reply` tool: accepts `workspace_id`, `request_id`, `reply` (enum: once/always/reject), optional `message`; calls the permission reply API
- [ ] All tools validate required parameters before making API calls
- [ ] All tools return structured success/error results
- [ ] `request_id` is validated (must start with `que_` or `per_` as appropriate)
- [ ] Tools are registered in `NewServer`

## Technical Details

### HTTPClient Implementation

```go
func (c *HTTPClient) QuestionReply(ctx context.Context, workspaceID, requestID string, answers [][]string) error {
    body := map[string]interface{}{"answers": answers}
    path := fmt.Sprintf("/api/v1/workspaces/%s/question/%s/reply", workspaceID, requestID)
    return c.doJSON(ctx, http.MethodPost, path, body, nil)
}

func (c *HTTPClient) QuestionReject(ctx context.Context, workspaceID, requestID string) error {
    path := fmt.Sprintf("/api/v1/workspaces/%s/question/%s/reject", workspaceID, requestID)
    return c.doJSON(ctx, http.MethodPost, path, nil, nil)
}

func (c *HTTPClient) PermissionReply(ctx context.Context, workspaceID, requestID, reply, message string) error {
    body := map[string]interface{}{"reply": reply}
    if message != "" {
        body["message"] = message
    }
    path := fmt.Sprintf("/api/v1/workspaces/%s/permission/%s/reply", workspaceID, requestID)
    return c.doJSON(ctx, http.MethodPost, path, body, nil)
}
```

### ID Validation for MCP

# US-16.7: MCP SendMessage Question Detection

**Epic:** 16 - Agent Input Requests
**Priority:** High
**Depends on:** US-16.0, US-16.6
**Layer:** Backend

## User Story

As an MCP client sending a message to the agent, I need to know when the agent asks a question (instead of completing normally), so that I can answer the question and let the agent continue.

## Acceptance Criteria

- [ ] When `SendMessage` is waiting for `session.idle` via SSE and a `question.asked` event arrives for the same session, it returns early with the question data
- [ ] The return value clearly indicates this is a question (not a normal response)
- [ ] The MCP caller can then use `session_question_reply` to answer, and call `SendMessage` again (or a new tool) to get the final response
- [ ] If a `permission.asked` event arrives, it is auto-approved (headless mode) and `SendMessage` continues waiting for idle
- [ ] Timeout still applies — if neither idle nor question arrives within the timeout, fallback to history

## Technical Details

### Modified `SendMessage` in `client.go`

The SSE parsing loop currently breaks on `session.idle`. Add detection for `question.asked`:

### Return Format for Questions

When a question is detected, `SendMessage` returns a structured JSON string:

```json
{
  "type": "question",
  "request": {
    "id": "que_abc",
    "sessionID": "ses_xyz",
    "questions": [...]
  }
}
```

The MCP tool handler wraps this in a `mcp.NewToolResultText(...)` so the calling LLM sees the question and can decide how to respond.

### MCP Caller Flow

```
1. Caller: session_message(workspace_id, session_id, "Create a project")
2. MCP returns: {"type":"question","request":{"id":"que_abc","questions":[{"question":"What language?","options":[...]}]}}
3. Caller reads the question, decides answer
4. Caller: session_question_reply(workspace_id, "que_abc", [["Go"]])
5. Caller: session_message(workspace_id, session_id, "") ← empty message to resume waiting
   OR: caller just waits and calls session_history later
```

**Alternative (simpler)**: After replying to the question, the caller can just call `session_history` to get the final result once the agent finishes. No need to re-subscribe to SSE.
