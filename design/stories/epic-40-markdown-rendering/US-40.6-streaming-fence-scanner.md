# US-40.6: Fix streaming fence scanner — replace line-count heuristic

**Epic:** 40 — Markdown Rendering Overhaul
**Status:** Planning
**Effort:** Small (0.5d)
**Depends on:** None (isolated pure function, no dependency on other stories)
**Blocked by:** Nothing

---

## Problem

`MessagePart.tsx:89-93` applies a streaming fence fix to prevent an in-progress code block from swallowing the rest of the message:

```ts
if (isStreaming) {
  const fenceCount = (text.match(/^```/gm) || []).length;
  if (fenceCount % 2 !== 0) {
    text += "\n```";
  }
}
```

This counts lines that start with exactly three backticks. It is incorrect for:

1. **Four-backtick fences** — ` ```` ` — CommonMark allows fences of 3 or more backticks. A message containing ` ```` code ```` ` has two four-backtick fences but the regex only counts three-backtick lines, finding zero matches and incorrectly appending a three-backtick closing fence.

2. **Tilde fences** — `~~~` — CommonMark allows `~` as the fence character. A tilde fence is counted as zero by the backtick regex.

3. **Fences with language info strings** — ` ```python` — the regex `/^```/gm` matches these correctly (line starts with three backticks). This case works.

4. **Mismatched fence characters** — If an opening ` ```` ` (4 backticks) is followed by the heuristic's appended ` ``` ` (3 backticks), the markdown parser does not close the fence. CommonMark requires the closing fence to use the same character and at least as many characters as the opening fence.

5. **Nested fences in blockquotes** — `> ``` ` — the `^` anchor in `/^```/gm` does not match after `> `. These fences are invisible to the heuristic.

The result of cases 1–4: a mid-stream code block that uses a non-standard fence variant is never closed, and everything after the opening fence is swallowed into the code block for the rest of the message.

---

## User Story

As a user watching an assistant message stream in, all code fence variants (3+ backtick, 4-backtick, tilde, tilde with language info) are handled correctly. The rest of the message text is never erroneously included inside a code block.

---

## Assumptions (validate before implementing)

1. CommonMark spec: a fenced code block opens with a line of 3+ identical fence characters (`` ` `` or `~`), and closes with a line of the same character and at least as many characters — **validated:** CommonMark spec §4.5.
2. `react-markdown` uses `remark` which follows the CommonMark spec — **validated:** remark-parse documentation.
3. The streaming text fix runs before `react-markdown` parses — **validated:** `MessagePart.tsx:88-94` mutates `text` before passing to `ReactMarkdown`.
4. Fences in blockquotes (`> ``` `) are rare in LLM output and acceptable to leave as edge cases — **assumption:** not validated, but LLMs rarely produce blockquote-fenced code. Document as known limitation.

## Implementation

### Replace the heuristic with `closeOpenFence()`

**File:** `frontend/src/components/chat/MessagePart.tsx`

Add a pure function above the `MessagePart` component:

```ts
/**
 * If the text ends with an unclosed CommonMark fenced code block, append
 * the appropriate closing fence. Used during streaming to prevent an
 * in-progress code block from swallowing the rest of the message.
 *
 * Handles:
 *   - 3+ backtick fences (```python, ````go, etc.)
 *   - 3+ tilde fences (~~~, ~~~~)
 *   - Language info strings on opening fence
 *
 * Known limitation: fences inside blockquotes (> ```) are not detected.
 * LLMs rarely produce these; handling them would require a full parser.
 */
export function closeOpenFence(text: string): string {
  // A fence line: 3+ identical fence chars (` or ~) optionally followed by
  // a language info string, with no other fence chars on the line.
  const fenceLine = /^(`{3,}|~{3,})[^`~]*$/m;

  let openChar: string | null = null;
  let openLen = 0;

  for (const line of text.split("\n")) {
    const match = fenceLine.exec(line);
    if (!match) continue;

    const fenceStr = match[1];
    const char = fenceStr[0];
    const len = fenceStr.length;

    if (openChar === null) {
      // Opening fence: record character and length
      openChar = char;
      openLen = len;
    } else if (char === openChar && len >= openLen) {
      // Closing fence: must match opening character, length >= opening length
      openChar = null;
      openLen = 0;
    }
    // A fence using a different character or shorter length inside an open
    // fence is part of the code content — ignore it.
  }

  if (openChar !== null) {
    // Unclosed fence: append the minimum valid closing fence
    return text + "\n" + openChar.repeat(openLen);
  }

  return text;
}
```

## CommonMark fence rules implemented

| Rule | Implementation |
|---|---|
| Opening fence: 3+ identical chars | `fenceStr = /^(`{3,}|~{3,})/` match |
| Closing fence: same char as opening | `char === openChar` check |
| Closing fence: length ≥ opening | `len >= openLen` check |
| Fence char: backtick or tilde only | Pattern `[^`~]*$` ensures no mixed chars on the line |
| Language info string ignored | `[^`~]*$` allows any suffix after the fence chars |
| Nesting: fence inside open fence is content | Inner fence chars are only processed if they could close the current open fence |

---

## Failure Modes

**FM-1: Fences inside blockquotes (`> ``` `).**
The line `> ```python` does not match `/^(`{3,}|~{3,})/m` because of the `> ` prefix. These fences are invisible to the scanner and will cause the same swallowing behaviour as before.
*Mitigation:* LLMs very rarely produce blockquote-fenced code. Document as known limitation. A fix would require stripping blockquote prefixes before scanning, which is complex for marginal gain.

**FM-2: Code content that looks like a fence.**
A line inside a code block that starts with `` ``` `` (e.g., a markdown example showing a fence) would be matched by the scanner. This is correct CommonMark behaviour — the scanner mirrors what the parser does. The scanner correctly handles this because: inside an open backtick fence, a line starting with backticks is only a closing fence if it uses the same character and has length ≥ the opening fence.

**FM-3: Very long messages with many code blocks.**
`text.split("\n")` on a large message is O(N) in message length. For typical LLM output (< 10,000 lines), this is < 1ms. No performance concern.

**FM-4: Empty text.**
`closeOpenFence("")` returns `""` — no open fence, no change. Safe.

**FM-5: Text with only whitespace lines.**
Lines that are all whitespace do not match the fence regex. `openChar` remains `null`. Returns text unchanged. Safe.

## Test Plan

Add a dedicated `describe("closeOpenFence")` block. This is a pure function — fully unit-testable with no mocks needed.

```tsx
import { closeOpenFence } from "./MessagePart";

describe("closeOpenFence", () => {
  it("returns unchanged text when no fences are present", () => {
    expect(closeOpenFence("no code here")).toBe("no code here");
  });

  it("returns unchanged text when fences are balanced", () => {
    const text = "```go\nfunc main(){}\n```";
    expect(closeOpenFence(text)).toBe(text);
  });

  it("closes an open 3-backtick fence", () => {
    expect(closeOpenFence("```go\nfunc main(){}")).toBe("```go\nfunc main(){}\n```");
  });

  it("closes an open 4-backtick fence with 4 backticks, not 3", () => {
    expect(closeOpenFence("````go\nfunc main(){}")).toBe("````go\nfunc main(){}\n````");
  });

  it("closes an open tilde fence", () => {
    expect(closeOpenFence("~~~python\nprint('hi')")).toBe("~~~python\nprint('hi')\n~~~");
  });

  it("closes an open 4-tilde fence", () => {
    expect(closeOpenFence("~~~~sh\necho hello")).toBe("~~~~sh\necho hello\n~~~~");
  });

  it("does not close a fence with mismatched character (backtick vs tilde)", () => {
    // Tilde fence can only be closed by tilde
    const text = "~~~\ncode\n```";
    const result = closeOpenFence(text);
    expect(result).toBe("~~~\ncode\n```\n~~~");
  });

  it("does not close with shorter fence than opening", () => {
    // ````  (4) can only be closed by 4+ backticks; ``` (3) is content
    const text = "````\ncode\n```\nmore";
    const result = closeOpenFence(text);
    expect(result).toBe("````\ncode\n```\nmore\n````");
  });

  it("handles multiple balanced fences correctly", () => {
    const text = "```go\ncode1\n```\n```py\ncode2\n```";
    expect(closeOpenFence(text)).toBe(text);
  });

  it("closes the second open fence in a mixed sequence", () => {
    const text = "```go\ncode1\n```\ntext\n```py\ncode2";
    expect(closeOpenFence(text)).toBe("```go\ncode1\n```\ntext\n```py\ncode2\n```");
  });

  it("handles empty string", () => {
    expect(closeOpenFence("")).toBe("");
  });

  it("handles text with only a fence opening and no newline", () => {
    expect(closeOpenFence("```")).toBe("```\n```");
  });

  it("handles fence with no language info string", () => {
    expect(closeOpenFence("```\ncode")).toBe("```\ncode\n```");
  });
});
```
