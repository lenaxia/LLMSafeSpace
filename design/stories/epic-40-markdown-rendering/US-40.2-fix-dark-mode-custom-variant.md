# US-40.2: Fix dark mode — CSS variable layer and @custom-variant dark declaration

**Epic:** 40 — Markdown Rendering Overhaul
**Status:** Planning
**Effort:** Small (0.5d)
**Depends on:** None
**Blocked by:** Nothing
**GitHub Issue:** #125

---

## Problem

Dark mode is broken at two independent layers. Both must be fixed together — fixing only one leaves the app partially or fully broken depending on the user's OS setting.

### Layer 1: Tailwind `dark:` utilities ignore the `.dark` class

Tailwind v4 defaults the `dark:` variant to `@media (prefers-color-scheme: dark)`. When a project uses class-based dark mode (toggling `class="dark"` on `<html>`), the following declaration is required:

```css
@custom-variant dark (&:where(.dark, .dark *));
```

This declaration is absent from `frontend/src/styles/index.css`.

`ThemeProvider.tsx:63` correctly toggles the class:

```ts
document.documentElement.classList.toggle("dark", r === "dark");
```

But without `@custom-variant dark`, Tailwind ignores the class entirely. Every `dark:` utility across the entire application — in every component — responds to the OS `prefers-color-scheme` instead.

### Layer 2: CSS custom properties also ignore the `.dark` class

The design token variables (`--color-background`, `--color-foreground`, `--color-muted`, `--color-border`, etc.) are defined in `index.css` under `@media (prefers-color-scheme: dark)` only. There is no `.dark { }` block. These variables drive every `bg-background`, `text-foreground`, `border-border`, and `bg-muted` usage across the entire app.

When the user manually sets the app to dark while their OS is in light mode:
- The `@media` block does not fire (OS is light)
- The CSS variables stay at their light values
- `bg-background` is white, `text-foreground` is near-black, `bg-muted` is light grey
- Every component background and text colour is wrong regardless of Tailwind `dark:` utilities

This is the deeper of the two bugs. The `@custom-variant` fix alone does not help here.

### Combined impact matrix

| Scenario | Expected | Actual (both layers broken) |
|---|---|---|
| OS: light, app: dark | Dark backgrounds, light text, dark `dark:` utilities | Light backgrounds, light text, no `dark:` utilities |
| OS: dark, app: light | Light backgrounds, dark text, no `dark:` utilities | Dark backgrounds, dark text, `dark:` utilities active |
| OS: dark, app: system | Dark everything | Works (OS and class agree by accident) |
| OS: light, app: system | Light everything | Works (OS and class agree by accident) |

Manual override is fully broken in both directions. System mode works by coincidence.

## User Story

As a user who has manually set the app theme to "Dark" while my OS is in light mode, I see dark background colours, light text, and correct dark styling everywhere in the app. The manual override works independently of the OS preference.

## Assumptions (validate before implementing)

1. `ThemeProvider.tsx` sets `class="dark"` on `document.documentElement` — **validated:** `ThemeProvider.tsx:63`.
2. Tailwind v4 uses `@custom-variant` in CSS for custom variant declaration — **validated:** https://tailwindcss.com/docs/dark-mode#toggling-dark-mode-manually.
3. The `&:where(.dark, .dark *)` selector form is the correct Tailwind v4 syntax — **validated:** directly from Tailwind v4 docs example code.
4. No other dark mode mechanism exists in the codebase (no `data-theme` attribute, no second class name) — **validated:** grep of `ThemeProvider.tsx` shows only `classList.toggle("dark", ...)`.
5. CSS custom properties are set only under `@media (prefers-color-scheme: dark)` with no `.dark` class block — **validated:** full read of `index.css` confirms no `.dark { }` block exists.
6. The `@media (prefers-color-scheme: dark)` block must be retained to handle the "System" theme option — **validated:** `ThemeProvider` sets the `dark` class based on `window.matchMedia` for system mode, but the CSS media query also fires for any consumer that doesn't go through React (e.g. pre-React paint). Keeping both ensures correctness in all scenarios.

## Implementation

### Step 1: Add `@custom-variant dark` to `index.css`

**File:** `frontend/src/styles/index.css`

Add immediately after `@import "tailwindcss"`, before any `@theme` block:

```css
@import "tailwindcss";
@custom-variant dark (&:where(.dark, .dark *));
```

The `:where()` pseudo-class has zero specificity, preventing dark mode styles from winning unexpected specificity battles. This is the form from the official Tailwind v4 documentation.

### Step 2: Add `.dark` CSS variable block to `index.css`

**File:** `frontend/src/styles/index.css`

The existing `@media (prefers-color-scheme: dark)` block sets CSS custom properties for the OS-level system preference. Add a parallel `.dark` block immediately after it that sets the same variables, driven by the class:

```css
/* Manual dark mode override — driven by class="dark" on <html>.
 * Must mirror @media (prefers-color-scheme: dark) exactly.
 * ThemeProvider sets/removes this class on every theme change.
 * The media query block below handles the "System" case when the
 * class has not yet been set (pre-React paint). */
.dark {
  --color-background: hsl(222.2 84% 4.9%);
  --color-foreground: hsl(210 40% 98%);
  --color-muted: hsl(217.2 32.6% 17.5%);
  --color-muted-foreground: hsl(215 20.2% 65.1%);
  --color-popover: hsl(222.2 84% 4.9%);
  --color-popover-foreground: hsl(210 40% 98%);
  --color-card: hsl(222.2 84% 4.9%);
  --color-card-foreground: hsl(210 40% 98%);
  --color-border: hsl(217.2 32.6% 17.5%);
  --color-input: hsl(217.2 32.6% 17.5%);
  --color-primary: hsl(210 40% 98%);
  --color-primary-foreground: hsl(222.2 47.4% 11.2%);
  --color-secondary: hsl(217.2 32.6% 17.5%);
  --color-secondary-foreground: hsl(210 40% 98%);
  --color-accent: hsl(217.2 32.6% 17.5%);
  --color-accent-foreground: hsl(210 40% 98%);
  --color-destructive: hsl(0 62.8% 30.6%);
  --color-destructive-foreground: hsl(210 40% 98%);
  --color-ring: hsl(212.7 26.8% 83.9%);
}
```

The values are identical to the existing `@media (prefers-color-scheme: dark)` `:root` block. The media query block is retained as-is — it handles the "System" theme when the OS is dark and ensures the app looks correct before React mounts (avoiding flash of wrong theme).

### No changes to `ThemeProvider.tsx`

The `ThemeProvider` implementation is correct. It:
- Reads preference from `localStorage` (`lsp-theme`)
- Syncs from the API on mount
- Toggles `class="dark"` on `document.documentElement`
- Listens for OS `prefers-color-scheme` changes when theme is "System"
- Exports `useTheme()` hook with `resolved: "light" | "dark"` for components that need the current theme programmatically

No changes needed.

## Relationship to Shiki (US-40.3)

US-40.3 wires shiki's dual-theme CSS using the `.dark` class selector:

```css
html.dark .shiki,
html.dark .shiki span {
  color: var(--shiki-dark) !important;
  background-color: var(--shiki-dark-bg) !important;
}
```

This plain CSS selector works regardless of whether `@custom-variant dark` is declared — it is not a Tailwind utility. However, without the `.dark` CSS variable block added in Step 2, the surrounding UI (backgrounds, borders, text) remains light-themed even when shiki correctly renders dark code blocks. US-40.2 must land in the same PR so the whole app is consistent.

## Failure Modes

**FM-1: `@custom-variant dark` is placed after `@theme` in the CSS file.**
The declaration order matters. `@custom-variant` must appear before any rule that uses the `dark:` variant. Placing it after `@import "tailwindcss"` and before `@theme` is correct.
*Mitigation:* The implementation specifies the exact location.

**FM-2: The `.dark` variable block and the `@media` block diverge over time.**
If a developer adds a new CSS variable to the `@media` block but forgets to add it to the `.dark` block, manual dark mode will be missing that variable while system dark mode has it.
*Mitigation:* The comment on the `.dark` block explicitly states it must mirror the media query block exactly. A future improvement would be to consolidate to a single source of truth (the `.dark` block) and derive the media query from it, but that requires a CSS preprocessor and is out of scope here.

**FM-3: A component uses `prefers-color-scheme` directly.**
If any component uses `@media (prefers-color-scheme: dark)` in a CSS-in-JS block or inline style, it will continue to follow the OS preference. Grep of `src/` confirms this only appears in `index.css` itself — no components use it directly.

**FM-4: Flash of wrong theme on initial load (FOUT).**
`ThemeProvider` sets the dark class via a React `useEffect` on mount. There is a brief window (~one frame) between page load and React mount where the OS `@media` preference applies. For users with OS=light, app=dark, this means a flash of light theme before dark kicks in.
*Mitigation:* The standard fix is an inline `<script>` in `<head>` that synchronously reads `localStorage` and sets the class before React loads. This is a known pre-existing issue, not introduced by this story, and is deferred as a future improvement.
