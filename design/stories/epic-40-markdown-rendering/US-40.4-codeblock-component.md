# US-40.4: CodeBlock component — language label, copy button, not-prose, wordWrap

**Epic:** 40 — Markdown Rendering Overhaul
**Status:** Planning
**Effort:** Medium (1d)
**Depends on:** US-40.3 (shiki `highlight()` function)
**Blocked by:** Nothing once US-40.3 is done

---

## Problem

`ReactMarkdown` in `MessagePart.tsx` has no custom `components` prop. Fenced code blocks render as plain `<pre><code>` with no syntax highlighting, no language label, and no per-block copy button. Three missing UX elements:

1. **No syntax highlighting** — US-40.3 provides the `highlight()` function; this story wires it into the render path
2. **No language label** — users cannot tell what language a block is without reading the code
3. **No per-block copy button** — only the whole-message copy button exists in `MessageBubble`

Additionally, once a custom `pre`/`code` renderer is introduced:

4. **`prose` styles will conflict** — Tailwind Typography applies `background`, `padding`, `border-radius`, and `margin` to `<pre>` and `<code>`. Shiki injects its own background via inline style. Result without mitigation: double-padded blocks with mismatched backgrounds.
5. **`codeBlockWordWrap` setting will break** — the Tailwind arbitrary-variant selector `[&_pre]:whitespace-pre-wrap` on the prose container will not reach into `CodeBlock`'s own `<pre>` element.

---

## User Story

As a user reading an assistant message with a code block, I see:
- Token-colored syntax highlighting for the declared language
- A language label in the top-left corner of the block
- A copy button in the top-right corner that copies the raw code and shows a check mark for 2 seconds
- My word-wrap preference applied consistently to all code blocks

---

## Assumptions (validate before implementing)

1. `ReactMarkdown`'s `components` prop accepts a `pre` renderer that receives children including a `code` element — **validated:** react-markdown docs, `components.pre` receives `{ children, ...props }` where children is the `<code>` element.
2. The `code` element inside `pre` has `className="language-<lang>"` when a language is declared — **validated:** react-markdown source; the class is set from the fence info string.
3. `not-prose` is a valid Tailwind Typography escape hatch in v4 — **validated:** Tailwind Typography docs.
4. `dangerouslySetInnerHTML` with shiki output is safe — shiki emits only `<pre>`, `<code>`, and `<span style="...">` elements. No `href`, `src`, event handlers, or `<script>` tags are possible in shiki's output format. This is documented in code.
5. `useUserSetting("codeBlockWordWrap", false)` returns the current word-wrap preference — **validated:** existing usage in `MessagePart.tsx:81`.
6. The inline code path (single backtick, no fence) must NOT go through `CodeBlock` — **validated:** distinguished by the absence of a newline in the code content and the absence of a `language-*` class.

## Implementation

### Step 1: Add `CodeBlock` component to `MessagePart.tsx`

**File:** `frontend/src/components/chat/MessagePart.tsx`

Add the following component above the `MessagePart` function. It imports `highlight` from `../../lib/shiki` and `useState`/`useEffect`/`useCallback`/`useRef` from react.

```tsx
interface CodeBlockProps {
  code: string;
  lang: string | null;
  wordWrap: boolean;
}

function CodeBlock({ code, lang, wordWrap }: CodeBlockProps) {
  const [highlightedHtml, setHighlightedHtml] = useState<string | null>(null);
  const [copied, setCopied] = useState(false);
  const timerRef = useRef<ReturnType<typeof setTimeout>>(undefined);

  useEffect(() => {
    // Highlighting is deferred: render plain text first, replace with
    // highlighted HTML once the async call resolves. For cached languages
    // this is imperceptible. Streaming guard (isStreaming) is in MessagePart
    // — this component only receives code when highlighting is appropriate.
    if (!lang) return;
    let cancelled = false;
    highlight(code, lang).then((html) => {
      if (!cancelled) setHighlightedHtml(html);
    });
    return () => { cancelled = true; };
  }, [code, lang]);

  const handleCopy = useCallback(async () => {
    try {
      await navigator.clipboard.writeText(code);
      clearTimeout(timerRef.current);
      setCopied(true);
      timerRef.current = setTimeout(() => setCopied(false), 2000);
    } catch {
      // clipboard write failed — stay in idle state
    }
  }, [code]);

  useEffect(() => () => clearTimeout(timerRef.current), []);

  const CopyIcon = copied ? Check : Copy;
  const copyLabel = copied ? "Copied" : "Copy code";

  return (
    // not-prose: opt out of Tailwind Typography's pre/code rules so they
    // don't double-pad or conflict with shiki's inline background styles.
    <div className="not-prose my-4 rounded-md overflow-hidden border border-border">
      {/* Header bar: language label left, copy button right */}
      <div className="flex items-center justify-between px-3 py-1.5 bg-muted/60 border-b border-border">
        <span className="text-xs font-mono text-muted-foreground">
          {lang ?? ""}
        </span>
        <button
          type="button"
          onClick={handleCopy}
          aria-label={copyLabel}
          className={cn(
            "rounded p-0.5 transition-all text-muted-foreground/70",
            "hover:text-muted-foreground hover:scale-110 active:scale-95",
            copied && "text-green-500",
          )}
        >
          <CopyIcon className="h-3.5 w-3.5" />
        </button>
      </div>

      {/* Code body: shiki HTML or plain fallback */}
      {highlightedHtml ? (
        // shiki output is safe: contains only <pre><code><span style="..."> elements.
        // No href, src, event handlers, or <script> tags are possible in shiki's
        // output format. dangerouslySetInnerHTML is appropriate here.
        <div
          className={cn(
            "overflow-x-auto touch-manipulation text-sm [&_pre]:p-4 [&_pre]:m-0",
            wordWrap && "[&_pre]:whitespace-pre-wrap [&_pre]:break-words",
          )}
          dangerouslySetInnerHTML={{ __html: highlightedHtml }}
        />
      ) : (
        <pre
          className={cn(
            "overflow-x-auto touch-manipulation p-4 text-sm font-mono",
            wordWrap && "whitespace-pre-wrap break-words",
          )}
        >
          <code>{code}</code>
        </pre>
      )}
    </div>
  );
}
```

### Step 2: Add custom `components` prop to `ReactMarkdown`

**File:** `frontend/src/components/chat/MessagePart.tsx`

Replace the existing `ReactMarkdown` call for assistant text parts (lines 101–103):

```tsx
// Before:
<ReactMarkdown remarkPlugins={[remarkGfm]} rehypePlugins={[rehypeSanitize]}>
  {text}
</ReactMarkdown>

// After:
<ReactMarkdown
  remarkPlugins={[remarkGfm]}
  rehypePlugins={[rehypeSanitize]}
  components={{
    pre({ children }) {
      // Extract code content and language from the child <code> element.
      // React.Children utilities are legacy — use a direct null-safe cast.
      const child = (Array.isArray(children) ? children[0] : children) as
        | React.ReactElement<{ className?: string; children?: string }>
        | null
        | undefined;
      const className = child?.props?.className ?? "";
      const lang = className.startsWith("language-")
        ? className.slice("language-".length)
        : null;
      const code = String(child?.props?.children ?? "").trimEnd();
      // If the structure is unexpected (no extractable code), fall through
      // to the plain pre renderer rather than crashing the bubble.
      if (!code && !lang) {
        return <pre className="overflow-x-auto p-4 text-sm font-mono">{children}</pre>;
      }
      return <CodeBlock code={code} lang={lang} wordWrap={wordWrap} isStreaming={isStreaming ?? false} />;
    },
    code({ className, children, ...props }) {
      // Inline code (single backtick): no language- class.
      // Block code inside a fenced fence is handled by the pre renderer
      // above — this branch only fires for inline code.
      return (
        <code className={cn("break-all", className)} {...props}>
          {children}
        </code>
      );
    },
  }}
>
  {text}
</ReactMarkdown>
```

### Step 4: Remove now-redundant arbitrary-variant selector for `[&_:not(pre)>code]:break-all`

The prose container's `[&_:not(pre)>code]:break-all` selector targets inline code. With the custom `code` renderer added in Step 2, inline code now receives `break-all` via the renderer's `cn("break-all", className)`. The prose container selector is now redundant.

It is safe to leave it (no conflict), but removing it reduces noise. Remove from `MessagePart.tsx:98`:

```tsx
// Before:
"[&_pre]:overflow-x-auto [&_pre]:touch-manipulation [&_table]:block [&_table]:overflow-x-auto [&_table]:touch-manipulation [&_:not(pre)>code]:break-all",

// After:
"[&_table]:block [&_table]:overflow-x-auto [&_table]:touch-manipulation",
```

The `[&_pre]:overflow-x-auto` and `[&_pre]:touch-manipulation` selectors are also now redundant since `CodeBlock` handles its own `<pre>` overflow. Remove those too. The table selectors remain because tables are still handled inside `prose` without a custom renderer.

## Interaction with `rehype-sanitize`

`rehype-sanitize` runs as a rehype plugin in the `ReactMarkdown` pipeline. It sanitizes the markdown AST before it reaches the `components` renderer. The `components.pre` renderer is called by React after the pipeline — it operates on React elements, not on HTML. Shiki's `dangerouslySetInnerHTML` injects HTML directly into the DOM, bypassing the rehype pipeline.

This is safe because shiki's output format is constrained: `<pre class="shiki">`, `<code>`, and `<span style="color: ...">` only. No attributes that could be XSS vectors (`href`, `src`, `onerror`, `onclick`, etc.) appear in shiki output. This is documented in the comment in the component.

`rehype-sanitize` continues to protect all other markdown output (headings, links, images, raw HTML blocks) — only the specific code block content is handled separately.

## Why `useEffect` + `useState` for async highlighting

`ReactMarkdown`'s `components` prop receives synchronous React render functions. They cannot be async. Two approaches exist:

**Option A (chosen): `useState` + `useEffect`**
- Render plain text immediately (first render is synchronous, fast)
- `useEffect` fires after mount, calls `highlight()` async
- On resolution, `setHighlightedHtml(html)` triggers one re-render with highlighted output
- For cached languages (all subsequent uses): `highlight()` resolves near-instantly, re-render is imperceptible
- For first-use languages: brief flash of plain text → highlighted (~50–200ms for grammar download)

**Option B: `use()` React hook with a Promise (rejected)**
React 19 `use()` can suspend on a Promise, enabling async component rendering. This would avoid the two-render approach. Rejected because:
- It requires a `Suspense` boundary around every `CodeBlock` instance
- Suspense fallback for a code block (a loading spinner) is worse UX than briefly showing unstyled code
- It introduces complexity for marginal gain — the flash is only on first use per language per session

## Failure Modes

**FM-1: `pre` renderer receives unexpected children structure.**
`react-markdown` always renders fenced code blocks as `<pre><code className="language-...">...</code></pre>`, so the first child of `pre` is reliably a `<code>` element. The null-safe cast `(Array.isArray(children) ? children[0] : children)` handles both a single element and an array with one element. If `code` and `lang` are both empty after extraction (which should not happen in normal react-markdown output), the guard `if (!code && !lang)` renders a plain `<pre>{children}</pre>` fallback rather than an empty `CodeBlock`. No crash, no blank space.

**FM-2: shiki HTML wraps content in an outer `<pre>` which conflicts with `CodeBlock`'s own `<pre>` fallback.**
Shiki `codeToHtml` returns a full `<pre class="shiki"><code>...</code></pre>` structure. When rendered via `dangerouslySetInnerHTML`, this produces a `<pre>` inside `CodeBlock`'s `<div>`. The `[&_pre]:p-4 [&_pre]:m-0` selector on the wrapper div handles the shiki `<pre>` padding correctly.

**FM-3: `code` content passed to `highlight()` has a trailing newline.**
Fenced code blocks always end with a newline before the closing fence. `String(child?.props?.children ?? "").trimEnd()` removes trailing whitespace, preventing a spurious blank line at the bottom of highlighted output.

**FM-4: `wordWrap` prop is stale in the `useEffect` closure.**
The `useEffect` dependency array is `[code, lang]`. `wordWrap` is not in the dependency array because highlighting does not depend on `wordWrap` — word-wrap is a CSS class applied to the container, not a rendering concern. The `wordWrap` prop is read directly in the JSX render, not in the effect. No stale closure issue.

**FM-5: Timer leak if `CodeBlock` unmounts during the 2s copy timeout.**
The `useEffect(() => () => clearTimeout(timerRef.current), [])` cleanup handles this. Identical pattern to `MessageBubble.tsx:66`.
