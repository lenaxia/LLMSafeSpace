# US-40.7: Fix PWA cache — exclude grammar chunks from precache, add LRU eviction

**Epic:** 40 — Markdown Rendering Overhaul
**Status:** Planning
**Effort:** Small (0.5d)
**Depends on:** US-40.3 (shiki must be installed so grammar chunks exist in build output)
**Blocked by:** Nothing once US-40.3 is done

---

## Problem

`vite.config.ts` configures the PWA service worker with:

```ts
workbox: {
  globPatterns: ["**/*.{js,css,html,svg}"],
  ...
}
```

`globPatterns` defines the **precache manifest** — files downloaded and cached at service worker install time, before the user has interacted with the app. After shiki is added, every language grammar chunk (`@shikijs/langs/go`, `@shikijs/langs/python`, etc.) is a `.js` file in the build output. The glob `**/*.js` would match all of them.

Shiki ships grammars for 300+ languages. Even if only 30 languages are bundled by Vite's tree-shaking, that could be 30+ additional files (~5–30 KB each) force-downloaded at install time, before the user has ever seen a code block in a language that uses them.

This defeats shiki's lazy-loading model entirely and unnecessarily bloats the service worker installation cost.

The existing `runtimeCaching` configuration only has one entry: `env.json`. There is no caching strategy for shiki grammar chunks, so they fall through to the default network-only strategy. After the first use, the browser's HTTP cache may cache them, but:
- The HTTP cache is not guaranteed (depends on server headers)
- The HTTP cache is separate from the service worker cache — offline use will fail
- There is no LRU eviction to bound cache growth

---

## User Story

As a user installing the app as a PWA, the installation is fast and does not download grammar files for languages I haven't used yet. When I do use a language, its grammar is downloaded once and available offline on subsequent visits. The cache does not grow unboundedly.

## Implementation

### Step 1: Run build and identify shiki chunk names

After US-40.3 is implemented:

```bash
npm run build
ls dist/assets/*.js | head -30
```

Shiki language grammar chunks will have names like `_shikijs_langs_go-[hash].js`. The core app chunks will have names like `index-[hash].js`, `vendor-[hash].js`, `query-[hash].js`.

The goal: precache only core app chunks; runtime-cache grammar chunks.

### Step 2: Update `vite.config.ts`

**File:** `frontend/vite.config.ts`

```ts
// Before:
workbox: {
  globPatterns: ["**/*.{js,css,html,svg}"],
  navigateFallback: "/index.html",
  navigateFallbackDenylist: [/^\/api/],
  runtimeCaching: [
    {
      urlPattern: /^\/env\.json$/,
      handler: "NetworkFirst",
      options: { cacheName: "env-config", expiration: { maxEntries: 1 } },
    },
  ],
},

// After:
workbox: {
  // Precache only core app chunks (index, vendor, query, CSS, HTML, SVG).
  // Language grammar chunks (shiki) are excluded and handled by runtimeCaching
  // below. This prevents force-downloading all grammar files at SW install time.
  globPatterns: ["**/*.{css,html,svg}", "**/index*.js", "**/vendor*.js", "**/query*.js"],
  navigateFallback: "/index.html",
  navigateFallbackDenylist: [/^\/api/],
  runtimeCaching: [
    {
      urlPattern: /^\/env\.json$/,
      handler: "NetworkFirst",
      options: { cacheName: "env-config", expiration: { maxEntries: 1 } },
    },
    {
      // Shiki language grammar chunks and other async JS chunks:
      // cache on first use (CacheFirst), evict LRU after 50 entries or 30 days.
      // This covers all /assets/*.js files not precached above.
      urlPattern: ({ url }: { url: URL }) =>
        url.pathname.startsWith("/assets/") && url.pathname.endsWith(".js"),
      handler: "CacheFirst",
      options: {
        cacheName: "async-chunks",
        expiration: {
          maxEntries: 50,
          maxAgeSeconds: 30 * 24 * 60 * 60, // 30 days
        },
      },
    },
  ],
},
```

### Chunk naming caveat

The `globPatterns` globs `**/index*.js`, `**/vendor*.js`, `**/query*.js` match by Vite chunk name prefix. These names are set by the existing `manualChunks` config:
- `vendor` — React, react-dom, react-router
- `query` — TanStack Query
- `index` — main app entry

After shiki is added, shiki's core may produce a chunk with a different prefix (e.g. `shiki-[hash].js`). If the shiki core chunk should be precached (for offline startup), add `**/shiki*.js` to the glob patterns. Determine this by inspecting build output.

**Recommended approach:** inspect the build output after US-40.3 and adjust `globPatterns` to explicitly include the core shiki chunk by prefix if it exists. Language grammar chunks (e.g. `_shikijs_langs_*.js`) should NOT be in `globPatterns`.

### Step 3: Verify build output

```bash
npm run build
# Inspect dist/assets/ to verify:
# 1. Core chunks (index, vendor, query) are in the precache manifest
# 2. Language grammar chunks are NOT in the precache manifest
# 3. Language grammar chunks ARE matched by the runtimeCaching URL pattern

# Check precache manifest:
cat dist/sw.js | grep -o '"url":"[^"]*"' | head -30
```

## Cache strategy rationale

| Cache | Strategy | Why |
|---|---|---|
| Core app chunks | Precache (install-time) | Must be available offline from first visit |
| `env.json` | NetworkFirst (existing) | Config must be fresh; offline fallback acceptable |
| Language grammar chunks | CacheFirst + LRU eviction | Download once on first use, serve from cache forever after; evict after 50 entries or 30 days |

**Why CacheFirst for grammar chunks:**
Grammar files are content-addressed (Vite includes a content hash in the filename). Once a grammar file is cached, the cached version is always correct — the URL will change if the grammar is updated. `CacheFirst` is safe and optimal for content-addressed assets.

**Why 50 entries, 30 days:**
A typical developer using the app will encounter 10–20 distinct languages. 50 entries provides a generous buffer for edge cases (unusual languages, multiple shiki chunk files per language) without allowing the cache to grow indefinitely. 30 days covers most active usage patterns; infrequently-used language grammars that haven't been accessed in a month are worth re-downloading.

## Failure Modes

**FM-1: `globPatterns` is too narrow and misses a core chunk.**
If a new chunk is added by a future dependency that doesn't match `**/index*.js`, `**/vendor*.js`, or `**/query*.js`, it won't be precached. The runtime cache fallback will still serve it (`CacheFirst` matches all `/assets/*.js`), but it won't be available at offline startup.
*Mitigation:* Add an explicit comment in `vite.config.ts` explaining the precache/runtime split and requiring a review of `globPatterns` when `manualChunks` is updated.

**FM-2: `maxEntries: 50` is too small and evicts frequently-used grammars.**
If a user regularly uses more than 50 distinct language grammars (unusual but possible in a polyglot environment), LRU eviction will evict older entries and they'll be re-downloaded. Not a correctness problem — just an extra network request.
*Mitigation:* 50 is a generous ceiling for real-world usage. Increase to 100 if monitoring shows frequent eviction.

**FM-3: The runtimeCaching URL pattern matches non-grammar JS chunks.**
The pattern `url.pathname.startsWith("/assets/") && url.pathname.endsWith(".js")` matches all `/assets/*.js` files. This includes any async chunks that are not grammar files (e.g., `react-diff-viewer-continued` lazy chunk). These chunks are small and content-addressed, so `CacheFirst` is correct for them too.

**FM-4: `vite-plugin-pwa` does not support function-form `urlPattern`.**
Workbox supports both regex and function forms for `urlPattern`. `vite-plugin-pwa` passes the config through to Workbox directly. The function form `({ url }) => ...` is valid Workbox syntax.

## Test Plan

No unit tests are needed for Workbox/PWA configuration. Verification is build-time inspection:

```bash
# Build
npm run build

# Verify precache manifest (should NOT include shiki grammar chunks)
node -e "
const sw = require('fs').readFileSync('dist/sw.js', 'utf8');
const match = sw.match(/self\.__WB_MANIFEST\s*=\s*(\[.*?\])/s);
const manifest = JSON.parse(match[1]);
const shikiEntries = manifest.filter(e => e.url.includes('shikijs') || e.url.includes('langs'));
console.log('Shiki entries in precache:', shikiEntries.length, '(expected: 0)');
console.log('Total precache entries:', manifest.length);
"

# Manual end-to-end: serve the app offline (disable network in DevTools)
# Verify: app loads from service worker
# Verify: on first code block render, grammar chunk downloads appear in Network tab
# Verify: on second render of same language, no network request (served from cache)
```
