# US-40.8: Fix hardcoded useDarkTheme in diff viewer

**Epic:** 40 — Markdown Rendering Overhaul
**Status:** Planning
**Effort:** Trivial (< 0.25d)
**Depends on:** US-40.2 (dark mode must actually work for this to be meaningful)
**Blocked by:** Nothing once US-40.2 is done

---

## Problem

`MessagePart.tsx:61` passes `useDarkTheme` hardcoded to `true` to `ReactDiffViewer`:

```tsx
<ReactDiffViewer
  oldValue={oldStr}
  newValue={newStr}
  splitView={false}
  useDarkTheme          // always true, regardless of app theme
  hideLineNumbers={false}
  styles={{ ... }}
/>
```

Before US-40.2 this was invisible: dark mode was broken app-wide, so the diff viewer's always-dark appearance was consistent with the rest of the (always-OS-driven) UI. Once US-40.2 lands, dark mode works correctly throughout the app. A user with app theme set to "Light" will see dark diff blocks inside light chat bubbles — a jarring inconsistency.

---

## User Story

As a user reading tool output that contains a file edit (shown as a diff), the diff viewer matches the current app theme: dark in dark mode, light in light mode.

---

## Assumptions (validate before implementing)

1. `react-diff-viewer-continued` accepts a boolean `useDarkTheme` prop — **validated:** `MessagePart.tsx:61` already uses it.
2. `useTheme()` from `ThemeProvider` is importable in `MessagePart.tsx` — **validated:** `ThemeProvider.tsx` exports `useTheme()` and it is used in `AppearanceTab.tsx` and `UserSettingsTab.tsx`.
3. `useTheme().resolved` returns `"light" | "dark"` synchronously on every render — **validated:** `ThemeProvider.tsx:24-25`; `resolved` is a `useState` value initialised from `getSystemTheme()` or the stored preference before any API call.

---

## Implementation

**File:** `frontend/src/components/chat/MessagePart.tsx`

### Step 1: Import `useTheme`

```ts
import { useTheme } from "../../providers/ThemeProvider";
```

### Step 2: Read resolved theme in `MessagePart`

`useTheme()` must be called at the component level, not inside a sub-function. Add alongside the existing `useUserSetting` call:

```tsx
export function MessagePart({ part, isUser, isStreaming }: Props) {
  const wordWrap = useUserSetting("codeBlockWordWrap", false);
  const { resolved } = useTheme();   // add this line
  // ...
}
```

### Step 3: Pass `resolved` to `ToolDiffView`

Update `ToolDiffView`'s props interface and usage:

```tsx
// Before:
function ToolDiffView({ oldStr, newStr }: { oldStr: string; newStr: string }) {
  return (
    <Suspense fallback={...}>
      <div ...>
        <ReactDiffViewer
          ...
          useDarkTheme
          ...
        />
      </div>
    </Suspense>
  );
}

// After:
function ToolDiffView({ oldStr, newStr, isDark }: { oldStr: string; newStr: string; isDark: boolean }) {
  return (
    <Suspense fallback={...}>
      <div ...>
        <ReactDiffViewer
          ...
          useDarkTheme={isDark}
          ...
        />
      </div>
    </Suspense>
  );
}
```

Pass from the call site inside `MessagePart`:

```tsx
<ToolDiffView
  oldStr={...}
  newStr={...}
  isDark={resolved === "dark"}
/>
```

---

## Why not use `useTheme()` directly inside `ToolDiffView`

`ToolDiffView` is a function defined at module scope inside `MessagePart.tsx`. Calling `useTheme()` inside it is valid React (hooks can be called in any function component), but it adds an extra hook call per tool diff render. Reading `resolved` at the `MessagePart` level and passing it as a prop is simpler, more explicit, and consistent with how `wordWrap` is already threaded through.

---

## Failure Modes

**FM-1: `useTheme()` is called before `ThemeProvider` mounts.**
`useTheme()` throws if called outside `ThemeProvider`. `MessagePart` is always rendered inside `AppShell`, which wraps `ThemeProvider`. No risk.
*Validation:* Confirmed by reading `router.tsx` and `AppShell.tsx` structure — `ThemeProvider` wraps the entire app tree.

**FM-2: `resolved` flickers on initial load.**
`ThemeProvider` initialises `resolved` synchronously from `localStorage` or `getSystemTheme()` — no async delay. No flicker.
