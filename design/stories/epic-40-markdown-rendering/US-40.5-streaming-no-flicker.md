# US-40.5: No flicker during streaming — skip highlighting while isStreaming

**Epic:** 40 — Markdown Rendering Overhaul
**Status:** Planning
**Effort:** Small (0.5d)
**Depends on:** US-40.4 (`CodeBlock` component exists)
**Blocked by:** Nothing once US-40.4 is done

---

## Problem

`CodeBlock` (introduced in US-40.4) uses `useEffect` + `useState` to asynchronously call `highlight()`. The component renders plain text first, then re-renders with highlighted HTML once the promise resolves.

During streaming, `MessagePart` re-renders on every incoming token. Each re-render that includes a code block unmounts and remounts `CodeBlock` (because the code content prop changes). Each remount fires `useEffect` again. Each effect call fires an async `highlight()` call. The sequence for a streaming code block is:

```
token arrives → CodeBlock remounts → plain text rendered → highlight() called
token arrives → CodeBlock remounts → plain text rendered → highlight() called (previous cancelled)
token arrives → CodeBlock remounts → plain text rendered → highlight() called (previous cancelled)
... (one cycle per token in the code block)
...
streaming ends → CodeBlock remounts → plain text rendered → highlight() called → highlighted HTML rendered
```

Each `plain text → highlighted` transition is a visible flash. For a 30-line code block streamed over ~3 seconds, this produces dozens of flashes. The user experience is visually distracting.

Additionally, each `highlight()` call during streaming fires a dynamic `import()` for the language grammar if it has not been cached yet. The first call downloads and parses the grammar; subsequent cancelled calls still hit the module cache check. This is low overhead but unnecessary.

---

## User Story

As a user watching an assistant message stream in containing a code block, the code block displays as plain unstyled text while streaming. Once streaming completes, it transitions once to highlighted output. There are no intermediate flashes between styled and unstyled states.

---

## Assumptions (validate before implementing)

1. `isStreaming` prop is available in `MessagePart` and passed through to `CodeBlock` — **validated:** `MessagePart` already accepts `isStreaming?: boolean` (line 77).
2. The streaming fence fix (US-40.6) ensures the code block fence is always closed before `CodeBlock` is rendered — **validated by design:** US-40.6 runs on the raw text before `ReactMarkdown` parses it, so by the time `CodeBlock`'s `pre` renderer is called, the fence is always syntactically complete.
3. `useEffect` with `[code, lang, isStreaming]` dependency array will fire when `isStreaming` flips from `true` to `false` — **validated:** React effect re-runs when any dependency changes.

## Implementation

### Step 1: Pass `isStreaming` to `CodeBlock`

**File:** `frontend/src/components/chat/MessagePart.tsx`

Update `CodeBlockProps`:

```tsx
interface CodeBlockProps {
  code: string;
  lang: string | null;
  wordWrap: boolean;
  isStreaming: boolean;  // add
}
```

Update the `pre` renderer in the `ReactMarkdown` `components` prop to pass `isStreaming`:

```tsx
pre({ children }) {
  try {
    const child = Children.only(children) as React.ReactElement<{
      className?: string;
      children?: string;
    }>;
    const className = child?.props?.className ?? "";
    const lang = className.startsWith("language-")
      ? className.slice("language-".length)
      : null;
    const code = String(child?.props?.children ?? "").trimEnd();
    return <CodeBlock code={code} lang={lang} wordWrap={wordWrap} isStreaming={isStreaming ?? false} />;
  } catch {
    return <pre className="overflow-x-auto p-4 text-sm font-mono">{children}</pre>;
  }
},
```

### Step 2: Guard `useEffect` in `CodeBlock`

**File:** `frontend/src/components/chat/MessagePart.tsx`

In the `CodeBlock` component's `useEffect`, add an early return when streaming:

```tsx
useEffect(() => {
  // Do not highlight during streaming. The code content changes on every
  // token, which would cause a highlight() call per token and a visible
  // plain→highlighted flash on every update.
  // Wait until streaming is complete before calling highlight().
  if (isStreaming || !lang) return;

  let cancelled = false;
  highlight(code, lang).then((html) => {
    if (!cancelled) setHighlightedHtml(html);
  });
  return () => { cancelled = true; };
}, [code, lang, isStreaming]);
```

Add `isStreaming` to the dependency array. When `isStreaming` flips from `true` to `false`, the effect fires with the final code content and calls `highlight()` exactly once.

### Step 3: Reset highlighted state when code changes during streaming

When a message transitions from non-streaming to streaming (e.g. a retry or edit scenario), the `highlightedHtml` state may contain stale highlighted output from a previous render. Add a reset:

```tsx
useEffect(() => {
  if (isStreaming) {
    setHighlightedHtml(null);
  }
}, [isStreaming]);
```

This ensures the code block returns to the plain fallback if streaming restarts for the same message part.

In practice, streaming never restarts on an existing message part in the current architecture. This guard is defensive.

## Behaviour summary

| State | `highlight()` called? | Rendered output |
|---|---|---|
| `isStreaming=true` | No | Plain `<pre><code>` |
| `isStreaming=false`, first render | Yes, once | Plain text → highlighted (one transition) |
| `isStreaming=false`, subsequent renders (same code) | No (effect deps unchanged) | Highlighted (stable) |
| `isStreaming` flips `true→false` | Yes, once | Plain text → highlighted (one transition) |

---

## Failure Modes

**FM-1: `isStreaming` is `undefined` (not passed from `MessageList`).**
`isStreaming?: boolean` is optional. The `?? false` in the `pre` renderer coerces it to `false`. If `isStreaming` is never set, `CodeBlock` behaves as non-streaming and highlights immediately. Acceptable fallback — the flash issue only occurs when messages are genuinely streaming.

**FM-2: Long code block is still partially displayed as plain text after streaming ends.**
The transition from plain to highlighted takes one `useEffect` + `highlight()` async cycle after `isStreaming` flips to `false`. For cached languages, this is < 1ms. For a cold first-use language grammar download, this is 50–200ms. The user may notice a brief delay on the very first occurrence of a new language. Acceptable — this is the designed behaviour and far better than per-token flashing.

**FM-3: `code` prop changes while `isStreaming=false` (non-streaming edit).**
If the code content changes after streaming ends (e.g. message editing, which is not a current feature), `useEffect` fires again and re-highlights. Correct behaviour.

- [ ] While `isStreaming=true`, no call to `highlight()` is made (verified by mock assertion in tests)

## Test Plan

Add to the `CodeBlock` describe block in `MessagePart.test.tsx`:

```tsx
it("does not call highlight() while isStreaming=true", () => {
  render(<MessagePart
    part={{ type: "text", text: "```go\nfunc main(){}\n```" }}
    isUser={false}
    isStreaming={true}
  />);
  expect(mockHighlight).not.toHaveBeenCalled();
});

it("calls highlight() exactly once when isStreaming transitions to false", async () => {
  const { rerender } = render(<MessagePart
    part={{ type: "text", text: "```go\nfunc main(){}\n```" }}
    isUser={false}
    isStreaming={true}
  />);
  expect(mockHighlight).not.toHaveBeenCalled();

  rerender(<MessagePart
    part={{ type: "text", text: "```go\nfunc main(){}\n```" }}
    isUser={false}
    isStreaming={false}
  />);
  await waitFor(() => expect(mockHighlight).toHaveBeenCalledTimes(1));
  expect(mockHighlight).toHaveBeenCalledWith("func main(){}", "go");
});

it("does not re-highlight if code is unchanged after streaming ends", async () => {
  const { rerender } = render(<MessagePart
    part={{ type: "text", text: "```go\nfunc main(){}\n```" }}
    isUser={false}
    isStreaming={false}
  />);
  await waitFor(() => expect(mockHighlight).toHaveBeenCalledTimes(1));

  // Re-render with same props
  rerender(<MessagePart
    part={{ type: "text", text: "```go\nfunc main(){}\n```" }}
    isUser={false}
    isStreaming={false}
  />);
  expect(mockHighlight).toHaveBeenCalledTimes(1); // still 1, not 2
});
```
