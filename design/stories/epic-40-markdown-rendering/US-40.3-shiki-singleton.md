# US-40.3: Add syntax highlighting — shiki singleton with JS engine

**Epic:** 40 — Markdown Rendering Overhaul
**Status:** Planning
**Effort:** Medium (1d)
**Depends on:** US-40.2 (dark mode CSS vars use `.dark` class selector)
**Blocked by:** Nothing (US-40.2 is one line and ships first)

---

## Problem

`react-markdown` emits `<pre><code class="language-go">...</code></pre>` for fenced code blocks. The `language-go` class is present in the DOM but nothing reads it. No syntax highlighter is installed. Code renders as unstyled monospace regardless of language.

No custom `components` prop is passed to `ReactMarkdown` in `MessagePart.tsx`, so there is no hook point for a custom renderer either.

---

## User Story

As a user reading an assistant message containing a fenced code block with a declared language, I see token-colored syntax highlighting that matches my current app theme (light or dark), making code significantly more readable.

---

## Approach: shiki with JavaScript regex engine, createHighlighter from bundle/full

**Library chosen:** `shiki` v4.2.0

**Why shiki over alternatives:**
- 300+ languages supported (TextMate grammar-based, same engine as VS Code)
- Dual-theme support via CSS custom properties — one render pass, theme-switching via CSS class, no re-render needed
- JavaScript regex engine option eliminates WASM download entirely
- On-demand language loading via `h.loadLanguage(import('@shikijs/langs/<lang>'))` — one download per language per session, cached in the browser module registry, shared across all workspaces in the tab
- No language ceiling — all 300+ languages available without any pre-configuration

**Why not highlight.js / rehype-highlight:**
- All 190+ language grammars bundled upfront (~900 KB) vs shiki's on-demand loading
- Lower quality token output (regex-based, not TextMate grammar-based)

**Why not react-syntax-highlighter:**
- Largest bundle of all three options (ships both Prism and hljs)
- Less actively maintained

## Assumptions (validate before implementing)

1. `shiki` v4.2.0 is the current latest — **validated:** `npm info shiki dist-tags` shows `latest: 4.2.0`.
2. `shiki/engine/javascript` exports `createJavaScriptRegexEngine` — **validated:** `npm info shiki exports` confirms `'./engine/javascript': './dist/engine-javascript.mjs'`.
3. `createHighlighter` from `shiki/bundle/full` accepts and forwards the `engine` option to `createHighlighterCore` — **validated:** read `@shikijs/core/dist/index.mjs` source; `createBundledHighlighter` does `engine: options.engine ?? engine()` — the passed engine is used; the default engine factory is only called as fallback when engine is absent.
4. `createHighlighter` from `shiki/bundle/full` supports `h.loadLanguage('go')` with a string — language name strings are resolved against the bundle's built-in language map — **validated:** tested locally; `h.loadLanguage('go')` and `h.loadLanguage('python')` both succeed and produce correct highlighted HTML.
5. `codeToHtml` from `shiki/bundle/web` does NOT reliably accept an `engine` option — **validated:** `bundle/web` unconditionally imports `engine-oniguruma` (WASM) internally; passing `engine` is syntactically accepted but ignored. Confirmed by code inspection of `bundle-web.mjs` and by testing: outputs with and without the option are identical.
6. `import('@shikijs/langs/${lang}')` dynamic template literal does NOT work in Vite 5 production builds — **validated:** Vite 5 docs explicitly state dynamic imports must start with `./` or `../`; bare package specifier `@shikijs/langs/` with a runtime variable is not statically analyzable and will not be bundled. `shiki/bundle/full` is the correct Vite-compatible abstraction for all-language support.
7. Shiki dual-theme mode outputs `--shiki-light` / `--shiki-dark` CSS custom property variables on `<span>` elements, and `--shiki-light-bg` / `--shiki-dark-bg` on the `<pre>` element — **validated:** actual shiki output inspected locally.
8. `!important` is required in the dark mode CSS override — **validated:** shiki emits inline `style` attributes on `<pre>` and `<span>`. Inline styles have higher specificity than any class-based selector. Without `!important`, the dark mode selector is overridden by shiki's own inline styles.
9. `h.getLoadedLanguages()` returns an array of already-loaded language identifiers — **validated:** tested; returns `['javascript', 'go', 'js', 'cjs', 'mjs']` after loading JS and Go.
10. US-40.2 is landed so `.dark` class is toggled and CSS variables are correctly set — **assumed:** both land in the same PR.

## Implementation

### Step 2: Create `src/lib/shiki.ts`

**File:** `frontend/src/lib/shiki.ts` (new file)

```ts
import { createHighlighter } from "shiki/bundle/full";
import { createJavaScriptRegexEngine } from "shiki/engine/javascript";

// Module-level singleton. Created once on first import, reused for the
// lifetime of the tab. Never disposed — the highlighter must outlive any
// individual component.
//
// shiki/bundle/full includes all 347 languages as lazy async chunks.
// Themes are pre-loaded at init time (small, always needed).
// Languages are loaded on demand via h.loadLanguage(string) — shiki resolves
// the string against its bundled language map and loads the corresponding
// async chunk. Each language chunk is downloaded at most once per session;
// Vite/Rollup handles the chunk splitting at build time.
//
// Using createHighlighter (not codeToHtml shorthand) so that we can pass
// engine: createJavaScriptRegexEngine() — eliminating the WASM download.
// Validated: createHighlighter forwards engine to createHighlighterCore via
// `engine: options.engine ?? engine()` in @shikijs/core source.
const highlighterPromise = createHighlighter({
  themes: ["github-light", "github-dark"],
  langs: [],
  engine: createJavaScriptRegexEngine(),
});

/**
 * Highlight a code string for the given language using shiki.
 *
 * Language grammars are loaded on demand from shiki's bundled language set
 * (shiki/bundle/full, 347 languages) and cached in the highlighter instance.
 * The first call for a new language downloads one async chunk; all subsequent
 * calls are synchronous.
 *
 * Returns dual-theme HTML: the <pre> element carries --shiki-light-bg /
 * --shiki-dark-bg inline style vars; each <span> token carries
 * --shiki-light / --shiki-dark. The active theme is selected by CSS rules
 * in index.css using the html.dark class. No re-render is needed when the
 * user switches themes.
 *
 * Returns null if the language is unknown or loading/highlighting fails
 * for any reason. Callers must render a plain <pre><code> fallback on null.
 *
 * @param code - Raw code string (not HTML-encoded)
 * @param lang - Language identifier (e.g. "go", "python", "yaml")
 */
export async function highlight(code: string, lang: string): Promise<string | null> {
  try {
    const h = await highlighterPromise;
    if (!h.getLoadedLanguages().includes(lang)) {
      await h.loadLanguage(lang as Parameters<typeof h.loadLanguage>[0]);
    }
    return h.codeToHtml(code, {
      lang,
      themes: { light: "github-light", dark: "github-dark" },
      defaultColor: false,
    });
  } catch {
    // Unknown language, network failure, or grammar parse error.
    // Caller renders plain <pre><code> as fallback.
    return null;
  }
}
```

#### Design decisions explained

**`shiki/bundle/full` + `createHighlighter` instead of `createHighlighterCore` + dynamic template literal import:**
`createHighlighterCore` with `import('@shikijs/langs/${lang}')` (a dynamic template literal with a bare package specifier) does not work in Vite 5 production builds. Vite's bundler only supports dynamic imports starting with `./` or `../`. A bare specifier `@shikijs/langs/` with a runtime variable is not statically analyzable and will not be chunked. `shiki/bundle/full` is the correct Vite abstraction: all 347 language grammars are pre-chunked at build time by Rollup, and `h.loadLanguage(string)` at runtime triggers loading of only the requested chunk.

**`createHighlighter` instead of `codeToHtml` shorthand:**
`shiki/bundle/full`'s `codeToHtml` shorthand also unconditionally defaults to Oniguruma (WASM). `createHighlighter` accepts and correctly forwards the `engine` option (validated in source: `engine: options.engine ?? engine()`). The JS engine is only used when explicitly passed; without it, the default factory `engine()` is called — which returns Oniguruma.

**`h.loadLanguage(lang as ...)` cast:**
`h.loadLanguage` from `shiki/bundle/full` accepts `BundledLanguage` (a string union of 347 language names) or a language object. Passing an unknown string at runtime requires a cast since TypeScript cannot verify the string is in the union. The `try/catch` handles any invalid language at runtime.

**`getLoadedLanguages()` check before `loadLanguage()`:**
Calling `loadLanguage()` for an already-loaded language is a no-op in shiki, but the check avoids the function call overhead on the hot path. After first use, `highlight()` resolves in one microtask (`await highlighterPromise`, which is already resolved).

**`defaultColor: false`:**
With `defaultColor: false`, shiki emits only CSS custom property references — no inline `color` or `background-color` values. No inline style specificity conflict with the dark mode CSS override. The CSS block in `index.css` is mandatory and ships in the same commit.

### Step 3: Add shiki CSS variable wiring to `index.css`

**File:** `frontend/src/styles/index.css`

Add after the `@custom-variant dark` declaration and `.dark` variable block:

```css
/* Shiki syntax highlighting — dual-theme via CSS custom properties.
 *
 * With defaultColor: false, shiki emits no inline color/background values.
 * The <pre class="shiki"> element has --shiki-light-bg / --shiki-dark-bg
 * vars in its inline style. Each <span> token has --shiki-light /
 * --shiki-dark vars.
 *
 * !important is required: shiki emits inline style attributes which have
 * higher specificity than any class-based selector without !important.
 *
 * The html.dark selector mirrors ThemeProvider's classList.toggle("dark").
 * No media query here — dark mode is class-driven, matching the rest of
 * the app's theme system. */
html.dark .shiki,
html.dark .shiki span {
  color: var(--shiki-dark) !important;
  background-color: var(--shiki-dark-bg) !important;
}
```

Note: The light theme requires no CSS. With `defaultColor: false` and no light-mode override, tokens have no `color` or `background-color` applied — which means they inherit from the prose container (correct for light mode). Only the dark override is needed.

**Why `html.dark` rather than `.dark`:**
Specificity. `html.dark .shiki` is more specific than `.shiki` alone, ensuring the override wins over any other `.shiki`-targeting rules. `html.dark` matches `<html class="dark">` exactly as set by `ThemeProvider`.

**Why `!important`:**
Shiki emits inline `style="color:#24292e;--shiki-dark:#e1e4e8"` on `<pre>` and `style="color:#D73A49;--shiki-dark:#F97583"` on each `<span>`. These inline styles have specificity `(1,0,0,0)` — higher than any class selector. Without `!important`, the dark theme CSS is silently overridden. This is the approach documented by shiki for class-based dark mode.

## How language loading works at runtime

1. User opens the app — shiki core and themes load (chunked by Vite from `shiki/bundle/full`)
2. User receives a message with a ```` ```go ```` code block
3. `CodeBlock` calls `highlight(code, "go")`
4. `highlight()` awaits `highlighterPromise` (resolves immediately after first use)
5. `"go"` is not in `getLoadedLanguages()` — `h.loadLanguage("go")` fires
6. Shiki resolves `"go"` against its bundle map, triggers the async chunk download (~20 KB)
7. Grammar is parsed and stored in the highlighter instance
8. `h.codeToHtml(...)` runs synchronously, returns highlighted HTML
9. All future `highlight(code, "go")` calls: `"go"` is in `getLoadedLanguages()`, skip the load, run `codeToHtml` directly
10. Different language ```` ```python ```` follows the same path — independent chunk, independent cache

The highlighter's loaded-language cache is per app session. The PWA service worker (US-40.7) adds a persistent `CacheFirst` layer so downloaded language chunks survive tab reloads.

## Other Options Considered

**Option A: `codeToHtml` shorthand from `shiki/bundle/web` or `shiki/bundle/full` (rejected)**
Simpler call site — no singleton management. Rejected because the shorthand manages its own internal singleton and does not allow engine configuration. The engine option is accepted syntactically but ignored; both `bundle/web` and `bundle/full` shorthands default to Oniguruma (WASM).

**Option B: `createHighlighterCore` + `import('@shikijs/langs/${lang}')` dynamic template literal (rejected)**
Works in Node.js but **fails in Vite 5 production builds**. Vite's bundler requires dynamic imports to start with `./` or `../`. A bare package specifier `@shikijs/langs/` with a runtime variable is not statically analyzable by Rollup and the chunks will not be produced. Confirmed by Vite 5 documentation.

**Option C: Pre-load a fixed set of languages at app startup (rejected)**
Pre-load Go, Python, Bash, YAML, TypeScript, JSON, SQL at init time. Guarantees zero latency for common languages but: forces ~200 KB download at startup before any code block is seen; creates a hard ceiling; any language outside the pre-loaded set still has first-use latency.

**Option D: Web Worker for highlighting (future enhancement, not in scope)**
Shiki's regex work can be offloaded to a Web Worker to avoid blocking the main thread. For code blocks < ~500 lines, main-thread highlighting takes < 10ms and is imperceptible. Noted as a future optimisation.

## Failure Modes

**FM-1: Unknown language — `h.loadLanguage('unknownlang')` throws.**
`shiki/bundle/full` throws `ShikiError: Language 'unknownlang' is not included in this bundle` for unknown strings. The try/catch in `highlight()` catches it and returns `null`. `CodeBlock` renders plain `<pre><code>` fallback. Confirmed by testing.

**FM-2: Network offline — language chunk download fails.**
Same failure path as FM-1. Returns `null`, fallback renders. Once connectivity is restored, the next call succeeds.

**FM-3: JS engine incompatibility with a grammar.**
The JavaScript engine is ~95% compatible. For the ~5% of grammars using Oniguruma-specific regex features, it falls back gracefully or throws. The try/catch handles the throw.

**FM-4: `defaultColor: false` + missing CSS = invisible tokens.**
If the `index.css` shiki block is accidentally omitted, tokens render with no colour (only CSS vars, no fallback). Text is still readable (monospace, browser default colour) but unstyled. The CSS block ships in the same commit as `shiki.ts` — they must not be separated.

**FM-5: Highlighter singleton is awaited by many concurrent `CodeBlock` mounts.**
On a page load with many messages containing code, multiple `CodeBlock` instances may call `highlight()` concurrently. All `await highlighterPromise` calls resolve to the same singleton — no double-initialisation. Concurrent `loadLanguage()` calls for the same language are safe: shiki's internal deduplication prevents double-loading.
