# US-40.1: Fix prose/heading styles — install @tailwindcss/typography

**Epic:** 40 — Markdown Rendering Overhaul
**Status:** Planning
**Effort:** Small (0.5d)
**Depends on:** None
**Blocked by:** Nothing

---

## Problem

`MessagePart.tsx:97` applies `prose prose-sm dark:prose-invert max-w-none` to the `ReactMarkdown` output container. These classes do nothing because `@tailwindcss/typography` is not installed and not registered as a Tailwind plugin.

`react-markdown` correctly emits semantic HTML elements:
- `# Heading` → `<h1>`
- `## Heading` → `<h2>`
- `- item` → `<ul><li>`
- `> quote` → `<blockquote>`
- `` `code` `` → `<code>`

But without the Typography plugin, every one of these elements renders at body text size with no visual differentiation. The markdown structure is parsed correctly; the output is just unstyled.

Additionally, the existing hand-written table CSS in `index.css:90-98` (`.prose table`, `.prose th`, `.prose td`) is only needed because the plugin is absent. Once the plugin is installed, Tailwind Typography handles table styling natively.

---

## User Story

As a user reading an assistant message that contains markdown headings, I see visually distinct heading elements with correct size, weight, and spacing. `# H1` is large and bold, `## H2` is medium, `### H3` is smaller — consistent with standard markdown rendering expectations.

## Assumptions (validate before implementing)

1. `@tailwindcss/typography@0.5.20` supports Tailwind v4 — **validated:** `peerDependencies: { tailwindcss: ">=3.0.0 || >=4.0.0 || insiders" }`.
2. Tailwind v4 uses `@plugin` directive in CSS (not `tailwind.config.js`) — **validated:** confirmed in Tailwind v4 docs and `vite.config.ts` which uses `@tailwindcss/vite`.
3. The existing `prose prose-sm dark:prose-invert max-w-none` class usage in `MessagePart.tsx` is correct — **validated:** matches Tailwind Typography docs for a compact responsive prose block with dark mode inversion.
4. `dark:prose-invert` requires `@custom-variant dark` to work correctly with class-based dark mode — **validated:** see US-40.2. US-40.1 installs the plugin; US-40.2 wires dark mode. Both must land in the same PR for `dark:prose-invert` to work.

## Implementation

### Step 1: Install the package

```bash
npm install @tailwindcss/typography
```

Confirmed latest version: `0.5.20`.

### Step 2: Register the plugin in `index.css`

**File:** `frontend/src/styles/index.css`

Add after `@import "tailwindcss"`:

```css
@import "tailwindcss";
@plugin "@tailwindcss/typography";
```

This is the Tailwind v4 plugin registration mechanism. No `tailwind.config.js` is needed or used in this project.

### Step 3: Remove the now-redundant hand-written table CSS

**File:** `frontend/src/styles/index.css`

Remove lines 90–98:

```css
/* Markdown table borders — REMOVE: covered by @tailwindcss/typography */
.prose table {
  border-collapse: collapse;
}
.prose th,
.prose td {
  border: 1px solid var(--color-border);
  padding: 0.4em 0.8em;
}
```

Tailwind Typography provides its own table styling. The hand-written rules were a workaround for the missing plugin and are now redundant. Keeping them would create specificity conflicts.

**Caveat:** Tailwind Typography's default table borders use its own colour tokens (`--tw-prose-td-borders`), not `var(--color-border)`. The visual result will be slightly different (Typography's default is a grey that matches its prose palette). If exact border colour matching with the app theme is required, the typography theme can be customised via `@theme`:

```css
/* Optional: override Typography's border colour to match app theme */
@theme {
  --tw-prose-td-borders: var(--color-border);
  --tw-prose-invert-td-borders: var(--color-border);
}
```

Include this override unless the default Typography borders are acceptable.

### Step 4: No component changes required

`MessagePart.tsx:97` already has the correct class string:

```tsx
"prose prose-sm dark:prose-invert max-w-none",
```

The arbitrary-variant overrides on the same element are also correct and remain unchanged:

```tsx
"[&_pre]:overflow-x-auto [&_pre]:touch-manipulation [&_table]:block [&_table]:overflow-x-auto [&_table]:touch-manipulation [&_:not(pre)>code]:break-all",
```

## Other Options Considered

**Option A: Use a CSS reset + hand-written heading styles (rejected)**
Write `.prose h1`, `.prose h2`, etc. manually. This is what the hand-written table CSS was doing partially. Rejected — Tailwind Typography exists precisely to avoid this; maintaining custom prose styles is ongoing maintenance burden.

**Option B: Apply prose styles to a different element (rejected)**
Wrap `ReactMarkdown` output in a custom element instead of using `prose`. Rejected — `prose` is the correct abstraction; changing the element would require rewriting all the downstream arbitrary-variant selectors.

## Failure Modes

**FM-1: Typography plugin conflicts with existing component styles.**
Tailwind Typography applies opinionated styles to all descendant elements inside `.prose`, including links, images, and code. If any component inside a chat bubble has custom styling that conflicts, it may be overridden.
*Mitigation:* The `not-prose` escape hatch is available on any element that should not inherit prose styles. US-40.4 (`CodeBlock`) uses `not-prose` explicitly for this reason.

**FM-2: `dark:prose-invert` colours don't match the app's dark theme.**
Typography's dark inversion uses its own palette. The `dark:prose-invert` utility inverts all prose colour tokens. If the resulting colours don't match the app's `--color-foreground` / `--color-background` variables, the text will look off.
*Mitigation:* Override specific prose tokens in `@theme` to match the app's CSS variables. The most likely mismatch is the prose body colour (`--tw-prose-body` vs `--color-foreground`). Verify visually after implementing and override as needed.

**FM-3: Table border colour mismatch (noted above).**
Covered by the optional `@theme` override in Step 3.
