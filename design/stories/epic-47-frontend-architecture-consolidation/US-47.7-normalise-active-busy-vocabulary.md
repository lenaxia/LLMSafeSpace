# US-47.7 — Normalise active/busy vocabulary (backend)

## Summary

The same concept ("the agent has an in-flight turn") is reported as `"busy"` by the SSE event (`SessionStatusEvent.status`) and `AgentSessionInfo.status`, but as `"active"` by `SessionListItem.status` (from `GET /workspaces/:id/sessions`). The frontend silently translates between the two (`SessionActivityProvider.tsx:78,196`). Any future code reading `SessionListItem.status` must "know" that `"active"` means busy.

## Root Cause

The session-index table (`session_index`) stores `status` as a free-text column. The API proxy writes `"active"` when the session enters the busy set and `"idle"` on idle transition. The SSE event and the workspace-status endpoint use the opencode-native `"busy"`/`"idle"` vocabulary.

## Change

Normalise the session-index write path to use `"busy"` instead of `"active"`. This is a one-word change in the proxy's session-index recorder, plus a migration for existing rows:

1. Find where `session_index.status` is set to `"active"` (the proxy's `onSessionActive` / session-index recorder).
2. Change to `"busy"`.
3. Migration `000037`: `UPDATE session_index SET status = 'busy' WHERE status = 'active'`.
4. Remove the translation in `SessionActivityProvider.tsx:78,196` (treat `"busy"` directly, not `"active"`).
5. Update `SessionListItem.status` type from bare `string` to `"idle" | "busy"` (`api/types.ts:71`).

## Acceptance Criteria

1. `GET /workspaces/:id/sessions` returns `status: "busy"` (not `"active"`) for sessions in the opencode busy set.
2. `SessionListItem.status` is typed as `"idle" | "busy"`.
3. `SessionActivityProvider` no longer translates `"active"` → busy.
4. Migration applies cleanly (round-trip + idempotent).
5. All backend + frontend tests pass.

## Validation

Confirm that no other consumer of `SessionListItem.status` relies on the `"active"` string value. Grep frontend for `=== "active"` in session contexts and update.
