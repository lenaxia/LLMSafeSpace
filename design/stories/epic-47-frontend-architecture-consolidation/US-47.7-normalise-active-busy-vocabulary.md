# US-47.7 — Normalise active/busy vocabulary (backend)

## Summary

The same concept ("the agent has an in-flight turn") is reported as `"busy"` by the SSE event (`SessionStatusEvent.status`), `AgentSessionInfo.status` (from `GET /workspaces/:id/status`), and `"active"` by `SessionListItem.status` (from `GET /workspaces/:id/sessions`). The frontend silently translates between the two (`SessionActivityProvider.tsx:78,196`). Any future code reading `SessionListItem.status` must "know" that `"active"` means busy.

## Root Cause (validated)

The `"active"` value is set **at runtime** in the router, NOT stored in the database. `session_index` has no `status` column (verified — migration `000003` creates the table with only `workspace_id`, `session_id`, `title`, `last_message_at`, `message_count`, timestamps).

The computation is in `router.go:836-842`:
```go
activeIDs := proxyHandler.GetActiveSessions(workspaceID)
// ...
for i := range sessions {
    if _, ok := activeSet[sessions[i].ID]; ok {
        sessions[i].Status = "active"   // ← should be "busy"
    }
}
```

`GetActiveSessions` reads from the `wsstate.Store` (Redis-backed as of Epic 45). The value is a simple string literal — no migration needed, no data migration.

## Change

One-word change in `api/internal/server/router.go:840`:
```go
// before
sessions[i].Status = "active"
// after
sessions[i].Status = "busy"
```

Then remove the translation in the frontend:
- `SessionActivityProvider.tsx:78` — treat `"busy"` directly (not `"active"`)
- `SessionActivityProvider.tsx:196` — write `"busy"` to the cache (not `"active"`)
- `SessionListItem.status` type from bare `string` to `"idle" | "busy"` (`api/types.ts:71`)

**No database migration needed** — the status is computed at runtime, not persisted.

## Acceptance Criteria

1. `GET /workspaces/:id/sessions` returns `status: "busy"` (not `"active"`) for sessions in the opencode busy set.
2. `SessionListItem.status` is typed as `"idle" | "busy"`.
3. `SessionActivityProvider` no longer translates `"active"` → busy.
4. All backend + frontend tests pass (update any test that asserts `status: "active"` on session list responses).
5. No migration needed.

## Validation

Confirm that no other consumer of `SessionListItem.status` relies on the `"active"` string value. Grep frontend for `=== "active"` in session contexts and update. Check backend tests that assert on session list response `status` field.
