# US-47.12 — Fold SecretsTab llm-provider into UserProviderCredentialsTab

| Field | Value |
|-------|-------|
| Epic | 47 — Frontend Architecture Consolidation |
| Priority | Medium |

## Summary

Two user-facing paths to store an LLM provider key (OpenAI, Anthropic, etc.):

1. **`UserProviderCredentialsTab`** ("Provider Keys" tab) — modern path: structured fields, model probing, explicit workspace bindings. Writes to `provider_credentials` table.
2. **`SecretsTab`** with type=`llm-provider` ("Secrets" tab) — older, weaker path: free-text value, no model probe, generic binding model. Writes to `user_secrets` table.

Both converge on the **same** `applyLLMProvider → FlushProviders` materializer and the same wire format in `agent-config.json` (`design/0040` §1, validated against backend `secrets.go:581-589`).

**Product decision needed:** migrate existing `llm-provider` secrets to `provider_credentials`, or ask users to re-enter?

## Change

### Remove `llm-provider` from SecretsTab creatable types
`SecretsTab.tsx:9` — remove `{ value: "llm-provider", ... }` from the type picker. Follow the existing pattern for `api-key` (already `legacyOnly: true` at `:14-19`): mark `llm-provider` as `legacyOnly` so existing rows still render but new ones cannot be created.

### Data migration (if migrating)
Backend migration `000038`:
```sql
-- Migrate llm-provider secrets to provider_credentials
INSERT INTO provider_credentials (id, owner_type, owner_id, name, provider, ciphertext, key_version, created_at, updated_at)
SELECT
  CONCAT('migrated-', s.id),
  'user',
  s.user_id,
  s.name,
  s.meta->>'provider',
  s.ciphertext,
  s.key_version,
  s.created_at,
  s.updated_at
FROM user_secrets s
WHERE s.type = 'llm-provider'
ON CONFLICT DO NOTHING;
```

Then migrate workspace bindings from `user_secret_bindings` to `workspace_credential_bindings` for the migrated credentials.

### If NOT migrating (ask users to re-enter)
Just mark `llm-provider` as `legacyOnly` in `SecretsTab.tsx`. Existing secrets continue to work (the materializer handles both tables). Users add new keys via "Provider Keys" tab. Over time, legacy secrets age out.

## Acceptance Criteria

1. `SecretsTab` no longer offers `llm-provider` as a creatable type.
2. Existing `llm-provider` secrets still render (as legacy) and still inject into agent-config.json.
3. `UserProviderCredentialsTab` is the sole path for new LLM provider keys.
4. If migrating: migration applies cleanly (round-trip + idempotent), bindings preserved.
5. All tests pass.

## Notes

The fold is a real story (~1–2 days with migration, trivial without) because the two paths write to different Postgres tables that both feed the same K8s Secret. The backend consolidation (US-38.13, PR #206) already unified the Go handler/store types for `provider_credentials` — this story is the frontend + data-layer completion.

The `AdminCredentialsTab` (already dead, removed by US-47.1) was a THIRD path whose backend table was dropped in migration `000015`. It is not part of this fold.
