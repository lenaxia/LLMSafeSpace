# US-47.4 — Add Tier-3 autoSuspend setting + workspace-creation resolution

## Summary

Add account-level (per-user) autoSuspend override. Currently autoSuspend is a Tier-2 instance setting only (`workspace.autoSuspend.enabled`, default true; `workspace.autoSuspend.idleTimeoutMinutes`, default 60). The user wants each user to be able to override the global default for their own workspaces.

**Resolution order at workspace creation:** explicit request → user setting (Tier 3) → instance setting (Tier 2) → hardcoded default.

## Backend Changes

### 1. Add Tier-3 setting definitions (`pkg/settings/schema.go`)

Add to `UserSettings()`:
```go
{Key: "workspace.autoSuspend.enabled", Tier: 3, Type: TypeBool, Default: true, Category: "Auto-Suspend", Label: "Auto-Suspend", Description: "Auto-suspend idle workspaces (account-level override)"},
{Key: "workspace.autoSuspend.idleTimeoutMinutes", Tier: 3, Type: TypeInt, Default: 60, Min: intPtr(5), Max: intPtr(10080), Category: "Auto-Suspend", Label: "Idle Timeout (min)", Description: "Idle timeout before auto-suspend (account-level override)"},
```

These mirror the Tier-2 definitions (`schema.go:66-67`) with the same defaults. The Tier-3 schema-driven settings UI (`UserSettingsTab`) will surface them automatically.

### 2. Inject `UserService` into `WorkspaceService`

`WorkspaceService` currently has `instanceSettings *settings.InstanceService` (`workspace_service.go:49`) but no user-settings accessor. Add:
```go
userSettings *settings.UserService
```
Update the constructor and `app.go` wiring.

### 3. Resolve autoSuspend with user override at workspace creation

In the existing autoSuspend resolution block (`workspace_service.go:867-877`), add user-setting resolution before falling back to instance:

```go
if crd.Spec.AutoSuspend == nil {
    autoSuspendEnabled := true
    idleTimeout := int64(86400)

    // User override (Tier 3) — falls through to instance (Tier 2) if not set
    if s.userSettings != nil {
        if v, err := s.userSettings.GetBool(ctx, userID, "workspace.autoSuspend.enabled"); err == nil {
            autoSuspendEnabled = v
        } else if v, err := s.instanceSettings.GetBool(ctx, "workspace.autoSuspend.enabled"); err == nil {
            autoSuspendEnabled = v
        }
        if v, err := s.userSettings.GetInt(ctx, userID, "workspace.autoSuspend.idleTimeoutMinutes"); err == nil && v > 0 {
            idleTimeout = int64(v) * 60
        } else if v, err := s.instanceSettings.GetInt(ctx, "workspace.autoSuspend.idleTimeoutMinutes"); err == nil && v > 0 {
            idleTimeout = int64(v) * 60
        }
    } else {
        // Fallback: instance settings only (backward compat)
        if v, err := s.instanceSettings.GetBool(ctx, "workspace.autoSuspend.enabled"); err == nil {
            autoSuspendEnabled = v
        }
        if v, err := s.instanceSettings.GetInt(ctx, "workspace.autoSuspend.idleTimeoutMinutes"); err == nil && v > 0 {
            idleTimeout = int64(v) * 60
        }
    }
    crd.Spec.AutoSuspend = &v1.WorkspaceAutoSuspend{ ... }
}
```

The `userID` is already available in `CreateWorkspace` from `workspace.Spec.Owner.UserID` (or the auth context). Validate which source is correct before implementing.

## Frontend Changes

The Tier-3 settings UI (`UserSettingsTab.tsx`) is schema-driven — adding the Tier-3 definitions to `schema.go` automatically renders the controls in the user's Preferences tab. No frontend code changes needed beyond verifying the form renders correctly.

This places autoSuspend in the **account-level** Preferences tab (per the user's product decision), NOT in the per-workspace settings drawer (removed by US-47.2).

## Acceptance Criteria

1. Tier-3 `workspace.autoSuspend.enabled` and `workspace.autoSuspend.idleTimeoutMinutes` appear in `GET /settings/user/schema`.
2. A user can set their own autoSuspend override via `PUT /settings/user/{key}`.
3. Creating a workspace with a user override set writes the user's value to `crd.Spec.AutoSuspend`.
4. Creating a workspace with no user override falls back to the instance setting.
5. Creating a workspace with neither user nor instance override falls back to the hardcoded default (enabled, 60 min).
6. The controller enforces the CRD spec value (existing behaviour — no controller changes needed).
7. Integration test: `TestCreateWorkspace_AutoSuspend_UserOverride` and `TestCreateWorkspace_AutoSuspend_InstanceFallback`.

## Validation Gates

1. Confirm `userID` is accessible in `CreateWorkspace` context (check how `workspace.Spec.Owner.UserID` is populated — it may come from auth middleware, not the request body).
2. Confirm the `UserService.GetBool` / `GetInt` return a distinguishable "not set" error (check `user_service.go:124` `get()` — it returns `fmt.Errorf("setting %s not found", key)` on miss).
3. Confirm the existing Tier-2 admin settings UI still works (adding Tier-3 definitions with the same key prefix should not conflict — verify `UserSettingIndex` at `schema.go:128` does not collide with `InstanceSettingIndex`).

## Notes

The Tier-3 settings use the SAME key as the Tier-2 settings (`workspace.autoSuspend.enabled`). This is intentional and consistent with how `preferredModel` (Tier 3) and other settings work — the tier determines who can set it, the key determines what it controls. The `UserService` and `InstanceService` have independent stores (`user_settings` table vs `instance_settings` table), so there is no key collision.
