# US-47.5 — Unify server-busy state source (ChatPage → SessionActivityProvider)

| Field | Value |
|-------|-------|
| Epic | 47 — Frontend Architecture Consolidation |
| Priority | Medium |
| Depends On | US-47.1 (dead-code sweep first, to avoid confusion) |
| Blocks | US-47.6 (primitives), US-47.9 (ChatPage perf fixes) |
| Estimated | 4–6 hours |
| Evidence | `design/0037` F1, P1 (5 validation gates) |
| Risk | Medium — touches the most-tested chat area |

## Summary

`ChatPage` maintains its own `serverBusy` state (`useState(false)` at `:205`) + `sseHasDrivenBusy` ref (`:207`) to track whether the current session's agent is actively processing. `SessionActivityProvider` already tracks this for ALL sessions via `busySessions` Map, fed by the same `session.status` SSE events. The two derivations consume the same signal via two independent SSE connections and produce two independent state stores for the same fact.

Unify by having `ChatPage` consume `useIsSessionBusy(sessionId)` from the provider, eliminating `serverBusy`, `sseHasDrivenBusy`, and the sync/reset effects that maintain them.

## Current State (validated)

| Concern | ChatPage | SessionActivityProvider |
|---------|----------|------------------------|
| State store | `serverBusy: boolean` (`:205`) | `busySessions: Map<sid, wid>` (`:24`) |
| SSE source | workspace stream via `handleSSEEvent` (`:549-568`) | user stream via `onEvent` (`:181-249`) |
| REST seed | `["workspace-status"]` → `status === "busy"` (`:251-255`) | `["sessions"]` → `status === "active"` (`:64-94`) |
| Race guard | `sseHasDrivenBusy` ref (`:207`) | `seededRef` Set (`:35`) |
| Reset on session change | `:49-51` | (per-session keyed map; no reset needed) |
| Reset on reconnect | `:702` | `:167-171` |

## Change

### Remove from ChatPage.tsx
- `serverBusy` state (`:205`)
- `sseHasDrivenBusy` ref (`:207`)
- The status-poll sync effect (`:249-255`)
- The `setServerBusy(true/false)` branches in `handleSSEEvent` (`:553-567`)
- The `sseHasDrivenBusy.current = false` in session-change effect (`:51`)
- The `sseHasDrivenBusy.current = false` in SSE-reconnect handler (`:702`)

### Add to ChatPage.tsx
```ts
import { useIsSessionBusy } from "../providers/SessionActivityProvider";
// ...
const isSessionBusy = useIsSessionBusy(sessionId ?? "");
```

### Update useChatStream consumption
```ts
// before
const { send, abort, streaming, ... } = useChatStream(activeWorkspaceId, sessionId, serverBusy);
// after
const { send, streaming, ... } = useChatStream(activeWorkspaceId, sessionId, isSessionBusy);
```

### Update isReconnectMode derivation
```ts
// before
useEffect(() => {
  if (serverBusy && !localStreaming) { isReconnectMode.current = true; }
}, [serverBusy, localStreaming]);
// after
useEffect(() => {
  if (isSessionBusy && !localStreaming) { isReconnectMode.current = true; }
}, [isSessionBusy, localStreaming]);
```

### Preserve
- `localStreaming` in `useChatStream` — it is genuinely distinct (optimistic, set on send before any SSE arrives). `streaming = localStreaming || serverBusy` becomes `streaming = localStreaming || isSessionBusy`.
- The `serverBusyRef` pattern inside `useChatStream` (`:20-21`) — it mirrors the busy value into a ref so the 60s timeout closure can read the current value. After the change, `serverBusyRef.current = isSessionBusy` (synced via `useEffect`). Same pattern, different source.

## Validation Gates (MANDATORY before implementation)

These are the 5 gates from `design/0037` P1. Each must be verified:

1. **REST-seed parity.** `ChatPage` seeds from `["workspace-status"]` (`status === "busy"`); the provider seeds from `["sessions"]` (`status === "active"`). Different endpoints, different cache-population timing. **Must verify** that mounting into a busy session (reconnect-mode path, `:280-285`) still works when busy comes from the provider's seed. The provider only seeds once `["sessions", wsId]` is in the cache; `ChatPage` triggers that fetch (`:92`).

2. **`useChatStream` timeout correctness.** `useChatStream` reads busy via a ref to decide interrupted-vs-slow at 60 s (`:97`). The ref-mirror pattern must be preserved.

3. **Reconnect semantics.** Provider clears busy on user-stream reconnect (`:167`); ChatPage re-polls workspace-status on workspace-stream reconnect (`:700-706`). After unification, `handleSSEReconnect` still needs to invalidate `["workspace-status"]` (for phase, not busy).

4. **`isReconnectMode` derivation** (`:280-285`) keys off `serverBusy`; update to key off `isSessionBusy`.

5. **Test impact.** `ChatPage.*.test.tsx` mock `useWorkspaceBusyCount` but the SSE busy path is tested via `handleSSEEvent`. Those tests assert `serverBusy`-driven behaviour through the captured SSE handler; they need rework to assert via the provider.

## Acceptance Criteria

1. `serverBusy` and `sseHasDrivenBusy` no longer exist in `ChatPage.tsx`.
2. Both the bouncing-dot indicator (ChatView) and the blue-circle indicator (Sidebar) read the same `busySessions` map.
3. Reconnect mode (`isReconnectMode`) still activates when mounting into a busy session.
4. The 60s timeout in `useChatStream` still distinguishes "interrupted" from "slow response".
5. All `ChatPage.*.test.tsx` pass (updated as needed).
6. `npm run typecheck && npm run test && npm run lint && npm run build` pass.
