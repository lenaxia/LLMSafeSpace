# US-47.6 — Consolidate loading/busy primitives (Spinner + BusyIndicator)

## Summary

The codebase has one `Spinner` primitive (`components/ui/Spinner.tsx`, for data-loading) but 7 inline `Loader2` usages that bypass it, conflating "agent working" (concept A, `text-blue-500`) with "data loading" (concept C, `text-muted-foreground`). Users cannot distinguish the two by shape — only by colour, which is not accessible.

## Change

1. **Keep `Spinner`** as the sole primitive for concept C (loading). Replace 4 inline `Loader2` loading usages:
   - `Sidebar.tsx:161` (create-workspace mutation)
   - `Sidebar.tsx:358` (workspace activating)
   - `Sidebar.tsx:384` (creating session)
   - `MessageList.tsx:127` (loading older messages)

2. **Introduce `BusyIndicator`** component for concept A (agent working). Two usages:
   - `Sidebar.tsx:371` (collapsed workspace busy count)
   - `Sidebar.tsx:792` (per-session busy)

   OR extend `Spinner` with a `tone="busy"` prop. The canonical busy shape/colour is a product decision — **ask user**: is `blue-500` intentional for busy, or should it match `muted-foreground` for consistency?

3. **Add `aria-label`** to both primitives. `Spinner` has it (`aria-label="Loading"`); inline `Loader2` and `StreamingIndicator` do not. Add `aria-label="Agent working"` to `BusyIndicator`; add `aria-live="polite"` to `StreamingIndicator` (per `design/0026` line 783).

4. **Update Sidebar tests** that assert `.animate-spin.text-blue-500` (`Sidebar.test.tsx:290,342`) to target a `data-busy` attribute instead — more robust than a colour class.

## Acceptance Criteria

1. No inline `Loader2` usages remain outside primitives.
2. `Spinner` and `BusyIndicator` (or `Spinner` with `tone`) are the only loading/busy glyphs.
3. Both have `aria-label`.
4. `StreamingIndicator` has `aria-live="polite"`.
5. All tests pass.
