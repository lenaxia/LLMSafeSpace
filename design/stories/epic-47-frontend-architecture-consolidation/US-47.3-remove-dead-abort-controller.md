# US-47.3 — Remove dead AbortController from useChatStream

| Field | Value |
|-------|-------|

| Priority | Medium |

| Blocks | None |

## Summary

`useChatStream` creates an `AbortController`, stores it in a ref, and exposes `abort()` which calls `abortRef.current?.abort()`. The UI wires a "Stop generating" button to it. But the controller's `signal` is **never passed to `sendAsync`** — the POST completes regardless. The actual agent-stop mechanism is `workspacesApi.abortSession` (called separately in `ChatPage.tsx:953`).

The `AbortController` is vestige from when `useChatStream` used HTTP streaming (where client-side abort matters). The architecture moved to POST + SSE, making client-side abort of the POST meaningless.

## Scope

### Remove from `useChatStream.ts`
- `abortRef` ref (`:15`)
- `abortRef.current = new AbortController()` (`:49`)
- `abortRef.current = null` (`:123`)
- The `abort` callback (`:129-131`)
- The `abort` export from the return object (`:142`)

### Update `ChatPage.tsx`
- Remove the `abort` call from the `onAbort` handler (`:954`) — `abortSession` at `:953` is the real mechanism
- Remove `abort` from the `useChatStream` destructure (`:258`)
- Add a comment at `:953` documenting that server-side `abortSession` is the real abort path

### Preserve
- The `onAbort` prop on `ChatView` (the Stop button still works via `abortSession`)
- The `streaming` state (still drives the Stop button visibility)

## Acceptance Criteria

1. `useChatStream` no longer creates or exposes an `AbortController`.
2. The Stop button still stops the agent (via `abortSession`).
3. All existing tests pass (update any test that mocks/asserts on `abort`).
4. `npm run typecheck && npm run test && npm run lint` pass.

## Out of scope

If client-side cancellation of the POST request is ever desired (e.g. to save bandwidth on a slow response), that is a separate enhancement: pass the signal into `messagesApi.sendAsync` and onward to `api.post`. Not part of this story — the current architecture doesn't benefit from it.
