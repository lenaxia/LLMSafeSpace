# US-14.1: Formalize OpenAPI Specification

**Epic:** 14 - Multi-Language SDKs & VS Code Extension
**Priority:** Critical
**Depends on:** None (existing API is the input)

## User Story

As an SDK generator pipeline, I need a complete, accurate OpenAPI 3.1 specification of the LLMSafeSpace REST API, so that I can produce typed clients for any language without manual intervention.

## Acceptance Criteria

- [ ] `sdks/openapi.yaml` exists and validates against OpenAPI 3.1 schema
- [ ] Every REST endpoint in `api/internal/server/router.go` is represented
- [ ] All request/response types from `pkg/types/types.go` are modeled as components/schemas
- [ ] Authentication schemes documented (Bearer JWT + `lsp_` API key)
- [ ] Error response format documented (consistent `{"error": "..."}` shape)
- [ ] Pagination parameters and response envelope documented
- [ ] Spec passes `openapi-generator validate` without warnings
- [ ] CI step: `make openapi-validate` fails if spec is invalid
- [ ] CI step: integration test verifies spec matches actual API responses (at least for key endpoints)

## Assumptions

| # | Assumption | Validation |
|---|-----------|-----------|
| A1 | All endpoints are defined in `api/internal/server/router.go` | Verified: `registerAuthRoutes`, `registerWorkspaceRoutes`, `registerProxyRoutes`, plus settings/secrets/credentials groups |
| A2 | All request/response types are in `pkg/types/types.go` | Verified: CreateWorkspaceRequest, AuthResponse, WorkspaceStatusResult, etc. all defined there |
| A3 | Error responses use `{"error": "string"}` format consistently | Verified: `gin.H{"error": ...}` used throughout router.go handlers |
| A4 | `openapi-typescript` and `oapi-codegen` support OpenAPI 3.1 | Unvalidated — must verify tool versions before selecting. Fallback: use OpenAPI 3.0.3 which has universal tool support |

## Technical Details

**Approach:**

The existing `api/internal/docs/swagger.go` is a stub — it contains `"paths": {}` and there are NO `@Router`, `@Summary`, or `@Param` annotations on any handler function. `swag init` will produce an empty spec. Therefore, the OpenAPI specification must be **hand-written** by reading:

1. `api/internal/server/router.go` — all route registrations and inline handlers
2. `pkg/types/types.go` — all request/response struct definitions
3. `api/internal/handlers/` — handler implementations for response codes and edge cases

This is a documentation task, not a code generation task.

**Security schemes:**

```yaml
securitySchemes:
  bearerAuth:
    type: http
    scheme: bearer
    bearerFormat: JWT
  apiKeyAuth:
    type: apiKey
    in: header
    name: Authorization
    description: "lsp_ prefixed API key in Bearer format"
```

**SSE endpoint documentation:**

`GET /workspaces/:id/events` is an SSE stream. OpenAPI cannot model SSE natively. Document it as:
- Response content-type: `text/event-stream`
- Description with event format examples
- Note: "Not usable by generated SDK clients — use language-specific SSE libraries"

**Proxy endpoint response shapes:**

The proxy endpoints (`/sessions/:sessionId/message`, `/sessions/:sessionId` GET, `/sessions/:sessionId/message` GET) return opencode's native response format — they are transparent proxies. The OpenAPI spec must:
1. Document these response schemas based on opencode's API (version-coupled)
2. Mark them with `x-opencode-proxy: true` extension to signal they track an upstream API
3. Include the opencode version these schemas were captured from

This is the one area where the spec may drift if opencode updates its response format. The CI drift detection test (hitting a real opencode instance) catches this.

**CI drift detection:**

Add a lightweight integration test that starts the API, hits 3-5 key endpoints, and validates the response shape matches the OpenAPI spec using a schema validator. This catches drift when handlers change but the spec isn't updated.
