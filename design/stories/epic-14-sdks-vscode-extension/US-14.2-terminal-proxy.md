# US-14.2: WebSocket Terminal Proxy Endpoint

**Epic:** 14 - Multi-Language SDKs & VS Code Extension
**Priority:** Critical
**Depends on:** None (builds on existing proxy infrastructure)

## User Story

As a VS Code extension user, I want to open a terminal connected to my sandbox's shell, so that I can run commands directly in the sandbox environment without leaving my IDE.

## Acceptance Criteria

- [ ] `POST /api/v1/workspaces/:id/terminal/ticket` returns a one-time ticket (short-lived, 30s expiry)
- [ ] `GET /api/v1/workspaces/:id/terminal?ticket=<ticket>` upgrades to WebSocket
- [ ] WebSocket proxies stdin/stdout/stderr to a shell process inside the sandbox pod
- [ ] Ticket-based auth: ticket is single-use, expires in 30s, tied to workspace ID and user ID
- [ ] Workspace ownership enforced (only the workspace owner can request a ticket)
- [ ] Workspace must be in Active phase (returns 409 if suspended/terminated)
- [ ] Terminal resizes supported (client sends resize control messages)
- [ ] Connection closes cleanly when sandbox pod terminates
- [ ] Multiple concurrent terminal sessions supported per workspace (max 5)
- [ ] Per-API-instance global limit: max 500 concurrent terminal connections
- [ ] Idle timeout: terminal disconnects after 30 minutes of no input (configurable)
- [ ] Integration test: connect, run `echo hello`, verify output, disconnect
- [ ] Helm chart RBAC updated: API ServiceAccount gets `pods/exec` permission
- [ ] `/bin/sh` verified executable in all runtime images under readOnlyRoot constraints

## Technical Details

**Authentication: One-Time Ticket Pattern**

Query-param tokens are a security risk (appear in logs, browser history, proxy caches). Instead, use a short-lived one-time ticket:

```
1. Client: POST /api/v1/workspaces/:id/terminal/ticket
   Headers: Authorization: Bearer <jwt_or_api_key>
   Response: {"ticket": "tkt_<random>", "expiresAt": "2026-05-29T17:31:00Z"}

2. Client: GET /api/v1/workspaces/:id/terminal?ticket=tkt_<random>
   → WebSocket upgrade (no Authorization header needed)
   → Ticket consumed (single-use), validated against workspace ID + user ID
```

Ticket storage: Redis with 30s TTL. Key: `terminal:ticket:<ticket_value>`, Value: `{userID, workspaceID}`.

**Protocol:**

Client connects via WebSocket. Messages are JSON-framed:

```json
// Client → Server: stdin data
{"type": "input", "data": "ls -la\n"}

// Client → Server: terminal resize
{"type": "resize", "cols": 120, "rows": 40}

// Server → Client: stdout/stderr data
{"type": "output", "data": "total 4\ndrwxr-xr-x ..."}

// Server → Client: terminal exited
{"type": "exit", "code": 0}

// Server → Client: error
{"type": "error", "message": "sandbox not running"}
```

**Implementation approach:**

The API server uses `k8s.io/client-go/tools/remotecommand` to exec into the sandbox pod via the Kubernetes API server. This is the same mechanism `kubectl exec` uses — traffic flows: API → kube-apiserver → kubelet → container. No direct network path to the pod is needed.

```go
// api/internal/handlers/terminal.go

func (h *TerminalHandler) HandleTicket(c *gin.Context) {
    // 1. Validate auth (JWT/API key via standard middleware)
    // 2. Validate workspace ownership
    // 3. Check workspace phase == Active
    // 4. Generate ticket (crypto/rand, 32 bytes, hex-encoded, prefixed "tkt_")
    // 5. Store in Redis: key=terminal:ticket:<ticket>, value={userID, workspaceID}, TTL=30s
    // 6. Return ticket + expiry
}

func (h *TerminalHandler) HandleTerminal(c *gin.Context) {
    // 1. Extract ticket from query param
    // 2. Validate ticket in Redis (atomic GET+DELETE — single-use)
    // 3. Verify ticket.workspaceID matches :id param
    // 4. Check global connection count < 500
    // 5. Check per-workspace connection count < 5
    // 6. Get workspace CRD → extract pod name from status
    // 7. Upgrade to WebSocket (gorilla/websocket)
    // 8. Create SPDY exec connection to pod (remotecommand)
    // 9. Bridge WebSocket ↔ SPDY streams (stdin/stdout/stderr)
    // 10. Handle resize messages
    // 11. Decrement connection counts on disconnect
}
```

**Pod name resolution:**

The terminal handler needs the pod NAME (not IP) for exec. The workspace CRD status already includes `PodName` — the controller populates it at `controller/internal/workspace/controller.go:181` when the pod is created, and clears it on suspend.

Use the same pattern as the existing proxy handler: fetch the workspace CRD via `h.k8sClient.LlmsafespaceV1().Workspaces(h.namespace).Get(workspaceID, ...)`, then read `workspace.Status.PodName`.

```go
workspace, err := h.k8sClient.LlmsafespaceV1().Workspaces(h.namespace).Get(workspaceID, metav1.GetOptions{})
if err != nil {
    // 404
    return
}
if workspace.Status.Phase != v1.WorkspacePhaseActive || workspace.Status.PodName == "" {
    // 409 — workspace not active
    return
}
podName := workspace.Status.PodName
podNamespace := workspace.Status.PodNamespace
```

**Route registration:**

```go
// In router.go:

// Ticket endpoint — on the authenticated workspace group (requires JWT/API key)
workspaceGroup.POST("/:id/terminal/ticket", terminalHandler.HandleTicket)

// WebSocket endpoint — on the ROOT router (NOT the auth-protected group).
// Auth is handled by the one-time ticket, not by JWT/API key headers.
// WebSocket upgrade requests cannot carry Authorization headers in browsers.
router.GET("/api/v1/workspaces/:id/terminal", terminalHandler.HandleTerminal)
```

**Why separate groups:** The ticket endpoint needs standard auth (user must prove identity to get a ticket). The WebSocket endpoint uses the ticket AS its auth — putting it behind the auth middleware would require the client to send both a ticket AND a JWT, defeating the purpose of the ticket pattern.

**Idle timeout:**

A goroutine monitors the last input timestamp. If no `input` message received for 30 minutes (configurable via instance settings), send `{"type": "error", "message": "idle timeout"}` and close the WebSocket.

**Shell verification task:**

Before implementation, run:
```bash
docker run --rm --read-only --user 1000:1000 <runtime-base-image> /bin/sh -c "echo ok"
```
If this fails, the runtime Dockerfile needs `/bin/sh` in a non-root-writable location (it should already be there as part of the base image layer).

**Dependencies:**

- `github.com/gorilla/websocket` — add as direct dependency in go.mod
- `k8s.io/client-go/tools/remotecommand` (already available via client-go)

**Note on exec transport:** `remotecommand` traditionally uses SPDY. Kubernetes 1.29+ supports WebSocket-based exec (KEP-4006). Since this project uses client-go v0.32.3 (K8s 1.32), prefer `remotecommand.NewWebSocketExecutor` if the cluster supports it, with SPDY fallback. This is handled automatically by `remotecommand.NewSPDYExecutor` in recent client-go versions which negotiate the best transport.
