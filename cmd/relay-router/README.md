# relay-router

In-cluster HTTP router that distributes workspace traffic across healthy relay VMs over HTTP with per-VM token auth (WireGuard was removed in worklog 0447). Runs as a single-replica Deployment (no sidecar, PSA `restricted`-compliant).

## Responsibilities

1. **Weighted relay selection** â€” OCI gets 100% of traffic when healthy; GCP receives traffic only during OCI failure or rotation
2. **Health checking** â€” Probes each relay `/healthz` every 15s; marks unhealthy after 3 consecutive failures (45s)
3. **429 detection (two-tier)** â€” Immediate probe on first 429; storm detection via 5-minute rolling window at 50% threshold
4. **Failover** â€” When OCI goes down, all traffic routes to GCP
5. **Fallback mode** â€” When all relays are down, proxies directly to upstream at 0.5 req/s, max 1 concurrent

## Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/healthz` | 200 OK â€” router health |
| `GET` | `/metrics` | Prometheus metrics |
| `*` | `/*` | Reverse proxy to selected relay (or fallback) |

On the relay path the router injects `X-Relay-Token` from the selected peer's token. In direct fallback mode it does **not** set the header (the upstream is not token-gated).

| Metric | Type | Description |

| `relay_router_requests_total{relay,status}` | counter | Requests routed per relay by HTTP status |

## Peer ConfigMap

The controller writes the `relay-router-peers` ConfigMap in this JSON format (`controller/internal/relay/router_configmap.go`):

```json
{
  "relays": [
    {"id": "oci-1", "endpoint": "203.0.113.10:8080", "provider": "oci", "state": "healthy", "token": "<per-vm-secret>"},
    {"id": "gcp-1", "endpoint": "198.51.100.5:8080", "provider": "gcp", "state": "healthy", "token": "<per-vm-secret>"}
  ]
}
```

`endpoint` is the relay VM's public IP/host:port that the router dials over HTTP. `token` is the per-VM shared secret presented in `X-Relay-Token`. Tokens are generated by the controller and stored in the `relay-vm-tokens` Secret.
